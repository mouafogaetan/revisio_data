Pratique essentielle #1

🧹 Libérer les ressources avec `using`

En C#, certains objets accèdent à des ressources système (fichiers, connexions DB, flux réseau). Ils implémentent l'interface IDisposable. Le mot-clé using garantit que Dispose() est appelé même en cas d'exception.

// ❌ Mauvaise pratique : oublier de libérer
StreamReader reader = new StreamReader("data.txt");
string content = reader.ReadToEnd(); // risque de fuite mémoire si exception

// ✅ Bonne pratique : bloc using
using (StreamReader reader = new StreamReader("data.txt"))
{
    string content = reader.ReadToEnd();
    Console.WriteLine(content);
} // Dispose automatique, même si exception survient

// Syntaxe simplifiée (C# 8+) :
using StreamReader reader2 = new StreamReader("log.txt");
string log = reader2.ReadToEnd();
// Dispose à la fin du bloc englobant

📌 À retenir : Utilisez `using` pour tout objet implémentant `IDisposable` (fichiers, requêtes SQL, clients HTTP). Évitez les fuites mémoire !

Piège fréquent

🚫 Éviter `catch (Exception ex)` vide ou trop large

Capturer Exception générique sans traitement masque les erreurs critiques (problèmes mémoire, stack overflow...).

// ❌ TRÈS MAUVAIS : avaler toutes les exceptions
try {
    int division = 10 / int.Parse("0");
} catch (Exception) {
    // rien du tout → erreur silencieuse, impossible à debuguer
}

// ✅ acceptable si on loggue ET on relance ou gère proprement
try {
    int division = 10 / int.Parse("0");
} catch (DivideByZeroException ex) {
    Console.WriteLine($"Erreur mathématique: {ex.Message}");
    // gestion spécifique
} catch (FormatException ex) {
    Console.WriteLine($"Format invalide: {ex.Message}");
} catch (Exception ex) when (ex is not OutOfMemoryException) {
    // on loggue et on propage
    Console.WriteLine($"Erreur inattendue: {ex}");
    throw; // préserve la pile d'appels
}

✅ Règle d'or : Ne capturer que les exceptions que vous pouvez traiter. Sinon, laissez-les remonter.

Bonnes pratiques

📝 Journaliser les exceptions (Logging)

En production, la console n’est pas suffisante. Utiliser un système de logs (ILogger, fichiers, séries). Toujours inclure le message et la pile d'appels.

// Exemple simple avec fichier texte ou console enrichie
try {
    int[] nombres = {1,2,3};
    Console.WriteLine(nombres[5]);
} catch (IndexOutOfRangeException ex) {
    // Logging vers fichier ou outil
    File.AppendAllText("error.log", $"{DateTime.Now}: {ex}\n");
    Console.Error.WriteLine("Erreur: index hors bornes. Consultez error.log");
    // Option : notifier l'admin ou afficher un message utilisateur
}

// Avec un vrai logger (simulation)
public static void LogError(Exception ex, string contexte) {
    // on enregistre TOUTE l'exception (StackTrace, Message)
    Console.WriteLine($"[ERREUR][{contexte}] {ex.GetType().Name} : {ex.Message}");
    // Dans un projet réel → Serilog, NLog, ILogger
}

À ne pas faire : `catch(Exception ex) { Console.WriteLine(ex.Message); }` sans contexte, sans la stack trace. En production, on veut des détails complets.

Synthèse pratique

🛠️ Mise en situation : lecture fichier robuste

Application des trois bonnes pratiques ensemble : using + catch spécifique + logging.

public static string LireFichierSecurise(string chemin)
{
    try
    {
        using (StreamReader sr = new StreamReader(chemin))
        {
            return sr.ReadToEnd();
        }
    }
    catch (FileNotFoundException ex)
    {
        LogError(ex, $"Fichier {chemin} introuvable");
        return null; // gestion métier
    }
    catch (UnauthorizedAccessException ex)
    {
        LogError(ex, "Accès refusé au fichier");
        throw new SecurityException("Impossible de lire les données", ex);
    }
    catch (Exception ex) when (ex is IOException || ex is ArgumentException)
    {
        LogError(ex, "Erreur d'entrée/sortie");
        throw; // on remonte l'exception préservant la pile
    }
    // Pas de catch(Exception) général sans when → les vrais bugs critiques remontent
}

private static void LogError(Exception ex, string contexte)
{
    // Logger vers fichier ou console enrichie
    Console.WriteLine($"[ERREUR][{DateTime.Now}] {contexte} : {ex}");
    // Dans une vraie app : File.AppendAllText("logs/app.log", ...)
}

✅ Le code respecte : using (libération fichier) – catch spécifique – logging détaillé – aucune capture aveugle.

📌 Mémo – Les 3 règles d’or

1. 🔁 Utiliser `using` pour `IDisposable`

Flux, fichiers, connexions DB → toujours `using` ou `await using` pour ne pas saturer les ressources.

2. ⚠️ Éviter `catch (Exception)` vide / général

Ne masquez pas les erreurs système. Attrapez les exceptions spécifiques que vous savez traiter. Sinon, laissez propager.

3. 📋 Logger correctement

Toujours enregistrer la stack trace, le type d’exception, le contexte. Un simple `Console.WriteLine(ex.Message)` est insuffisant en production.

💪 Un code robuste = anticiper les échecs + libérer les ressources + garder des traces
