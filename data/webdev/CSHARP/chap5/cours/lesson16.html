Modificateurs d'accès

Mot-cléAccès autorisé
publicAccessible partout (dans et hors de l'assembly)
privateAccessible uniquement dans la classe contenante
protectedAccessible dans la classe et ses classes dérivées
internalAccessible dans le même projet (assembly)
protected internalprotected OU internal (union)
private protectedAccessible dans la classe ou les dérivées du même assembly
⚡ Règle d'or : par défaut, rendez vos membres private, et exposez uniquement ce qui est nécessaire avec public ou des propriétés.
📌

readonly : l’immutabilité contrôlée

Un champ readonly ne peut être modifié qu’au moment de la déclaration ou dans le constructeur. Après, plus rien ne peut le changer.

public class CompteBancaire
{
    private readonly string _numeroCompte;
    public readonly DateTime DateCreation;   // champ readonly public (peu courant)
    
    public CompteBancaire(string numero)
    {
        _numeroCompte = numero;
        DateCreation = DateTime.Now;
    }
    // Impossible de modifier _numeroCompte ailleurs !
}
🔒 Sécurité Prévention des modifications accidentelles

Encapsulation & Propriétés

Ne pas exposer directement des champs public. Utilisez des propriétés avec get/set pour contrôler l'accès.

// Mauvaise pratique ❌
public class Etudiant {
    public string nom;  // champ public dangereux
}

// Bonne pratique ✅
public class Etudiant {
    private string _nom;
    public string Nom {
        get { return _nom; }
        set { 
            if(!string.IsNullOrWhiteSpace(value))
                _nom = value;
        }
    }
    
    // Propriété automatique avec validation privée possible
    public int Age { get; private set; }  // setter privé
    
    public Etudiant(int age) => Age = age;
}
✔️ Principe : champs privés, exposés via des méthodes/propriétés → contrôle total.

📐 Schéma d'encapsulation

Design simplifié

Une classe expose une interface publique, mais cache ses détails internes.

🔹 private → interne à la classe (cœur métier)
🔹 public → interface visible (API de la classe)

⚙️ Conseils pratiques pour l'encapsulation

  • Toujours commencer par private pour les champs. Exposer via propriétés si nécessaire.
  • Utiliser readonly pour les valeurs qui ne changent jamais après construction.
  • Privilégier les propriétés automatiques quand aucune logique n'est requise : public string Nom { get; set; }.
  • Éviter les champs publics (sauf constantes public const).
  • Pour les dépendances internes, utiliser private protected pour l’héritage contrôlé.
  • Encapsulation = réduction de l’impact des changements internes sur le reste du code.
🧠 À retenir : "Ce qui est caché peut changer sans casser le code client."

Récapitulatif

  • ✔️ internal : au sein du même projet
