📌 Rappel : type valeur (struct) vs type référence (class)

struct → Type valeur
  • Stocké dans la pile (stack) ou inline.
  • Copie intégrale lors de l'affectation.
  • Plus performant pour de petites données.
  • Ne peut pas être null (sauf Nullable<T>).
class → Type référence
  • Stocké dans le tas (heap), référence sur la pile.
  • Affectation = copie de la référence (pas l'objet).
  • Partage de données entre variables.
  • Peut être null.
// Exemple struct
struct Point {
    public int X, Y;
}
Point p1 = new Point { X = 5, Y = 10 };
Point p2 = p1;  // COPIE TOTALE
p2.X = 99;
Console.WriteLine(p1.X); // 5 (indépendant)

// Exemple class
class Personne {
    public string Nom;
}
Personne alice = new Personne { Nom = "Alice" };
Personne bob = alice;   // partage référence
bob.Nom = "Robert";
Console.WriteLine(alice.Nom); // "Robert" (modifié!)

⚙️ Quand utiliser struct plutôt que class ?

✅Petites structures de données (moins de 16 octets) comme un couple coordonnées, RGB, Intervalle.
✅Immutable par nature (ex: DateTime, Guid, complexe mathématique).
✅Éviter l'allocation tas pour des milliers d'instances temporaires (performances jeux vidéo, calculs).
❌Éviter si la structure doit être modifiée fréquemment (effet de bord).
❌Ne pas utiliser pour des données volumineuses (> 16-24 octets) car copie coûteuse.

📌 Règle d’or Microsoft : une structure doit représenter une seule valeur atomique et idéalement être immutable.

✨ Les record : le meilleur des deux mondes

Introduits avec C# 9, parfaits pour les DTO, données immutables, value-based equality

Un record est un type référence (class record) ou un type valeur (struct record) qui bénéficie de l'égalité structurelle (comparaison par valeurs) et d'une syntaxe concise pour l'immutabilité.

// Record class (immuable par défaut avec init-only)
public record Personne(string Nom, int Age);

var p1 = new Personne("Alice", 25);
var p2 = new Personne("Alice", 25);
Console.WriteLine(p1 == p2); // True ✅ (comparaison par valeurs)

// Expression 'with' pour copie modifiée
var p3 = p1 with { Age = 26 };
Console.WriteLine(p3); // Personne { Nom = Alice, Age = 26 }
✅ Égalité par valeur – plus besoin de surcharger Equals/GetHashCode.
✅ Déconstruction – var (nom, age) = personne;
✅ Immutabilité – thread-safe, idéal pour APIs.
✅ Performance – syntaxe plus lisible pour POCOs.

Comparaison : Même exemple avec struct, class, record

Imaginons un point 2D simple. Voyons le comportement à la copie et à l’égalité.

💡 Rappel : record est un type référence (class) mais avec comportement value-based. Vous pouvez aussi déclarer record struct pour un type valeur encore plus léger.

⚠️ Pièges et bonnes pratiques

  • Ne pas créer de struct mutable : modifier un champ d'une struct copiée n'affecte que la copie → confusion.
  • Attention aux grandes structures : passer en paramètre une grosse struct est plus coûteux qu'une référence (classe).
  • Préférer record pour les modèles de données : API, couche métier, DTO.
  • Utilisez readonly struct pour garantir l'immutabilité et améliorer les performances.
  • Si vous héritez : struct ne supporte pas l'héritage. Utilisez class ou record (record peut hériter d'un autre record).
🏆
En résumé pour le dev web / Unity / backend
- struct : petits types immutables (Vector2, Couleur)
- class : comportement riche, entités avec identité
- record : données immutables, comparaison par valeur, JSON friendly

🧠 Exercice rapide

Que va afficher ce code ? 🤔

struct S { public int Val; }
class C { public int Val; }
var s1 = new S { Val = 10 };
var s2 = s1;
s2.Val = 20;
var c1 = new C { Val = 10 };
var c2 = c1;
c2.Val = 20;
Console.WriteLine($"{s1.Val} / {c1.Val}");
10 / 20 (struct copie, classe référence)

🎓 Explication: struct s2 est une copie indépendante. class c2 partage la référence.
