NX bit & DEP

Le bit NX (Never eXecute) ou DEP (Data Execution Prevention) empêche l’exécution de code dans les zones mémoire réservées aux données.

💡 Principe : Une page mémoire peut être marquée comme « exécutable » ou « non exécutable ». Le processeur refuse d’exécuter des instructions dans une page non-exécutable.

🎯 Utilité

Contre les attaques par buffer overflow : l’attaquant injecte du code malveillant dans la pile ou le tas, mais le CPU refuse de l’exécuter si la zone est non-exécutable.

⚙️ Implémentation

Support matériel (AMD / Intel) + support OS (Windows, Linux, macOS). Sur Linux, on parle de XD bit (eXecute Disabled).

🎲

ASLR (Address Space Layout Randomization)

L’ASLR mélange aléatoirement l’emplacement des régions mémoire (code, librairies, pile, tas) à chaque exécution d’un processus.

🔁 Pourquoi ? Sans ASLR, les attaquants connaissent exactement les adresses des fonctions systèmes (ex: system()). Avec l’aléatoire, l’exploitation est quasi impossible.

📌 Exemple concret : Sous Linux, on peut vérifier avec cat /proc/self/maps → les adresses changent à chaque exécution.

Canaris (Stack cookies)

Valeur secrète placée entre les tampons et les données critiques (adresse de retour). Avant de quitter une fonction, on vérifie que le cookie n’a pas été modifié.

+-----------------+
| Buffer          |  ← zone vulnérable
+-----------------+
| CANARY (secret) |  ← valeur aléatoire
+-----------------+
| Adresse retour  |  ← protégée
+-----------------+

Si un débordement écrase le canari, le programme détecte l’attaque et s’arrête (*** stack smashing detected ***).

🧠 Protection des pages mémoire : droits d’accès (lecture/écriture/exécution) sur chaque page via la MMU. Exemple : mprotect() sous Linux.
✅ Avantage
Stop aux buffer overflow classiques
⚠️ Contournement
Attaques par fuite du canari (nécessite autre faille)

🛡️ Mise en pratique & visualisation

🔍 Vérifier l’ASLR sous Linux :

cat /proc/sys/kernel/randomize_va_space

0 = désactivé, 1 = randomisation partielle, 2 = randomisation complète (par défaut moderne).

📉 Protection DEP sous Windows :

Active par défaut pour les programmes compatibles. Option /NXCOMPAT lors de la compilation.

💡 Ces mécanismes ne sont pas infaillibles mais élèvent considérablement le niveau de sécurité. Les attaques modernes (ROP) contournent partiellement NX, mais ASLR rend ROP très complexe.

🚫 NX/DEP
Empêche exécution de code en zone données
🎲 ASLR
Randomise les adresses mémoire
🍪 Canaris
Détecte corruption de la pile
✍️

Petit exercice

Situation : Un programme vulnérable est compilé avec protection NX activée + ASLR complète + stack canary. Un attaquant tente un buffer overflow classique pour exécuter un shellcode dans la pile.

❓ Question : Pourquoi l’attaque échouera-t-elle probablement ?

  • Le NX bit empêche l'exécution du shellcode dans la pile (zone non exécutable).
  • Le canari détecte l’écrasement de l’adresse de retour et termine le processus avant le saut.
  • ASLR rend les adresses des gadgets ROP difficiles à prévoir même si l’attaquant tente de contourner NX.
✅ Réponse: Grâce à la combinaison des trois mécanismes, le shellcode n’est pas exécutable, le canari saute et ASLR complexifie toute tentative avancée.
🔒

Protection en profondeur

NX, ASLR, canaris et protection des pages sont devenus des standards dans les OS modernes (Windows 11, Linux, macOS). Ils forment une barrière solide contre les exploits mémoire.

📘 En résumé : chaque mécanisme agit sur une facette différente de l’attaque — exécution, adressage, intégrité de la pile.
