🔍 Détection : Algorithme du graphe d'attente

Le système maintient un graphe d'allocation-requête (ou graphe d'attente). Un cycle dans ce graphe signifie qu'un deadlock est présent.

📌 Principe :

  • ✔️ Sommets = processus (P) et ressources (R).
  • ✔️ Arêtes demande : P → R (P attend R).
  • ✔️ Arêtes allocation : R → P (R est allouée à P).
  • ✔️ Un cycle = interblocage.

Exemple cyclique (deadlock)

P1 → R1 → P2 → R2 → P1

Pas de cycle → pas de deadlock

P1 → R1 → P2   (P2 n'attend plus)
⚙️ Algorithme de détection périodique : lancé à intervalles réguliers ou lors d’une forte attente. Complexité O(n²) pour n processus.

📈 Détection avec plusieurs ressources de chaque type

Si plusieurs instances d'une ressource existent, on utilise un algorithme similaire à celui de l'évitement mais basé sur les demandes en attente.

1. Initialiser Work = Available, Finish[i] = false pour tout processus i
2. Trouver i tel que: Finish[i] = false et Request_i <= Work
   Si trouvé: Work = Work + Allocation_i ; Finish[i]=true ; répéter
3. Si Finish[i] = false pour certains i → deadlock détecté

💡 Cet algorithme ressemble à l'algorithme du banquier mais ici on détecte l'état mort.

🎯 Exemple pratique : 3 processus, 2 ressources A et B. Si un cycle apparaît, on déclenche la récupération.

⚙️ Récupération après deadlock

Une fois le blocage détecté, il faut casser le cycle. Deux grandes familles :

🔪 1. Terminaison de processus

Annuler un ou plusieurs processus du cycle. Deux sous-stratégies :

  • Annuler tous – simple mais coûteux.
  • Annuler un par un jusqu'à briser le cycle (choix basé sur priorité, temps restant, ressources utilisées).

🔄 2. Préemption (reprise)

Retirer une ressource à un processus et la donner à un autre. Il faut gérer le rollback (retour à un état sûr).

✨ Technique utilisée dans certains systèmes transactionnels.

⚠️ Attention : la reprise nécessite de sauvegarder l’état des processus (checkpoints).

🗄️ Deadlocks dans les bases de données

Dans les SGBD, les transactions utilisent des verrous (locks) pour assurer l'isolation. Le protocole 2PL (Two-Phase Locking) garantit la sérialisabilité mais peut causer des interblocages.

🔒 Exemple classique :

Transaction T1: Verrou X sur A → attend verrou Y sur B
Transaction T2: Verrou Y sur B → attend verrou X sur A

➡️ Deadlock détecté par le SGBD !

🚨 Mécanismes courants :

  • Wait-Die : une transaction plus ancienne attend, la plus jeune est annulée.
  • Wound-Wait : la transaction plus âgée "blesse" (annule) la plus jeune.
  • Timeout : si une transaction attend trop longtemps → on l'annule.
📌 MySQL/InnoDB utilise la détection de deadlock + choix d'une victime pour résoudre automatiquement.

🧮 Mise en pratique : Détection pas à pas

Prenons 3 processus (P0,P1,P2) et 2 ressources (R1 : 2 instances, R2 : 1 instance).

ProcessusAlloué (R1,R2)Demande (R1,R2)
P0(1,0)(0,1)
P1(1,1)(1,0)
P2(0,0)(1,0)

Disponible = (0,0) → aucun processus ne peut satisfaire sa demande → deadlock généralisé (P0, P1, P2 bloqués).

🧠 L'algorithme de détection conclut que tous les processus sont dans l'ensemble non terminé → interblocage.

🛠️ Stratégies avancées de récupération

🎲 Critères de sélection de la victime
  • Priorité du processus
  • Nombre de ressources détenues
  • Nombre de cycles déjà exécutés
  • Coût du rollback (si transaction base de données)
⏪ Rollback / Checkpoint

On peut ramener le processus à un état antérieur sûr (checkpoint) puis le relancer. Technique coûteuse mais efficace.

Souvent utilisé dans les systèmes temps réel critiques.

✨ Exemple sous Unix/Linux : aucun système ne résout automatiquement les deadlocks entre processus utilisateur, mais on peut les tuer avec kill -9 [PID].

🧠 Récapitulatif : Détection vs Prévention vs Évitement

ApprocheAvantagesInconvénients
Détection + récupérationN'impose pas de restrictions a prioriOverhead de détection, risque de pertes de données
Prévention (invalider une condition)Garantie sans deadlockUtilisation inefficace des ressources
Évitement (Banker)Flexible, sécuritéNécessite connaissance des besoins max, coûteux

🎯 En pratique : les OS modernes utilisent souvent la préemption sur les verrous (mutex) avec timeout, et les SGBDs emploient la détection.

Windows : Détection limitée Linux : lockdep (outil de détection) Oracle : Deadlock graph

✏️ Mini exercice de réflexion

Soit un système avec 4 processus (P1,P2,P3,P4) et 3 ressources R1, R2, R3. Le graphe d'attente montre un cycle : P1→R1→P2→R2→P3→R3→P1. Que concluez-vous ?

✅ Il y a un interblocage généralisé. P1, P2, P3 sont en deadlock circulaire. La solution : terminer ou préempter un de ces processus.

📌 En résumé

  • La détection repose sur l'analyse du graphe d'attente.
  • Une fois détecté, on annule des processus ou on préempte des ressources.
  • Les bases de données gèrent les deadlocks via le 2PL et des algorithmes comme Wait-Die.
  • Connaître ces mécanismes est essentiel pour l'administration système.
