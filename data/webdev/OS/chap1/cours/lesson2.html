Leçon 2 – Chapitre 1

🏛️ Architecture des OS

Mode noyau vs mode utilisateur, architectures et appels système

Durée : 45 min Niveau : Débutant Protection et performances

"Le noyau est le cœur du système d'exploitation, il gère tout avec des privilèges maximum."

🎯 Objectifs de la leçon

  • Comprendre la différence entre mode noyau (kernel mode) et mode utilisateur (user mode).
  • Distinguer les architectures de noyau : monolithique, micro-noyau, hybride, modulaire.
  • Découvrir les appels système (system calls) et leur rôle d'interface entre les applications et le noyau.
La séparation entre mode utilisateur et mode noyau est essentielle pour la stabilité et la sécurité du système.

🔐 Mode noyau vs Mode utilisateur

Mode noyau (Ring 0)

  • Accès illimité à tout le matériel
  • Exécute les instructions privilégiées
  • Accès direct à la mémoire et aux périphériques
  • Une erreur peut planter tout le système
  • Réservé au noyau et aux pilotes

Mode utilisateur (Ring 3)

  • Accès limité aux ressources
  • Doit passer par des appels système
  • Ne peut pas exécuter d'instructions privilégiées
  • Une erreur ne tue que le processus concerné
  • Réservé aux applications normales
Transition : Pour passer du mode utilisateur au mode noyau, on utilise une interruption ou un appel système.

🔹 Monolithique : tout le noyau dans un seul bloc (Linux, BSD) → performances élevées mais complexe.

🔹 Micro-noyau : noyau minimal, services dans l'espace utilisateur → plus stable mais plus lent.

🔹 Hybride : compromis entre les deux (Windows, macOS XNU).

🔹 Modulaire : chargement/déchargement dynamique des modules (Linux moderne).

📞 Les appels système (System Calls)

Les appels système sont l'interface entre les applications en mode utilisateur et le noyau. Ils permettent d'effectuer des opérations privilégiées (lecture/écriture fichier, création processus, communication réseau).

// Exemple d'appel système en C (lecture d'un fichier)
#include <fcntl.h>
#include <unistd.h>

int fd = open("fichier.txt", O_RDONLY); // Appel système open()
char buffer[100];
read(fd, buffer, 100); // Appel système read()
close(fd); // Appel système close()

📋 Exemples d'appels système courants :

open() - ouvrir un fichier read() / write() - lire/écrire fork() - créer un processus exec() - exécuter un programme malloc() - allocation mémoire socket() - communication réseau

⚙️ Mécanisme d'un appel système

sequenceDiagram participant App as Application
(Mode utilisateur) participant API as API (libc) participant Noyau as Noyau
(Mode noyau) participant Mat as Matériel App->>API: open("fichier.txt", O_RDONLY) API->>Noyau: Interruption/TRAP Noyau->>Mat: Vérifie permissions Mat-->>Noyau: OK Noyau-->>API: descripteur de fichier API-->>App: fd (entier)

Étapes :

  1. L'application appelle une fonction de la bibliothèque standard (ex: libc).
  2. La fonction déclenche une interruption logicielle (trap) vers le noyau.
  3. Le CPU passe en mode noyau.
  4. Le noyau exécute le service demandé (vérifications, opération matérielle).
  5. Le noyau retourne le résultat et repasse en mode utilisateur.

✏️ Exercice & Conclusion

Exercice guidé :

  1. Citez deux avantages du micro-noyau par rapport au noyau monolithique.
  2. Pourquoi une application ne peut-elle pas accéder directement au disque dur ?
  3. Quel appel système utiliseriez-vous pour créer un nouveau processus sous Linux ?
Voir les réponses

1. Micro-noyau : plus stable (les services sont isolés, une panne ne plante que le service), plus sécurisé (moins de code en mode noyau), plus facile à étendre.

2. Une application ne peut pas accéder directement au disque dur car elle est en mode utilisateur, sans privilèges. Elle doit passer par un appel système qui bascule en mode noyau, où le pilote de disque peut effectuer l'opération en toute sécurité.

3. L'appel système fork() (ou clone() sous Linux) permet de créer un nouveau processus.

Récapitulatif :

  • Mode noyau : privilèges maximum, accès direct au matériel.
  • Mode utilisateur : accès limité, doit passer par des appels système.
  • Architectures : monolithique (Linux), micro-noyau (QNX), hybride (Windows).
  • Les appels système sont le pont entre les applications et le noyau.
