🔁 HTTP est sans état (stateless)

Chaque requête est indépendante. Le serveur ne se souvient pas d’un utilisateur entre deux requêtes. Problème ? Impossibilité de savoir si l’utilisateur est connecté, ou d'afficher son panier.
Solutions : Cookies et Sessions.

Client → Serveur
[GET /profile]
Serveur : "Je ne te connais pas" 😕
+ cookie/session → "Ah! c'est toi NDI" ✅

Les cookies : petits fichiers texte stockés côté client, envoyés automatiquement à chaque requête. Les sessions : données stockées côté serveur, identifiées par un identifiant unique (sessionId) contenu dans un cookie.

📦 Installation & Configuration

npm install express-session

Configuration minimale avec express-session :

const session = require('express-session');
app.use(session({
  secret: 'unSecretTresLongEtAleatoire',
  resave: false,
  saveUninitialized: true,
  cookie: { secure: false, maxAge: 60000 } // 1 minute
}));
secret : signe le cookie, sécurité essentielle.
resave : évite de sauvegarder si inchangé.
saveUninitialized : créer une session même vide.
En développement, secure: false (HTTP). En production, active secure: true (HTTPS).

✍️ Manipuler les sessions

L'objet req.session permet de stocker des données utilisateur.

// Route de connexion fictive
app.post('/login', (req, res) => {
  const { username } = req.body;
  req.session.user = { username, role: 'apprenant' };
  req.session.isLoggedIn = true;
  res.send('Connecté ✅');
});

// Vérification d'authentification
app.get('/dashboard', (req, res) => {
  if (req.session.isLoggedIn) {
    res.json({ message: `Bienvenue ${req.session.user.username}` });
  } else {
    res.status(401).send('Non authentifié');
  }
});
Les données de session sont conservées tant que le cookie n’expire pas ou que l’utilisateur ne supprime pas ses cookies.

🍪 Paramètres avancés des cookies

cookie: {
  maxAge: 24 * 60 * 60 * 1000, // 1 jour
  httpOnly: true,   // inaccessible via JavaScript (XSS protection)
  secure: true,     // seulement HTTPS (prod)
  sameSite: 'lax'   // protection CSRF
}
httpOnly : empêche les scripts malveillants d’accéder au cookie.
secure : cookie envoyé uniquement sur HTTPS.
sameSite : réduit les attaques CSRF.
⚙️ Destruction de session : req.session.destroy(err => { ... }) pour la déconnexion côté serveur.

⚖️ Sessions stateful vs JWT stateless

CritèreSessions (stateful)JWT (stateless)
StockageCôté serveur (mémoire/Redis)Côté client (token)
RévocationFacile (supprimer session)Complexe (blacklist)
ScalabilitéNécessite un store partagéTrès scalable (pas de session)
SécuritéMoins vulnérable XSS (httpOnly)Vulnérable XSS si stocké localStorage
💡 Quand choisir ? Sessions pour les apps web classiques (monolites), JWT pour API distribuées / microservices ou mobile.

📀 Stockage persistant des sessions

En production, la session en mémoire (MemoryStore par défaut) n’est pas adaptée (perte au redémarrage, non partagée). On utilise connect-redis ou connect-mongo.

npm install connect-redis redis

const RedisStore = require('connect-redis')(session);
app.use(session({
  store: new RedisStore({ host: 'localhost', port: 6379 }),
  secret: 'superSecret',
  resave: false,
  saveUninitialized: false
}));
Ainsi, toutes vos instances de serveur partagent la même session → idéal pour la scalabilité horizontale.

🔐 Mini-Projet : Authentification par session

// Fichier server.js simplifié
app.post('/login', (req, res) => {
  if (req.body.pwd === 'secret') {
    req.session.userId = 1;
    req.session.role = 'admin';
    res.json({ ok: true });
  } else res.status(401).json({ error: 'Mauvais mot de passe' });
});

app.get('/profile', (req, res) => {
  if (!req.session.userId) return res.status(401).send('Non connecté');
  res.send(`Profil de l'utilisateur ${req.session.userId}`);
});

app.post('/logout', (req, res) => {
  req.session.destroy((err) => {
    res.clearCookie('connect.sid'); // supprime le cookie
    res.send('Déconnecté');
  });
});

À tester avec Postman : après login, le cookie est automatiquement renvoyé.

🛡️ Bonnes pratiques sessions

  • Utilisez un secret fort et gardez-le dans les variables d’environnement.
  • Activez secure: true, httpOnly: true, sameSite: 'strict' en production.
  • Ne stockez pas d’informations sensibles en clair dans le cookie (juste l’ID).
  • Expirez les sessions inactives (maxAge) et proposez un bouton de déconnexion.
  • Pour les très hautes charges, utilisez un store Redis.
Attention : saveUninitialized: true peut générer des cookies même pour les visiteurs non authentifiés, cela peut être évité avec false selon le besoin.
