📋 Définir endpoints et modèles

Avant d'écrire du code, on dessine l'architecture de l'API : quelles ressources ? quelles actions ?

Exemple : API de blog

📄 Ressource : Article

  • GET /articles → liste
  • GET /articles/:id → détail
  • POST /articles → créer
  • PUT /articles/:id → modifier
  • DELETE /articles/:id → supprimer

👤 Ressource : Utilisateur

  • POST /register → inscription
  • POST /login → connexion
  • GET /profile → profil
// Modèle Mongoose pour Article const articleSchema = new Schema({ title: String, content: String, author: { type: Schema.Types.ObjectId, ref: 'User' }, createdAt: Date });

⚙️ Choisir les bonnes technologies

Un projet API complet nécessite plusieurs briques. Voici un choix standard et pourquoi :

🌐 Serveur : Express.js (simple, flexible, énorme communauté)
🗄️ Base de données : MongoDB (flexibilité) ou PostgreSQL (relations)
🔐 Authentification : JWT (stateless, idéal pour API)
✅ Validation : Joi ou Zod
📝 Documentation : Swagger (OpenAPI 3.0)
🧪 Tests : Jest + Supertest

💡 Le choix dépend du projet : pour des données très structurées avec relations complexes → PostgreSQL. Pour de la rapidité et flexibilité → MongoDB.

🔗 Relations entre données (One-to-Many, Many-to-Many)

Deux approches principales :

📦 Embedding (document imbriqué)

On stocke les données liées à l'intérieur du document parent.

{ "_id": "user123", "name": "Alice", "addresses": [ { "street": "123 Rue", "city": "Yaoundé" } ] }

✅ Rapide en lecture, ❌ Risque de duplication.

🔗 Référence (normalisation)

On stocke l'ID de l'objet lié.

// Article { "_id": "art1", "title": "Node.js", "authorId": "user123" } // User { "_id": "user123", "name": "Alice" }

✅ Flexible, ❌ Requêtes supplémentaires (populate).

💡 Astuce : utiliser populate() (Mongoose) ou JOIN (SQL) pour reconstituer les données.

📝 Swagger / OpenAPI : Documentation automatique

Swagger génère une interface interactive qui permet de tester les endpoints directement depuis le navigateur.

// Installation npm install swagger-jsdoc swagger-ui-express // swaggerOptions.js const swaggerOptions = { definition: { openapi: '3.0.0', info: { title: 'API Blog', version: '1.0.0', description: 'API pour un système de blog' }, servers: [{ url: 'http://localhost:3000' }] }, apis: ['./routes/*.js'] };

✅ Swagger lit vos commentaires JSDoc dans les routes et génère la doc automatiquement.

✍️ Exemple : Documenter une route avec JSDoc

/** * @swagger * /api/articles: * get: * summary: Récupère tous les articles * tags: [Articles] * responses: * 200: * description: Liste des articles * content: * application/json: * schema: * type: array * items: * $ref: '#/components/schemas/Article' */ router.get('/articles', articleController.getAll);

Après configuration, visitez http://localhost:3000/api-docs pour voir l'interface Swagger UI.

🎯 Documentation = gain de temps pour les développeurs front-end et les testeurs.

🧪 Exercice : Concevoir une API de gestion de tâches

À faire en binôme :

  • 🔹 Étape 1 : Lister les endpoints nécessaires (CRUD tâches, utilisateurs, authentification).
  • 🔹 Étape 2 : Dessiner les modèles (MongoDB ou PostgreSQL) et leurs relations.
  • 🔹 Étape 3 : Choisir les technologies et justifier.
  • 🔹 Étape 4 : Écrire la documentation Swagger pour au moins 3 routes.
// Modèle Task
{
  "title": "string",
  "completed": false,
  "userId": "ObjectId (reference vers User)"
}
// Endpoints :
POST /register, POST /login
GET /tasks, POST /tasks, PUT /tasks/:id, DELETE /tasks/:id
// Relations : User 1---* Task (un utilisateur a plusieurs tâches)
// Technos : Express, MongoDB, JWT, Joi, Swagger

📌 Points clés à retenir

✅ Endpoints = actions sur les ressources (GET, POST, PUT, DELETE).
✅ Modèles = schémas de données (Mongoose ou Sequelize).
✅ Relations : embedding (NoSQL) ou référence (SQL/NoSQL).
✅ Swagger = documentation interactive et standardisée.
🌟 Une bonne conception en amont évite 80% des bugs en production !
