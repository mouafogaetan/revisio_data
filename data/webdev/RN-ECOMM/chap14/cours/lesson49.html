Objectifs de la leçon

  • Préparer les assets (icônes, écrans de chargement, bannières) pour le Play Store.
  • Générer un bundle AAB signé avec EAS Build.
  • Configurer la fiche Play Store (description, captures d'écran, catégorie).
  • Gérer les tracks de déploiement : Internal, Alpha, Beta et Production.
À la fin, vous saurez publier votre application React Native sur le Google Play Store de A à Z.

1. Préparation des assets

Avant de générer le bundle, il faut préparer les ressources graphiques :

Icône de l'application
  • Format PNG (transparent ou fond blanc)
  • Tailles: 512x512 px (Play Store) et 48,72,96,144,192 px (app)
  • Outil recommandé: expo-icon-maker ou npx expo optimize
Écran de chargement (Splash)
  • Fond blanc ou adapté au thème
  • Logo centré (taille ~36dp)
  • Configuration dans app.json → splash
Astuce : Utilisez expo-optimize pour compresser les images sans perte de qualité visible.

2. Bundle AAB avec EAS Build

L'Android App Bundle (AAB) est le format requis par Google Play depuis août 2021. Pour générer un AAB signé avec Expo :


# 1. Installer EAS CLI
npm install -g eas-cli

# 2. Se connecter à Expo
eas login

# 3. Configurer le projet (créer eas.json)
eas build:configure

# 4. Générer la clé de signature (une seule fois)
eas credentials

# 5. Lancer le build de production (AAB)
eas build --platform android --profile production
Important : La clé de signature (.jks) doit être conservée précieusement. Sans elle, impossible de mettre à jour l'application sur le Play Store.

3. Fiche Play Store

Éléments à préparer pour la fiche de présentation :

  • Titre court (max 30 caractères) et titre long (max 80 caractères)
  • Description courte (80 caractères) et description complète (4000 caractères)
  • Captures d'écran : minimum 2, recommandé 8 (formats téléphone, tablette, TV)
  • Bannière feature graphic : 1024 x 500 px
  • Icône haute résolution : 512 x 512 px
  • Catégorie (ex: Shopping, Productivité, Jeux)
Les applications avec des descriptions détaillées et des captures d'écran de qualité ont $+30\%$ de taux de conversion (téléchargements vs vues).

4. Gérer les tracks (canaux de diffusion)

Google Play Console permet de déployer progressivement :

Internal track

Test interne : jusqu'à 100 testeurs (par email). Déploiement rapide en quelques minutes. Idéal pour les tests de dernière minute.

Alpha track

Tests fermés : jusqu'à 2000 testeurs. Nécessite inscription via lien ou groupe Google. Bon pour les bêta-testeurs externes.

Beta track

Tests ouverts : tout utilisateur peut rejoindre via lien. Permet de valider la stabilité avec une audience large.

Production track

Diffusion publique : disponible pour tous les utilisateurs du Play Store. Déploiement progressif recommandé (ex: 5%, 20%, 50%, 100%).

5. Déploiement progressif et rollback

Stratégie recommandée pour minimiser les risques :

📊 Pourcentage d'utilisateurs touchés Durée de surveillance

Étape 1 : 5% → surveiller 24h (crash rate, avis utilisateurs)

Étape 2 : 20% → surveiller 48h

Étape 3 : 50% → surveiller 72h

Étape 4 : 100% → déploiement complet

Rollback : En cas de problème critique, vous pouvez promouvoir une version précédente en production en quelques clics depuis la Console.

6. Checklist pré-publi­cation

Version Code & Version Name cohérentes (ex: versionCode=5, versionName="1.2.0")
Politique de confidentialité (lien URL obligatoire)
Test sur au moins 10 appareils virtuels (Firebase Test Lab)
Niveaux API min et target cibles (min SDK 24, target 33+)
Droits et permissions justifiées (ex: pourquoi la caméra ?)
Tarif : 25$ (paiement unique pour compte développeur)
Le délai de validation par Google est de quelques heures à 3 jours.
