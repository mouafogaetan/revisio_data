📦 Provider standard : le plus simple

Par défaut, on déclare un service dans providers: [UsersService]. C'est un sucre syntaxique pour :

{ provide: UsersService, useClass: UsersService }

NestJS crée une instance du service et l'injecte là où il est demandé. Mais parfois on a besoin de plus de flexibilité :

  • Utiliser une valeur simple (chaîne, nombre, objet) → useValue
  • Remplacer une classe par une autre → useClass
  • Créer dynamiquement le provider → useFactory

🔹 useValue : injecter une constante ou une configuration

// app.module.ts import { Module } from '@nestjs/common'; export const APP_CONFIG = 'APP_CONFIG'; @Module({ providers: [ { provide: APP_CONFIG, useValue: { env: 'development', port: 3000, apiUrl: 'https://api.example.com', }, }, ], }) export class AppModule {} // config.service.ts import { Injectable, Inject } from '@nestjs/common'; import { APP_CONFIG } from './app.module'; @Injectable() export class ConfigService { constructor(@Inject(APP_CONFIG) private config) {} get(key: string) { return this.config[key]; } }

💡 Cas d'usage : injecter une configuration statique, un mock dans les tests, ou une bibliothèque tierce non compatible avec l'injection standard.

🔹 useClass : substitution de classe

Permet de dire à NestJS : "quand on demande le token T, donne une instance de cette autre classe".

// mail.service.ts export interface MailService { send(to: string, message: string): void; } export class SmtpMailService implements MailService { send(to: string, message: string) { console.log(`SMTP to ${to}: ${message}`); } } export class MockMailService implements MailService { send(to: string, message: string) { console.log(`[MOCK] ${to}`); } } // app.module.ts @Module({ providers: [ { provide: 'MAIL_SERVICE', useClass: process.env.NODE_ENV === 'production' ? SmtpMailService : MockMailService, }, ], })

✨ Utile pour basculer entre implémentations réelles et mocks selon l'environnement.

🔹 useFactory : provider dynamique

// database.provider.ts import { createConnection } from 'typeorm'; export const databaseProviders = [ { provide: 'DATABASE_CONNECTION', useFactory: async (configService: ConfigService) => { const connection = await createConnection({ type: 'postgres', host: configService.get('DB_HOST'), port: configService.get('DB_PORT'), username: configService.get('DB_USER'), password: configService.get('DB_PASSWORD'), database: configService.get('DB_NAME'), }); return connection; }, inject: [ConfigService], }, ];

⚙️ useFactory permet de créer le provider de manière asynchrone. La factory reçoit les dépendances listées dans inject.

⚠️ Important : la factory peut être async. NestJS attendra sa résolution avant d'injecter le provider.

⏳ Providers asynchrones avancés

On peut combiner useFactory avec des dépendances asynchrones.

// config.provider.ts export const configProvider = { provide: 'CONFIG', useFactory: async () => { const data = await fetch('https://config-server.com/my-app.json'); return data.json(); }, }; // utilisation dans un service @Injectable() export class AppService { constructor(@Inject('CONFIG') private config: any) {} }
✅ Bonnes pratiques : Toujours gérer les erreurs dans les factories async (try/catch), et utiliser des tokens explicites.

🏷️ Tokens personnalisés avec @Inject()

Quand on utilise provide avec une chaîne ou un symbole, on doit utiliser @Inject() pour injecter.

// constants.ts export const CONNECTION_TOKEN = 'CONNECTION_TOKEN'; // database.module.ts providers: [ { provide: CONNECTION_TOKEN, useValue: mongoose.connect('mongodb://localhost:27017/mydb'), }, ] // user.service.ts import { Inject, Injectable } from '@nestjs/common'; import { CONNECTION_TOKEN } from './constants'; @Injectable() export class UserService { constructor(@Inject(CONNECTION_TOKEN) private connection) {} }
📌 Pourquoi ? Les tokens évitent les collisions de noms et améliorent la clarté du code.
