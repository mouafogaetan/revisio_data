Tests E2E avec Supertest

Valider toute l'application : endpoints HTTP, base de données et scénarios complets

🧪 supertest ⚙️ Test.createTestingModule 🗄️ base de test

🎯 Objectifs de la leçon

  • ✓ Configurer les tests end-to-end (e2e) avec supertest dans NestJS.
  • ✓ Tester des endpoints HTTP complets : requêtes, réponses, codes HTTP.
  • ✓ Utiliser une base de données dédiée aux tests (isolation, nettoyage).
  • ✓ Simuler des flux utilisateur (création → récupération → modification).
  • ✓ Appliquer les bonnes pratiques pour des tests e2e robustes et rapides.
💡 Pourquoi e2e ? Ils valident que tous les composants (contrôleurs, services, DB, pipes, guards) fonctionnent ensemble comme attendu.
🧪 1. Supertest & setup

Configurer l'environnement de tests e2e

Supertest permet de simuler des requêtes HTTP sur votre application NestJS sans réellement écouter un port. On utilise Test.createTestingModule pour créer un module d'intégration.

// test/app.e2e-spec.ts
import { Test, TestingModule } from '@nestjs/testing';
import { INestApplication } from '@nestjs/common';
import * as request from 'supertest';
import { AppModule } from '../src/app.module';

describe('AppController (e2e)', () => {
  let app: INestApplication;

  beforeAll(async () => {
    const moduleFixture: TestingModule = await Test.createTestingModule({
      imports: [AppModule],
    }).compile();

    app = moduleFixture.createNestApplication();
    await app.init();
  });

  afterAll(async () => {
    await app.close();
  });

  it('/ (GET)', () => {
    return request(app.getHttpServer())
      .get('/')
      .expect(200)
      .expect('Hello World!');
  });
});
📌 Bon à savoir : on utilise beforeAll pour initialiser l'application une fois avant tous les tests, et afterAll pour fermer proprement.

📬 Tester des endpoints REST

On peut enchaîner les requêtes : créer une ressource, la lire, la mettre à jour et la supprimer.

it('should create a user and then fetch it', async () => {
  const createDto = { name: 'Alice', email: 'alice@example.com' };
  
  const createResponse = await request(app.getHttpServer())
    .post('/users')
    .send(createDto)
    .expect(201);
  
  const userId = createResponse.body.id;
  
  const getResponse = await request(app.getHttpServer())
    .get(`/users/${userId}`)
    .expect(200);
  
  expect(getResponse.body.name).toEqual('Alice');
});

it('should return 404 for non-existent user', () => {
  return request(app.getHttpServer())
    .get('/users/99999')
    .expect(404);
});
✅ .expect(201) → création réussie
✅ .send() → corps de la requête
🗄️ 2. Base de données dédiée

Isoler les tests avec une DB temporaire

Pour éviter les interférences, on peut utiliser une base de test (ex: SQLite en mémoire ou PostgreSQL dédié). Avec TypeORM, il suffit de modifier la configuration.

// avant les tests, utiliser un fichier .env.test ou override
const moduleFixture = await Test.createTestingModule({
  imports: [AppModule],
})
.overrideProvider(TypeOrmModule)
.useValue(TypeOrmModule.forRoot({
  type: 'sqlite',
  database: ':memory:',
  synchronize: true,
  entities: [User, Product],
}))
.compile();
💡 Pratique : Nettoyer la base avant chaque test avec beforeEach (vider les tables ou utiliser des transactions rollback).

🔐 Tester des routes protégées (JWT)

On simule une connexion, on récupère un token, puis on l'utilise dans le header Authorization.

it('should access protected profile', async () => {
  // 1) Login
  const loginRes = await request(app.getHttpServer())
    .post('/auth/login')
    .send({ email: 'test@mail.com', password: 'pass123' })
    .expect(200);
  
  const token = loginRes.body.access_token;
  
  // 2) Appel à une route sécurisée
  return request(app.getHttpServer())
    .get('/users/profile')
    .set('Authorization', `Bearer ${token}`)
    .expect(200)
    .expect(res => {
      expect(res.body.email).toBe('test@mail.com');
    });
});
⚡ astuce: vous pouvez créer un utilisateur de test directement via le repository pendant beforeAll pour disposer d'un compte prêt à l'emploi.

🧹 Organisation & nettoyage des données

  • Utiliser beforeEach pour réinitialiser l'état (vider tables clés).
  • Créer des fixtures (données de test) reproductibles.
  • Grouper les tests par contexte avec describe().
  • Lancer les tests e2e avec un script dédié : "test:e2e": "jest --config ./test/jest-e2e.json".
// Exemple de remise à zéro avec TypeORM
beforeEach(async () => {
  const repo = app.get(getRepositoryToken(User));
  await repo.query('DELETE FROM user;'); // SQL direct
  // ou utiliser repo.clear()
});
✅ Objectif : chaque test doit être indépendant et reproductible.
