🌐 Route paramétrée : [FromRoute]

En API REST, on inclut souvent un identifiant directement dans le chemin :

GET /api/produits/42   → récupérer le produit avec l'id 42

Avec ASP.NET Core, on utilise l’attribut [FromRoute] pour lier une variable d’URL à un paramètre d’action.

💡 Exemple concret :

[HttpGet("{id}")]
public IActionResult GetProduit([FromRoute] int id)
{
    return Ok($"Produit reçu avec l'ID : {id}");
}
📌 Note : Le paramètre {id} dans le template de route correspond au nom du paramètre du contrôleur.

🔎 Query strings : [FromQuery]

Les paramètres après le ? dans l'URL servent à filtrer, paginer ou trier.

GET /api/produits?categorie=informatique&page=2&taille=10

Dans votre contrôleur, il suffit d'ajouter [FromQuery] :

public IActionResult GetProduits(
    [FromQuery] string? categorie,
    [FromQuery] int page = 1,
    [FromQuery] int taille = 10)
{
    return Ok($"Filtre: {categorie}, page {page}, taille {taille}");
}
✅ Paramètres optionnels ✅ Valeurs par défaut ✅ Compatible avec int, string, bool, etc.

⚙️ Paramètres optionnels et valeurs par défaut

Quand vous définissez un paramètre avec une valeur par défaut, ASP.NET Core le rend automatiquement optionnel.

// Le client peut ne pas fournir "statut", la valeur sera "actif"
public IActionResult Rechercher([FromQuery] string statut = "actif")
{
    return Ok($"Statut recherché: {statut}");
}

Pour un paramètre de route optionnel, mettez un ? dans le template :

[HttpGet("{id?}")]
public IActionResult GetById([FromRoute] int? id)
{
    if (id == null) return Ok("Liste complète");
    return Ok($"Détail ID {id}");
}
🧠 Bon à savoir : Avec les types référence (string, objet), la valeur par défaut est null. Pour les int, double, vous pouvez utiliser int? pour accepter null.

📘 Cas concret : API de produits

[ApiController]
[Route("api/[controller]")]
public class ProduitsController : ControllerBase
{
    [HttpGet("{id}")]
    public IActionResult GetById([FromRoute] int id)
    {
        return Ok(new { id, nom = "Ordinateur" });
    }

    [HttpGet]
    public IActionResult List(
        [FromQuery] string? categorie,
        [FromQuery] int page = 1,
        [FromQuery] int limite = 5)
    {
        // Simulons un résultat paginé
        return Ok(new { 
            categorie = categorie ?? "toutes",
            page, 
            limite,
            resultats = new[] { "Item1", "Item2" }
        });
    }
}
➡️ Requête GET /produits/7

→ id = 7 (FromRoute)

🔍 GET /produits?categorie=telephonie&page=2

→ categorie="telephonie", page=2, limite=5 (défaut)

✨ Astuces pratiques

  • Nommage cohérent : Le nom du paramètre de méthode doit correspondre à la variable de route ou au nom du champ dans la query string.
  • Utilisez des types simples : int, string, bool, DateTime, Guid, etc. La liaison de modèle fonctionne automatiquement.
  • FromRoute et FromQuery peuvent cohabiter : action GetByFilter(int id, [FromQuery] string search)
  • Gérer le cas null : Vérifiez si un paramètre optionnel est null pour éviter les exceptions.
  • Swagger les documente automatiquement : Tout paramètre FromRoute ou FromQuery apparaît dans la doc OpenAPI.

⚠️ Évitez de mettre des données sensibles dans l'URL (query strings) car elles sont loggées.

🎉 Récapitulons

[FromRoute]

Utilisé pour les valeurs obligatoires dans le chemin (ex: id, slug). Ex: /user/123 → id=123

[FromQuery]

Pour les critères de recherche, pagination, tri. Ex: ?page=3&nom=to
