📌 Étape 1 – Définition du projet

Choisir un domaine & modéliser la base de données

Pour construire une API complète, il faut d'abord un cas concret. Exemples :

📝 Blog : Articles, Catégories, Commentaires, Utilisateurs.
🛒 E-commerce : Produits, Commandes, Paniers, Clients.
✅ Gestion de tâches : Tâches, Projets, Statuts, Priorités.
👥 Gestion utilisateurs : Utilisateurs, Rôles, Permissions.

Modélisation des relations

Exemple pour un blog :

  • Un Utilisateur peut écrire plusieurs Articles (One-to-Many).
  • Un Article appartient à une Catégorie (Many-to-One).
  • Relation Many-to-Many entre Articles et Tags (table de liaison).
public class Article {
    public int Id { get; set; }
    public string Titre { get; set; }
    public int UtilisateurId { get; set; }
    public Utilisateur Auteur { get; set; }
    public ICollection<Tag> Tags { get; set; }
}

✨ L'architecture typique : Controllers → Services → Repository → DbContext → DTOs.

🏗️ Étape 2 – Architecture professionnelle

Séparer les responsabilités

Pourquoi des DTO ? Ils permettent de contrôler les données renvoyées, éviter les cycles de références et améliorer la sécurité.

public class ArticleDto {
  public int Id { get; set; }
  public string Titre { get; set; }
  public string NomAuteur { get; set; } // seulement ce dont le client a besoin
}

Utilisez AutoMapper pour simplifier les conversions entre Entité ↔ DTO.

⚡ Étape 3 – Implémentation fonctionnelle

Endpoints CRUD, authentification JWT, rôles

Avec les patterns vus précédemment, on code :

  • [HttpGet] pour récupérer la liste des ressources.
  • [HttpPost] avec [FromBody] pour créer.
  • [Authorize] + JWT pour protéger les endpoints sensibles.
  • Attributs de validation : [Required], [StringLength] – validation automatique via [ApiController].

🔐 Exemple d’ajout de rôle :

[Authorize(Roles = "Admin")]
[HttpDelete("{id}")]
public async Task<IActionResult> DeleteArticle(int id) { ... }

Pensez aussi à la journalisation (ILogger) et aux health checks (/health) pour la surveillance.

🧪 Étape 4 – Qualité & mise en ligne

Documenter, tester, déployer

📄 Swagger

Ajoutez AddSwaggerGen() et UseSwaggerUI(). Documentez avec les commentaires XML et [ProducesResponseType].

🧪 Tests xUnit + Moq

Testez vos services, repository mockés, et contrôlez les réponses HTTP avec WebApplicationFactory.

🐳 Déploiement

Utilisez dotnet publish, Docker, Azure App Service ou AWS. N'oubliez pas les variables d'environnement.

🏥 Health checks

AddHealthChecks().AddDbContextCheck() pour surveiller la base de données.

🚀 Mise en pratique

Exemple de projet : API de gestion de tâches

Pour vous entraîner, voici un plan d'action pour créer une API REST complète :

  1. Modèle : Tâche (Id, Titre, Description, EstTerminée, DateLimite, UtilisateurId). Utilisateur (Id, Nom, Email, MotDePasseHash, Role).
  2. Relations : un utilisateur possède plusieurs tâches (One-to-Many).
  3. CRUD : endpoints pour gérer les tâches, accessible uniquement aux utilisateurs authentifiés.
  4. JWT : login génère un token avec claims (role, userId).
  5. Validation : le titre est requis, longueur max 100 caractères.
  6. Filtres : possibilité de récupérer les tâches par utilisateur (query parameter).
  7. Documentation Swagger + tests unitaires sur le service Tâche.
💡 Conseil : Commencez par créer les entités puis le DbContext, les migrations, le service, enfin les contrôleurs.
