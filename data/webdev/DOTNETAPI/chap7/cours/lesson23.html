📄 Étape 1 : Commentaires XML pour documenter les endpoints

En ASP.NET Core, vous pouvez ajouter des balises XML dans vos contrôleurs et actions. Swagger les lira automatiquement pour enrichir la documentation.

/// <summary>
/// Récupère la liste de tous les produits.
/// </summary>
/// <returns>Une liste d'objets ProductDTO</returns>
/// <response code="200">Retourne la liste des produits</response>
[HttpGet]
public async Task<ActionResult<IEnumerable<ProductDTO>>> GetProducts()
{
    return Ok(await _productService.GetAllAsync());
}

✨ Activez la documentation XML dans .csproj : <GenerateDocumentationFile>true</GenerateDocumentationFile> et configurez Swagger.

🧠 Conseil : Décrivez chaque paramètre, le code de retour et les exceptions possibles. Vos collègues vous remercieront !

📌 [ProducesResponseType] : documenter les codes HTTP

Cet attribut permet de spécifier explicitement les retours HTTP possibles (200, 400, 404, 201...). Swagger les affiche clairement.

[HttpGet("{id}")]
[ProducesResponseType(StatusCodes.Status200OK)]
[ProducesResponseType(StatusCodes.Status404NotFound)]
[ProducesResponseType(StatusCodes.Status400BadRequest)]
public async Task<ActionResult<ProductDTO>> GetProductById(int id)
{
    if (id <= 0) return BadRequest();
    var product = await _productService.GetByIdAsync(id);
    return product is null ? NotFound() : Ok(product);
}
✅ 200 Succès
❌ 404 Non trouvé
⚠️ 400 Requête invalide

📋 Exemples de requêtes / réponses dans Swagger

Vous pouvez décorer vos DTO avec des exemples ou utiliser [SwaggerRequestExample] ou les balises XML étendues, ou encore [DefaultValue], [SwaggerResponseExample] (via extensions).

/// <example>1</example>
/// <response example="true">Objet produit complet</response>
[ProducesResponseType(typeof(ProductDTO), 200)]
[SwaggerResponseExample(200, typeof(ProductExampleProvider))]

⚡ Avec Swashbuckle.AspNetCore.Annotations : [SwaggerRequestBody] ou [SwaggerResponse] pour fournir des exemples concrets.

⚙️ Configuration complète dans Program.cs

builder.Services.AddSwaggerGen(c => 
{
    c.SwaggerDoc("v1", new OpenApiInfo { Title = "Mon API", Version = "v1" });
    var xmlFile = $"{Assembly.GetExecutingAssembly().GetName().Name}.xml";
    var xmlPath = Path.Combine(AppContext.BaseDirectory, xmlFile);
    c.IncludeXmlComments(xmlPath);
    // Ajout d'annotations
    c.EnableAnnotations(); // Swashbuckle.AspNetCore.Annotations
});

Pensez à installer les packages Swashbuckle.AspNetCore et Swashbuckle.AspNetCore.Annotations.

🔍 Après configuration, lancez l'API, accédez à /swagger : vous verrez vos commentaires XML et les détails des codes HTTP.

🏆 Bonnes pratiques & astuces

  • Documentez toujours les codes d’erreur (400, 401, 403, 404, 500).
  • Utilisez des constantes pour les messages d’exemple.
  • Ajoutez des descriptions pour les paramètres de route/query avec <param> XML.
  • Pour les en-têtes d’authentification, documentez la sécurité JWT avec AddSecurityDefinition dans Swagger.
  • Mettez à jour la documentation dès que l’API évolue !
