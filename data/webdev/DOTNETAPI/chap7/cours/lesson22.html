Documenter son API avec Swagger

Générez automatiquement une documentation interactive, testez vos endpoints et facilitez l’intégration.

🎯 Niveau : Intermédiaire ⏱️ Durée : 40 min 📘 Swagger / OpenAPI

📌 Objectifs d'apprentissage

À la fin de cette leçon, vous serez capable de :

  • ✓ Comprendre le rôle de Swagger / OpenAPI dans la documentation d’API.
  • ✓ Configurer Swagger dans une API ASP.NET Core (AddSwaggerGen, UseSwagger).
  • ✓ Exploiter Swagger UI pour tester les endpoints interactivement.
  • ✓ Personnaliser la documentation avec des informations sur l’API.

💡 Pourquoi documenter ? Une API bien documentée réduit les erreurs d’intégration, accélère la collaboration entre équipes et sert de référence unique.

📖 Swagger / OpenAPI en bref

OpenAPI (anciennement Swagger) est une spécification standard pour décrire les API REST. Le format est lisible par des humains et des machines (JSON/YAML).

Swagger UI est l’interface web interactive générée à partir de cette spécification : elle liste tous les endpoints, les paramètres attendus, et permet d’exécuter des requêtes directement depuis le navigateur.

🎯 Avantages pour un développeur :

  • Documentation toujours à jour (générée depuis le code).
  • Tests rapides sans outil externe (Postman).
  • Génération automatique de SDK clients pour différents langages.
// Exemple d’extrait de spécification OpenAPI (généré par ASP.NET Core)
{
  "openapi": "3.0.1",
  "paths": {
    "/api/products": {
      "get": {
        "summary": "Récupère tous les produits",
        "responses": { "200": { "description": "Succès" } }
      }
    }
  }
}

⚙️ Configurer Swagger (swagger.json + UI)

Les packages Swashbuckle.AspNetCore sont inclus par défaut dans le template Web API. Voici les étapes de configuration dans Program.cs :

var builder = WebApplication.CreateBuilder(args);
builder.Services.AddControllers();
// 1️⃣ Ajouter le générateur Swagger
builder.Services.AddEndpointsApiExplorer();
builder.Services.AddSwaggerGen();

var app = builder.Build();

// 2️⃣ Activer Swagger middleware (génère le fichier swagger.json)
if (app.Environment.IsDevelopment())
{
    app.UseSwagger();
    app.UseSwaggerUI(); // Interface interactive
}

app.UseAuthorization();
app.MapControllers();
app.Run();

📌 AddEndpointsApiExplorer est requis pour exposer les métadonnées des endpoints. UseSwaggerUI rend l’interface accessible sur /swagger.

💡 En production, vous pouvez désactiver Swagger UI ou le restreindre via app.Environment.IsDevelopment().

🎨 Personnaliser les métadonnées de l’API

On peut enrichir la documentation avec des informations globales (titre, description, version, contact).

builder.Services.AddSwaggerGen(options =>
{
    options.SwaggerDoc("v1", new OpenApiInfo
    {
        Title = "Mon API E-commerce",
        Version = "v1",
        Description = "API de gestion de produits et commandes",
        Contact = new OpenApiContact
        {
            Name = "Support Technique",
            Email = "support@exemple.com"
        }
    });
});

Cette information apparaîtra en haut de l’interface Swagger UI.

🏷️ Bonne pratique :

Indiquez toujours la version de l’API, un point de contact et une description claire pour les développeurs consommateurs.

✍️ Documenter les endpoints avec commentaires XML

Swagger peut extraire les commentaires XML pour enrichir la description des endpoints, paramètres et réponses.

  1. Activez la génération du fichier XML dans le fichier .csproj :
<PropertyGroup>
  <GenerateDocumentationFile>true</GenerateDocumentationFile>
  <NoWarn>$(NoWarn);1591</NoWarn>
</PropertyGroup>
  1. Configurez Swagger pour utiliser le fichier XML :
var xmlFile = $"{Assembly.GetExecutingAssembly().GetName().Name}.xml";
var xmlPath = Path.Combine(AppContext.BaseDirectory, xmlFile);
options.IncludeXmlComments(xmlPath);

3. Ajoutez des commentaires XML sur vos actions :

/// <summary>Récupère un produit par son identifiant</summary>
/// <param name="id">Identifiant unique du produit</param>
/// <response code="200">Produit trouvé</response>
/// <response code="404">Produit introuvable</response>
[HttpGet("{id}")]
public IActionResult GetProduct(int id) { ... }
✅ Résultat : Swagger UI affichera les descriptions, les types de paramètres et les codes de réponse directement dans l’interface.

🧪 Tester les endpoints via Swagger UI

Une fois votre API lancée, ouvrez https://localhost:5001/swagger (ou le port configuré). Vous verrez :

  • La liste de tous les contrôleurs/actions avec les méthodes HTTP (GET, POST, etc.).
  • La possibilité d’étendre chaque endpoint pour voir les paramètres.
  • Un bouton "Try it out" : vous pouvez saisir des valeurs et exécuter la requête directement.
  • Affichage des réponses (code HTTP, corps, en-têtes).

📌 Exemple de flux de test :

1. Cliquez sur POST /api/products → "Try it out".
2. Modifiez le corps JSON (ex: {"name": "Clavier"}).
3. Cliquez sur "Execute".
4. Visualisez la réponse et le code 201/200.

✅ Bonnes pratiques & pièges courants

  • ✔️ Toujours versionner votre documentation : options.SwaggerDoc("v1", ...) pour chaque version.
  • ✔️ Ne pas exposer Swagger UI en production sans protection : utilisez des conditions ou un middleware d’authentification.
  • ✔️ Documenter les codes de retour avec [ProducesResponseType] pour une meilleure clarté.
  • ❌ Éviter les descriptions obsolètes : mettez à jour les commentaires XML à chaque modification d’endpoint.
  • ❌ Ne pas publier le fichier .xml public en production (il reste sur le serveur, mais sans risque majeur).

⚠️ Piège fréquent :

Oublier d’appeler AddEndpointsApiExplorer() → SwaggerUI ne trouvera aucun endpoint. Vérifiez toujours cette ligne dans Program.cs.
