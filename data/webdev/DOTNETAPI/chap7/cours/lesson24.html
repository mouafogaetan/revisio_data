Tests unitaires &
intégration avec xUnit + Moq

Garantir la fiabilité de votre API grâce aux tests automatisés.

🎯 xUnit ⚙️ Moq 🧪 Tests d'intégration

🎯 Objectifs d'apprentissage

  • ✓ Créer un projet de tests avec xUnit dans une solution .NET.
  • ✓ Tester les services métier et le Repository pattern.
  • ✓ Utiliser Moq pour mocker les dépendances (DbContext, IRepository).
  • ✓ Assurer l'isolation des tests unitaires et valider la logique.
💡 Pourquoi tester ? Réduire les bugs, faciliter la refactorisation et documenter le comportement attendu.

📦 Mise en place – xUnit

Dans une API .NET, on ajoute un projet séparé de type xUnit Test Project.

📁 MonApi.sln
 ├── 📁 MonApi (projet API)
 └── 📁 MonApi.Tests (tests unitaires)
      ├── ServicesTests.cs
      ├── RepositoriesTests.cs
      └── Utils/

Installer les packages NuGet :

dotnet add MonApi.Tests package Microsoft.NET.Test.Sdk
dotnet add MonApi.Tests package xunit
dotnet add MonApi.Tests package xunit.runner.visualstudio
dotnet add MonApi.Tests package Moq
dotnet add MonApi.Tests package FluentAssertions   (optionnel)
🧠 Important : Une classe de test est publique et contient des méthodes annotées avec [Fact] ou [Theory].

🎭 Mocking avec Moq

On veut tester ProductService qui dépend d’un repository IProductRepository. On va mocker cette dépendance.

// Exemple : service qui retourne un produit par son Id
public class ProductService {
    private readonly IProductRepository _repo;
    public ProductService(IProductRepository repo) => _repo = repo;
    public async Task<ProductDto> GetProductAsync(int id) {
        var product = await _repo.GetByIdAsync(id);
        return product == null ? null : new ProductDto { Id = product.Id, Name = product.Name };
    }
}

// TEST avec Moq & xUnit
public class ProductServiceTests {
    [Fact]
    public async Task GetProductAsync_ReturnsDto_WhenProductExists() {
        // Arrange
        var mockRepo = new Mock<IProductRepository>();
        var fakeProduct = new Product { Id = 42, Name = "Laptop" };
        mockRepo.Setup(repo => repo.GetByIdAsync(42)).ReturnsAsync(fakeProduct);
        var service = new ProductService(mockRepo.Object);
        
        // Act
        var result = await service.GetProductAsync(42);
        
        // Assert
        Assert.NotNull(result);
        Assert.Equal("Laptop", result.Name);
        mockRepo.Verify(r => r.GetByIdAsync(42), Times.Once);
    }
}

✅ Moq permet de simuler le comportement sans toucher à la vraie base de données.

🌐 Tests d'intégration – WebApplicationFactory

Pour tester les endpoints complets (y compris pipeline, middleware, contrôleurs) on utilise Microsoft.AspNetCore.Mvc.Testing.

// Intégration : tester un contrôleur ProductsController
public class ProductsIntegrationTests : IClassFixture<WebApplicationFactory<Program>>
{
    private readonly HttpClient _client;
    public ProductsIntegrationTests(WebApplicationFactory<Program> factory)
    {
        _client = factory.CreateClient();
    }

    [Fact]
    public async Task GetProducts_ReturnsOk()
    {
        // Act
        var response = await _client.GetAsync("/api/products");
        // Assert
        Assert.Equal(HttpStatusCode.OK, response.StatusCode);
    }
}
  • WebApplicationFactory lance l’API en mémoire.
  • Permet de remplacer les services (ex: base InMemory pour EF Core).
  • Idéal pour valider les flux complets (authentification, validation, etc).

🗄️ Base de données InMemory

Pour éviter d'utiliser une vraie base SQL pendant les tests, on utilise Microsoft.EntityFrameworkCore.InMemory.

// Dans un fixture personnalisé
services.AddDbContext<AppDbContext>(options =>
    options.UseInMemoryDatabase("TestDb"));

// Exemple de test avec un repository réel en mémoire
[Fact]
public async Task AddProduct_SavesToDatabase()
{
    var options = new DbContextOptionsBuilder<AppDbContext>()
        .UseInMemoryDatabase(databaseName: Guid.NewGuid().ToString())
        .Options;
    using var context = new AppDbContext(options);
    var repo = new ProductRepository(context);
    
    var product = new Product { Name = "Test" };
    await repo.AddAsync(product);
    
    Assert.Equal(1, context.Products.Count());
}
📌 Chaque test doit avoir une base isolée (nouveau Guid) pour éviter les interférences.

✅ Récapitulatif & Bonnes pratiques

📌 Tests unitaires

  • Testez une seule classe/méthode à la fois.
  • Moq pour les dépendances externes.
  • Noms explicites : Méthode_Scénario_Attendu.

🌊 Tests d'intégration

  • WebApplicationFactory + InMemory DB.
  • Vérifiez les codes HTTP et payloads JSON.
  • Testez également les middlewares de gestion d'erreurs.
🧪 À retenir : Un projet sans tests est un projet qui finira par planter en production.
Avec xUnit + Moq, vous construisez des API robustes et maintenables.
