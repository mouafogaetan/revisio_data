Pourquoi le Repository ?

Il centralise les requêtes EF Core, évite la duplication de code et facilite les tests unitaires (mock du repository).

📦 Qu'est-ce que le pattern Repository ?

Le Repository est un design pattern qui agit comme une collection en mémoire de vos entités. Il fait le pont entre la couche métier (services/contrôleurs) et la couche d'accès aux données (EF Core).

Avantages clairs :
  • ✔️ Séparation des responsabilités (code plus propre)
  • ✔️ Centralisation de la logique de requêtage
  • ✔️ Facilite les tests unitaires (on peut mocker l’interface)
  • ✔️ Changement d’ORM sans impacter les contrôleurs
// Interface générique simple
public interface IRepository<T> where T : class
{
  Task<T> GetByIdAsync(int id);
  Task<IEnumerable<T>> GetAllAsync();
  Task AddAsync(T entity);
  void Update(T entity);
  void Delete(T entity);
  Task<bool> SaveChangesAsync();
}

⚙️ Implémentation avec EF Core

On crée une classe générique GenericRepository<T> qui utilise DbContext.

public class GenericRepository<T> : IRepository<T> where T : class
{
  protected readonly AppDbContext _context;
  protected readonly DbSet<T> _dbSet;

  public GenericRepository(AppDbContext context)
  { _context = context; _dbSet = context.Set<T>(); }

  public async Task<T> GetByIdAsync(int id) => await _dbSet.FindAsync(id);
  public async Task<IEnumerable<T>> GetAllAsync() => await _dbSet.ToListAsync();
  public async Task AddAsync(T entity) => await _dbSet.AddAsync(entity);
  public void Update(T entity) => _dbSet.Update(entity);
  public void Delete(T entity) => _dbSet.Remove(entity);
  public async Task<bool> SaveChangesAsync() => await _context.SaveChangesAsync() > 0;
}
On évite de répéter du code pour chaque entité (Product, Order, User…).

🔌 Enregistrement dans le conteneur DI

Dans Program.cs (ou Startup.cs), on enregistre le repository générique.

// Ajouter le DbContext (SQL Server par exemple)
builder.Services.AddDbContext<AppDbContext>(options =>
    options.UseSqlServer(builder.Configuration.GetConnectionString("Default")));

// Enregistrer le repository générique (Scoped ou Transient ?)
builder.Services.AddScoped(typeof(IRepository<>), typeof(GenericRepository<>));

// Optionnel : enregistrer un repository spécifique (ex: IProductRepository)
builder.Services.AddScoped<IProductRepository, ProductRepository>();
Durée de vie Scoped : le repository vit pendant toute la requête HTTP (recommandé avec DbContext).

🎮 Injection dans un Controller

Plus besoin d'instancier manuellement le DbContext. Le contrôleur reçoit l’interface via son constructeur.

[ApiController]
[Route("api/[controller]")]
public class ProductsController : ControllerBase
{
  private readonly IRepository<Product> _productRepo;

  public ProductsController(IRepository<Product> productRepo)
  { _productRepo = productRepo; }

  [HttpGet]
  public async Task<IActionResult> GetAll()
  {
    var products = await _productRepo.GetAllAsync();
    return Ok(products);
  }

  [HttpPost]
  public async Task<IActionResult> Create(Product product)
  {
    await _productRepo.AddAsync(product);
    await _productRepo.SaveChangesAsync();
    return CreatedAtAction(nameof(GetAll), new { id = product.Id }, product);
  }
}

✅ Tests unitaires & moindre couplage

Grâce à IRepository<T>, on peut mocker le repository dans un test xUnit :

// Exemple avec Moq
var mockRepo = new Mock<IRepository<Product>>();
mockRepo.Setup(repo => repo.GetAllAsync())
    .ReturnsAsync(new List<Product> { new Product { Id = 1, Name = "Test" } });
var controller = new ProductsController(mockRepo.Object);
var result = await controller.GetAll();
// Assert ...
Ne pas exposer IQueryable depuis le repository (limite le couplage) Repository spécifique = méthodes métier (ex: GetByCategory)
