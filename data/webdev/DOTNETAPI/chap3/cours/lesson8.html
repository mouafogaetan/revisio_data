Pattern Injection de dépendances

Comprendre le couplage faible, la testabilité et les durées de vie des services

Durée : 45 min Lesson 8 / Chapitre 3
Appuyez sur Suivant pour découvrir les objectifs

Objectifs de la leçon

1. Comprendre le couplage faible et la testabilité

Découvrez pourquoi l'injection de dépendances réduit les dépendances directes et facilite les tests unitaires.

2. Définir les durées de vie (Transient, Scoped, Singleton)

Maîtrisez le cycle de vie de chaque service et leur comportement dans une requête HTTP.

3. Choisir la bonne durée de vie selon le contexte

Savoir quand utiliser Transient, Scoped ou Singleton dans une application web réelle.

Compétence cible : Maîtriser le pattern DI dans ASP.NET Core

Qu'est-ce que le couplage faible ?

En programmation traditionnelle, une classe crée directement ses dépendances (ex: new MonService()). Cela crée un couplage fort : difficile de modifier ou tester indépendamment.

Injection de dépendances = une classe reçoit ses dépendances de l'extérieur (via constructeur, propriété).

Avantages concrets :

  • ✔️ Testabilité : On peut injecter un faux service (mock) pour isoler le test.
  • ✔️ Modularité : On peut échanger une implémentation sans modifier les clients.
  • ✔️ Maintenabilité : Code plus propre, moins de "new" cachés.

Le contrôleur dépend d'une abstraction, pas d'une implémentation concrète.

Durées de vie des services

Transient

Une nouvelle instance est créée chaque fois qu'elle est demandée.

✅ Idéal pour services légers et sans état.

Scoped

Une instance unique par requête HTTP (ou par scope).

✅ Parfait pour DbContext (Entity Framework) et services partagés durant une requête.

Singleton

Une seule instance pour toute l'application (partagée).

✅ Pour les configurations, caches, ou services thread-safe.
Durée de vieCréationExemple type
TransientÀ chaque injectionHelper, générateur de GUID
ScopedUne fois par requêteDbContext, service métier lié à la requête
SingletonUne seule au démarrageIConfiguration, logger, cache mémoire
À retenir : Si un service Singleton dépend d’un Scoped, cela crée un problème (capitaine évident) ! Toujours respecter la hiérarchie.

Choisir selon le contexte

Accès à la base de données

Utilisez Scoped pour partager le même DbContext durant une requête et éviter les conflits.

Service utilitaire / formatage

Transient est idéal pour des opérations légères et sans état, pas de risque de contamination.

Cache applicatif / configuration

Singleton pour conserver les données globales en mémoire.

Service métier avec état local (par requête)

Scoped, afin que chaque utilisateur possède sa propre instance pendant le traitement HTTP.

Règle d’or : Singleton → Scoped → Transient (un Singleton ne doit JAMAIS capturer un Scoped)

Enregistrement des services - Pratique

Dans Program.cs on configure le conteneur DI :

// Ajout des services avec différentes durées de vie
builder.Services.AddTransient<IEmailSender, SmtpEmailSender>();
builder.Services.AddScoped<IProductRepository, ProductRepository>();
builder.Services.AddSingleton<IConfiguration>(builder.Configuration);

// Exemple avec options
builder.Services.AddScoped<IOrderService, OrderService>();

💡 Injection dans un contrôleur :

public class ProductsController : ControllerBase
{
    private readonly IProductRepository _repository;
    private readonly IEmailSender _emailSender;

    public ProductsController(IProductRepository repo, IEmailSender emailSender)
    {
        _repository = repo;
        _emailSender = emailSender;
    }
}

Attention !

Singleton + Scoped = fuite mémoire / risque d'état incohérent – Évitez d'injecter un service Scoped dans un Singleton.
Transient pour des objets lourds ? Création répétée, attention aux performances. Privilégiez Scoped ou Singleton selon l'usage.
Testabilité maximale : Grâce à l'injection, nous pouvons remplacer un service réel par une implémentation factice (Mock) dans les tests unitaires xUnit.
"L' injection de dépendances est un pilier d'ASP.NET Core : elle rend le code extensible, testable et professionnel."

📌 Récapitulatif & Application

  • Le conteneur DI gère la création et la destruction des instances.
  • Transient : neuf à chaque demande.
  • Scoped : une fois par requête (le plus utilisé pour les APIs).
  • Singleton : unique pour toute l’application.
  • On injecte dans les contrôleurs via le constructeur → code propre.
À vous : Pour un service de logging (écriture dans fichier) partagé dans toute l'appli, quelle durée de vie choisiriez-vous ? (Réponse : Singleton généralement)
