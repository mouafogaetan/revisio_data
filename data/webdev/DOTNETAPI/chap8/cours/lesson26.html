🎯 Objectifs de la leçon

  • ✓ Comprendre pourquoi séparer l’API en couches (Controllers, Services, Repository, DTOs).
  • ✓ Implémenter le pattern Service pour isoler la logique métier.
  • ✓ Maîtriser les DTO (Data Transfer Objects) pour éviter l’exposition directe des entités.
  • ✓ Utiliser AutoMapper pour mapper automatiquement Entités ↔ DTOs.
💡 À la fin, vous saurez construire une API maintenable, testable et évolutive.

⚠️ Le problème du controller "gros" (Fat Controller)

Quand on met toute la logique métier, l’accès aux données et la validation directement dans le controller, on obtient :

  • Un code difficile à tester (besoin de la base réelle)
  • Du code dupliqué entre plusieurs endpoints
  • Une maintenance compliquée (tout est mélangé)
  • Des difficultés pour changer la source de données
// ❌ Mauvais exemple [ApiController] [Route("api/products")] public class ProductsController : ControllerBase { private readonly AppDbContext _context; public ProductsController(AppDbContext context) => _context = context; [HttpGet] public async Task GetAll() { var products = await _context.Products.ToListAsync(); return Ok(products); // expose directement l'entité ! } }

Solution : séparer en couches (Separation of Concerns).

🏗️ Architecture recommandée (Clean Architecture simplifiée)

🌐 Controllers
Réception requêtes, renvoi réponses
→
⚙️ Services
Logique métier, orchestration
→
🗄️ Repository
Accès aux données (EF Core)

📦 DTOs (Data Transfer Objects) :

Les contrôleurs ne renvoient jamais directement les entités. On utilise des DTOs pour contrôler exactement les données exposées.

// ProductDto.cs public class ProductDto { public int Id { get; set; } public string Name { get; set; } public decimal Price { get; set; } // pas de champ "InternalStock" par exemple }

📝 Mise en pratique : Service et DTO

1. Interface et Service

public interface IProductService { Task> GetAllProductsAsync(); } public class ProductService : IProductService { private readonly IProductRepository _repository; private readonly IMapper _mapper; public ProductService(IProductRepository repository, IMapper mapper) => (_repository, _mapper) = (repository, mapper); public async Task> GetAllProductsAsync() { var products = await _repository.GetAllAsync(); return _mapper.Map>(products); } }

2. Controller allégé

[ApiController] [Route("api/products")] public class ProductsController : ControllerBase { private readonly IProductService _service; public ProductsController(IProductService service) => _service = service; [HttpGet] public async Task GetAll() { var productsDto = await _service.GetAllProductsAsync(); return Ok(productsDto); } }

🔄 AutoMapper : simplifier les mappings

AutoMapper évite d’écrire manuellement `productDto.Name = product.Name`. Il suffit de déclarer un profil.

// 1. Installer AutoMapper.Extensions.Microsoft.DependencyInjection // 2. Créer un profil public class MappingProfile : Profile { public MappingProfile() { CreateMap<Product, ProductDto>(); CreateMap<CreateProductDto, Product>(); // pour la création } } // 3. Dans Program.cs builder.Services.AddAutoMapper(typeof(Program));
✅ AutoMapper supporte les conventions : propriétés de même nom sont automatiquement mappées.

🧪 Testabilité et avantages

  • On peut mocker le service dans les tests du controller.
  • On peut mocker le repository dans les tests du service.
  • Chaque couche a une responsabilité unique (SRP).
  • On peut changer la source de données (SQL → API externe) sans impacter le controller.
// Exemple test unitaire du service avec Moq var mockRepo = new Mock<IProductRepository>(); mockRepo.Setup(r => r.GetAllAsync()).ReturnsAsync(products); var service = new ProductService(mockRepo.Object, mapper); var result = await service.GetAllProductsAsync(); Assert.Equal(2, result.Count());
