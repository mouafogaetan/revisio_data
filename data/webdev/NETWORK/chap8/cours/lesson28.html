Chiffrement & Certificats

Cryptographie symétrique/asymétrique, PKI, certificats et TLS

🤔 Pourquoi chiffrer les communications ?

Sur Internet, les données traversent de nombreux équipements intermédiaires (routeurs, FAI, serveurs). Sans chiffrement, n'importe qui peut les lire.

PC 🖥️ → [Routeur FAI] → [Serveur DNS] → ... → 🌐 Serveur Web
  • Confidentialité : personne ne peut lire le message.
  • Intégrité : le message n'a pas été modifié en route.
  • Authentification : on est sûr de l'identité du serveur.

🔑 Chiffrement symétrique

Une seule clé secrète sert à chiffrer et déchiffrer.

Message clair → [Chiffrement] → Message chiffré → [Déchiffrement] → Message clair
(même clé des deux côtés)

Exemples : AES (Advanced Encryption Standard), DES, RC4.

Avantage : Très rapide.

Inconvénient : Comment partager la clé secrète à distance sans qu'elle soit interceptée ?

⚠️ Problème du « échange de clés » → solution : chiffrement asymétrique !

🔐 Chiffrement asymétrique (RSA)

Deux clés mathématiquement liées :

  • Clé publique : distribuée librement → pour chiffrer.
  • Clé privée : gardée secrète → pour déchiffrer.
Alice envoie sa clé publique à Bob → Bob chiffre avec la clé publique → Seule Alice peut déchiffrer.

Avantage : Pas besoin de partager une clé secrète à l'avance.

Inconvénient : Lourd et lent (100 à 1000 fois plus lent que AES).

En pratique : on utilise l'asymétrique pour échanger une clé symétrique (hybride).

📐 Petit aperçu RSA (simplifié)

RSA repose sur la difficulté de factoriser un grand nombre en deux nombres premiers.

$p = 61$, $q = 53$ (premiers)
$n = p \times q = 3233$
$\phi(n) = (p-1)(q-1) = 3120$
$e = 17$ (clé publique), $d = 2753$ (clé privée)
Chiffrer $m = 65$ : $c = m^e \mod n = 65^{17} \mod 3233 = 2790$
Déchiffrer $c$ : $m = c^d \mod n = 2790^{2753} \mod 3233 = 65$

En réalité, les clés RSA font 2048 ou 4096 bits (beaucoup plus grands).

📜 Certificats numériques

Un certificat lie une identité (ex: google.com) à une clé publique. Il est signé par une Autorité de Certification (CA).

Contenu typique (X.509) :
- Sujet : CN=google.com, O=Google LLC
- Clé publique (RSA 2048 bits)
- Période de validité
- Empreinte + signature de la CA

La PKI (Public Key Infrastructure) gère l'émission, la révocation et la vérification des certificats.

🔗 Chaîne de confiance

Votre navigateur contient une liste de CA racines préinstallées.

CA Racine (Trusted) │ signe ▼ CA Intermédiaire │ signe ▼ Certificat du site (google.com)
  • Le navigateur vérifie la signature du certificat avec la clé publique de la CA.
  • Si la signature est valide, le site est authentifié.
  • Les certificats auto-signés (pas de CA) déclenchent une erreur.

TLS 1.3 a simplifié le handshake (2 aller-retours au lieu de 3+).

🛡️ Vérification du certificat

  • Est-il signé par une CA de confiance ?
  • La date de validité est-elle correcte ?
  • Le nom du site (CN ou SAN) correspond-il au domaine visité ?
  • Le certificat n'est-il pas révoqué ? (via CRL ou OCSP)
⚠️ Si la vérification échoue, le navigateur affiche un avertissement « Votre connexion n'est pas privée ».

✍️ Signature numérique

Garantit l'intégrité et l'authenticité d'un document.

  1. Calcul d'un hachage (SHA-256) du message.
  2. Chiffrement du hachage avec la clé privée de l'émetteur → signature.
  3. Le récepteur déchiffre avec la clé publique et recalcule le hachage.
  4. Si hachage = signature déchiffrée → message authentique et non modifié.
signature = RSA( SHA-256(message) , clé_privée)

🌍 Let's Encrypt : la CA gratuite

Avant, les certificats étaient payants (comodo, digicert, ...). Let's Encrypt (2015) propose des certificats gratuits et automatisés.

  • Automatisé via ACME (ex: Certbot).
  • Certificats valables 90 jours (renouvellement auto).
  • Utilisé par des millions de sites (dont ce cours est hébergé via HTTPS).
🔓 Aujourd'hui, plus de 80% du trafic Web est chiffré (HTTPS).

📌 Ce qu'il faut retenir

✓ Chiffrement symétrique = même clé (rapide)
✓ Chiffrement asymétrique = clé publique/privée (lent)
✓ Certificat = identité + clé publique + signature CA
✓ PKI = gestion des certificats
✓ TLS = échange clé symétrique + vérification certificat
✓ HTTPS = HTTP + TLS
