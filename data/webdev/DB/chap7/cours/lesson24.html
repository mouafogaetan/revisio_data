🔒 Verrous partagés vs exclusifs

Les SGBD utilisent des verrous pour maintenir l'intégrité des données lorsque plusieurs utilisateurs accèdent/modifient les mêmes données simultanément.

🔓

Verrou partagé (SHARED)

  • Permet la lecture uniquement.
  • Plusieurs transactions peuvent avoir un verrou partagé sur la même ressource.
  • Empêche toute écriture tant qu'un verrou partagé est actif.
  • Ex: SELECT ... LOCK IN SHARE MODE (MySQL).
🔒

Verrou exclusif (EXCLUSIVE)

  • Permet la lecture et l'écriture (modification).
  • Une seule transaction peut détenir un verrou exclusif sur une ressource.
  • Bloque tous les autres verrous (partagés ou exclusifs).
  • Ex: UPDATE, DELETE, INSERT + SELECT ... FOR UPDATE.

🎨 Verrous en action (illustration):

Un verrou partagé permet la lecture simultanée ; un verrou exclusif est nécessaire pour modifier.

⛔ Interblocage (Deadlock)

Un deadlock se produit lorsque deux transactions (ou plus) attendent mutuellement qu'une ressource verrouillée par l'autre soit libérée. Aucune ne peut avancer, et le SGBD doit les annuler.

⚠️ Conditions nécessaires au deadlock

🔁 Exclusion mutuelle
⛓️ Détention et attente
🔃 Non préemption
🌀 Attente circulaire

Exemple classique :

🔁 T1 verrouille A et demande B ; T2 verrouille B et demande A → Interblocage !

🛠️ Résoudre et éviter les deadlocks

Les SGBD modernes détectent automatiquement les deadlocks et choisissent une transaction "victime" à annuler (rollback). Nous pouvons aussi adopter de bonnes pratiques.

1️⃣ Détection automatique

Le SGBD surveille les cycles d'attente. Ex: SHOW ENGINE INNODB STATUS (MySQL) ou SELECT * FROM pg_locks (PostgreSQL).

2️⃣ Timeout des transactions

Une transaction qui attend trop longtemps est automatiquement annulée (ex: innodb_lock_wait_timeout).

3️⃣ Bonnes pratiques pour éviter les deadlocks

  • Accéder aux ressources dans le même ordre (toujours table A puis table B).
  • Réduire la durée des transactions (COMMIT rapide).
  • Utiliser des indexes pour que les verrous soient plus précis (niveau ligne).
  • Éviter les interactions utilisateur au sein d'une transaction longue.

🔐 SELECT ... FOR UPDATE

Cette instruction permet de verrouiller explicitement les lignes sélectionnées en mode exclusif. Utile pour éviter les modifications concurrentes avant une mise à jour conditionnelle.

-- Exemple : réservation de billets sans dépassement de stock
BEGIN;
SELECT stock FROM produits WHERE id = 101 FOR UPDATE;
-- Si stock > 0 alors on diminue
UPDATE produits SET stock = stock - 1 WHERE id = 101;
COMMIT;

✅ Avantages:

  • Empêche les lectures sales / mises à jour perdues.
  • Garantit l'atomicité pour des opérations critiques.

⚠️ Inconvénients:

  • Risque accru de deadlock si mal utilisé.
  • Réduit la concurrence.
