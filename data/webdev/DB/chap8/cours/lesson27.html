⚙️ Qu'est-ce qu'un Trigger ?

Un trigger (déclencheur) est une procédure stockée spéciale qui s’exécute automatiquement lorsqu’un événement se produit sur une table : INSERT, UPDATE ou DELETE.

📌 Utilités principales :

  • ✓ Audit des modifications (journalisation).
  • ✓ Validation complexe avant insertion/modification.
  • ✓ Mise à jour automatique d’autres tables (mise à jour de totaux, statistiques).
  • ✓ Application de règles métier impossibles avec des contraintes standard.

🎨 Diagramme : un événement SQL → déclenche le trigger → exécute des actions.

📝 Créer un trigger : syntaxe & moments

La syntaxe générale (MySQL / PostgreSQL) :

CREATE TRIGGER nom_trigger
{BEFORE | AFTER} {INSERT | UPDATE | DELETE}
ON table
FOR EACH ROW
BEGIN
   -- instructions SQL
END;
⏱️ BEFORE

Exécuté avant l’événement. Permet de modifier les données (NEW) ou d’annuler l’opération avec un signal.

⏱️ AFTER

Exécuté après l’événement. Idéal pour la journalisation (historique), impossible de modifier la ligne courante.

💡 Important : FOR EACH ROW signifie que le trigger s’exécute pour chaque ligne affectée (utile pour les mises à jour en masse).

🔍 NEW et OLD : les enregistrements virtuels

Dans le corps d’un trigger, on accède aux valeurs avant et après modification grâce à deux pseudo-lignes :

TriggerNEWOLD
INSERT✅ contient les nouvelles valeurs❌ non disponible (NULL)
UPDATE✅ valeurs après modification✅ valeurs avant modification
DELETE❌ non disponible✅ valeurs avant suppression

Exemple : :NEW.salaire en Oracle, ou NEW.salaire en MySQL/PostgreSQL.

✅ On peut utiliser NEW pour vérifier une condition et empêcher l’opération (avec SIGNAL SQLSTATE).

📖 Exemple concret : trigger BEFORE UPDATE qui empêche de baisser un prix de plus de 30%.

📌 Exemples concrets de triggers

🛡️ 1. Trigger BEFORE INSERT pour validation métier

CREATE TRIGGER verifier_age_etudiant
BEFORE INSERT ON etudiants
FOR EACH ROW
BEGIN
   IF NEW.age < 16 THEN
      SIGNAL SQLSTATE '45000' 
      SET MESSAGE_TEXT = 'Âge minimum 16 ans';
   END IF;
END;

📝 2. AFTER UPDATE pour table d'audit (historique des salaires)

CREATE TRIGGER log_salaire_update
AFTER UPDATE ON employes
FOR EACH ROW
BEGIN
   INSERT INTO audit_employes(employe_id, ancien_salaire, nouveau_salaire, date_modif)
   VALUES (OLD.id, OLD.salaire, NEW.salaire, NOW());
END;

✨ Ainsi chaque modification est tracée sans effort supplémentaire dans l'application.

💡 Bonnes pratiques : évitez les triggers trop complexes ou qui appellent d’autres triggers (effet domino). Utilisez-les pour l’intégrité ou l’audit, mais préférez les contraintes quand c’est possible.

⚙️ Gérer, modifier et supprimer un trigger

Commandes essentielles pour l’administration des triggers :

  • DROP TRIGGER nom_trigger; → supprimer définitivement un trigger.
  • SHOW TRIGGERS; (MySQL) ou SELECT * FROM pg_trigger; (PostgreSQL) pour lister.
  • On ne peut pas modifier directement un trigger, il faut le supprimer puis le recréer.
  • Pensez à nommer clairement vos triggers : trg_before_update_salaire.

⚠️ Attention aux performances :

Un trigger s’exécute à chaque ligne modifiée, ce qui peut ralentir les gros traitements par lots. À utiliser avec discernement.

📌 Récapitulons & petit défi !

🎯 Automatisation 🔄 NEW / OLD ⏲️ BEFORE/AFTER 📊 Audit

Les triggers sont des outils puissants pour automatiser des règles métier complexes et tracer les modifications. Ils agissent silencieusement, mais doivent être bien documentés.

🧪 Exercice d’entraînement :

Créez une table produits(id, nom, prix, stock) et une table historique_prix. Écrivez un trigger AFTER UPDATE qui, si le prix change, insère une ligne dans historique_prix (id_produit, ancien_prix, nouveau_prix, date_modification).

Voir une piste de solution (MySQL) 
CREATE TRIGGER log_prix_change
AFTER UPDATE ON produits
FOR EACH ROW
BEGIN
  IF OLD.prix != NEW.prix THEN
    INSERT INTO historique_prix(produit_id, ancien_prix, nouveau_prix, date_changement)
    VALUES (OLD.id, OLD.prix, NEW.prix, NOW());
  END IF;
END;

🔁 N’oubliez pas de tester vos triggers avec des requêtes simples !
