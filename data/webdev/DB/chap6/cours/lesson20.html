🔍 Index fondamental🌲 Arbre équilibré

🌲 B-Tree (Arbre-B)

L'index par défaut dans MySQL, PostgreSQL, SQLite. Structure en arbre équilibré qui permet des recherches, tris et comparaisons très rapides (\( O(\log n) \)).

  • Idéal pour : Colonnes avec opérateurs < , > , = , BETWEEN , LIKE 'prefix%'
  • Exemple : CREATE INDEX idx_prix ON produits(prix);
  • 📌 Avantage : maintient l’ordre, parfait pour ORDER BY et les jointures.
⚡ HachageRecherche exacte

🔑 Index HASH

Utilise une fonction de hachage pour mapper directement la clé à un emplacement. Extrêmement rapide pour les égalités ( = ou IN ) mais ne supporte pas les recherches par intervalle ou le tri.

Hachage("client_42") → 0xA3F2
Accès direct à la ligne
  • 📌 Cas d'usage : tables de correspondance, cache, colonnes souvent recherchées avec égalité stricte (ex: email, identifiant unique).
  • ⚠️ Limites : Pas de recherche > , < , LIKE ; mémoire consommée selon fonction.
  • Exemple PostgreSQL : CREATE INDEX ON users USING HASH (email);
🧩 Bitmap – faible cardinalité

📊 Index Bitmap

Représente les valeurs sous forme de bits (chaque bit correspond à une ligne). Très performant pour les colonnes avec peu de valeurs distinctes (sexe, catégorie, statut).

Valeur "HOMME"ligne1ligne2ligne3
bitmap :101
Opérations bit à bit rapides pour combiner des conditions (AND / OR)
  • ✔️ Avantages : compression, idéal pour les entrepôts de données (Data Warehousing) et requêtes analytiques.
  • 📌 Exemple : Table "clients" avec colonne "statut" (actif/inactif) ou "pays" (peu de pays différents dans l'entrepôt).
  • ⚠️ Inconvénient : coût élevé en mise à jour (verrouillage de bitmap) – peu adapté aux transactions fréquentes.
🔎 Full-Text – recherche textuelle

📖 Index Full-Text (texte intégral)

Spécialement conçu pour la recherche dans de grandes quantités de texte. Permet de chercher des mots ou expressions avec MATCH ... AGAINST (MySQL) ou tsvector (PostgreSQL).

"Chercher tous les articles contenant 'base de données' et 'performance' "
  • 📌 Cas typique : moteur de recherche, blogs, documentation, gestion de produits (descriptions).
  • Fonctionnement : analyse lexicale (stop words, radicalisation), index inversé.
  • Exemple (MySQL) : CREATE FULLTEXT INDEX idx_titre ON articles(titre, contenu);
    Puis requête : WHERE MATCH(titre,contenu) AGAINST('SQL optimisation');

⚖️ Coûts & bonnes pratiques

Un index accélère les lectures mais ralentit les écritures (INSERT, UPDATE, DELETE). À utiliser avec discernement.

📉 Coût écriture

Chaque modification nécessite la mise à jour des index : + I/O, + verrous.

📈 Gain lecture

SELECT sur colonnes indexées : jusqu'à x100 plus rapide sur des millions de lignes.

🎯 Choisir le bon type :

  • B-Tree → requêtes courantes, colonnes avec ordre, clés étrangères.
  • Hash → égalité stricte, tables type cache, email ou identifiant.
  • Bitmap → colonnes avec peu de valeurs, entrepôts, reporting.
  • Full-text → recherche de mots clés dans textes longs.

✅ À retenir

B-Tree est le couteau suisse, Hash pour recherche exacte rapide, Bitmap pour faible cardinalité et Full-text pour le texte. Pensez toujours au compromis lecture/écriture.
