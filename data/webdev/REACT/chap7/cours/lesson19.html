🕳️ Props Drilling : Définition

Le props drilling est une situation où l'on transmet des données à travers plusieurs niveaux de composants qui n'ont pas besoin de ces données, uniquement pour qu'elles atteignent un composant profond.

La prop user traverse 3 composants inutiles avant d'arriver à destination !

Problèmes : code moins maintenable, composants difficiles à réutiliser, re-rendus inutiles.

🔴 Exemple problématique

// Composant App (niveau 1)
function App() {
  const [user, setUser] = useState({ name: 'Alice', role: 'admin' });
  return <Layout user={user} />;
}

// Composant Layout (niveau 2) - n'utilise PAS user, mais doit le transmettre
function Layout({ user }) {
  return (
    <div>
      <Header user={user} />
      <MainContent />
    </div>
  );
}

// Composant Header (niveau 3) - n'utilise PAS user non plus !
function Header({ user }) {
  return <UserMenu user={user} />;
}

// Composant UserMenu (niveau 4) - C'est ici qu'on a besoin de user !
function UserMenu({ user }) {
  return <div>Bienvenue {user.name}</div>;
}

Observation : Layout et Header transmettent user sans l'utiliser. Si la structure change, il faut modifier tous les composants intermédiaires !

📊 Quand le props drilling devient un problème ?

Niveaux profonds

3+ niveaux de composants intermédiaires

Restructuration fréquente

Déplacer un composant casse le chemin de props

Beaucoup de données

De nombreuses props traversent les mêmes composants

Composants intermédiaires

Ils reçoivent des props qu'ils n'utilisent pas

Si vous remarquez ces signes, il est temps de refactoriser !

✅ Solution 1 : La composition avec children

Au lieu de transmettre les props à travers les composants intermédiaires, on les construit au bon endroit et on les passe via children.

// Composant App (niveau 1)
function App() {
  const [user, setUser] = useState({ name: 'Alice', role: 'admin' });
  return (
    <Layout>
      <Header>
        <UserMenu user={user} />
      </Header>
    </Layout>
  );
}

// Layout devient générique - ne connaît plus user
function Layout({ children }) {
  return <div className="layout">{children}</div>;
}

// Header aussi devient générique
function Header({ children }) {
  return <header className="header">{children}</header>;
}

// UserMenu reçoit user directement !
function UserMenu({ user }) {
  return <div>Bienvenue {user.name}</div>;
}

Plus de props drilling ! Chaque composant reçoit uniquement ce dont il a besoin.

↓

La composition consiste à inverser la responsabilité : c'est le composant parent qui assemble les pièces.

🎨 Composition avec plusieurs "slots"

On peut passer plusieurs éléments via des props nommées (pas seulement children).

function Page({ header, sidebar, main, footer }) {
  return (
    <div>
      <header>{header}</header>
      <div className="flex">
        <aside>{sidebar}</aside>
        <main>{main}</main>
      </div>
      <footer>{footer}</footer>
    </div>
  );
}

// Utilisation
function App() {
  const [user, setUser] = useState({ name: 'Alice' });
  const [notifications, setNotifs] = useState([]);
  
  return (
    <Page
      header={<Logo />}
      sidebar={<UserMenu user={user} />}
      main={<Dashboard data={notifications} />}
      footer={<Footer />}
    />
  );
}

🔑 Chaque "slot" est construit au niveau où les données sont disponibles, pas plus tard !

🤔 Quand utiliser quelle approche ?

SituationSolution recommandée
1 ou 2 niveaux de profondeurProps drilling acceptable
3+ niveaux, données statiquesComposition (children / slots)
Données globales (thème, auth, langue)Context API (prochaine leçon)
État complexe partagéContext + useReducer ou Redux
Composants fortement couplésRefactoriser en composition

🚨 Signes que vous souffrez de props drilling

  • Vous modifiez 5 fichiers pour ajouter une nouvelle donnée
  • Les composants intermédiaires ont des props {...props} ou ...rest sans réelle intention
  • La refactorisation (déplacement d'un composant) casse tout
  • Les props traversent des composants qui n'en ont pas besoin
  • Il devient difficile de comprendre le flux de données

Si vous reconnaissez ces signes, la composition est votre meilleure amie !

🧪 Exercice : Refactoriser un code qui souffre

Code problématique :

function Dashboard() {
  const [theme, setTheme] = useState('light');
  return <Container theme={theme} onThemeChange={setTheme} />;
}

function Container({ theme, onThemeChange }) {
  return <Wrapper theme={theme} onThemeChange={onThemeChange} />;
}

function Wrapper({ theme, onThemeChange }) {
  return <Content theme={theme} onThemeChange={onThemeChange} />;
}

function Content({ theme, onThemeChange }) {
  return (
    <div className={theme}>
      <p>Contenu</p>
      <button onClick={() => onThemeChange('dark')}>Mode sombre</button>
    </div>
  );
}
function Dashboard() {
  const [theme, setTheme] = useState('light');
  const content = <Content theme={theme} onThemeChange={setTheme} />;
  return <Container>{content}</Container>;
}

function Container({ children }) {
  return <Wrapper>{children}</Wrapper>;
}

function Wrapper({ children }) {
  return <div className="wrapper">{children}</div>;
}

function Content({ theme, onThemeChange }) {
  return (
    <div className={theme}>
      <p>Contenu</p>
      <button onClick={() => onThemeChange('dark')}>Mode sombre</button>
    </div>
  );
}

✨ Container et Wrapper ne savent plus rien du thème !

📌 Ce qu'il faut retenir

Props Drilling

  • Props traversent des niveaux inutiles
  • Difficile à maintenir
  • Composants moins réutilisables

Composition

  • Construire les composants où les données sont disponibles
  • Utiliser children ou props nommées (slots)
  • Composants génériques et réutilisables

Règle d'or : Remonter la construction, pas les props !
