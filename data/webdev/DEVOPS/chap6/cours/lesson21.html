🌐 Exposer des applications dans Kubernetes

Services (ClusterIP, NodePort, LoadBalancer) & Ingress Controller

K8s Networking Leçon 21 DevOps

Découvrons comment exposer vos applications Kubernetes au monde extérieur, choisir le bon Service et utiliser un Ingress pour le routage HTTP/HTTPS.

🎯 Objectifs de la leçon

  • ✔️ Exposer des Pods avec les types de Service : ClusterIP, NodePort, LoadBalancer.
  • ✔️ Configurer un Ingress Controller pour le routage HTTP/HTTPS intelligent.
  • ✔️ Comprendre le DNS interne (CoreDNS / kube-dns) pour la communication entre services.
  • ✔️ Savoir choisir la bonne méthode d’exposition selon le contexte (dev, prod, cloud).

📡 Services Kubernetes : le pont vers l'extérieur

Les Pods sont éphémères et ont des adresses IP qui changent. Un Service fournit une IP stable et une découverte de service.

ClusterIP

Interne uniquement. Idéal pour communication microservices.

NodePort

Expose sur un port statique de chaque nœud (30000-32767). Accessible via <NodeIP>:NodePort.

LoadBalancer

Utilise un équilibreur de charge cloud (AWS, Azure, GCP). Donne une IP publique.

💡 À retenir : Chaque Service reçoit une adresse IP virtuelle et un nom DNS résolu par CoreDNS. Par exemple, mon-service.mon-namespace.svc.cluster.local

🔌 Démo : ClusterIP & NodePort en YAML

Voici comment créer un Service de type NodePort pour exposer une application web sur un port de vos nœuds :

apiVersion: v1
kind: Service
metadata:
  name: mon-app-nodeport
spec:
  type: NodePort
  selector:
    app: mon-app
  ports:
    - port: 80          # port du service
      targetPort: 8080  # port du conteneur
      nodePort: 31000   # optionnel, plage 30000-32767

Pour y accéder : curl http://<IP_Noeud>:31000 . En production, on préfère LoadBalancer sur cloud.

⚠️ NodePort n'est pas sécurisé directement sans firewall. Idéal pour tests ou environnements on-premise simples.

🚦 Ingress : le contrôleur de trafic HTTP

Un Ingress agit comme un reverse proxy pour exposer plusieurs services sous un même équilibreur de charge. Idéal pour le routage basé sur le nom d'hôte ou le chemin.

apiVersion: networking.k8s.io/v1
kind: Ingress
metadata:
  name: demo-ingress
spec:
  rules:
  - host: monapp.exemple.com
    http:
      paths:
      - path: /
        pathType: Prefix
        backend:
          service:
            name: web-service
            port:
              number: 80

👉 Pour utiliser Ingress, il faut d'abord installer un contrôleur (NGINX Ingress, Traefik, HAProxy...).

🌍 DNS interne kube-dns / CoreDNS

Dans Kubernetes, chaque Service reçoit automatiquement un enregistrement DNS. Les Pods peuvent communiquer en utilisant le nom du service plutôt que l'IP.

mon-service

Dans le même namespace

mon-service.mon-namespace

Dans un autre namespace

mon-service.mon-namespace.svc.cluster.local

Nom complet (FQDN)

✨ Astuce pratique :

Pour tester la résolution DNS depuis un Pod, utilisez nslookup mon-service ou cat /etc/resolv.conf dans le Pod.

☁️ LoadBalancer : exposition cloud native

Sur AWS, Azure ou GCP, le type LoadBalancer provisionne automatiquement un équilibreur de charge externe avec une IP publique.

apiVersion: v1
kind: Service
metadata: name: mon-app-lb
spec:
type: LoadBalancer
ports:
- port: 80
targetPort: 8080
selector:
app: mon-app

Une fois déployé, kubectl get svc affiche une colonne EXTERNAL-IP. Parfait pour la production.

AWS ELB Azure Load Balancer GCP Cloud Load Balancing

🧠 Vérifions vos acquis

1️⃣ Quel type de Service utiliseriez-vous pour rendre une application accessible depuis Internet sur un cloud public ?

✅ Réponse : LoadBalancer (ou Ingress avec un contrôleur). NodePort pour les tests.

2️⃣ Vrai ou faux ? Ingress permet le routage basé sur le chemin URL comme /api vers un service et /web vers un autre.

✅ Vrai ! L'Ingress est parfait pour le routage dynamique HTTP/HTTPS.

3️⃣ À quoi sert CoreDNS dans un cluster Kubernetes ?

✅ Il fournit la résolution DNS interne pour que les Pods trouvent les Services par leur nom.

💪 Petite commande utile : kubectl expose deployment mon-deploy --type=NodePort --port=80 --target-port=8080 pour exposer rapidement.
