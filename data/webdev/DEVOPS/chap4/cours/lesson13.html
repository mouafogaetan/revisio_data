Leçon 13 · GitLab CI/CD

Configurer une chaîne CI/CD avec GitLab

Maîtriser .gitlab-ci.yml, les runners, caches, artifacts et variables d'environnement.

🎯 Objectifs de la leçon

À la fin de ce cours, vous serez capable de :

  • 1 Configurer le fichier .gitlab-ci.yml en définissant stages, jobs et scripts.
  • 2 Comprendre et utiliser les runners GitLab (shared vs specific).
  • 3 Gérer les caches, artifacts et variables d'environnement pour optimiser les pipelines.

📄 Le fichier .gitlab-ci.yml

GitLab CI utilise un fichier YAML à la racine du dépôt pour décrire l’automatisation. Il définit les stages (étapes) et les jobs (tâches). Chaque job s’exécute dans un runner.

stages:
  - build
  - test
  - deploy

build_job:
  stage: build
  script:
    - echo "Compilation en cours..."
    - npm install
  only:
    - main

✨ Explication : Le pipeline s’exécute dans l’ordre : build, puis test, puis deploy. Chaque job a un script (commandes shell).

💡 À retenir : GitLab détecte automatiquement le fichier à chaque push ou MR. Les jobs parallèles si aucun besoin d'ordre.

🏃‍♂️ Les runners GitLab

Un runner est un agent qui exécute les jobs. GitLab fournit des runners partagés (shared) ou vous pouvez installer vos propres runners spécifiques.

Shared Runners

Fournis par GitLab.com ou l'instance. Disponibles immédiatement, mais limités en ressources (concurrence). Idéal pour projets open-source ou début.

Specific Runners

Propres à un projet ou un groupe. Vous contrôlez l'infra (Docker, VM, Kubernetes). Parfait pour des besoins spécifiques ou sécurité.

⚙️ Installation d’un runner spécifique : avec gitlab-runner register et un token depuis l’interface GitLab → Settings → CI/CD → Runners.

📦 Caches & Artifacts : accélérer les pipelines

Deux mécanismes essentiels pour ne pas refaire le travail à chaque fois.

🗄️ Cache

Conserve des dossiers entre les jobs du même pipeline (ex: node_modules/, vendor/). Défini via mot-clé cache.

cache:
  paths:
    - node_modules/

🎁 Artifacts

Fichiers générés par un job et persistants après exécution, utilisables dans des jobs suivants ou téléchargeables (ex: binaires, rapports de tests).

artifacts:
  paths:
    - dist/
  expire_in: 1 week
✅ Bonne pratique : Utiliser le cache pour les dépendances, les artifacts pour les livrables.

🔐 Variables d'environnement dans GitLab CI

Stocker des secrets (clés API, mots de passe) ou configurer dynamiquement les pipelines.

📍 Déclaration possible :

  • Dans le fichier .gitlab-ci.yml (variables globales ou par job)
  • Via l'interface GitLab : Settings → CI/CD → Variables (masquées / protégées)
variables:
  DEPLOY_ENV: "production"
  DATABASE_URL: ${DB_URL}   # héritée depuis UI

deploy_job:
  script:
    - echo "Déploiement sur $DEPLOY_ENV"
    - deploy --token $SECRET_TOKEN
🛡️ Les variables masquées (Masked) n'apparaissent pas dans les logs. Toujours protéger les secrets.

📁 Exemple complet : pipeline Node.js

stages:
  - install
  - test
  - build
  - deploy

cache:
  paths:
    - node_modules/

before_script:
  - npm install

install_deps:
  stage: install
  script:
    - echo "Dépendances déjà mises en cache"

test_app:
  stage: test
  script:
    - npm run test
  artifacts:
    reports:
      junit: reports/junit.xml

build:
  stage: build
  script:
    - npm run build
  artifacts:
    paths:
      - dist/

deploy_prod:
  stage: deploy
  script:
    - scp -r dist/ user@server:/var/www/
  only:
    - main
  variables:
    ENV: "production"

➡️ Ce pipeline installe les dépendances, teste, construit l'application et ne déploie que sur la branche main.

📌 Bonnes pratiques & astuces

🧪 Utiliser rules

Plutôt que only/except, privilégiez rules pour des conditions plus souples.

🏷️ Tags runners

Assignez des tags aux runners spécifiques pour garantir l’exécution sur certaines machines.

⏱️ Timeouts

Configurer un timeout par job pour éviter des pipelines bloqués.

📈 Pipeline as code

Versionnez toujours .gitlab-ci.yml avec le code source.
