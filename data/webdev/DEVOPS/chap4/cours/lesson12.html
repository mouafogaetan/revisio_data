🔧 GitHub Actions : définition

GitHub Actions est une plateforme d’automatisation intégrée directement à GitHub. Elle permet de créer des workflows (pipelines) pour construire, tester et déployer votre code automatiquement.

⚙️ Exemple : À chaque fois que vous poussez du code (push), GitHub Actions peut exécuter vos tests unitaires.
📁 Déclaration

Fichiers .yml dans .github/workflows/

⚡ Gratuit

2000 minutes/mois pour les comptes gratuits (public).

🏗️ Structure d'un workflow

Un workflow se compose de :

  • name : nom du workflow
  • on : événements déclencheurs (push, pull_request, schedule...)
  • jobs : ensemble de tâches exécutées en parallèle ou en séquence
  • runs-on : type de runner (ubuntu-latest, windows-latest, macos-latest)
  • steps : chaque étape utilise une action ou exécute des commandes shell
name: CI
on: [push, pull_request]
jobs:
  build:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - name: Run tests
        run: npm test

⏯️ Les triggers (déclencheurs)

Un workflow peut être déclenché par plusieurs événements GitHub :

push Déclenché quand vous poussez des commits sur une branche.
pull_request Déclenché lors de l'ouverture / mise à jour d'une PR.
schedule Planification (cron) : exécution périodique.
workflow_dispatch Déclenchement manuel depuis l'interface GitHub.

📌 Exemple schedule : schedule: - cron: '0 2 * * *' (tous les jours à 2h du matin)

🧩 Jobs, Steps & Actions

Job : ensemble d'étapes qui s'exécutent sur le même runner. Plusieurs jobs peuvent être parallélisés.

Step : une action ou une commande shell (run).

Action : une unité réutilisable (ex: actions/checkout@v4, actions/setup-node@v4).

📦 Job "test" (runs-on: ubuntu)
├── Step 1: checkout code 🧾
├── Step 2: setup Node.js 🟢
└── Step 3: run tests ✅

⭐ Les actions officielles du marketplace GitHub permettent d’étendre facilement vos pipelines.

📄 Exemple complet : CI pour une app Node.js

Créez le dossier .github/workflows/ci.yml :

name: Node.js CI

on:
  push:
    branches: [ main, develop ]
  pull_request:
    branches: [ main ]

jobs:
  build:
    runs-on: ubuntu-latest
    steps:
    - uses: actions/checkout@v4
    - name: Use Node.js 18
      uses: actions/setup-node@v4
      with:
        node-version: 18
    - run: npm ci
    - run: npm test
    - run: npm run build

✅ À chaque push, GitHub Actions installe les dépendances, lance les tests et construit l'application.

🏃 Runners GitHub

Un runner est un serveur qui exécute votre workflow. Types :

  • GitHub-hosted runners : hébergés par GitHub (Ubuntu, Windows, macOS) – simples et rapides
  • Self-hosted runners : vous gérez vos propres machines (sécurité, matériel spécifique)

💡 Pour un projet d'école, les runners hébergés suffisent largement !

Ils sont préconfigurés avec des outils comme Docker, Node, Python, Java, etc.

Spécifiez le runner avec runs-on: ubuntu-latest

🔐 Variables d'environnement & secrets

Pour sécuriser vos clés API ou configurer des valeurs, utilisez :

  • env : variables classiques dans le workflow
  • secrets : stockés dans GitHub (Settings → Secrets and variables) → masqués dans les logs
steps:
  - name: Déploiement
    env:
      API_KEY: ${{ secrets.MY_API_KEY }}
    run: ./deploy.sh

⚠️ Ne jamais écrire de secrets en clair dans le code !

📊 Visualisation & logs

Sur GitHub, allez dans l'onglet "Actions" de votre dépôt pour voir :

  • L'historique des exécutions
  • Le détail de chaque job et step
  • Les logs en temps réel (sorties console)
✅ Commit abc123
➡️ Job build
➡️ Step test passed
✔️ Succès / ❌ Échec

💬 En cas d'erreur, les logs vous aident à corriger rapidement.

🎓 Synthèse & bonnes pratiques

✅ Nommez clairement vos workflows et jobs.

✅ Utilisez actions/checkout@v4 pour récupérer le code.

✅ Cachez les dépendances (actions/cache) pour accélérer l’exécution.

✅ Limitez les déclencheurs inutiles (ex: sur certaines branches).

✅ Testez localement avec act (simulateur d'actions GitHub).
