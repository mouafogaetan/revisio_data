💡 Pourquoi ? Une faille de sécurité coûte cher. Mieux vaut la détecter tôt (quand le code est écrit) plutôt qu'en production.

🔐 Qu'est-ce que "Shift Left" ?

Dans un pipeline classique, la sécurité intervient souvent à la fin (tests de pénétration après déploiement). Shift Left signifie déplacer la sécurité vers la gauche (les premières phases) : planification, développement, intégration continue.

  • ✔️ Analyses de code automatiques
  • ✔️ Scans des dépendances
  • ✔️ Tests d'intrusion automatisés dans CI
  • ✔️ Révision des configurations Infrastructure as Code
[ Plan ] ---> [ Code ] ---> [ Build ] ---> [ Test ] ---> [ Release ]
🛡️ Sécurité continue (SAST, secret scan, dépendances)

✅ La sécurité devient partie prenante dès l'écriture du code

📌 Bénéfices concrets

Correction rapide (coût réduit), moins de vulnérabilités en production, conformité facilitée, et meilleure culture d’équipe.

🛠️ SAST vs DAST : Les alliés sécurité

📄

SAST (Static Analysis)

Analyse du code source sans exécution. Cherche des failles (injections SQL, XSS, failles logiques).

Exemple : SonarQube CodeQL

🟢 Idéal pour les pull requests.

🌐

DAST (Dynamic Analysis)

Teste l’application en cours d'exécution (environnement de test). Simule des attaques réelles.

OWASP ZAP Burp Suite

🔴 Détecte les failles de configuration, authentification etc.

⚙️ Intégration dans le pipeline CI/CD :

<-- SonarQube scan après "build" -->   <-- OWASP ZAP sur l'URL de staging -->

On bloque le déploiement si des vulnérabilités critiques sont détectées.

🔑 Gestion des secrets : Vault & AWS Secrets Manager

Ne jamais stocker de mots de passe, clés API ou tokens en clair dans le code ou les dépôts Git. Utiliser des gestionnaires dédiés !

🏛️HashiCorp Vault

Stockage sécurisé, rotation dynamique des secrets, accès basé sur politiques, audit logs. Idéal pour Kubernetes ou machines virtuelles.

☁️AWS Secrets Manager

Solution cloud natif AWS : rotation automatique des secrets pour RDS, Redshift. Très pratique pour les applications hébergées chez AWS.

# Exemple d'accès à Vault depuis un pipeline
$ vault kv get secret/application/db_password

✅ On injecte les secrets comme variables d’environnement au moment de l’exécution, jamais hardcodés.

📦 Pipeline DevSecOps : Exemple concret

1️⃣ Code Push 2️⃣ SAST (SonarQube) 3️⃣ Scan dépendances 4️⃣ Build / Test 5️⃣ DAST (OWASP ZAP) 6️⃣ Deploy (si ok)

✅ À chaque étape, la sécurité vérifie : pas de secrets exposés, vulnérabilités bloquantes → échec du pipeline.

Résultat :

Livraison continue, mais sécurisée. Moins de risques de violation de données, confiance renforcée.
