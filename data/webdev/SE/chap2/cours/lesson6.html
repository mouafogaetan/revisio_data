Le cahier des charges (Software Requirements Specification)

Un document officiel qui décrit toutes les exigences (fonctionnelles et non fonctionnelles) d’un logiciel. C'est le contrat entre le client et l’équipe de développement.

📌 Structure typique d'un SRS (norme IEEE 830)

  • Introduction : objectif, portée, définitions.
  • Description générale : perspectives du produit, contraintes, hypothèses.
  • Exigences spécifiques : fonctionnelles, non fonctionnelles (performance, sécurité, disponibilité).
  • Annexes et index : maquettes, glossaire, traçabilité.
✍️

Caractéristiques d’une bonne exigence

Claire, testable, réalisable, nécessaire, non ambiguë, cohérente, priorisée (MoSCoW).

📎

Exemple concret

« Le système doit permettre à un utilisateur de se connecter avec son email et mot de passe, avec un temps de réponse inférieur à 2 secondes. »

✅

Valider les exigences

La validation garantit que les exigences décrivent correctement ce que le client souhaite. On évite les mauvaises surprises en fin de projet.

🔍

Revues formelles (inspections)

L’équipe et les parties prenantes analysent le SRS ligne par ligne, détectent les erreurs, ambiguïtés ou incohérences. Très efficace pour la qualité.

🖌️

Prototypage (maquettes interactives)

Créer des maquettes cliquables (ex: Figma, papier) pour que l’utilisateur visualise le futur logiciel. Les retours précoces améliorent les exigences.

🧪

Tests d’acceptation (UAT)

Le client teste le logiciel dans un environnement réel ou de recette, en se basant sur les critères d’acceptation des user stories. Validation finale avant livraison.

💡 Bon à savoir : La validation répond à la question « Avons-nous construit le bon produit ? » alors que la vérification répond à « Avons-nous bien construit le produit ? ».

🔄

Gérer le changement des exigences

Les besoins évoluent ! Une bonne pratique de génie logiciel intègre un processus de contrôle des changements pour éviter le chaos.

📦

Contrôle de version des exigences

Comme pour le code, on versionne le document SRS (Git, outils de gestion des exigences comme Jira, Trello ou ReqIF). Chaque modification est tracée : date, auteur, raison.

⚙️

Comité de pilotage (CCB)

Change Control Board : évalue l'impact des demandes (coût, délai, technique) et décide d'accepter/refuser le changement. Évite le "scope creep".

Bonnes pratiques & mise en œuvre

✏️ Exemple : extrait de SRS pour une application de notes

3.2.1 Exigence fonctionnelle – Créer une note :
« L’utilisateur authentifié peut créer une note avec titre et contenu textuel. Le système enregistre automatiquement la date de création. »
Critère d’acceptation : La note doit être persistante après rafraîchissement.

📝 Validation par revue entre pairs (en classe)

Divisez-vous en groupes : un groupe joue le client, l’autre analyste. Inspectez un mini SRS pour un blog ou une boutique en ligne. Notez les ambiguïtés.

✅ Outils utiles pour le contrôle de version des exigences :
🔹 Git + Markdown 🔹 Jira / Confluence 🔹 IBM DOORS (environnement pro) 🔹 Notion / Trello avec checklist

💭 À retenir pour le projet final

Un bon SRS réduit les malentendus. Toujours faire valider le document par le client avant de commencer le code. Prévoyez un process de gestion des changements : utilisez un tableau de suivi des demandes.

  • Le SRS est un contrat détaillé (normes IEEE).
