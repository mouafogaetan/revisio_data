📌 Qu’est-ce qu’un test unitaire ?

Un test unitaire vérifie le comportement d’une petite portion de code (fonction, méthode) de manière isolée. Il valide que chaque brique fonctionne correctement avant l’intégration.

✅ Rapide ✅ Automatisable ✅ Documente le code

🔺 Pyramide des tests (stratégie)

Les tests unitaires constituent la base massive de la pyramide : nombreux, rapides, essentiels.

🧰 Frameworks xUnit (exemples)

☕

JUnit (Java)

@Test
void addition() {
    assertEquals(5, calc.add(2,3));
}
🐍

pytest (Python)

def test_addition():
    assert add(2,3) == 5
📜

Jest (JS)

test('addition', () => {
    expect(add(2,3)).toBe(5);
});

✅ Structure commune : Arrange, Act, Assert (préparer, exécuter, vérifier).

🔄 Test Driven Development (TDD)

Une discipline de développement : écrire le test AVANT le code

🔴 RED
→
🟢 GREEN
→
🔵 REFACTOR
  • 🔴 Red : Écrire un test qui échoue (fonctionnalité pas encore implémentée).
  • 🟢 Green : Écrire le minimum de code pour faire passer le test.
  • 🔵 Refactor : Nettoyer, améliorer la structure sans casser les tests.
💡 TDD réduit les bugs, force une conception modulaire et sert de documentation vivante.
"Red-Green-Refactor"

📝 Exemple pas à pas : TDD pour une fonction "diviser"

// Étape 1 - RED : test qui échoue

def test_division_par_zero():
    with pytest.raises(ZeroDivisionError):
        diviser(10, 0)

// Étape 2 - GREEN : code minimal

def diviser(a,b):
    if b == 0:
        raise ZeroDivisionError()
    return a/b

// Étape 3 - REFACTOR : ajouter gestion des types, clean

✅ Après refactoring, tous les tests restent Verts → confiance dans l’évolution du code.

🎭 Mocks & Stubs : simuler les dépendances

🪤 Stub

Fournit des réponses prédéfinies aux appels. Exemple : une base de données factice retourne toujours un utilisateur.

🎭 Mock

Vérifie les interactions : on s’assure qu’une méthode a bien été appelée avec certains paramètres.

Pourquoi mocker ?

Tester une fonction sans dépendre d’une API externe, d’une BDD réelle ou d’un service lent → tests plus rapides et fiables.

// Exemple avec Jest
const getUser = jest.fn().mockReturnValue({ name: "Alice" });
expect(getUser()).toEqual({ name: "Alice" });

✨ Bonnes pratiques des tests unitaires

  • ✔️Un test doit tester une seule chose (nom explicite).
  • ✔️Les tests doivent être indépendants (ordre non important).
  • ✔️Utiliser des noms longs mais descriptifs comme test_division_renvoie_float().
  • ✔️Exécuter les tests à chaque modification (CI / local).

💡 La couverture de code n’est pas une fin en soi : privilégier la qualité des assertions et les cas limites.

📚 Résumé & Vérification des acquis

✅ 1. À quoi sert un mock ?

👉 À simuler le comportement d’une dépendance et vérifier les interactions sans recourir à l’implémentation réelle.

✅ 2. Que signifie "Refactor" dans TDD ?

👉 Améliorer la structure du code (lisibilité, performance) sans modifier le comportement externe, avec tous les tests au vert.

✅ 3. Quel est l’avantage d’un framework xUnit ?

👉 Standardisation des tests, assertions riches, exécution automatisée et intégration facile dans les pipelines CI.
