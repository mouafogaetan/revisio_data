🎯 Objectifs de la leçon

À la fin de cette leçon, vous serez capable de :

  • ✅ Comprendre les styles architecturaux : monolithique, microservices, événementiel
  • ✅ Maîtriser l'architecture en couches (présentation, métier, persistance)
  • ✅ Découvrir l'architecture hexagonale (ports & adaptateurs) et l'architecture oignon (Onion)
💡 Pourquoi c'est important ? Une bonne architecture rend le logiciel facile à maintenir, évolutif et résistant aux changements.

🏛️ Qu'est-ce qu'un style architectural ?

C'est la structure globale d'un logiciel. Elle définit comment les composants interagissent et s'organisent. Choisir une architecture, c'est décider de la scalabilité, de la maintenabilité et de la fiabilité du système.

📦

Architecture monolithique

Toute l'application (interface, logique métier, base de données) est réunie en un seul bloc. Une seule base de code, un seul déploiement.

✅ Avantages : simple à développer, débug facile, performances rapides.
⚠️ Inconvénients : difficile à faire évoluer, tout est couplé, une panne peut tout bloquer.
   +-----------------------+
   |   UI + Métier + Données | 
   |      (MONOLITHE)       |
   +-----------------------+

🧩 Microservices

L'application est découpée en petits services indépendants, chacun avec sa propre base de données et son propre métier. Ils communiquent via des API (HTTP, message queue).

📦 Service utilisateurs
📦 Service commandes
📦 Service paiements

Chaque service indépendant, déployable séparément

  • ✅ Scalabilité : on ne scale que les services surchargés.
  • ✅ Résilience : un service tombe, les autres continuent.
  • ❌ Complexité opérationnelle (réseau, monitoring, orchestration).

⚡ Architecture événementielle (Event-driven)

Les composants produisent et réagissent à des événements (ex: "commande créée", "paiement validé"). Un bus d'événements ou un courtier (broker) centralise les messages.

📢 Producteur ➡️ Broker ➡️ 👂 Consommateur(s)
Découplage, réactivité en temps réel

Exemple typique : système de notification, IoT, applications financières.

À retenir : bonne pour les systèmes asynchrones et extensibles.

🥪 Architecture en couches (N-tier / Layered)

L'une des plus courantes. Organisation verticale :

🖥️ Couche Présentation (IHM, API)
⚙️ Couche Métier (logique, règles)
🗄️ Couche Persistance (accès données, base)

✔️ Chaque couche dépend de la couche inférieure → claire, facile à comprendre pour les débutants. ❌ Peut créer des fuites de dépendances si mal appliquée.

Exemple : application web classique (React → Spring → MySQL)

⬡ Architecture hexagonale (Ports & Adaptateurs)

Mise au point par Alistair Cockburn. Le métier (coeur) est isolé des détails techniques (base de données, API externes, interface utilisateur). On utilise des ports (interfaces) et des adaptateurs (implémentations concrètes).

🔌 [Adaptateur Web] → 🧠 [Cœur métier] ← 🔌 [Adaptateur BDD]

✅ Avantages : testabilité maximale, indépendance technologique, respect du principe d'inversion de dépendance (DIP). Souvent associée au DDD.

     /\
    /  \   <-- Adaptateur HTTP
   |    |
    \  /    
     \/     Cœur métier (indépendant)
     /\
    /  \   <-- Adaptateur SQL
   |    |
    \  /

🧅 Architecture Oignon (Onion)

Inspirée de l'hexagonale, elle pousse plus loin la séparation en cercles concentriques. Le modèle métier et les interfaces de domaine sont au centre. Les couches externes sont les infrastructures (BDD, UI, API).

Domaine
  • ✔️ Forte maintenabilité, aucun couplage vers l’extérieur.
  • ✔️ Idéal pour les applications complexes et évolutives.
  • ⚠️ Courbe d'apprentissage plus élevée.

📊 Comment choisir ?

StyleQuand l'utiliser ?
MonolithiquePetites équipes, prototypes, applications simples
MicroservicesGrandes équipes, forte scalabilité, besoin d'indépendance
ÉvénementielTraitements asynchrones, streaming, réactivité
En couchesApplications standards (web, desktop) – bon compromis
Hexagonale / OnionApplications avec forte logique métier, besoin de tests unitaires et évolutivité long terme
💡 Conseil pratique : On peut commencer avec une architecture en couches, puis évoluer vers microservices si le besoin s'en fait sentir.

✍️ Ce qu'il faut retenir

🔑 Points clés :

  • Architecture = organisation macroscopique du logiciel.
  • Monolithique : tout-en-un, simple mais rigide.
  • Microservices : découpage fin, indépendant.
  • Événementiel : réaction à des événements, asynchrone.
  • Couches classiques : structure logique standard.
  • Hexagonale & Onion : centrées métier, très testables.
