Branches : développement parallèle

Une branche est une ligne de développement indépendante. Elle permet de travailler sur une fonctionnalité sans perturber la version stable (main ou master).

git branch ma-fonctionnalité # créer une branche
git checkout ma-fonctionnalité # basculer dessus
git switch -c nouvelle-branche # créer + basculer (Git 2.23+) 
        main      o---o---o
                      \
        feature        o---o---o

Chaque commit est une évolution indépendante.

✅ Pourquoi utiliser des branches ? Isolation, expérimentation, réduction des conflits.

🔀 Fusion (merge) vs Rebase

Merge (fusion)

Crée un commit de fusion qui combine l'historique. Préserve le contexte mais peut complexifier l'historique.

git checkout main && git merge feature

Rebase (réapplication)

Réécrit l'historique en plaçant les commits de la branche feature à la suite de main. Historique linéaire et propre.

git checkout feature && git rebase main
⚠️ Règle d'or : Ne pas faire de rebase sur des branches publiques partagées (risque de désynchronisation).

En équipe, merge est souvent plus sûr ; le rebase est parfait pour nettoyer son historique local.

📐 Stratégies de branchement populaires

Git Flow

Idéal pour les projets avec versions long terme. Branches : main, develop, feature/, release/, hotfix/. Organisation stricte.

main (prod) ← release ← develop ← feature/*
GitHub Flow

Plus simple : branche main toujours déployable, on crée des branches de fonctionnalités, pull request, puis fusion.

main ← feature-branch (via PR) → déploiement continu
Recommandation débutant : Commencez par GitHub Flow, puis explorez Git Flow pour les gros projets.

⚔️ Gérer les conflits de fusion

Un conflit survient quand deux branches modifient la même portion de code. Git ne peut pas décider automatiquement.

<<<<<<< HEAD
print("Hello depuis main")
=======
print("Bonjour depuis feature")
>>>>>>> feature

🔧 Résolution manuelle : éditer le fichier, garder la version souhaitée, supprimer les marqueurs, puis :

git add . && git commit -m "Résolution conflit"
Outils: git mergetool ou IDE (VSCode) Astuce: communication en équipe réduit les conflits

🏷️ Tags & Versioning sémantique

Les tags sont des étiquettes sur un commit précis, souvent utilisées pour marquer les versions (v1.0.0, release).

git tag v1.0.0 # tag léger
git tag -a v2.0.0 -m "Version majeure" # tag annoté
git push origin --tags # envoyer les tags au remote
MAJEUR X.y.z
breaking changes
MINEUR x.Y.z
fonctionnalité rétrocompatible
CORRECTIF x.y.Z
correctif bugs

Versioning sémantique (SemVer) : MAJEUR.MINEUR.CORRECTIF. Ex: 2.1.0 → nouvelle fonctionnalité sans rupture.

📌 Commandes Git essentielles

git branch
lister les branches
git checkout -b feat/nom
créer + basculer
git merge --no-ff
merge sans fast-forward
git rebase -i HEAD~3
rebase interactif
git log --oneline --graph
visualiser l'historique
Bonnes pratiques:
  • Nommer les branches clairement : feature/login, fix/bug-paiement
  • Faire des petits commits atomiques et commentés
  • Fusionner fréquemment pour éviter l'éloignement
  • Ne jamais forcer le push (--force) sur une branche partagée sans accord

👥 Exemple concret (travail à deux)

Alice
1. git checkout -b fonction-panier
2. Écrit du code → commit
3. git push origin fonction-panier
4. Ouvre une Pull Request
Bob
1. git fetch
2. git checkout -b review fonction-panier
3. Teste, suggestion
4. Fusion après revue (merge)
Plateformes: GitHub, GitLab, Bitbucket offrent PR/MR pour faciliter le versioning collaboratif.
