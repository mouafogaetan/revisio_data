🎯 Objectifs de la leçon

  • Définir le périmètre et les exigences d’un projet logiciel
  • Rédiger des user stories et les prioriser (méthode MoSCoW)
  • Réaliser des diagrammes UML : cas d’utilisation, classes, séquence
  • Choisir une architecture technique adaptée (couches, microservices…)

Pourquoi ? Un bon projet commence par une conception rigoureuse. Cela évite les erreurs coûteuses et améliore la qualité.

📌 1. Définir le périmètre & les exigences

Avant de coder, on doit répondre à trois questions :

  • Quoi ? (fonctionnalités)
  • Pour qui ? (utilisateurs cibles)
  • Contraintes ? (budget, délais, sécurité)

🔹 Exigences fonctionnelles vs non fonctionnelles

Fonctionnelle : “L’utilisateur peut ajouter un produit au panier”

Non fonctionnelle : “Le temps de réponse doit être inférieur à 2 secondes”

À vous : Pour une application de livraison de repas, donnez un exemple des deux types d’exigences.

📝 2. User stories & MoSCoW

Formule magique :

"En tant que [rôle], je veux [action] afin de [bénéfice]"

Exemple : En tant que client, je veux filtrer les produits par prix afin de trouver rapidement une bonne affaire.

⭐ Priorité MoSCoW :

Must (indispensable)
Should (important)
Could (optionnel)
Won't (exclu)
✅ Bon à savoir : Le MoSCoW aide à livrer rapidement l’essentiel (MVP).

📊 3. Diagrammes UML (les 3 clés)

🎭 Cas d’utilisation – qui fait quoi ? (acteurs + fonctionnalités)
🏛️ Diagramme de classes – structure des données (entités, relations)
⏱️ Diagramme de séquence – interactions chronologiques entre objets
Outils gratuits : Draw.io, Lucidchart (version gratuite), PlantUML.

Ces diagrammes facilitent la communication entre développeurs et non-techniciens.

🎭 Cas d’utilisation – Exemple e-commerce

Acteurs : Client, Admin, Système de paiement

  • Consulter catalogue
  • Ajouter au panier
  • Passer commande
  • Gérer les produits (admin)
Dessinez sur papier les relations <> ou <>

Les cas d’utilisation guident la conception des tests d’acceptation.

🏗️ 4. Choisir l’architecture technique

Architecture en couches (classique) :

Présentation (UI)
Métier (logique)
Persistance (base de données)

🔹 Autres styles : microservices, hexagonale (ports & adaptateurs), événementiel.

💡 Pour un petit projet : architecture en couches. Pour une grande équipe : microservices.

📌 Récapitulatif des étapes

  1. Recueillir les besoins (entretiens, personas).
  2. Rédiger les user stories + critères d’acceptation.
  3. Prioriser avec MoSCoW → définir le MVP.
  4. Modéliser : diagrammes de cas d’utilisation, classes, séquence.
  5. Choisir l’architecture technique (couches, bases de données).
  6. Documenter le tout (cahier des charges).
Conseil : Validez toujours les diagrammes avec le client avant de coder !

✏️ Exercice – Application “Gestion de bibliothèque”

Énoncé : Vous devez concevoir une application permettant aux usagers de réserver des livres et aux bibliothécaires de gérer le stock.

  • Rédigez 3 user stories (différents rôles).
  • Proposez un diagramme de classes simplifié (Livre, Usager, Réservation).
  • Citez une exigence non fonctionnelle importante (ex: sécurité, disponibilité).
Rédigez vos réponses sur une feuille ou dans votre outil de notes.
