📈 Auto-scaling : Élasticité automatique

Le cloud ajuste les ressources en temps réel selon la charge (CPU, réseau, requêtes).

✔️ Scaling out/in (horizontal)

Ajout ou suppression de serveurs (VMs, conteneurs) selon métriques.

⚙️ Scaling up/down (vertical)

Changement de taille d'instance (plus de RAM/CPU).

Exemple avec AWS Auto Scaling Group : défini une métrique (CPU > 70%) → lance des EC2 supplémentaires.

🔍 Rightsizing : la bonne taille pour chaque ressource

Beaucoup de projets sur-provisionnent. Analyser l'utilisation permet d'économiser jusqu'à 40%.

💰 Instances Spot / Low Priority

Récupérez la capacité inutilisée des fournisseurs cloud avec une remise de 60 à 90%.

⚡

Parfait pour : batch processing, analyse de données, environnements de test, workloads tolérants aux interruptions.

AWS Spot
Économie jusqu'à 90%
Azure Spot VM
Low-priority, éviction possible
GCP Preemptible
Durée max 24h, prix réduit
⚠️ Attention: AWS peut interrompre l'instance avec un préavis de 2 minutes. Architecturez vos jobs pour être résilients (checkpoints).

⚙️ Politiques d'auto-scaling

🎯 Scaling basé sur métriques

CPU, mémoire, requêtes SQS, latence. Exemple: si CPU > 75% pendant 5 min → +2 instances.

📅 Scaling planifié

Pour les charges prévisibles (pic à 9h, fin de mois). Réduction manuelle programmée.

⚖️ Predictive scaling (ML)

AWS Auto Scaling Predictive, Azure Autoscale basé sur historique. Anticipe les pics.

📉 Identifier les dérives de coûts

Les ressources oubliées sont le premier ennemi du budget cloud.

  • ✔️ Snapshots et AMI non nettoyés
  • ✔️ Load balancers sans backend actif
  • ✔️ Adresses IP élastiques inutilisées
  • ✔️ Espaces de stockage de logs volumineux
🚨 Alertes budgétaires : AWS Budgets, Azure Cost Alerts, GCP Budgets & alerts → configuration indispensable !
Exemple de calcul d'économie : \[ \text{Économie} = (\text{Prix OnDemand} - \text{Prix Spot}) \times \text{Heures d'exécution} \]

🧠 Optimisation continue : FinOps & Culture

🏷️ Tagging obligatoire

Tagger chaque ressource (environnement, projet, équipe) pour analyser les coûts précisément.

📦 Suppression automatique

Utiliser AWS Instance Scheduler, ou des fonctions Lambda qui éteignent les ressources hors heures ouvrées.

🔄 Rightsizing automatique

AWS Compute Optimizer, Azure Advisor, GCP Rightsizing recomm. changent les types d'instances.

📈 Savings Plans / RI

Pour les workloads stables, s'engager 1 ou 3 ans = -30% à -60%.

💡 Conseil enseignant: Activez toujours les rapports de coûts hebdomadaires et partagez avec l'équipe.

📝 Exercice pratique (maison)

Scénario : Vous avez une application web qui subit un pic de 10h à 12h et un autre de 18h à 20h. Le reste du temps, l'utilisation CPU est très faible.
