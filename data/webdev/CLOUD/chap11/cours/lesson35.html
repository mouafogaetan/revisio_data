Monitoring, Optimisation & Résilience

Configurer la supervision, maîtriser les coûts et assurer la haute disponibilité

✨ Dernière étape pratique du parcours Cloud ✨

🎯 Objectifs de la leçon

✓ Configurer le monitoring et les alertes — Mettre en place des métriques, journaux et notifications pour suivre l’état du système.
✓ Optimiser les coûts et documenter l'architecture — Utiliser les bons outils pour réduire la facture et garder une trace claire.
✓ Présenter le projet et démontrer la résilience — Simuler une panne et prouver que l’application tient le coup !
💡 Résilience : Capacité d’un système à continuer de fonctionner même quand un composant tombe en panne (ex: redondance, bascule automatique).

📊 Surveiller son cloud pour éviter les surprises

Dans le cloud, on ne peut pas “voir” les serveurs physiques. On utilise donc des métriques, logs et alertes.

  • CloudWatch (AWS) / Azure Monitor / Cloud Monitoring (GCP) : collectent CPU, mémoire, trafic réseau.
  • Métriques personnalisées : temps de réponse, erreurs applicatives.
  • Alertes : envoient un email, SMS ou déclenchent une action automatique (ex: redémarrer un service).
  • Tableaux de bord : visualisation en temps réel de l’état de santé.

Exemple : baromètres de santé d’une infrastructure cloud

💰 Optimiser les coûts & documentation

🏷️

Stratégies financières

  • Supprimer les ressources inutilisées (instances oubliées).
  • Utiliser instances réservées pour les charges stables.
  • Activer l’auto-scaling pour suivre la demande réelle.
  • Analyser avec AWS Cost Explorer ou Azure Cost Management.
📄

Documenter l’architecture

  • Diagramme des composants (VPC, sous-réseaux, LB, bases).
  • Lire la configuration Terraform comme documentation vivante.
  • Note sur les décisions d’architecture (ADR).
✅ Rightsizing : adapter la taille des machines à leurs besoins réels (pas trop grosses, pas trop petites).

🛡️ Démontrer la résilience : simuler une panne

Une application cloud résiliente supporte l'arrêt d'un serveur, d'une zone ou d'un conteneur.

$ kubectl delete pod mon-app-pod --namespace=production
→ Nouveau pod recréé automatiquement (ReplicaSet)
Load Balancer redirige le trafic → aucune coupure !
📌 Astuce présentation : tuer une instance EC2 ou un pod et montrer que l’application répond toujours.
🔁 Bascule multi-AZ : configurer au moins 2 zones de disponibilité.

✨ En résumé : clés pour une production sereine

📡
Monitoring proactif – Dashboards et alertes sur les métriques vitales.
💰
FinOps – Suivi budgétaire, rightsizing, instances spot pour les batchs.
🧪
Tests de résilience – Chaos engineering (simuler des pannes pour renforcer l’architecture).
📚
Documentation vivante – Diagrammes, README et code IaC pour partager la connaissance.
Une bonne supervision + optimisation + résilience = application cloud professionnelle !

🔔 Exemple concret d'alerte CloudWatch

{
  "AlarmName": "CPU-Utilisation-Haute",
  "MetricName": "CPUUtilization",
  "Threshold": 80,
  "ComparisonOperator": "GreaterThanThreshold",
  "EvaluationPeriods": 2,
  "Actions": ["arn:aws:sns:...notification"]
}

💡 À faire pour votre projet :

  • Configurer une alerte si le coût dépasse 50€/mois.
  • Afficher un dashboard en temps réel sur Grafana ou CloudWatch.
  • Simuler une montée en charge (stress test) et observer l’auto-scaling.
