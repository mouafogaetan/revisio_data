🔄 CI/CD en bref

Intégration Continue (CI)

Les développeurs fusionnent fréquemment leur code. À chaque push, on exécute automatiquement : build + tests unitaires/intégration.

✅ Détection rapide des erreurs
Déploiement Continu (CD)

Une fois le code validé, on le déploie automatiquement sur des environnements (staging, puis production).

✅ Livraisons fiables et fréquentes

🏗️ Le pipeline cloud : code source → build → test → déploiement sur cloud (VM, conteneur ou serverless)

☁️⚡

AWS CodePipeline

Service entièrement géré pour automatiser les phases de livraison.

  • Source : CodeCommit, GitHub, S3
  • Build : AWS CodeBuild (compilation, tests)
  • Deploy : Elastic Beanstalk, ECS, S3, Lambda
# Exemple simplifié de pipeline (structure)
pipeline:
  source: github
  build: codebuild
  deploy: 
    provider: ecs
    image: mon-app:latest

💡 Intégration native avec CloudWatch pour monitoring.

🔷

Azure DevOps Pipelines

Solution complète (CI/CD) avec YAML ou interface graphique. Supporte n’importe quel langage.

✔️ Intégration parfaite avec GitHub & Azure Repos
✔️ Déploiement sur Azure Web Apps, AKS, VM, Functions
✔️ Environnements multi-stages (dev, test, prod) avec approvals
trigger: main
pool: ubuntu-latest
steps:
- script: npm install
- task: AzureWebApp@1
☁️💨

Google Cloud Build

Exécution de builds basée sur des steps conteneurisés, ultra rapide et scalable.

  • Définition d’un fichier cloudbuild.yaml
  • Déclencheurs (trigger) sur dépôt GitHub / Cloud Source Repositories
  • Déploiements natifs sur Cloud Run, GKE, App Engine, Firebase
steps:
- name: 'gcr.io/cloud-builders/docker'
  args: ['build', '-t', 'gcr.io/$PROJECT_ID/mon-app', '.']
- name: 'gcr.io/cloud-builders/kubectl'
  args: ['set', 'image', 'deployment/mon-app', 'app=gcr.io/$PROJECT_ID/mon-app']

🌍 Environnements multiples : dev → staging → prod

Une bonne pratique DevOps pour réduire les risques.

🛠️
Dev
Dernières modifications, tests automatisés
🧪
Staging
Identique à la prod, recette finale, validation métier
🚢
Production
Déploiement progressif (blue/green, canary)

💡 Les pipelines cloud permettent d’enchaîner ces étapes avec des gates manuelles ou automatiques.

📋 Exemple de pipeline multi-cloud (culture)

GitHub Actions (CI) + AWS CodeDeploy (CD).

name: Deploy to AWS
on: push to main
jobs:
  build:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - name: Build & test
        run: npm ci && npm test
      - name: Deploy to EB
        uses: aws-actions/elastic-beanstalk-deploy@v2
🔁 À retenir : Chaque fournisseur propose sa solution, mais les concepts restent identiques (déclencheur, étapes, actions).

📈 Monitoring & rollback automatique

Un bon pipeline intègre des vérifications post-déploiement.

  • ✅ CloudWatch / Azure Monitor / Cloud Monitoring pour détecter les erreurs.
  • ✅ En cas d’échec de santé → rollback vers version précédente.
  • ✅ Stratégies : Blue/Green ou Canary pour réduire l’impact.

“Automatiser sans surveillance n’est pas suffisant ; il faut observer et réagir.”
