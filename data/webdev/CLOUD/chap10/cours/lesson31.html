Surveillance & Monitoring Cloud

CloudWatch · Azure Monitor · Google Cloud Monitoring

Module Cloud Computing — Leçon 31

🎯 Objectifs de la leçon

  • Utiliser les services de monitoring majeurs : CloudWatch (AWS), Azure Monitor, Cloud Monitoring (GCP).
  • Configurer métriques, logs et alertes pour surveiller l’état de santé de l’infrastructure.
  • Créer des tableaux de bord (dashboards) afin de visualiser clairement les performances et détecter les anomalies.
💡 Pourquoi c'est crucial ? Le monitoring vous permet d'anticiper les pannes, d'optimiser les ressources et d'assurer une haute disponibilité.

☁️ Les trois grandes solutions de monitoring

📊 AWS CloudWatch

Collecte des métriques (CPU, réseau, disque), journaux (Logs) et alarmes. Surveillance d'EC2, RDS, Lambda, etc.

✔️ Métriques personnalisées • Dashboards • Anomaly detection
🔷 Azure Monitor

Solution unifiée pour applications, VMs, conteneurs. Inclut Application Insights, Log Analytics.

✔️ Alertes intelligentes • Workbooks • Intégration avec Microsoft Sentinel
📈 GCP Cloud Monitoring

Basé sur Metrics Explorer, alertes et dashboards. Parfait pour GKE, BigQuery, et services Google.

✔️ SLI/SLO natifs • Tracing • Intégration Prometheus

📌 Principe commun :

Chaque cloud fournit des métriques prédéfinies, la possibilité d'envoyer des logs applicatifs et de déclencher des alertes (e-mails, SMS, webhooks) quand un seuil est dépassé.

⚙️ Configurer la surveillance

🔍 1. Métriques & Logs

Activez la collecte des logs détaillés (ex: agent CloudWatch, Azure Log Analytics, Ops Agent sur GCP). Les métriques de base (CPU, mémoire, requêtes réseau) sont automatiques. Pour les applications personnalisées, envoyez des métriques via SDK ou API.

Exemple AWS CLI : aws cloudwatch put-metric-data --namespace "MonApp" --metric-name "TempsReponse" --value 120

🚨 2. Alertes intelligentes

Créez des seuils statiques ou dynamiques (anomaly detection). Déclenchez des actions : envoi vers SNS, webhook PagerDuty, Azure Action Groups, ou exécution de fonctions serverless.

CPU > 85% pendant 5 min Erreurs HTTP 5xx > 10/min

📊 3. Tableaux de bord

Centralisez les widgets pour visualiser les tendances. Exemple: créer un dashboard CloudWatch avec graphes de latence, utilisation disque, nombre de requêtes. Azure Workbooks ou GCP Dashboards permettent des vues personnalisables.

📈 Dashboards & Bonnes pratiques

🖥️ Exemple de dashboard centralisé

Un tableau de bord de supervision cloud typique contient :

  • Latence globale des API (p50, p99)
  • Nombre d'instances actives / auto-scaling events
  • Statut des bases de données managées
  • Alertes récentes et logs d'erreurs
🔵 CPU moyen (pool web) :32%
🟢 Requêtes / sec :1,280
🔴 Erreurs 500 :3 (alerte seuil >5)

✅ Bonnes pratiques

  • Configurer des alertes non seulement sur les ressources mais aussi sur les quotas.
  • Utiliser la centralisation des logs (CloudWatch Logs Insights, Log Analytics).
  • Créer des dashboards partagés pour les équipes DevOps.

⚠️ Pièges à éviter

  • Ne pas surveiller les coûts (budgets Cloud + alertes).
  • Oublier les logs d'application critiques → corrélation difficile.
  • Trop d'alertes peu pertinentes → lassitude (alert fatigue).
