Riverpod : gestion d’état
moderne dans Flutter

Remplacez Provider avec plus de puissance, de sécurité et de simplicité.

🎯 3 objectifs clés ⚡ Providers & ref.watch 🧠 Type-safe

📌 Objectifs de la leçon

1

Comprendre les avantages de Riverpod par rapport à Provider : absence de dépendance au contexte, sécurité de typage, et modularité.

2

Créer des providers concrets : StateProvider, FutureProvider, StreamProvider pour gérer différents types d’états.

3

Utiliser ref.watch, ref.read et ref.listen pour interagir avec l’état et réagir aux changements efficacement.

💡 Pourquoi Riverpod ? Il résout les problèmes de Provider (BuildContext obligatoire, dépendances fragiles) tout en offrant une meilleure testabilité.

✨ Pourquoi Riverpod dépasse Provider

Provider ❌
  • Dépend du BuildContext
  • Difficile à combiner plusieurs providers
  • Risque d'erreur si contexte perdu
  • Consommation moins flexible
Riverpod ✅
  • Pas besoin de BuildContext → plus sûr
  • Composition facile des providers
  • Lisibilité accrue, typage fort
  • ref.watch / ref.read / ref.listen explicites
// Exemple Provider (context obligatoire)
final counterProvider = StateProvider<int>((ref) => 0);
// Consommation dans un widget :
ref.watch(counterProvider); // Pas de context !

📦 Les trois providers essentiels

🔹 StateProvider<T>

Pour gérer un état simple et modifiable (ex: compteur, booléen, texte).

final compteurProvider = StateProvider<int>((ref) => 0);
// Dans le widget : ref.watch(compteurProvider);
// Modifier : ref.read(compteurProvider.notifier).state++;

🔸 FutureProvider<T>

Parfait pour les appels API asynchrones avec gestion automatique du chargement/erreur.

final userProvider = FutureProvider<User>((ref) async {
  return await api.fetchUser();
});

🔹 StreamProvider<T>

Pour réagir à un flux de données (Firebase, WebSockets, bases de données temps réel).

final messagesProvider = StreamProvider<List<Message>>((ref) {
  return messageService.getMessagesStream();
});

🎛️ Les 3 méthodes clés de ref

ref.watch

Écoute un provider et reconstruit le widget automatiquement quand l'état change.

final count = ref.watch(counterProvider);
ref.read

Lit l’état une seule fois, sans écouter les modifications. Utiliser dans des callbacks.

ref.read(counterProvider.notifier).increment();
ref.listen

Exécute une fonction lorsque le provider change, sans rebuild (idéal pour toast, navigation).

ref.listen<int>(counterProvider, (previous, next) {
  if(next == 5) showNotification("Objectif atteint !");
});
💡 Bonnes pratiques : utiliser watch dans la méthode build, read dans les gestes (onPressed), et listen pour les effets secondaires.

📱 Exemple concret : compteur avec Riverpod

// 1. Déclaration du provider
final compteurProvider = StateProvider<int>((ref) => 0);

// 2. Widget utilisateur
class CounterPage extends ConsumerWidget {
  @override
  Widget build(BuildContext context, WidgetRef ref) {
    final compteur = ref.watch(compteurProvider);   // écoute
    return Scaffold(
      body: Center(
        child: Column(
          children: [
            Text('Valeur : $compteur'),
            ElevatedButton(
              onPressed: () => ref.read(compteurProvider.notifier).state++,
              child: Text('Incrémenter'),
            ),
          ],
        ),
      ),
    );
  }
}

🔁 Note : Dans un projet réel, on utilise ConsumerWidget ou ConsumerStatefulWidget pour accéder à ref. Riverpod fournit sa propre extension sans dépendre de BuildContext.

Flux réactif : Provider → watch → mise à jour automatique de l'interface

🧩 Combinaison de providers & exercice

Riverpod permet de combiner plusieurs providers facilement grâce à ref.watch dans un autre provider.

final filterProvider = StateProvider<String>((ref) => 'all');
final filteredTodosProvider = Provider<List<Todo>>((ref) {
  final todos = ref.watch(todosProvider);
  final filter = ref.watch(filterProvider);
  return applyFilter(todos, filter);
});
✏️ Exercice pratique (à faire chez vous)

Créez une app simple qui affiche une liste d'utilisateurs depuis une API fakestore. Utilisez FutureProvider pour charger les données et afficher un indicateur de chargement. Puis implémentez un champ de recherche avec StateProvider et combinez-les dans un Provider dérivé pour filtrer les résultats.

Utiliser final searchProvider = StateProvider((ref) => ""); puis final filteredUsersProvider = Provider((ref) => filterUsers(ref.watch(usersProvider), ref.watch(searchProvider)));
