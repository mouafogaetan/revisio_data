⚙️ Git Hooks : automatisez vos workflows

Les Git Hooks sont des scripts exécutés automatiquement avant ou après certaines actions Git (commit, push, etc.). Ils permettent d’appliquer des contrôles qualité, d’exécuter des tests ou de formater le code.

📁 Emplacement : .git/hooks/ (exemples : pre-commit.sample, pre-push.sample)

Pour activer un hook, il suffit de créer un fichier sans extension (ex: pre-commit) et de le rendre exécutable (chmod +x).

🔹 pre-commit

Se déclenche avant de créer un commit. Idéal pour linter le code, vérifier les espaces, exécuter des tests rapides.

🔹 pre-push

Se déclenche avant un git push. Permet d'exécuter la suite complète de tests ou de rejeter un push si des règles ne sont pas respectées.

✏️ Exemple de hook pre-commit (bash) :

#!/bin/sh
# Vérifier les erreurs de syntaxe Python
flake8 .
if [ $? -ne 0 ]; then
    echo "❌ Erreur de linting, commit annulé"
    exit 1
fi

📦 Git Submodules : réutiliser des dépôts externes

Un sous-module Git permet d’inclure un autre dépôt Git dans votre projet tout en conservant son historique indépendant. Utile pour partager des bibliothèques communes ou des configurations.

# Ajouter un sous-module
git submodule add https://github.com/example/librairie.git libs/librairie

# Cloner un projet avec ses sous-modules
git clone --recursive https://github.com/monprojet.git

# Mettre à jour tous les sous-modules
git submodule update --remote

✅ Avantages

  • Isolation des versions
  • Réutilisation propre
  • Suivi de commits précis

⚠️ Précautions

  • Complexité supplémentaire
  • Attention aux références détachées

🏷️ Versioning sémantique (SemVer) et tags Git

Le versioning sémantique (SemVer) est un standard de nommage des versions : MAJEUR.MINEUR.CORRECTIF

  • MAJEUR : changement incompatible (breaking change)
  • MINEUR : ajout fonctionnel rétrocompatible
  • CORRECTIF : correction de bug rétrocompatible

git tag -a v1.2.0 -m "Version 1.2.0" # créer un tag annoté
git push origin --tags # pousser les tags

On peut utiliser git describe pour obtenir la dernière version à partir d’un tag. Exemple : v1.2.0-3-g1a2b3c4 (3 commits après le tag).

💡 Astuce DevOps : Dans un pipeline CI/CD, on peut automatiquement générer un numéro de version basé sur les tags pour les artefacts.

🌿 Stratégies de branchement : Git Flow vs GitHub Flow

📘 Git Flow (modèle classique)

Branches principales : main (production) et develop (intégration). Branches support : feature/*, release/*, hotfix/*.

Idéal pour les projets avec cycles de release planifiés.

main ●----------------------●
develop ●---●---●---●---●---●
feature ●---●

🚀 GitHub Flow (simplifié)

Une seule branche principale main. Les fonctionnalités sont développées dans des branches feature/ et mergées via Pull Request. On déploie depuis main.

Idéal pour la livraison continue (CI/CD) et les petites équipes agiles.

main ●---●---●---●---●
feature ●---●
PR: review → merge

🤔 Comment choisir ?

Git Flow est plus lourd mais adapté aux applications avec versions multiples. GitHub Flow est plus simple et efficace pour les déploiements fréquents.

🛠️ Bonnes pratiques et outils complémentaires

  • Conventional Commits : standardiser les messages de commit (feat:, fix:, docs:) pour générer automatiquement des changelogs.
  • Git Hooks partagés : utiliser husky (Node.js) ou pre-commit (framework Python) pour partager les hooks avec toute l'équipe.
  • CI/CD avec tags : déclencher un déploiement automatiquement lorsqu'un tag est poussé.
  • Gestion des dépendances internes : préférer les sous-modules ou les packages managers (npm, pip) avec registre privé.
