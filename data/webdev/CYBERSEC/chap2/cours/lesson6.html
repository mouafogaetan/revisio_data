🔐 Leçon 6 • Cybersécurité

Fonctions de hachage
et sécurisation des mots de passe

Comprendre le hachage, les algorithmes modernes, et comment protéger efficacement les mots de passe dans les systèmes.

Bases critiques pour développeurs et administrateurs

🎯 Objectifs de la leçon

1️⃣

Comprendre les propriétés fondamentales du hachage : déterministe, non réversible, résistance aux collisions.

2️⃣

Découvrir les algorithmes modernes (SHA-256, SHA-3, bcrypt, Argon2) et leurs usages.

3️⃣

Utiliser les hachages pour stocker des mots de passe : salage, itération (sel + pepper).

💡 Pourquoi le hachage est-il essentiel ? Il empêche qu'un attaquant découvre les mots de passe même s'il accède à la base de données.

🧠 Propriétés des fonctions de hachage

⚡
Déterministe : une même entrée produit toujours la même empreinte (hash).
🔒
Non réversible (à sens unique) : impossible de retrouver le message original à partir du hash.
🧩
Résistance aux collisions : deux entrées différentes ne doivent pas donner le même hash.
📐 h(x) = y — facile à calculer, mais impossible d'inverser mathématiquement (grâce à la perte d'information).

"Mot de passe" → [Fonction Hachage] → 5e884898da...

✅ même mot de passe → même hash   |   ❌ impossible de déchiffrer

⚙️ Algorithmes de hachage cryptographiques

SHA-256 / SHA-3

Famille Secure Hash Algorithm. SHA-256 produit 256 bits. Utilisé pour signatures, intégrité, blockchain.

bcrypt

Conçu pour mots de passe, intégre un sel et un facteur de coût (itérations lentes).

Argon2 (vainqueur PHC)

Moderne, résistant aux GPU/ASIC, paramètres : mémoire, temps, parallélisme. Recommandé OWASP.

Scrypt / PBKDF2

Alternatives robustes pour dérivation de clés. PBKDF2 utilisé dans WPA2, etc.

⚠️ Ne jamais utiliser MD5 ou SHA-1 pour des mots de passe (considérés vulnérables).

🧂 Stockage sécurisé des mots de passe : sel & itération

Le hachage seul ne suffit pas à cause des attaques par tables arc-en-ciel (pré-calculs). On ajoute :

  • Sel (salt) : valeur aléatoire unique pour chaque utilisateur, concaténée au mot de passe avant hachage. Empêche les tables pré-calculées.
  • Itérations / facteur de coût : appliquer la fonction de hachage plusieurs milliers de fois (ex: bcrypt coût 10-12). Ralentit les attaques par force brute.
  • Pepper (secret) : clé secrète globale stockée séparément (hors base de données).
hash = bcrypt( mot_de_passe + sel_unique, facteur_coût )
stockage en base : (hash, sel, coût)
✅ Bonne pratique : utiliser Argon2id ou bcrypt avec sel généré automatiquement.

🛡️ Pourquoi le sel rend les mots de passe robustes ?

Utilisateur A

motdepasse123

+ sel_A: "f7s9D"

hash_A = 9c8df...

→

Utilisateur B

motdepasse123

+ sel_B: "2kLp9"

hash_B = a7F3e...

✨ Même mot de passe → hash complètement différent grâce au sel. Une attaque par table arc-en-ciel devient impossible.

🔁 Sans sel : hash identiques → vulnérable
🧂 Avec sel : hash uniques → sécurisé
⚙️ Itérations : ralentit le craquage

🔑 Recommandations pratiques pour les mots de passe

✔️ Longueur & complexité : Au moins 12 caractères, mélange de minuscules, majuscules, chiffres, symboles.
✔️ Gestionnaires de mots de passe (Bitwarden, KeePass) pour générer et stocker des mots de passe uniques.
✔️ Double authentification (2FA/MFA) : réduire les risques même si hash compromis.
✔️ Politique de rotation intelligente : ne pas forcer trop souvent, éviter la réutilisation.

📌 Résumé & application concrète

Dans un projet web / développement :

  • Ne jamais stocker de mot de passe en clair.
  • Utiliser Argon2id ou bcrypt (librairies : bcrypt pour Node/Python/PHP).
  • Le sel doit être aléatoire et d’au moins 16 octets.
  • Vérifier la robustesse des politiques de mot de passe côté utilisateur.
  • Mettre en œuvre le hachage aussi pour l’intégrité des données (fichiers, logs).

💡 Exercice possible : comparer les temps de génération bcrypt vs SHA-256, et comprendre l’importance des itérations.
