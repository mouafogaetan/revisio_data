🐳 Conteneurs & Kubernetes : nouveaux défis

Les conteneurs partagent le noyau de l’hôte. Une faille de sécurité peut permettre à un attaquant de compromettre tout le cluster. Kubernetes simplifie l’orchestration, mais étend la surface d’attaque.

⚠️ Risques fréquents

  • Images obsolètes avec vulnérabilités
  • Privilèges excessifs (root dans le conteneur)
  • Mauvaise isolation des namespaces
  • Accès non autorisés à l’API Kubernetes

🛡️ Objectifs de sécurisation

  • Scan des vulnérabilités (CI/CD)
  • Politiques restrictives (PodSecurity)
  • Isolation runtime (gVisor, Kata)
  • Least privilege & RBAC strict

📦 1. Sécuriser les images conteneur

Avant tout déploiement, il faut analyser les images à la recherche de CVE (vulnérabilités connues). Des outils comme Trivy, Clair ou Grype scannent les couches OS et les bibliothèques.

# Exemple avec Trivy
trivy image monapp:latest --severity HIGH,CRITICAL

Bonnes pratiques : utiliser des images minimales (distroless, alpine), mettre à jour régulièrement, intégrer le scan dans le pipeline CI/CD.

⚙️ 2. Politiques de sécurité K8s

Kubernetes propose Pod Security Admission (PSA) (remplace PodSecurityPolicy) et des moteurs comme OPA/Gatekeeper pour des règles personnalisées.

Pod Security Standards

Trois niveaux: Privileged, Baseline, Restricted. Interdit par exemple d’exécuter en root, d’utiliser hostNetwork, etc.

enforce: restricted

OPA Gatekeeper

Permet d'écrire des contraintes personnalisées (Rego). Ex: vérifier que toutes les images proviennent d'un registry approuvé.

Schéma: Contrôle d'admission avant déploiement

🛡️ 3. Isolation des conteneurs

Linux namespaces (PID, NET, MNT, UTS) et cgroups limitent les ressources, mais des vulnérabilités d’évasion persistent. Kubernetes renforce l’isolation :

  • gVisor : noyau en espace utilisateur, isolement fort.
  • Kata Containers : chaque pod = VM légère.
  • Runtime security : Falco détection des comportements anormaux (exécution de commandes suspectes).

🧠 À retenir : même avec des conteneurs, appliquer "Security Context" dans Kubernetes (runAsNonRoot, readOnlyRootFilesystem, capabilities drop).

Exemple de SecurityContext :
securityContext:
  runAsNonRoot: true
  capabilities: { drop: ["ALL"] }

✨ Bonnes pratiques & checklist

✓Utiliser des images signées (cosign, Notary)
✓Network Policies pour isoler les pods
✓Audit des RBAC, moindre privilège
✓Mises à jour régulières du cluster (k8s, containerd)
✓Monitoring des logs avec Falco / audit log
✓Sécuriser etcd (chiffrement des secrets)

📌 Rappel: Le principe de défense en profondeur s’applique aussi aux conteneurs !

🏋️ Exercice & Réflexion

Scénario : Vous déployez une application web sur un cluster Kubernetes. L'image utilisée tourne avec l'utilisateur root, et aucun PodSecurityPolicy n’est défini. L’application a une vulnérabilité RCE (Remote Code Execution).

❓ Question : Quels risques majeurs et quelles solutions immédiates appliqueriez-vous ?

Éléments de réponse : risque d’évasion du conteneur, compromission du nœud. Solutions : scan d’image, appliquer SecurityContext avec runAsNonRoot, activer Pod Security Standard (restricted), implémenter OPA pour interdire les pods privilégiés.

📌 Récapitulatif

  • Scanner les images (Trivy, Grype) avant déploiement.
  • Configurer Pod Security Admission / OPA Gatekeeper.
  • Isolation renforcée avec gVisor / Kata.
  • Limiter les privilèges (SecurityContext, drop capabilities).
  • Surveiller les comportements runtime avec Falco.
