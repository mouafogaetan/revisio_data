🎯 Objectifs de la leçon

À la fin de ce cours, vous serez capable de :

  • ✅ Comprendre la répartition (IaaS, PaaS, SaaS) – Différencier les trois grands modèles de services cloud.
  • ✅ Identifier les responsabilités du fournisseur vs client – Qui gère quoi en matière de sécurité, de systèmes et de données ?
  • ✅ Éviter les erreurs de configuration – Reconnaître les erreurs classiques (accès publics, stockages exposés).
💡 À retenir : le "Shared Responsibility Model" est la clé pour sécuriser vos ressources dans le cloud.

☁️ 1. Les trois grands modèles

IaaS

Infrastructure as a Service

Serveurs, stockage, réseau. Vous gérez l’OS, les applis, les données. Ex: AWS EC2, Azure VM, GCP Compute.

🔧 Contrôle total → responsabilité élevée

PaaS

Platform as a Service

Plateforme pour déployer vos apps sans gérer l’infra sous-jacente. Ex: Heroku, Google App Engine, Azure App Service.

⚙️ Focus sur le code et les données

SaaS

Software as a Service

Logiciel prêt à l’emploi via navigateur. Ex: Gmail, Office 365, Dropbox.

📦 Responsabilité limitée à la configuration utilisateur

Plus on monte, moins on gère d’infrastructure, mais moins on a de contrôle sécurité bas niveau.

🔐 2. Qui fait quoi ? Modèle de responsabilité partagée

Le fournisseur cloud sécurise l’infrastructure du cloud ; le client est responsable de la sécurité dans le cloud.

Couche / élément Fournisseur Cloud Client
Centres de données✅ physique, climatisation, accès❌
Hyperviseur / Compute✅Configuration (groupes de sécurité)
Système d'exploitation (IaaS)❌✅ patchs, durcissement, antivirus
Réseau virtuel / pare-feu✅ (mise à dispo)✅ règles ACL, configuration, NACL
Données & applications❌✅ chiffrement, gestion des accès, backups
Identités (IAM)(plateforme)✅ gérer utilisateurs, rôles, MFA
⚠️ Erreur classique : laisser un bucket de stockage (ex: S3) accessible publiquement. Toujours vérifier les droits !

⚠️ 3. Éviter les erreurs de configuration

La mauvaise compréhension du périmètre de responsabilité est la cause n°1 de fuites de données.

🚫
Stockage exposé en lecture public – buckets cloud ou bases de données accessibles à tout internaute. (Responsabilité client)
⚠️
Ports SSH/RDP ouverts sur 0.0.0.0/0 – attaques par force brute directes. Utilisez des groupes de sécurité restrictifs.
🔑
Clés d’API ou mots de passe en clair dans le code – utiliser des coffres type AWS Secrets Manager.

💡 Bon réflexe : Appliquer le principe du "moindre privilège" et auditer régulièrement la configuration.

📌 Récapitulatif & Bons outils

✔️ Responsabilités client

  • Gestion des identités (IAM)
  • Chiffrement des données sensibles
  • Pare-feu applicatif & règles réseau
  • Sauvegardes & plans reprise (RPO/RTO)
  • Configuration des bases de données

🏢 Responsabilités fournisseur

  • Sécurité physique des datacenters
  • Disponibilité des API et services
  • Mises à jour de l'hyperviseur
  • Protection contre DDoS infrastructure
  • Certifications (ISO 27001, SOC…)
🔎 Pour approfondir : Consultez les modèles de responsabilité AWS, Azure, Google Cloud – souvent schématisés dans leur documentation.

🧠 Auto-évaluation express

1️⃣ Dans le modèle IaaS, qui est responsable de l'installation des correctifs de sécurité sur le système d'exploitation ?
✅ Réponse : Le client (vous). Le fournisseur gère le matériel, mais l'OS est sous votre responsabilité.
2️⃣ Une entreprise utilise Gmail (SaaS). Peut-elle configurer un pare-feu au niveau réseau du fournisseur ?
✅ Non, le fournisseur gère toute la couche infrastructure, le client configure les politiques de partage et l'authentification.
3️⃣ Pourquoi faut-il éviter de laisser un bucket de stockage public sans restriction ?
✅ Cela expose toutes les données à Internet, violation de confidentialité (responsabilité du client).
🚀 Souvenez-vous : Le cloud ne signifie pas "sécurité gérée pour vous" — c'est un partenariat !
