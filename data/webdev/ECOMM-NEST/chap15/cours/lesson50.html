Leçon 50 / 50

🚀 Intégration finale & déploiement

De l'API modulaire à la mise en production réelle

✨ Plateforme e-commerce complète – Stack NestJS + PostgreSQL + Redis ✨

🎯 Objectifs de la leçon

  • ✅ Intégrer tous les modules (auth, produits, panier, commande, paiement) dans une API cohérente
  • ✅ Tester l'API avec des outils comme Postman / Insomnia (scénario complet)
  • ✅ Documenter proprement l'API et générer la documentation interactive
  • ✅ Déployer sur un cloud provider (VPS, Render, etc.) avec bonnes pratiques

🔍 Résultat final : une API prête pour le frontend, sécurisée, testée et accessible en ligne.

🧩 1. Cohérence de l'API

Architecture modulaire

Tous les domaines (Users, Products, Cart, Orders, Payments, Reviews, Wishlist) sont réunis autour de NestJS avec des DTOs communs, guards d’authentification et intercepteurs globaux.

✅ Intégration : chaque module expose ses endpoints REST, l'ensemble forme une API unifiée via le routeur principal.

// Exemple d'import dans app.module.ts @Module({ imports: [AuthModule, ProductsModule, CartModule, OrdersModule, PaymentsModule, ...], controllers: [AppController], })

🧪 2. Tester l'API complète

Simuler un parcours utilisateur complet : inscription ➜ connexion ➜ ajout panier ➜ validation commande ➜ paiement fictif avec Stripe/PayPal.

📌 Scénario type pour Postman / Insomnia :

  1. POST /auth/register → récupérer token JWT
  2. GET /products → afficher catalogue
  3. POST /cart/add → ajouter produit
  4. POST /orders/create → créer commande
  5. POST /payments/stripe/intent → initier paiement

✅ Vérifier la validation des stocks, la mise à jour du statut de commande, l’envoi d’email en background (file d’attente Bull).

💡 Astuce : utiliser des tests automatisés supertest en plus des tests manuels avec Postman.

📚 Documentation interactive

Grâce à @nestjs/swagger, générez automatiquement une documentation OpenAPI accessible via /api/docs.

// Exemple de décorateur pour endpoint produit @ApiOperation({ summary: 'Récupérer tous les produits' }) @ApiResponse({ status: 200, description: 'Liste paginée.' }) @Get() findAll() { ... }

✔️ Les développeurs frontend peuvent tester les endpoints, voir les schémas DTO et authentifier via JWT directement dans Swagger UI.

📘 Conseil : documentez aussi les exceptions et codes d'erreur pour une meilleure adoption.

☁️ 3. Déploiement production

Mettre l’API en ligne avec un environnement robuste (VPS, Docker, ou plateforme PaaS). Étapes typiques :

🐳 1. Conteneurisation

Dockerfile multi-stage + docker-compose (app, Postgres, Redis, Nginx).

⚙️ 2. CI/CD

Automatisation tests, builds et déploiement via GitHub Actions ou GitLab CI.

🌍 3. Hébergement VPS

DigitalOcean, Scaleway, ou Render pour un déploiement simplifié.

🔐 4. Variables secrètes

Gérer .env en production avec JWT_SECRET, Stripe keys, etc.

# Exemple de commandes sur VPS (Ubuntu)
git pull origin main
npm run build
pm2 start dist/main.js --name ecommerce-api
pm2 save

✅ Pensez à configurer un reverse proxy (Nginx) pour SSL (Let's Encrypt) et équilibrage.

📈 Surveillance & logs

bonnes pratiques

Après déploiement, assurez-vous la fiabilité grâce à :

  • Sentry : capture des erreurs en temps réel.
  • Winston/Pino : logs structurés (fichiers, Datadog, ou Papertrail).
  • PM2 / New Relic : métriques CPU, mémoire, uptime.
  • Health checks : endpoint /health pour vérifier DB, Redis, etc.
// Exemple endpoint santé
@Get('health')
healthCheck() { return { status: 'ok', timestamp: new Date() }; }
✨ Bonus : configurer des alertes (Slack/email) lors de panne critique.
