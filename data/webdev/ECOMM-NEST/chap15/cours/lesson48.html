Mise en production & CI/CD

Déployer votre API e-commerce NestJS avec Docker, automatiser les tests, et livrer en production comme un pro !

👨‍🏫 Classe: Développement Web | Matière: API e-commerce

🎯 Objectifs de la leçon

  • ✓ Exécuter les tests automatiquement avant déploiement (CI)
  • ✓ Déployer sur un VPS (DigitalOcean, Scaleway) avec sécurité
  • ✓ Mettre en place les migrations automatiques en production
  • ✓ Comprendre l'intégration continue et le pipeline Git

💡 À la fin, vous saurez livrer une API robuste sur un serveur réel avec des mises à jour fluides !

🤖 1. Exécution automatique des tests (CI)

L'intégration continue (CI) permet de lancer les tests à chaque push sur GitHub/GitLab. Cela évite de déployer du code cassé.

# Exemple de workflow GitHub Actions (.github/workflows/test.yml)
name: CI Tests
on: [push, pull_request]
jobs:
  test:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v3
      - uses: actions/setup-node@v3
        with:
          node-version: 18
      - run: npm ci
      - run: npm run test
      - run: npm run test:e2e
      - run: npm run build

✅ Avant tout déploiement, on exige que les tests unitaires, e2e et le build passent.

🎯 Astuce: Utiliser des seeds en base de test pour des environnements reproductibles.

🌍 2. Déployer sur un VPS

Un VPS (Virtual Private Server) vous donne une machine Linux. Étapes clés :

  • 🔧 Créer un droplet (Ubuntu 22.04) chez DigitalOcean / Scaleway
  • 🔑 Configurer SSH avec clés et désactiver root password
  • 🐳 Installer Docker et Docker Compose sur le VPS
  • 📦 Copier les fichiers (via git clone ou rsync)
# Sur le VPS
git clone https://github.com/votre-repo/ecommerce-api.git
cd ecommerce-api
cp .env.production .env
docker-compose -f docker-compose.prod.yml up -d
Nginx reverse proxy Certbot SSL

🐳 Dockerfile multi-stage & docker-compose

Une image légère et sécurisée pour NestJS :

# Dockerfile
FROM node:18-alpine AS builder
WORKDIR /app
COPY package*.json ./
RUN npm ci
COPY . .
RUN npm run build

FROM node:18-alpine
WORKDIR /app
COPY --from=builder /app/dist ./dist
COPY --from=builder /app/node_modules ./node_modules
COPY package*.json ./
EXPOSE 3000
CMD ["node", "dist/main"]

docker-compose.prod.yml avec app, postgres, redis :

version: '3.8'
services:
  api:
    build: .
    ports:
      - "3000:3000"
    env_file: .env
    depends_on:
      - postgres
      - redis
  postgres:
    image: postgres:15
    volumes:
      - pgdata:/var/lib/postgresql/data
  redis:
    image: redis:7-alpine
volumes:
  pgdata:

🔄 Migrations automatiques

Ne jamais exécuter synchronize: true en production. Utilisez les migrations TypeORM.

// datasource.ts (config)
export default new DataSource({
  type: 'postgres',
  url: process.env.DATABASE_URL,
  migrations: ['dist/migrations/*.js'],
  migrationsRun: false,
});

💡 Script dans package.json :

"migrate:prod": "npm run build && typeorm migration:run -d dist/config/datasource.js"

Dans le pipeline ou au démarrage du container : npm run migrate:prod avant de lancer le serveur.

⚠️ Toujours sauvegarder la base avant une migration sur un site en production !

⚙️ Pipeline CD - Déploiement Automatique

Après validation des tests, on peut déployer automatiquement sur le VPS via GitHub Actions ou GitLab CI.

# Exemple de job CD (GitHub Actions)
deploy:
  needs: test
  runs-on: ubuntu-latest
  steps:
    - name: Checkout code
      uses: actions/checkout@v3
    - name: Deploy to VPS
      uses: appleboy/ssh-action@v0.1.5
      with:
        host: ${{ secrets.VPS_HOST }}
        username: ${{ secrets.VPS_USER }}
        key: ${{ secrets.SSH_PRIVATE_KEY }}
        script: |
          cd /app/ecommerce-api
          git pull origin main
          docker compose -f docker-compose.prod.yml down
          docker compose -f docker-compose.prod.yml build
          docker compose -f docker-compose.prod.yml up -d
          docker exec api npm run migrate:prod

✅ Zéro downtime possible en utilisant des stratégies blue-green ou plusieurs réplicas.

Tests avant merge
