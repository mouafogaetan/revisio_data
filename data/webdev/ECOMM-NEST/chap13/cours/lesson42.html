Tests unitaires & d'intégration

Garantir la fiabilité de l'API e-commerce avec NestJS — mocks, couverture, et tests e2e

⚙️ Jest 🧪 Supertest 🎯 80% coverage

🎯 Objectifs de la leçon

✓ Tester les services avec des mocks (isolation complète)
✓ Tester les contrôleurs isolés et leurs dépendances
✓ Atteindre 80% de couverture de code (Jest coverage)

💡 Pourquoi tester ? Une API e-commerce gère l'argent, les stocks, et les données clients. Un bug peut coûter cher. Les tests automatiques protègent contre les régressions.

🧪 Tests unitaires : Service (exemple CartService)

On teste la logique métier en mockant le repository TypeORM et les dépendances externes.

// cart.service.spec.ts
const mockCartRepo = {
  findOne: jest.fn(),
  save: jest.fn(),
  create: jest.fn(),
};

describe('CartService', () => {
  let service: CartService;

  beforeEach(async () => {
    const module = await Test.createTestingModule({
      providers: [
        CartService,
        { provide: getRepositoryToken(Cart), useValue: mockCartRepo },
        { provide: ProductService, useValue: { findOne: jest.fn() } }
      ],
    }).compile();
    service = module.get<CartService>(CartService);
  });

  it('devrait ajouter un produit au panier', async () => {
    const mockProduct = { id: 'p1', price: 100, stock: 5 };
    mockCartRepo.findOne.mockResolvedValue(null); // panier inexistant
    const result = await service.addToCart('user1', 'p1', 2);
    expect(mockCartRepo.save).toHaveBeenCalled();
    expect(result.items[0].quantity).toBe(2);
  });
});
✨ Avantage : rapide, fiable, pas de base de données réelle.

🎮 Contrôleur isolé : simuler la requête/réponse

On mock le service associé et on vérifie que le contrôleur appelle les bonnes méthodes et retourne les bons statuts HTTP.

// products.controller.spec.ts
const mockProductsService = {
  findAll: jest.fn().mockResolvedValue([{ id: 1, name: 'T-shirt' }]),
  create: jest.fn().mockImplementation(dto => ({ id: 99, ...dto }))
};

describe('ProductsController', () => {
  let controller: ProductsController;

  beforeEach(async () => {
    const module = await Test.createTestingModule({
      controllers: [ProductsController],
      providers: [{ provide: ProductsService, useValue: mockProductsService }],
    }).compile();
    controller = module.get<ProductsController>(ProductsController);
  });

  it('doit retourner la liste des produits', async () => {
    const result = await controller.findAll();
    expect(result).toHaveLength(1);
    expect(mockProductsService.findAll).toHaveBeenCalled();
  });

  it('doit créer un produit et renvoyer 201', async () => {
    const dto = { name: 'Nouveau', price: 50 };
    const output = await controller.create(dto);
    expect(output.id).toBe(99);
  });
});

🌐 Tests d'intégration : supertest + base de test

On teste les endpoints HTTP réels avec une vraie instance NestJS et une base de données dédiée (SQLite ou PostgreSQL de test).

// app.e2e-spec.ts
import * as request from 'supertest';
import { Test } from '@nestjs/testing';
import { AppModule } from '../src/app.module';

describe('Auth e2e', () => {
  let app: INestApplication;

  beforeAll(async () => {
    const moduleRef = await Test.createTestingModule({
      imports: [AppModule],
    }).compile();
    app = moduleRef.createNestApplication();
    await app.init();
  });

  it('POST /auth/register - crée un nouvel utilisateur', async () => {
    const res = await request(app.getHttpServer())
      .post('/auth/register')
      .send({ email: 'test@mail.com', password: '123456' })
      .expect(201);
    expect(res.body.email).toBe('test@mail.com');
  });

  afterAll(async () => {
    await app.close();
  });
});

🔄 Test du flux critique : authentification → panier → commande → paiement

Simuler un scénario utilisateur complet garantit que tous les modules fonctionnent ensemble.

it('flux complet e-commerce', async () => {
  // 1. Inscription
  const { body: user } = await request(app)
    .post('/auth/register')
    .send({ email: 'client@test.com', password: 'pass' });
  
  // 2. Connexion + JWT
  const loginRes = await request(app)
    .post('/auth/login')
    .send({ email: 'client@test.com', password: 'pass' });
  const token = loginRes.body.access_token;

  // 3. Ajout produit au panier
  await request(app)
    .post('/cart/add')
    .set('Authorization', `Bearer ${token}`)
    .send({ productId: 'prod_123', quantity: 2 })
    .expect(201);

  // 4. Validation du panier et passage commande
  const orderRes = await request(app)
    .post('/orders')
    .set('Authorization', `Bearer ${token}`)
    .send({ shippingAddress: 'Yaoundé, Cameroun' })
    .expect(201);
  expect(orderRes.body.status).toBe('pending');
});
✅ Ce type de test détecte les régressions lors des refactorisations majeures.

📌 Bonnes pratiques pour les tests (NestJS)

🧩 Isolation

Tester chaque module séparément, utiliser jest.mock() pour les appels externes (Stripe, email).

🧹 Nettoyage DB

Avant chaque test, tronquer les tables ou utiliser une transaction rollback.

⚡️ Fixtures

Créer des données de test réutilisables (usines / factories) pour gagner du temps.

📈 Coverage ciblée

Se concentrer sur les parties métier: calculs de réduction, validation de stock, etc.

"Un test qui échoue vaut mieux qu'un bug en production" – sagesse DevOps 😉

// Extrait du service
calculateTotal(items: CartItem[], coupon?: Coupon): number {
  let subtotal = items.reduce((sum, i) => sum + i.price * i.quantity, 0);
  if (coupon?.type === 'percent') subtotal *= (1 - coupon.value/100);
  else if (coupon?.type === 'fixed') subtotal -= coupon.value;
  return subtotal > 0 ? subtotal : 0;
}
