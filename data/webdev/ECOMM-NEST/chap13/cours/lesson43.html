⚙️ Configurer l’environnement de test

Fichier test/app.e2e-spec.ts ou test/jest-e2e.json. Utiliser une base de données séparée (ex: ecommerce_test) pour ne pas polluer les données de développement.

// avant tous les tests, on initialise l'app Nest avec un module de test
let app: INestApplication;

beforeAll(async () => {
  const moduleFixture = await Test.createTestingModule({
    imports: [AppModule],
  }).compile();
  app = moduleFixture.createNestApplication();
  await app.init();
});

afterAll(async () => {
  await app.close();
});

📌 On peut surcharger DB_NAME via .env.test ou en utilisant ConfigModule avec un fichier spécifique.

🔄 Tester un endpoint GET avec Supertest

import * as request from 'supertest';

it('GET /products should return paginated products', async () => {
  const response = await request(app.getHttpServer())
    .get('/products?page=1&limit=5')
    .expect(200);
  expect(response.body.data).toBeInstanceOf(Array);
  expect(response.body.total).toBeDefined();
});

✨ Points clés :

  • request(app.getHttpServer()) envoie de vraies requêtes HTTP.
  • .expect(200) vérifie le code status.
  • On utilise Jest pour les assertions (expect(...)).

🗄️ Isoler et nettoyer la base de test

Avant chaque test, on vide les tables ou on utilise une transaction rollback.

// Option 1 : reset via TypeORM
beforeEach(async () => {
  await dataSource.query('TRUNCATE users, products, carts, orders CASCADE');
});

// Option 2 : seed data minimales avant chaque test
beforeEach(async () => {
  await seedTestData();
});

// Option 3 : utiliser un module de test avec .env.test
// configurer TypeORM pour utiliser 'synchronize: true' en test (⚠️ production non)

💡 Astuce : utiliser DataSource pour tronquer les tables dans l’ordre des dépendances (éviter les erreurs de clés étrangères).

🛒 Tester le flux complet e-commerce

it('should complete order flow', async () => {
  // 1. Inscription
  const user = { email: 'test@ex.com', password: '12345678', firstName: 'Jean', lastName: 'Dupont' };
  await request(app.getHttpServer()).post('/auth/register').send(user).expect(201);

  // 2. Login
  const loginRes = await request(app.getHttpServer()).post('/auth/login').send({ email: user.email, password: user.password }).expect(200);
  const token = loginRes.body.accessToken;

  // 3. Ajouter produit au panier
  await request(app.getHttpServer()).post('/cart/add').set('Authorization', `Bearer ${token}`).send({ productId: 1, quantity: 2 }).expect(201);

  // 4. Valider commande
  const orderRes = await request(app.getHttpServer()).post('/orders/checkout').set('Authorization', `Bearer ${token}`).expect(201);
  expect(orderRes.body.status).toBe('pending');

  // 5. Simuler paiement (webhook Stripe...)
  // ...
});

📌 On utilise le token JWT dans le header Authorization pour les routes protégées.

🎭 Mocker Stripe / PayPal / Email

Pendant les tests, on ne veut pas appeler les vraies API de paiement. On utilise des mocks NestJS.

// Dans le module de test, remplacer le vrai service par un mock
const moduleFixture = await Test.createTestingModule({
  imports: [AppModule],
}).overrideProvider(PaymentService)
  .useValue({
    processPayment: jest.fn().mockResolvedValue({ success: true, transactionId: 'mock_123' })
  })
.compile();

// Le webhook Stripe peut être simulé en envoyant une requête POST avec un payload fictif.
💡 On mock également l’envoi d’email (Nodemailer) pour éviter de polluer les vrais comptes.

🧪 Exercice :

Écrivez un test E2E qui :
1. Crée un utilisateur via /auth/register
2. Se connecte et récupère un token
3. Ajoute un produit (avec ID existant en base de test) au panier
4. Passe commande
5. Vérifie que le statut de la commande est 'pending' et que le panier est vide après checkout.

🔍 Voir solution (extrait)

On suit le même pattern que slide 6, avec des assertions supplémentaires :
- Vérifier que le panier avant checkout contient l’item.
- Après checkout, appeler GET /cart → vérifier que items.length === 0.
- Utiliser une transaction rollback ou truncate entre chaque test.
