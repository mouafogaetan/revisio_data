📋 Lister tous les utilisateurs

Endpoint protégé : GET /admin/users. Réservé aux administrateurs.

// users.controller.ts
@Get('admin/users')
@UseGuards(JwtAuthGuard, RolesGuard)
@Roles('admin')
async getAllUsers(@Query() paginationDto: PaginationDto) {
  return this.userService.findAll(paginationDto);
}

// user.service.ts
async findAll({ page, limit }: PaginationDto) {
  const [users, total] = await this.userRepo.findAndCount({
    skip: (page - 1) * limit,
    take: limit,
    select: ['id', 'email', 'firstName', 'lastName', 'isActive', 'roles'],
    relations: ['roles']
  });
  return { data: users, total, page, limit };
}

📌 On n’affiche jamais le mot de passe (même hashé) dans la liste.

🔒 Bloquage temporaire d’un compte

On ajoute une colonne isActive dans l’entité User. Un compte inactif ne peut pas se connecter.

// PATCH /admin/users/:id/block
@Patch('admin/users/:id/block')
@Roles('admin')
async blockUser(@Param('id') id: number) {
  await this.userService.update(id, { isActive: false });
  return { message: 'Utilisateur bloqué' };
}

// PATCH /admin/users/:id/unblock
async unblockUser(@Param('id') id: number) {
  await this.userService.update(id, { isActive: true });
}
💡 Dans le guard JWT, on vérifie aussi user.isActive === true pour refuser l’accès.

🗑️ Suppression de compte (soft delete)

On utilise @DeleteDateColumn() pour marquer un utilisateur comme supprimé sans perdre l’historique des commandes.

// user.entity.ts
@DeleteDateColumn()
deletedAt: Date;

// user.service.ts
async softDelete(id: number) {
  await this.userRepo.softDelete(id);
}

// Dans les requêtes classiques, exclure les utilisateurs supprimés
// TypeORM le fait automatiquement avec find()
// Pour voir les supprimés : withDeleted()

📌 Avantages : conservation des commandes associées (pas d’erreur de clé étrangère), possibilité de restauration.

🔄 Changer les rôles (promouvoir client → vendeur)

Les rôles sont stockés dans une entité Role avec relation ManyToMany. L’admin peut ajouter/supprimer des rôles.

// PATCH /admin/users/:id/roles
async updateRoles(@Param('id') id: number, @Body() { roleNames }: { roleNames: string[] }) {
  const user = await this.userRepo.findOne({ where: { id }, relations: ['roles'] });
  const roles = await this.roleRepo.findBy({ name: In(roleNames) });
  user.roles = roles;
  await this.userRepo.save(user);
}

// Exemple : donner le rôle 'seller' à un client
// { "roleNames": ["customer", "seller"] }
✨ Important : on conserve le rôle customer même quand on ajoute seller.

📦 Voir l’historique des commandes

L’admin peut consulter les commandes de n’importe quel utilisateur (pour support client).

// GET /admin/users/:id/orders
@Get('admin/users/:id/orders')
@Roles('admin')
async getUserOrders(@Param('id') id: number, @Query() paginationDto: PaginationDto) {
  return this.orderService.findByUser(id, paginationDto);
}

// La méthode findByUser existe déjà (lesson21)
// On peut ajouter des filtres : statut, période, etc.

📌 Utile pour gérer les litiges, remboursements ou problèmes de livraison.

🧪 Exercice :

Un vendeur se plaint qu’un client a posté un avis abusif. En tant qu’admin, vous devez :
1. Bloquer temporairement le compte client.
2. Consulter l’historique des commandes du client pour vérifier qu’il a bien acheté le produit.
3. Supprimer l’avis incriminé (autre endpoint).
Décrivez les appels API nécessaires.

🔍 Voir solution

1. PATCH /admin/users/123/block
2. GET /admin/users/123/orders?productId=456 (ou filtrage côté backend)
3. DELETE /admin/reviews/789 (avec permission admin)
Tous ces endpoints doivent être protégés par le rôle 'admin'.
