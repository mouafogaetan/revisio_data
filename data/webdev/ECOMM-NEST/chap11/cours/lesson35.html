💡 Pourquoi ? Empêcher les attaques par brute force, les abus de scraping, et garantir la stabilité du serveur.

⏳ Définition

Le Rate limiting (ou limitation de débit) est une technique qui contrôle le nombre de requêtes qu'un client peut envoyer à une API dans un intervalle de temps donné.

Exemple : 10 requêtes / minute pour un endpoint de recherche.

✅ Avantages :
  • Prévention des attaques DDoS
  • Protection contre le brute force (login)
  • Équité entre utilisateurs
  • Réduction des coûts serveur
⚠️ Sans throttler :
  • Un bot peut tenter 1M de mots de passe
  • Surcharge mémoire / CPU
  • Expérience dégradée pour tous

📦 Installation du module

npm install --save @nestjs/throttler

🔧 Configuration globale (app.module.ts)

import { ThrottlerModule } from '@nestjs/throttler';

@Module({
  imports: [
    ThrottlerModule.forRoot({
      ttl: 60,       // durée en secondes
      limit: 10,     // nombre max de requêtes
    }),
  ],
})
export class AppModule {}
📌 ttl = 60 signifie une fenêtre de 60 secondes. limit = 10 : maximum 10 requêtes par fenêtre.

🔒 Limiter les tentatives de connexion

Sur la route de connexion (/auth/login), on souhaite autoriser seulement 5 tentatives par minute par IP pour éviter le bruteforce.

import { Throttle } from '@nestjs/throttler';

@Controller('auth')
export class AuthController {
  @Post('login')
  @Throttle({ default: { ttl: 60, limit: 5 } })
  async login(@Body() loginDto: LoginDto) {
    // validation du mot de passe
  }
}
✨ Décoration personnalisée 📌 S'applique uniquement à ce contrôleur
🚨 Si l'utilisateur dépasse 5 requêtes, NestJS renvoie automatiquement une erreur 429 Too Many Requests.

🌐 Limiter par IP ou par utilisateur

Le module Throttler utilise par défaut l'adresse IP. Mais on peut personnaliser le tracker pour utiliser l'ID utilisateur (authentifié).

Créer un CustomThrottlerGuard :

export class UserThrottlerGuard extends ThrottlerGuard {
  protected async getTracker(req: Request): Promise<string> {
    const user = req.user; // après JWT
    return user ? user.id : req.ip;
  }
}
💡 Avantage : un utilisateur connecté peut utiliser plusieurs appareils (IP différente) mais reste limité globalement par son ID.

📊 Différentes stratégies

Fenêtre fixe

Nombre de requêtes maximum par intervalle fixe (ex: 100 requêtes / heure). Simple mais peut créer des pics en bordure.

Fenêtre glissante

Plus précis, basé sur un historique glissant. NestJS Throttler utilise une approche basée sur le cache (fenêtre fixe avec stockage).

Token bucket

Un "seau" se remplit à un taux constant. Permet des bursts temporaires.

Par endpoint

Configurations différentes : login strict, catalogue permissif.

🛒 Mise en pratique : API e-commerce

1 Endpoint POST /auth/login → 5 requêtes / minute (IP)
2 GET /products → 100 requêtes / minute (IP) pour éviter le scraping
3 POST /orders → 10 requêtes / minute par utilisateur (pour éviter les doubles commandes accidentelles) 
// Configuration avancée avec plusieurs limites
ThrottlerModule.forRoot([
  { name: 'short', ttl: 1000, limit: 3 },    // 3 req/sec
  { name: 'medium', ttl: 60000, limit: 20 }, // 20 req/min
  { name: 'long', ttl: 3600000, limit: 100 },// 100 req/heure
])

⚠️ Gestion de l'erreur 429

Quand la limite est atteinte, NestJS renvoie :

{ "statusCode": 429, "message": "Too Many Requests" }

Bonnes pratiques :

  • Ajouter des headers X-RateLimit-Limit, X-RateLimit-Remaining, Retry-After
  • Informer l'utilisateur d'attendre avant de réessayer
  • Ne pas bloquer les requêtes légitimes (mettre des limites raisonnables)
  • Utiliser un stockage distribué (Redis) pour les applications en cluster

⚡ Redis pour throttler en production

Par défaut, Throttler stocke les compteurs en mémoire (Map). Pour plusieurs instances, utilisez @nestjs/throttler-storage-redis.

import { ThrottlerModule } from '@nestjs/throttler';
import { ThrottlerStorageRedisService } from '@nestjs/throttler-storage-redis';

ThrottlerModule.forRoot({
  storage: new ThrottlerStorageRedisService({
    host: 'localhost',
    port: 6379,
  }),
  ttl: 60,
  limit: 10,
})

✅ Permet un throttler fiable même si plusieurs pods tournent.
