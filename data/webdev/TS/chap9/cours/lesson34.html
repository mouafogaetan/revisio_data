ESLint + Prettier + tsc --noEmit pour un code propre et fiable

🎯 Objectifs de la leçon

  • ✓ Mettre en place un flux de développement professionnel
💡 Pourquoi ? Un code bien formaté et sans erreurs est plus facile à maintenir en équipe.

🧐 Pourquoi deux outils ?

ESLint

Analyse statique du code : détecte les erreurs, les mauvaises pratiques, applique des règles de style (guillemets, points-virgules…).

🔍 Exemple : interdit var, force const/let.

Prettier

Formateur automatique : indentation, longueur de ligne, virgules, espaces… sans réfléchir.

✨ Exemple : --write reformate tout le projet.

Ils sont complémentaires et souvent utilisés ensemble.

🔧 ESLint avec TypeScript

Le parser standard d'ESLint ne comprend pas la syntaxe TS. Il faut donc utiliser :

npm install --save-dev eslint @typescript-eslint/parser @typescript-eslint/eslint-plugin

Exemple de configuration .eslintrc.json :

{
  "parser": "@typescript-eslint/parser",
  "plugins": ["@typescript-eslint"],
  "extends": [
    "eslint:recommended",
    "plugin:@typescript-eslint/recommended"
  ],
  "rules": {
    "semi": ["error", "always"],
    "quotes": ["error", "single"]
  }
}
📌 Lancer : npx eslint . --ext .ts

✨ Prettier : formateur universel

Installation :

npm install --save-dev prettier

Fichier de config .prettierrc :

{
  "semi": true,
  "singleQuote": true,
  "tabWidth": 2,
  "trailingComma": "es5",
  "printWidth": 100
}

Commandes utiles :

  • npx prettier --check . → vérifie sans modifier
  • npx prettier --write . → formate tous les fichiers
✅ Intégration avec VS Code : extension Prettier + format on save.

📋 tsc --noEmit : validation sans pollution

tsc --noEmit vérifie les types mais ne génère PAS de fichiers JavaScript. Idéal pour les CI/CD ou pré-commit.

# Dans le dossier du projet
npx tsc --noEmit
OptionEffet
--noEmitAucun fichier de sortie (JS, .d.ts, sourcemaps).
--skipLibCheckIgnore la vérification des types dans les fichiers .d.ts (accélère).

🎯 Utilisation typique : avant chaque commit ou dans un hook Git (husky).

⚡ Workflow complet

Package.json avec scripts pratiques :

{
  "scripts": {
    "lint": "eslint . --ext .ts",
    "lint:fix": "eslint . --ext .ts --fix",
    "format": "prettier --write .",
    "type-check": "tsc --noEmit",
    "validate": "npm run type-check && npm run lint && npm run format"
  }
}
🔁 Automatisation → on peut lier ces commandes à un hook pre-commit avec husky + lint-staged.

📌 Exemples d’erreurs courantes détectées

  • @typescript-eslint/no-unused-vars → variables déclarées mais jamais utilisées.
  • @typescript-eslint/explicit-function-return-type → exige le type de retour explicite.
  • @typescript-eslint/no-explicit-any → interdit l'utilisation de any.
// Mauvais code détecté par ESLint
let x = 10;    // 'x' is never reassigned. Use const instead.
function test(a) { return a; }  // Missing return type, 'a' has implicit any.

Prettier corrige automatiquement l'indentation, les guillemets, etc.

🖥️ VS Code : productivité maximale

Extensions recommandées :

  • ESLint (surligne les erreurs en direct)
  • Prettier (formate au Ctrl+S)

Paramètres utilisateur (settings.json) :

{
  "editor.formatOnSave": true,
  "editor.defaultFormatter": "esbenp.prettier-vscode",
  "editor.codeActionsOnSave": {
    "source.fixAll.eslint": true
  }
}
✨ Ainsi chaque sauvegarde applique Prettier + corrige les problèmes ESLint auto-fixables.
