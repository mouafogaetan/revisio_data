🔧 tsconfig.json – Mode strict et bonnes pratiques

Le mode "strict": true active la vérification de type maximale. Voici une configuration robuste pour une application réelle :

{
  "compilerOptions": {
    "target": "ES2020",
    "module": "commonjs",
    "lib": ["ES2020"],
    "outDir": "./dist",
    "rootDir": "./src",
    "strict": true,                     // active strictNullChecks, noImplicitAny, etc.
    "esModuleInterop": true,
    "skipLibCheck": true,
    "forceConsistentCasingInFileNames": true,
    "baseUrl": ".",
    "paths": {
      "@/*": ["src/*"],
      "@models/*": ["src/models/*"]
    },
    "resolveJsonModule": true,
    "noUnusedLocals": true,
    "noUnusedParameters": true
  },
  "include": ["src/**/*"],
  "exclude": ["node_modules", "dist"]
}

✨ Explication rapide : paths + baseUrl permettent d'importer avec des alias (ex: import { User } from '@/models').

📦 Typer une API REST – Exemple avec une TodoList

Avant d'écrire les routes, on définit les modèles (interfaces) pour garantir la cohérence des données.

// src/models/todo.model.ts
export interface Todo {
  id: number;
  title: string;
  completed: boolean;
  createdAt: Date;
  userId?: number;   // optionnel
}

export type CreateTodoDTO = Omit<Todo, 'id' | 'createdAt'>;
export type UpdateTodoDTO = Partial<Pick<Todo, 'title' | 'completed'>>;

// Réponse API standard
export interface ApiResponse<T> {
  success: boolean;
  data?: T;
  message?: string;
}

🎯 Bonne pratique :

Séparez les types dans un dossier models/ ou types/. Utilisez Omit, Partial et Pick pour éviter la duplication.

🌐 API REST – Express + TypeScript (exemple complet)

Installation : npm init -y && npm i express && npm i -D typescript @types/express ts-node nodemon

// src/index.ts
import express, { Request, Response } from 'express';
import { Todo, CreateTodoDTO, ApiResponse } from './models/todo.model';

const app = express();
app.use(express.json());

let todos: Todo[] = [{ id: 1, title: 'Apprendre TS', completed: false, createdAt: new Date() }];

app.get('/todos', (req: Request, res: Response<ApiResponse<Todo[]>>) => {
  res.json({ success: true, data: todos });
});

app.post('/todos', (req: Request<{}, {}, CreateTodoDTO>, res: Response<ApiResponse<Todo>>) => {
  const newTodo: Todo = {
    id: todos.length + 1,
    ...req.body,
    createdAt: new Date(),
    completed: false
  };
  todos.push(newTodo);
  res.status(201).json({ success: true, data: newTodo });
});

app.listen(3000, () => console.log('🚀 API TypeScript sur port 3000'));

✅ On typage les Request et Response avec les génériques pour une sécurité maximale.

⚛️ Application React avec TypeScript (Vite + TS)

Créez un projet : npm create vite@latest mon-app -- --template react-ts. Exemple de composant avec props typées et useState typé :

// src/components/TodoItem.tsx
import { useState } from 'react';

interface Todo {
  id: number;
  text: string;
  done: boolean;
}

interface TodoItemProps {
  todo: Todo;
  onToggle: (id: number) => void;
}

const TodoItem = ({ todo, onToggle }: TodoItemProps) => {
  const [isHovered, setIsHovered] = useState<boolean>(false);

  return (
    <div
      onMouseEnter={() => setIsHovered(true)}
      onMouseLeave={() => setIsHovered(false)}
      className={`p-2 ${isHovered ? 'bg-gray-100' : ''}`}
    >
      <input
        type="checkbox"
        checked={todo.done}
        onChange={() => onToggle(todo.id)}
      />
      <span className={todo.done ? 'line-through' : ''}>{todo.text}</span>
    </div>
  );
};

export default TodoItem;

🎨 Typage explicite des useState, des props et des événements (onChange, onClick).

🏗️ Notions avancées : composite et references

Pour les gros projets (monorepo), on peut diviser le code en plusieurs sous-projets TypeScript.

// tsconfig.base.json
{
  "compilerOptions": {
    "composite": true,
    "declaration": true,
    "strict": true
  }
}

// packages/api/tsconfig.json
{
  "extends": "../../tsconfig.base.json",
  "references": [{ "path": "../shared" }],
  "compilerOptions": { "outDir": "./dist" }
}

⚡ "composite": true permet à TypeScript d'accélérer les compilations et de gérer les dépendances entre projets via references.

🧠 Bonnes pratiques – Projet TypeScript réussi

  • ✅ Toujours activer "strict": true dans tsconfig.json.
  • ✅ Éviter any – utiliser unknown ou des types plus précis.
  • ✅ Organiser les types par domaine (models/, types/, interfaces/).
  • ✅ Mettre en place ESLint dès le début avec la config recommandée.
  • ✅ Utiliser tsc --noEmit dans les pipelines CI pour la vérification des types.
  • ✅ Pour React : typer les refs avec useRef<HTMLDivElement>(null) et les events avec React.MouseEvent.

⚠️ Attention : ne jamais laisser @ts-ignore en production. Préférez @ts-expect-error si vraiment nécessaire.

🚀 Mise en pratique – Appli de gestion de tâches

Architecture recommandée : dossier backend/ (API Express TS) et frontend/ (React TS).
Partagez les types via un dossier shared/ (interfaces communes).

📁 structure

project/
├── backend/
│   ├── src/
│   └── tsconfig.json
├── frontend/
│   ├── src/
│   └── tsconfig.json
└── shared/
    └── types/
        └── todo.ts

🧩 shared/todo.ts

export interface Todo {
  id: number;
  title: string;
  completed: boolean;
}

Les deux projets importent depuis shared pour une cohérence absolue !

🔁 N'oubliez pas d'ajouter "paths" dans chaque tsconfig.json pour résoudre l'import @shared/*.
