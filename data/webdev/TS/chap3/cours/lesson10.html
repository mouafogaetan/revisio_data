💡 Pourquoi c'est utile ? Une même fonction peut se comporter intelligemment selon qu'on lui passe un nombre, une chaîne, ou plusieurs paramètres.

📚 Qu'est-ce qu'une surcharge de fonction ?

En TypeScript, la surcharge permet de définir plusieurs signatures de types pour la même fonction. L'implémentation réelle (une seule) doit être compatible avec toutes les signatures.

⚠️ Important : la surcharge est statique (au niveau des types). En JavaScript après compilation, il n’y a qu’une seule fonction. TypeScript utilise les signatures pour vérifier les appels.

// Signatures de surcharge (disponibles pour l'appelant)
function afficher(valeur: string): void;
function afficher(valeur: number, unite?: string): void;

// Implémentation réelle (corps)
function afficher(valeur: string | number, unite?: string): void {
  if (typeof valeur === "string") {
    console.log("Texte :", valeur);
  } else {
    console.log(`Nombre : ${valeur} ${unite ?? ''}`);
  }
}

// Appels valides ✅
afficher("Bonjour");        // utilise 1ère signature
afficher(42, "kg");         // utilise 2ème signature
// afficher(true); -> ❌ Erreur (aucune signature ne correspond)

👉 La dernière signature (avec string | number) est l'implémentation, inaccessible directement de l'extérieur. On ne voit que les deux premières signatures.

✍️ Comment rédiger une surcharge ?

📌 Étapes :
  1. Écrire une ou plusieurs signatures (sans corps, point-virgule).
  2. Écrire la signature d'implémentation (avec paramètres de type union, et corps).
  3. Dans le corps, utiliser le type narrowing (typeof, instanceof, etc.).
🧩 Règle d'or :

La signature d'implémentation doit être compatible avec toutes les surcharges. Typiquement des paramètres avec types union (string | number) et paramètres optionnels.

// Exemple concret : combiné ou somme
function combine(a: number, b: number): number;       // surcharge 1
function combine(a: string, b: string): string;       // surcharge 2

function combine(a: any, b: any): any {               // implémentation
  if (typeof a === 'number' && typeof b === 'number') return a + b;
  if (typeof a === 'string' && typeof b === 'string') return a.concat(b);
  throw new Error("Arguments incompatibles");
}

const total = combine(10, 20);     // ✅ number
const phrase = combine("Hello ", "Cameroun"); // ✅ string

🌍 Application concrète : tarifs mobiles

Dans une application de rechargement (Orange, MTN, Cameroon), on veut une fonction calculerTarif qui accepte soit un montant numérique (forfait), soit un code promo (string) et retourne un prix final.

// Signatures
function calculerTarif(montant: number): number;
function calculerTarif(code: string, reduction?: number): number;

// Implémentation
function calculerTarif(param: number | string, reduction?: number): number {
  if (typeof param === "number") {
    // Forfait classique sans code
    return param >= 1000 ? param * 0.95 : param;
  } else {
    // Code promo : param est une chaîne
    const remiseBase = reduction ?? 200; // 200 FCFA de réduction par défaut
    if (param === "PROMO2025") return 5000 - remiseBase;
    if (param === "ETUDIANT") return 3000 - remiseBase;
    return 4000; // tarif standard
  }
}

// Appels typés ✅
console.log(calculerTarif(1200));   // 1140 (remise 5%)
console.log(calculerTarif("PROMO2025", 500)); // 4500 FCFA
console.log(calculerTarif("ETUDIANT")); // 2800 FCFA

💬 La surcharge permet au développeur d'avoir une autocomplétion claire : soit montant (number), soit code promo (string).

⚠️ Pièges & meilleures pratiques

❌
Oublier que l'implémentation n'est pas une signature publique – La signature d’implémentation doit être plus large (union / optionnels).
❌
Trop de surcharges complexes – Privilégier des unions ou des paramètres objets si plus de 3-4 variations.
✅
Toujours garder le narrowing (typeof) dans le corps. TypeScript ne peut pas déduire automatiquement quelle surcharge a été appelée.
✅
Ordre des surcharges : Mettre les signatures les plus spécifiques en premier. TypeScript résout la surcharge dans l'ordre.

🧪 À vous de jouer ! (Exercice)

Complétez la surcharge de la fonction formaterDate pour qu’elle puisse accepter soit un timestamp (number) soit une chaîne "YYYY-MM-DD". Écrivez le code en respectant les règles TypeScript.

// ✍️ Écrivez les signatures ici
function formaterDate(valeur: number): string;
function formaterDate(valeur: string): string;

// Implémentation
function formaterDate(valeur: number | string): string {
  if (typeof valeur === "number") {
    return new Date(valeur).toLocaleDateString("fr-FR");
  } else {
    return new Date(valeur).toLocaleDateString("fr-FR");
  }
}

✅ Solution expliquée :

// Signatures :
function formaterDate(timestamp: number): string;
function formaterDate(dateString: string): string;

// Implémentation (unique)
function formaterDate(valeur: number | string): string {
  const date = typeof valeur === "number" 
    ? new Date(valeur) 
    : new Date(valeur);
  return date.toLocaleDateString("fr-FR", { 
    year: 'numeric', month: 'long', day: 'numeric' 
  });
}

// Exemple d'utilisation
console.log(formaterDate(1704067200000)); // 1er janvier 2024
console.log(formaterDate("2025-12-25"));  // 25 décembre 2025

✨ La surcharge rend l'API flexible tout en garantissant qu'on passe bien soit un nombre, soit une chaîne de date.

🔁 Surcharge vs union simple ?

❌ Sans surcharge (union classique)
function saluer(nom: string | string[], age?: number): string {
  if(Array.isArray(nom)) return nom.join(", ");
  return `Bonjour ${nom}`;
}
// Pas de vérification fine: on peut appeler saluer(42) ❌ erreur? non, mais TS n’interdit pas
✅ Avec surcharge (contrôle strict)
function saluer(nom: string): string;
function saluer(noms: string[]): string;

function saluer(nom: any): string {
  if(Array.isArray(nom)) return nom.join(", ");
  return `Bonjour ${nom}`;
}
// saluer(42) => ❌ Type 'number' n'est pas assignable

👍 Plus précis, messages d'erreur clairs.

📖 Conclusion : La surcharge est idéale quand une fonction doit avoir des comportements spécifiques selon des types d'entrée très différents et que l'on souhaite des messages d'erreur explicites.

📌 Ce qu'il faut retenir

  • La surcharge permet d'avoir plusieurs signatures pour une même fonction.
  • Une seule implémentation doit être compatible avec toutes les surcharges.
  • TypeScript utilise la liste des surcharges pour valider les appels, pas l'implémentation.
  • Utile pour les librairies, API, ou fonctions polymorphiques (ex: `createElement`, `convertir`).
