📊 Leçon 15 : Passage du MCD au MLD

Informatique - Terminale C & D (Scientifiques) | Conception de bases de données (MERISE)

📌 Règles de transformation (MCD → MLD)

1. Principe général

Le Modèle Logique des Données (MLD) est obtenu à partir du Modèle Conceptuel des Données (MCD) en appliquant des règles précises. Chaque entité et chaque association du MCD se traduit par une ou plusieurs tables (relations) dans le MLD. On y définit les clés primaires (identifiant unique) et les clés étrangères (lien entre tables).

  • Entité → devient une table. Ses propriétés deviennent des colonnes.
  • Identifiant de l'entité → devient clé primaire (PRIMARY KEY).
  • Association → selon ses cardinalités, se transforme en table ou en clé étrangère.

2. Transformation des associations binaires

Le traitement d’une association dépend de ses cardinalités maximales (1,1 ou 1,n / 0,n). Voici les trois cas principaux à maîtriser pour l’examen (baccalauréat) :

  • Cas 1 : Association de type (1,1) - (1,n) → On ajoute une clé étrangère dans la table de l’entité côté (1,1) pointant vers l’autre entité.
  • Cas 2 : Association de type (0,n) - (0,n) → On crée une table d’association contenant les clés primaires des deux entités (clé composée).
  • Cas 3 : Association de type (1,1) - (1,1) → Fusion possible ou clé étrangère dans l’une des deux tables (selon contraintes).
📐 Illustration des règles de passage (à générer par IA)

3. Clés primaires et étrangères

Dans le MLD, chaque table doit avoir une clé primaire (une ou plusieurs colonnes dont la valeur est unique pour chaque ligne). Une clé étrangère est une colonne qui fait référence à la clé primaire d’une autre table, matérialisant le lien entre tables.

  • PRIMARY KEY (nom_colonne) : garantit l’unicité.
  • FOREIGN KEY (colonne) REFERENCES table(clé_prim) : crée la relation.
  • Une clé étrangère peut être NULL si la cardinalité minimale est 0.

💡 Exemples d’application (cas baccalauréat)

Exemple 1 : Association (1,1) - (0,n) : CLIENT passe COMMANDE

📌 MCD : Entité CLIENT (id_client, nom, adresse) – Entité COMMANDE (num_cmd, date). Association "Passe" : CLIENT (1,1) — Passe — (0,n) COMMANDE. Un client passe 0 à plusieurs commandes ; une commande est passée par un seul client.

✅ Règle appliquée : Côté (1,1) est CLIENT. On ajoute une clé étrangère dans la table du côté (0,n), c'est-à-dire dans COMMANDE.
📋 MLD obtenu :
CLIENT (id_client, nom, adresse) → clé primaire : id_client
COMMANDE (num_cmd, date, #id_client) → clé primaire : num_cmd, clé étrangère : id_client référence CLIENT(id_client).
💬 Astuce : La clé étrangère se place toujours du côté où la cardinalité maximale est "n" (plusieurs) dans le cas 1,n - 1,1.

Exemple 2 : Association (0,n) - (0,n) : ETUDIANT suit COURS

🎓 MCD : Entité ETUDIANT (id_etud, nom, niveau) – Entité COURS (code_cours, intitulé). Association "Suit" : (0,n) ETUDIANT — suit — (0,n) COURS (un étudiant suit plusieurs cours, un cours suivi par plusieurs étudiants).

✅ Règle : Créer une table d’association (table de liaison).
📋 MLD :
ETUDIANT (id_etud, nom, niveau)
COURS (code_cours, intitulé)
SUIVRE ( #id_etud, #code_cours, date_inscription )
Clé primaire de SUIVRE : (id_etud, code_cours) – clé composée. Chaque clé étrangère référence respectivement ETUDIANT et COURS.
💡 La table d'association peut contenir d'autres attributs (ex: date_inscription, note).

Exemple 3 : Association réflexive (employé - manager)

🧑‍💼 Une entité EMPLOYE (id_emp, nom) avec association "Gère" : (0,1) EMPLOYE — gère — (0,n) EMPLOYE. Un employé peut manager plusieurs employés, un employé a au plus un manager.

✅ Transformation : On ajoute une clé étrangère dans la même table pointant vers la clé primaire.
EMPLOYE (id_emp, nom, #id_manager) avec id_manager référence id_emp.
(Cardinalités : 0,n côté subordonné, 0,1 côté manager).
🧠 Bien identifier le rôle de chaque participation. Une clé étrangère peut être NULL si aucun manager.

Règles de passage en un clin d’œil

Type d'association (Cardinalités) Transformation MLD
(1,1) - (1,n)Clé étrangère dans la table de l’entité côté (1,n) référençant l’entité (1,1).
(1,1) - (0,n)Même règle : clé étrangère dans la table côté (0,n) (car max = n).
(0,n) - (0,n)Table d’association avec clé primaire composée des deux clés étrangères.

✅ À retenir : Une association "plusieurs-à-plusieurs" (n-n) nécessite toujours une table intermédiaire.

⚠️ Pièges fréquents (Bac & examens)

  • Erreur classique : Oublier de matérialiser une association (0,n)-(0,n) par une table d’association → risque de redondance et d’anomalies.
  • Confusion clé primaire/clé étrangère : La clé primaire identifie de manière unique une ligne. La clé étrangère établit le lien mais peut être dupliquée.
  • Cardinalités non respectées : Mettre une contrainte NOT NULL sur une clé étrangère alors que la cardinalité minimale est 0 (optionnel).
  • Transformer une association (1,n)-(1,n) : Ce cas suit la même règle que (0,n)-(0,n) : création d’une table de liaison (car max = n des deux côtés).
  • Oubli des identifiants : Dans le MLD, chaque table doit avoir une clé primaire explicitement définie.

✍️ Exercice d’entraînement (type examen)

Sujet : Gestion d’une bibliothèque

🏫 MCD partiel : Entité LIVRE (isbn, titre, auteur) – Entité ADHERENT (num_adherent, nom, prenom). Association "Emprunt" : un adhérent peut emprunter plusieurs livres (0,n), un livre est emprunté par au plus un adhérent à la fois (0,1). De plus, l’emprunt possède une date_emprunt et date_retour.

👉 Question : Donner le MLD correspondant (tables, clés primaires, clés étrangères).
✅ Correction :
• LIVRE (isbn, titre, auteur) → clé primaire : isbn.
• ADHERENT (num_adherent, nom, prenom) → clé primaire : num_adherent.
• Association (0,n) ADHERENT — (0,1) LIVRE : on ajoute la clé étrangère du côté "0,1" (LIVRE) ou bien on crée une table séparée ? Ici cardinalité max (0,1) du côté livre, max (0,n) du côté adhérent. Règle : (0,n)-(0,1) se traite comme (1,n)-(0,1) : on place la clé étrangère dans la table côté (0,1) ? Attention : La méthode classique: On place la clé étrangère dans la table où la cardinalité maximale est 1 (côté optionnel) ? Non, revoyons : Dans association binaire où l’une des cardinalités max = 1 et l’autre = n, la clé étrangère se trouve dans la table de l’entité qui porte la cardinalité max = n (côté plusieurs). Ici, côté adhérent max = n (plusieurs emprunts), côté livre max = 1 (0 ou 1 emprunt). La clé étrangère est donc placée dans la table LIVRE : on ajoute une colonne #num_adherent (qui peut être NULL car 0 emprunt possible). Mais aussi, l’emprunt porte des dates (attributs portés par l’association). Quand l’association porte des attributs, même si (0,n)-(0,1) on préfère créer une table d’association pour éviter de perdre date_emprunt, date_retour.
📌 MLD final (recommandation examen avec attributs d’association) :
LIVRE (isbn, titre, auteur)
ADHERENT (num_adherent, nom, prenom)
EMPRUNT (#isbn, #num_adherent, date_emprunt, date_retour)
Clé primaire : isbn (car un livre ne peut être emprunté qu’une seule fois à un instant donné).
Ou bien clé primaire composée (isbn, num_adherent) si on autorise historique. Mais la cardinalité (0,1) sur livre limite. Pour les examens, privilégiez la table d’association avec clé primaire (#isbn) car un livre ne peut avoir qu’un emprunt actif.
💬 Astuce examen : Quand l’association porte des attributs (date_emprunt, …), même si les cardinalités le permettraient sans table, on crée une table pour respecter la forme normale.
