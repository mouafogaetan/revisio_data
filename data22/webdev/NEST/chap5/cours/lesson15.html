🛡️ Qu'est-ce qu'un Guard ?

Un Guard est une classe annotée par @Injectable() qui implémente l'interface CanActivate. Il détermine si une requête doit être traitée par le contrôleur ou non.

📌 Position dans le cycle de vie :

Middleware → Guard → Interceptor (avant) → Pipe → Contrôleur → Interceptor (après) → Filtre d'exception

✨ Rôle principal :

  • Authentification : Vérifier que l'utilisateur est connecté (ex: JWT valide).
  • Autorisation : Vérifier les droits (rôle admin, permissions spécifiques).
  • Retourne true → accès autorisé ; false → accès refusé (exception 403).
import { Injectable, CanActivate, ExecutionContext } from '@nestjs/common';

@Injectable()
export class ExampleGuard implements CanActivate {
  canActivate(context: ExecutionContext): boolean {
    // Logique de vérification
    return true; // ou false
  }
}

🔑 Utiliser AuthGuard() avec JWT

NestJS fournit AuthGuard() via @nestjs/passport. Il utilise la stratégie JWT pour valider le token.

// products.controller.ts
import { Controller, Get, UseGuards } from '@nestjs/common';
import { AuthGuard } from '@nestjs/passport';

@Controller('products')
@UseGuards(AuthGuard('jwt'))  // Protection au niveau contrôleur
export class ProductsController {
  
  @Get()
  findAll() {
    return ['product1', 'product2'];
  }

  @Get('public')
  @UseGuards(AuthGuard('jwt'))  // Protection au niveau méthode
  findPublic() {
    return 'Donnée protégée';
  }
}
✅ Résultat : Si le token est absent ou invalide → 401 Unauthorized.

👑 Créer un guard personnalisé : @Roles

Nous allons créer un guard qui vérifie si l'utilisateur connecté possède un rôle spécifique (admin, moderator, user).

1. Décorateur @Roles()

// roles.decorator.ts
import { SetMetadata } from '@nestjs/common';
export const ROLES_KEY = 'roles';
export const Roles = (...roles: string[]) => SetMetadata(ROLES_KEY, roles);

2. Guard RolesGuard

// roles.guard.ts
import { Injectable, CanActivate, ExecutionContext } from '@nestjs/common';
import { Reflector } from '@nestjs/core';
import { ROLES_KEY } from './roles.decorator';

@Injectable()
export class RolesGuard implements CanActivate {
  constructor(private reflector: Reflector) {}

  canActivate(context: ExecutionContext): boolean {
    const requiredRoles = this.reflector.getAllAndOverride<string[]>(ROLES_KEY, [
      context.getHandler(),
      context.getClass(),
    ]);
    if (!requiredRoles) return true;
    
    const { user } = context.switchToHttp().getRequest();
    return requiredRoles.some((role) => user.roles?.includes(role));
  }
}

⚙️ Application du guard @Roles()

// admin.controller.ts
import { Controller, Get, UseGuards } from '@nestjs/common';
import { Roles } from '../auth/roles.decorator';
import { RolesGuard } from '../auth/roles.guard';
import { AuthGuard } from '@nestjs/passport';

@Controller('admin')
@UseGuards(AuthGuard('jwt'), RolesGuard)  // Combinaison des guards
export class AdminController {
  
  @Get('dashboard')
  @Roles('admin')    // Seuls les administrateurs peuvent accéder
  getDashboard() {
    return { message: 'Bienvenue admin' };
  }

  @Get('moderate')
  @Roles('admin', 'moderator')  // Admin ou moderateur autorisés
  moderateContent() {
    return { message: 'Contenu modéré' };
  }
}
💡 Astuce : L'ordre des guards compte ! On vérifie d'abord l'authentification (JWT), ensuite l'autorisation (rôles).

🔏 Permissions avancées

On peut généraliser avec des permissions granulaires (lecture, écriture, suppression).

// permissions.decorator.ts
export const Permissions = (...permissions: string[]) => SetMetadata('permissions', permissions);

// permissions.guard.ts
@Injectable()
export class PermissionsGuard implements CanActivate {
  constructor(private reflector: Reflector) {}
  canActivate(context: ExecutionContext): boolean {
    const requiredPermissions = this.reflector.get<string[]>('permissions', context.getHandler());
    if (!requiredPermissions) return true;
    const { user } = context.switchToHttp().getRequest();
    return requiredPermissions.every(perm => user.permissions?.includes(perm));
  }
}

// Utilisation
@Post('articles')
@Permissions('write:articles')
createArticle() { }
🧠 À retenir : La combinaison AuthGuard + RolesGuard/PermissionsGuard est une architecture robuste.

✅ Bonnes pratiques & pièges à éviter

✔️ À faire

  • Centraliser la logique d'authentification
  • Utiliser des guards globaux avec app.useGlobalGuards()
  • Toujours injecter le Reflector pour les métadonnées
  • Combiner plusieurs guards (auth + roles)

❌ À éviter

  • Mettre la logique métier dans un guard
  • Oublier d'injecter les dépendances
  • Hardcoder les rôles dans le guard
  • Négliger les tests unitaires du guard
⚡ Exemple de guard global dans main.ts :
app.useGlobalGuards(new AuthGuard('jwt'), new RolesGuard(reflector));

📝 Résumé & défi

📌 Ce qu'il faut retenir :

  • Un guard décide si une requête est autorisée (canActivate).
  • AuthGuard('jwt') valide le token et attache l'utilisateur à req.user.
  • Les guards personnalisés utilisent Reflector pour lire les métadonnées (@Roles).
  • On peut chaîner plusieurs guards.

🔧 Défi personnel :

Créez un guard OwnerGuard qui vérifie que l'utilisateur connecté est bien le propriétaire d'une ressource (ex: article). Utilisez les paramètres de la requête (userId, articleId).

💡 Indice : utilisez context.switchToHttp().getRequest() pour accéder aux paramètres de route.
