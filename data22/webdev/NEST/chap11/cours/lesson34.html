🌍 Étape 1 : Choisir un domaine & définir le besoin

Avant d'écrire la moindre ligne de code, il faut identifier le métier. Exemples populaires :

  • Blog : articles, commentaires, utilisateurs, catégories.
  • E-commerce : produits, commandes, panier, utilisateurs, catégories.
  • Réseau social : publications, likes, amis, commentaires.

📝 Questions à se poser :

  • Quelles sont les ressources principales ? (ex: Article, User, Comment)
  • Comment ces ressources interagissent-elles ?
  • Quelles opérations sont nécessaires (CRUD, recherches, actions spéciales) ?
📌 Domaine Blog
User 👤 → Article 📝 ← Commentaire 💬

🗂️ Étape 2 : Entités & relations (TypeORM)

Une fois le domaine choisi, on le traduit en classes TypeScript décorées par TypeORM.

// article.entity.ts
@Entity()
export class Article {
  @PrimaryGeneratedColumn()
  id: number;

  @Column()
  title: string;

  @Column('text')
  content: string;

  @ManyToOne(() => User, user => user.articles)
  author: User;

  @OneToMany(() => Comment, comment => comment.article)
  comments: Comment[];
}
@ManyToOne : plusieurs articles appartiennent à un auteur.
@OneToMany : un article possède plusieurs commentaires.

📡 Étape 3 : Planification des endpoints RESTful

Pour chaque ressource principale (ex: articles), on définit les routes standards :

MéthodeURLDescription
GET/articlesListe tous les articles
GET/articles/:idRécupère un article
POST/articlesCrée un article
PATCH/articles/:idMet à jour partiellement
DELETE/articles/:idSupprime un article
GET/articles/:id/commentsRécupère commentaires d'un article

💡 Respectez les conventions REST : noms de ressources au pluriel, utilisation des codes HTTP appropriés (200, 201, 204, 404).

🏗️ Organisation modulaire du projet

NestJS encourage une architecture propre : découpage par module (feature). Pour une API blog :

src/
├── modules/
│   ├── users/          (user.module, controller, service, entity)
│   ├── articles/       (article.module, controller, service, entity)
│   ├── comments/       (commentaires)
│   └── auth/           (authentification JWT)
├── common/             (pipes, guards, interceptors, dtos)
└── main.ts

✅ Chaque module expose son propre contrôleur et service, importe TypeOrmModule.forFeature([Entity]) pour le repository.

🛠️ Exemple concret : créer un article avec son auteur

Dans le service, on utilise le repository TypeORM pour les opérations avancées :

// article.service.ts
async create(createArticleDto: CreateArticleDto, userId: number) {
  const user = await this.userRepository.findOneBy({ id: userId });
  if (!user) throw new NotFoundException();
  const article = this.articleRepository.create({
    ...createArticleDto,
    author: user,
  });
  return this.articleRepository.save(article);
}

On utilise les DTO avec class-validator pour la validation des entrées. Les relations sont gérées via l'affectation d'entités.

🔁 Bonnes pratiques : Ne jamais exposer directement les entités, utiliser des DTO pour les réponses.

🔒 Ajouter validation, authentification & exceptions

  • ValidationPipe global pour vérifier les DTO (class-validator).
  • Authentification JWT avec passport : protéger les routes sensibles (création, modification).
  • Guards personnalisés pour les rôles (admin/user).
  • Exception filters pour formater les erreurs.
@UseGuards(JwtAuthGuard)
@Post()
createArticle(...) { }

Seul un utilisateur authentifié peut créer un article. On récupère l'utilisateur via @Req() ou un décorateur personnalisé.

📚 Documenter l'API avec Swagger + Tests

Une API professionnelle doit être documentée et testée :

📖 Swagger / OpenAPI

Ajoutez @ApiTags, @ApiOperation, @ApiResponse et @ApiProperty dans les DTO.

🧪 Tests unitaires & e2e

Avec Jest et supertest, testez chaque endpoint et simulez les repositories.

npm run test:e2e → Supertest + NestJS Testing

☁️ Déploiement & Réflexions finales

Déployez votre API REST sur des plateformes cloud simples :

  • Railway / Render : intégration Git, variables d'environnement.
  • Heroku (legacy) ou Fly.io.
  • Utiliser Docker pour containeriser l'application + base de données PostgreSQL.
