🧩 Module dynamique : définition

Un DynamicModule est un module qui expose une méthode statique (ex: register, forRoot) retournant une configuration de module personnalisée. Cela permet de paramétrer le module au moment de l'import.

📌 Pourquoi utile ?

Imaginez un module de base de données : on veut pouvoir passer l'URL de connexion, les entités, etc. Sans modules dynamiques, on serait obligé d'utiliser des variables globales ou de répéter la configuration. Les modules dynamiques offrent une API claire.

// Exemple simplifié
@Module({})
export class DatabaseModule {
  static forRoot(options: DbOptions): DynamicModule {
    return {
      module: DatabaseModule,
      providers: [
        { provide: 'CONFIG', useValue: options },
        DatabaseService,
      ],
      exports: [DatabaseService],
    };
  }
}

⚙️ register(), forRoot(), forFeature()

Dans l'écosystème NestJS, plusieurs conventions existent :

📦 register()

Configurer un module avec des options spécifiques à un contexte. Utilisé pour des modules "feature" légers.

🏛️ forRoot()

Configuration globale une seule fois dans l'application (ex: connexion DB). On l'appelle dans le module racine.

🔁 forFeature()

Utilisé pour enregistrer des providers spécifiques à une fonctionnalité (ex: repository pour une entité).

🌍 registerAsync()

Version asynchrone pour charger des configurations depuis des services ou fichiers.

Exemple concret : TypeOrmModule.forRoot() (global) et TypeOrmModule.forFeature([User]) (par entité).

🛠️ Créer un module dynamique : Exemple LoggerModule

Nous voulons un module de logging qui accepte un préfixe et un niveau de log.

// logger.module.ts
import { DynamicModule, Module } from '@nestjs/common';

interface LoggerOptions {
  prefix?: string;
  level?: 'info' | 'warn' | 'error';
}

@Module({})
export class LoggerModule {
  static register(options: LoggerOptions): DynamicModule {
    return {
      module: LoggerModule,
      providers: [
        {
          provide: 'LOGGER_OPTIONS',
          useValue: options,
        },
        LoggerService,
      ],
      exports: [LoggerService],
    };
  }
}

// logger.service.ts
@Injectable()
export class LoggerService {
  constructor(@Inject('LOGGER_OPTIONS') private options: LoggerOptions) {}
  log(message: string) { console.log(`[${this.options.prefix}] ${message}`); }
}

Utilisation dans un module : LoggerModule.register({ prefix: 'API', level: 'info' })

🌐 @Global : rendre un module disponible partout

Lorsqu'un module est décoré avec @Global(), ses exports sont automatiquement disponibles dans tous les modules de l'application, sans avoir à les importer à nouveau.

import { Global, Module } from '@nestjs/common';

@Global()
@Module({
  providers: [SharedService],
  exports: [SharedService],
})
export class SharedModule {}

✅ Dès lors, n'importe quel module peut injecter SharedService sans ajouter SharedModule dans ses imports.

⚠️ Attention: les modules globaux doivent être utilisés avec parcimonie (ex: config, logger, database). Un abus rend le code moins modulaire.

⏳ Configuration asynchrone (registerAsync, forRootAsync)

Parfois, les options de configuration proviennent d'une source externe (API, fichier .env asynchrone). NestJS permet de passer des fabriques asynchrones.

static registerAsync(options: { 
  useFactory: (...args) => Promise<LoggerOptions>,
  inject?: any[]
}): DynamicModule {
  return {
    module: LoggerModule,
    providers: [
      {
        provide: 'LOGGER_OPTIONS',
        useFactory: options.useFactory,
        inject: options.inject || [],
      },
      LoggerService,
    ],
    exports: [LoggerService],
  };
}

📘 Utilisation typique : ConfigService qui va lire un fichier .env de manière asynchrone pour fournir les options.

📁 .env ➡️ ConfigService ➡️ registerAsync() ➡️ DynamicModule

🏆 Bonnes pratiques + Cas concret @nestjs/config

  • Nommer clairement : forRoot (global), register (indépendant), forFeature (spécifique).
  • Utiliser des tokens d'injection uniques pour éviter les collisions.
  • Favoriser les modules dynamiques pour les bibliothèques partagées.
  • Le module @nestjs/config utilise ConfigModule.forRoot() : c'est un excellent exemple de module dynamique et global.

imports: [ ConfigModule.forRoot({ isGlobal: true, envFilePath: '.env' }) ]

👉 En interne, il utilise une méthode forRoot qui renvoie un DynamicModule avec des providers configurés et l'option global: true.

📌 Synthèse & Mémorisation

✔️ Module dynamique = méthode statique retournant DynamicModule
✔️ @Global() partage les exports sans imports répétés.
✔️ forRoot → configuration globale unique, register → config locale.
✔️ registerAsync pour les configurations asynchrones (DB, API).

🧪 Auto-évaluation :

À votre avis, pourquoi TypeOrmModule.forRoot() ne doit être appelé qu'une seule fois ?

Réponse : Parce que forRoot configure la connexion globale (pool de connexions) pour toute l'application. Un appel multiple créerait plusieurs connexions conflictuelles.
