🔍 Pourquoi une gestion d'état ?

Avec setState(), partager des données entre écrans devient vite compliqué (ascenseur). Provider simplifie cela en s'appuyant sur InheritedWidget (un widget natif qui propage les données dans l'arbre).

✅ Avantages de Provider :

  • Pas de boilerplate (contrairement à BLoC pur)
  • Facile à apprendre pour débuter
  • Permet d'écouter des parties précises du widget tree → performances
  • Recommandé officiellement par Google (documentation Flutter)

🧠 InheritedWidget : c'est le héros discret !

Provider utilise InheritedWidget sous le capot pour diffuser les données à tous les widgets descendants sans avoir à passer manuellement par le constructeur.

📦 Installer Provider

# Étape 1 : ajouter dans pubspec.yaml

dependencies:
  flutter:
    sdk: flutter
  provider: ^6.1.1   # dernière version stable

# Étape 2 : importer dans votre fichier Dart

import 'package:provider/provider.dart';

Puis exécutez flutter pub get pour récupérer le package. Simple comme bonjour ! 🌟

📱 Petit rappel: Provider fonctionne aussi bien sur Android, iOS, web et desktop.

🏭 Créer un modèle avec ChangeNotifier

On crée une classe qui étend ChangeNotifier. Chaque fois qu'une donnée change, on appelle notifyListeners().

// modele_compteur.dart
import 'package:flutter/material.dart';

class CompteurModel extends ChangeNotifier {
  int _compteur = 0;
  
  int get compteur => _compteur;
  
  void incrementer() {
    _compteur++;
    notifyListeners();   // 🔔 réveille les widgets qui écoutent
  }
  
  void decrementer() {
    _compteur--;
    notifyListeners();
  }
}

Puis on fournit ce modèle à l'arbre :

void main() {
  runApp(
    ChangeNotifierProvider(
      create: (context) => CompteurModel(),
      child: MyApp(),
    ),
  );
}
💡 Bon à savoir : ChangeNotifierProvider écoute la durée de vie du modèle et le dispose automatiquement.

👂 Utiliser Consumer & Provider.of

Deux façons principales d'accéder aux données :

1️⃣ Consumer<T> (recommandé pour les reconstructions fines)

Consumer<CompteurModel>(
  builder: (context, compteurModel, child) {
    return Text('Valeur : ${compteurModel.compteur}');
  },
);

Seul le widget Consumer se reconstruit quand notifyListeners() est appelé. Très performant !

2️⃣ Provider.of<T>(context)

final compteurModel = Provider.of<CompteurModel>(context);
// ou avec listen: false si on ne veut pas écouter les changements
final compteurModel = Provider.of<CompteurModel>(context, listen: false);

Utile dans les callbacks (onPressed) pour éviter des reconstructions intempestives.

🚀 Exemple concret - Compteur avec Provider

Voici l'implémentation typique :

// Notre page principale
class MonApp extends StatelessWidget {
  @override
  Widget build(BuildContext context) {
    return MaterialApp(
      home: Scaffold(
        appBar: AppBar(title: Text('Provider Counter')),
        body: Center(
          child: Column(mainAxisSize: MainAxisSize.min, children: [
            Consumer<CompteurModel>(
              builder: (_, model, __) => Text(
                '${model.compteur}',
                style: TextStyle(fontSize: 48, fontWeight: FontWeight.bold),
              ),
            ),
            Row(mainAxisAlignment: MainAxisAlignment.center, children: [
              ElevatedButton(
                onPressed: () => context.read<CompteurModel>().incrementer(),
                child: Icon(Icons.add),
              ),
              SizedBox(width: 20),
              ElevatedButton(
                onPressed: () => context.read<CompteurModel>().decrementer(),
                child: Icon(Icons.remove),
              ),
            ])
          ]),
        ),
      ),
    );
  }
}
🧪 Essayez mentalement : chaque clic incrémente ou décrémente, seul le texte se met à jour grâce à Consumer, les boutons restent inchangés (performances).
