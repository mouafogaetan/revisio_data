Gérer l’état simple avec setState()

Apprenez à rendre vos applications Flutter interactives et réactives en maîtrisant la méthode setState().

⚡ Hot reload
🎯 État local
🔄 Rebuild intelligent
📱 Cliquez sur les flèches pour naviguer • Flutter pour débutants

🎯 Objectifs de la leçon

  • ✓ Gérer l’état simple d’un widget avec setState()
  • ✓ Comprendre les limites de cette approche (problème de propagation dans l’arbre)
  • ✓ Créer une application compteur interactive et l’animer manuellement
  • ✓ Différencier StatelessWidget (immuable) et StatefulWidget (état mutable)

💡 Pourquoi c’est important ?

setState() est la porte d'entrée vers la réactivité. Elle permet de rafraîchir l'interface utilisateur après modification des données internes d’un widget.

🧠 Qu’est-ce que l’état (state) ?

L’état d’une application représente l’ensemble des données qui peuvent changer au cours du temps et qui influencent l’interface utilisateur.

📌 StatelessWidget

Widget immuable → une fois construit, il ne change pas (icône, texte fixe).

Ex: Text('Bonjour')
🔄 StatefulWidget

Widget mutable → possède un objet State qui peut être modifié via setState().

Ex: compteur, formulaire, toggle
   StatefulWidget                      State
  ┌─────────────┐                   ┌─────────────┐
  │ MonCompteur │ ── créer ───────► │ _MonCompteur│
  └─────────────┘                   │   int count │
                                    │   setState(){│
                                    │     count++; │
                                    │   }          │
                                    └─────────────┘

⚙️ Comment fonctionne setState() ?

Lorsque vous appelez setState(() { ... }), Flutter exécute le callback, puis déclenche une reconstruction du widget (sa méthode build est rappelée).

// Exemple simple de compteur
int _count = 0;

void _increment() {
  setState(() {
    _count++;
  });
}

// Dans la méthode build :
Text('$_count'),
ElevatedButton(
  onPressed: _increment,
  child: Text('Ajouter')
)

🔁 Cycle de vie rapide :

1. L'utilisateur interagit (clic) → 2. setState modifie les données → 3. build() est exécuté avec les nouvelles valeurs → 4. L'écran se met à jour.

📱 Exemple concret : compteur interactif

Voici un extrait de code Flutter complet pour créer un compteur qui s'incrémente à chaque pression.

import 'package:flutter/material.dart';

class CounterApp extends StatefulWidget {
  @override
  _CounterAppState createState() => _CounterAppState();
}

class _CounterAppState extends State<CounterApp> {
  int _counter = 0;

  void _increment() {
    setState(() {
      _counter++;
    });
  }

  @override
  Widget build(BuildContext context) {
    return Scaffold(
      appBar: AppBar(title: Text('Mon Compteur')),
      body: Center(
        child: Column(
          children: [
            Text('Valeur : $_counter', style: TextStyle(fontSize: 48)),
            ElevatedButton(onPressed: _increment, child: Text('Incrémenter'))
          ]
        )
      )
    );
  }
}
✅ Chaque clic → _counter change → interface reflète la nouvelle valeur.

⚠️ Limites de setState()

setState() est parfait pour un état local à un widget, mais dans des applications complexes, il montre ses faiblesses :

  • 🔁 Propagation difficile : si plusieurs widgets doivent partager la même donnée, vous devez faire remonter l'état (lifting state up) ce qui devient vite lourd.
  • 🧩 Code spaghetti : à grande échelle, setState éparpillé rend le code difficile à maintenir et tester.
  • ⚡ Reconstructions inutiles : sans optimisation, tout le widget est reconstruit même si seule une petite partie change.

💡 Solution : introduction aux gestionnaires d'état (Provider, Riverpod, BLoC) vus plus tard.

Pour des widgets simples et indépendants, setState reste parfait.

📝 Récapitulatif & bonnes pratiques

✅setState() est réservé aux StatefulWidget, ne l’utilisez jamais dans un StatelessWidget.
✅Gardez le callback simple et rapide (évitez les opérations lourdes).
✅Appelez setState uniquement sur des variables dont le changement nécessite un rafraîchissement UI.
✅Pour des états partagés entre plusieurs écrans, préférez les solutions comme Provider, Riverpod ou BLoC (voir chapitres suivants).
