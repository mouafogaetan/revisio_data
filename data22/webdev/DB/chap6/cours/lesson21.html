🔧 Pourquoi analyser ses requêtes ? Le rôle d’EXPLAIN

Imaginez que votre application web charge les données de plus en plus lentement. Parfois, une simple requête SQL met plusieurs secondes ! EXPLAIN est un outil magique qui montre comment le SGBD exécute votre requête : il révèle les accès aux tables, l'ordre des jointures et surtout le type de parcours.

EXPLAIN SELECT * FROM clients WHERE ville = 'Douala';

📢 Avec EXPLAIN, on voit :

  • Si la requête utilise un index ou fait un seq scan (parcours complet de la table).
  • Le nombre estimé de lignes examinées.
  • Les tables jointes et leur ordre.

📊 Interpréter les types de scan

🔴 Sequential Scan (Seq Scan)

Parcourt chaque ligne de la table, une à une. Très lent pour les grandes tables sans filtre efficace.

=> type = ALL dans MySQL / Seq Scan dans PostgreSQL

🟢 Index Scan

Utilise un index pour trouver rapidement les lignes correspondantes. Rapide sur des conditions WHERE ciblées.

=> Extra: Using index

🔷 Bitmap Scan / Index Only Scan

Combine plusieurs index, lit des blocs de données de manière efficace. Souvent un bon compromis.

📌 Note : EXPLAIN ANALYZE donne des statistiques réelles (temps, nombre de lignes) mais attention, cela exécute vraiment la requête.

💡 Quand privilégier un Index Scan ?

Lorsque la condition WHERE filtre un petit pourcentage des lignes (ex: recherche par identifiant, email, date précise). Les index évitent de lire toute la table.

⚡ Optimiser les requêtes lentes : bonnes pratiques

1️⃣ Créer des index pertinents

Index sur les colonnes utilisées dans WHERE, JOIN, ORDER BY.

2️⃣ Éviter SELECT *

Ne lire que les colonnes nécessaires réduit les données déplacées.

3️⃣ Analyser les jointures

Vérifier l’ordre des tables et que les clés étrangères sont indexées.

4️⃣ Réécrire les sous-requêtes

Parfois un JOIN est plus efficace qu’une sous-requête.

-- Exemple : créer un index pour améliorer un scan lent
CREATE INDEX idx_clients_ville ON clients(ville);
EXPLAIN SELECT * FROM clients WHERE ville = 'Yaoundé'; -- Désormais type = ref ou index scan

📝 Étude de cas : Requête lente → requête rapide

Scénario : une table commandes avec 500 000 lignes. Recherche des commandes d’un client sans index.

EXPLAIN SELECT * FROM commandes WHERE client_id = 1045;
❌ AVANT : Seq Scan sur commandes → 500k lignes analysées, lenteur extrême.
✅ APRÈS : Création d’un index CREATE INDEX idx_cmd_client ON commandes(client_id); → Index Scan, quelques millisecondes.

📈 Message à retenir : EXPLAIN vous sauve la vie. Si vous voyez "Seq Scan" sur une table volumineuse, demandez-vous si un index peut être ajouté.

📌 Récapitulatif & astuces finales

  • EXPLAIN est indispensable pour débusquer les goulots d’étranglement.
  • Interprétez les types : ALL / Seq Scan = ⚠️ danger sur grandes tables.
  • index / ref / const = 👍 bonne utilisation d’index.
  • Utilisez des index composites si plusieurs colonnes sont filtrées souvent ensemble.
  • Pensez à la maintenance : ANALYZE tables pour mettre à jour les statistiques.
🌟 Avec ces outils, vos applications web seront plus rapides et vos utilisateurs contents !
