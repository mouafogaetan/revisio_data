🔍 Sous-requêtes dans WHERE

1️⃣ Opérateur IN

Chercher les lignes dont la valeur correspond à une liste retournée par une sous-requête.

SELECT nom, prenom FROM etudiants 
WHERE id_classe IN (SELECT id FROM classes WHERE niveau = 'L3');

📌 Explication : On récupère d'abord les IDs des classes de niveau L3, puis on sélectionne les étudiants correspondants.

2️⃣ EXISTS / NOT EXISTS

Teste si la sous-requête renvoie au moins une ligne. Très efficace pour les conditions d’existence.

SELECT nom FROM produits p
WHERE EXISTS (SELECT 1 FROM commandes c WHERE c.produit_id = p.id);

💡 Renvoie les produits qui ont au moins une commande passée.

🎓 Conseil: EXISTS est souvent plus rapide que IN quand la sous-requête est volumineuse, car il s'arrête dès la première correspondance.

⚖️ ANY, ALL et comparaisons

ANY

Comparer une valeur à n'importe quelle valeur de la liste retournée.

SELECT * FROM employes 
WHERE salaire > ANY (SELECT salaire FROM employes WHERE service = 'IT');

→ Salaire supérieur à au moins un salaire du service IT.

ALL

Comparer à toutes les valeurs de la sous-requête.

SELECT nom, prix FROM produits 
WHERE prix > ALL (SELECT prix FROM produits WHERE categorie = 'Entrée');

→ Produits plus chers que TOUS les produits de la catégorie "Entrée".

📝 Rappel : = ANY équivaut à IN.
> ALL signifie strictement supérieur au maximum de la sous-requête.

📈 Sous-requêtes scalaires dans SELECT

Une sous-requête peut retourner une seule valeur (une ligne, une colonne) et être utilisée comme colonne calculée.

SELECT nom, (SELECT AVG(prix) FROM produits) AS prix_moyen_general
FROM produits;

🔹 Affiche chaque produit avec le prix moyen général de tous les produits (répété sur chaque ligne).

Exemple concret :

SELECT employe.nom, employe.salaire, (SELECT AVG(salaire) FROM employe) as salaire_moyen_societe FROM employe;

Utile pour comparer chaque salaire à la moyenne générale.

🗃️ Sous-requêtes dans FROM (table dérivée)

On traite le résultat d'une sous-requête comme une table temporaire. Cela permet de chaîner des traitements.

SELECT dept.nom_departement, moy_emp.salaire_moyen
FROM departements dept
JOIN (SELECT departement_id, AVG(salaire) as salaire_moyen
      FROM employes GROUP BY departement_id) moy_emp
ON dept.id = moy_emp.departement_id;
✅ Clarté : On sépare la logique d'agrégation du reste.
✅ Performances : Le moteur SQL matérialise parfois la table dérivée.
💡 Bon à savoir : On peut donner un alias obligatoire à la table dérivée (ici "moy_emp").

🔄 Sous-requêtes corrélées

Une sous-requête corrélée fait référence à une colonne de la requête principale. Elle est exécutée pour chaque ligne de la requête externe.

SELECT nom, salaire FROM employes e1
WHERE salaire > (SELECT AVG(salaire) FROM employes e2 WHERE e2.service = e1.service);

📌 Cette requête trouve les employés qui gagnent plus que la moyenne de leur propre service. La sous-requête dépend du service de la ligne courante (e1.service).

📌 Non-corrélée

Indépendante, exécutée une seule fois → plus rapide.

⚠️ Corrélée

Réevaluée pour chaque ligne, attention aux grosses tables.

🏆 Exemples & Bonnes pratiques

  • Exemple métier : Trouver les clients dont le montant total des commandes dépasse la moyenne générale :
    SELECT client_id, SUM(montant) FROM commandes GROUP BY client_id HAVING SUM(montant) > (SELECT AVG(total_par_client) FROM (...))
  • Utiliser EXISTS plutôt que IN lorsque la sous-requête renvoie beaucoup de NULLs ou pour tester l'existence.
  • Préférer les jointures pour des jeux de données très volumineux, les sous-requêtes peuvent être plus lisibles mais parfois moins optimisées.
  • Toujours tester ses sous-requêtes individuellement pour s'assurer qu'elles retournent le bon type de données.

🎯 Résumé : Une sous-requête enrichit vos requêtes SQL pour des analyses avancées.
