Problèmes de concurrence

Quand plusieurs utilisateurs ou applications manipulent la même base de données en même temps, trois problèmes classiques peuvent survenir :

📖 Dirty Read

Lecture de données non encore validées (COMMIT). Si la transaction écrit des données puis annule (ROLLBACK), l'autre transaction a lu des données « sales » qui n'ont jamais existé.

🔄 Non-repeatable Read

Au sein d'une même transaction, une requête identique retourne des résultats différents car une autre transaction a modifié et validé ces données entre-temps.

👻 Phantom Read

De nouvelles lignes apparaissent (insertion) ou disparaissent dans le résultat d'une même requête, à cause d'une autre transaction qui a ajouté/supprimé des enregistrements.

Niveaux d'isolation (SQL standard)

Niveau Dirty Read Non-repeatable Read Phantom Read
READ UNCOMMITTED✔️ Possible✔️ Possible✔️ Possible
READ COMMITTED❌ Impossible✔️ Possible✔️ Possible
REPEATABLE READ❌ Impossible❌ Impossible✔️ Possible (dépend SGBD)
SERIALIZABLE❌ Impossible❌ Impossible❌ Impossible

💡 Résumé simple :

  • READ UNCOMMITTED : le moins isolé, maximum de performances mais risque élevé.
  • READ COMMITTED : empêche les lectures sales (utilisé par PostgreSQL par défaut).
  • REPEATABLE READ : garantit la répétabilité (MySQL InnoDB par défaut).
  • SERIALIZABLE : le plus strict, comme si les transactions s'exécutaient une par une.
⚖️

Comment choisir le bon niveau ?

Le choix dépend du besoin de cohérence des données vs les performances attendues.

📌 Exemple : système bancaire

Pour un virement bancaire, on ne peut tolérer ni dirty read ni phantom. On utilisera SERIALIZABLE ou REPEATABLE READ avec précautions.

📊 Exemple : dashboard analytique

Pour un rapport de ventes en temps réel, quelques écarts mineurs sont acceptables, on peut utiliser READ COMMITTED pour de meilleures performances.

🔧 Configuration pratique sous MySQL :
SET SESSION TRANSACTION ISOLATION LEVEL READ COMMITTED;
SELECT @@transaction_isolation;

📝

Points clés à retenir

✓ L'isolation garantit que les transactions concurrentes n'interfèrent pas de manière inattendue.
✓ Plus le niveau d'isolation est élevé, plus on perd en concurrence (plus de verrous).
✓ PostgreSQL utilise par défaut READ COMMITTED ; MySQL (InnoDB) utilise REPEATABLE READ par défaut.
✓ Pour éviter les phénomènes fantômes, on peut passer à SERIALIZABLE ou utiliser des verrouillages explicites.

"Une bonne compréhension des niveaux d'isolation permet d'équilibrer performance et fiabilité dans vos applications."
