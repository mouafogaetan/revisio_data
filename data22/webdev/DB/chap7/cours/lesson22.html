Transactions ACID

Gérer l'intégrité des données en environnement multi-utilisateur

⚡ Atomicité 🔗 Cohérence 🛡️ Isolation 💾 Durabilité

🎯 Objectifs de la leçon

À la fin de ce cours, vous serez capable de :

1.

Comprendre le concept de transaction : une unité de travail atomique en base de données.

2.

Maîtriser les propriétés ACID (Atomicité, Cohérence, Isolation, Durabilité).

3.

Utiliser BEGIN, COMMIT, ROLLBACK pour contrôler les transactions en SQL.

💡 Pourquoi c'est important ? Imaginez un virement bancaire : si une étape échoue, tout doit être annulé !

🧩 Qu'est-ce qu'une transaction ?

Une transaction est une séquence d'opérations SQL (SELECT, INSERT, UPDATE, DELETE) qui s'exécute comme une seule unité logique de travail.

✨ Soit TOUTES les opérations réussissent ✨
✨ Soit AUCUNE n'est appliquée (comme si rien ne s'était passé) ✨

✅ Exemple concret

Virement bancaire : Débiter compte A (UPDATE) et Créditer compte B (UPDATE). Si le crédit échoue, le débit est annulé.

❌ Sans transaction

Argent débité mais jamais ajouté → perte de données. Les transactions évitent ce chaos !

📌 En SQL : BEGIN TRANSACTION; ... COMMIT; ou ROLLBACK;

🔐 Les propriétés ACID

Quatre piliers qui garantissent la fiabilité des transactions.

A - Atomicité

Tout ou rien. Si une opération échoue, la transaction entière est annulée (ROLLBACK).

C - Cohérence

Une transaction valide transforme la base d'un état cohérent à un autre (contraintes respectées).

I - Isolation

Transactions simultanées n'interfèrent pas. Chacune s'exécute comme si elle était seule.

D - Durabilité

Une fois validée (COMMIT), la transaction persiste même en cas de panne.

⚡ Atomicité & ROLLBACK

L'atomicité garantit qu'une transaction est indivisible. Si une erreur survient, tout est annulé.

-- Exemple: Achat en ligne
BEGIN;
UPDATE stock SET quantite = quantite - 1 WHERE produit_id = 10;
UPDATE commande SET total = total + 25 WHERE id = 5;
-- Si la ligne suivante échoue (contrainte) → ROLLBACK automatique possible
ROLLBACK; -- Annule les 2 modifications
-- ou COMMIT; pour valider définitivement

💻 Commande ROLLBACK : utilisé dans les blocs de programmation (procédures) ou quand une condition n'est pas remplie. En environnement SQL interactif, on peut l'exécuter manuellement.

🔁 BEGIN → Opérations → Si OK → COMMIT ❌ Si erreur → ROLLBACK

🔄 Isolation & problèmes de concurrence

L'isolation évite les interférences quand plusieurs utilisateurs manipulent les mêmes données en même temps.

⚠️
Dirty Read : Lire une donnée pas encore validée (puis annulée).
👀
Non-repeatable read : relire la même donnée, elle a changé entre temps.
👻
Phantom read : de nouvelles lignes apparaissent lors d'une seconde lecture.
🔐 Les SGBD offrent des niveaux d'isolement (Read Uncommitted, Read Committed, Repeatable Read, Serializable) pour équilibrer performance et fiabilité.

💾 COMMIT et Durabilité

COMMIT : valide définitivement toutes les modifications de la transaction.

Une fois le COMMIT exécuté, les changements sont permanents (même en cas de coupure électrique).

📀 Durabilité en pratique

Le SGBD écrit les modifications sur le disque (journaux de transactions) avant d'accuser réception.

🛠️ Exemple SQL
BEGIN;
INSERT INTO factures VALUES (101, 299.99);
COMMIT; ✔️ définitif.
💡 "Une fois commit, vos données sont en sécurité"

🔗 Cohérence des données

Avant et après une transaction, toutes les contraintes d'intégrité (clés primaires, étrangères, UNIQUE, CHECK) doivent être respectées.

-- Exemple : si on ajoute une commande dont l'ID client n'existe pas → violation FK

-- La transaction sera rejetée et annulée (ROLLBACK implicite).

La cohérence garantit que les règles métier sont toujours vraies.

🛠️ Syntaxes SQL principales

-- Début de transaction
START TRANSACTION; ou BEGIN WORK;
-- Opérations
UPDATE comptes SET solde = solde - 100 WHERE id=1;
UPDATE comptes SET solde = solde + 100 WHERE id=2;
-- Vérification (si tout ok)
COMMIT; -- validation
-- En cas de problème
ROLLBACK; -- annulation complète
🔹 BEGIN : démarre une nouvelle transaction
🔹 COMMIT : rend les changements permanents
🔹 ROLLBACK : annule toutes les modifications depuis BEGIN

📝 Récapitulons & Petit exercice

  • Une transaction = unité atomique de travail.
  • ACID = Atomicité, Cohérence, Isolation, Durabilité.
  • BEGIN pour démarrer, COMMIT/ROLLBACK pour valider/annuler.
  • Transactions essentielles pour les applis critiques (banque, réservations).

🧪 À vous de jouer (réflexion) :

Imaginez un système de réservation de vols. Deux clients tentent de réserver le dernier siège. Comment les transactions ACID évitent-elles une surréservation ?

Voir un indice

L'isolation et le verrouillage garantissent qu'une seule transaction verra le siège disponible, l'autre attendra ou échouera proprement. Atomicité empêche une réservation partielle.
