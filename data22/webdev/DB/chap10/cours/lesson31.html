👥 Gestion des utilisateurs
et des privilèges

Apprenez à créer des utilisateurs, accorder des droits et appliquer le principe du moindre privilège pour sécuriser vos bases de données.

📌 CREATE USER 🔑 GRANT / REVOKE 🛡️ Moindre privilège

🎯 Objectifs de la leçon

1️⃣

Créer des utilisateurs avec CREATE USER et gérer l’authentification.

2️⃣

Accorder des privilèges précis avec GRANT et les révoquer avec REVOKE.

3️⃣

Appliquer le principe du moindre privilège pour sécuriser les accès.

4️⃣

Visualiser les droits existants et gérer les rôles utilisateurs.

💡 Pourquoi c'est important ? Une mauvaise gestion des droits peut entraîner des fuites de données ou des suppressions accidentelles. La sécurité commence par les utilisateurs !

👤 1. Créer des utilisateurs

Chaque personne ou application qui accède à la base doit avoir un compte utilisateur distinct.

-- Syntaxe de base (MySQL / PostgreSQL)
CREATE USER 'nom_utilisateur'@'hôte' IDENTIFIED BY 'mot_de_passe_secure';
-- Exemple : utilisateur 'webapp' qui se connecte depuis localhost
CREATE USER 'webapp'@'localhost' IDENTIFIED BY 'SuperP@ssw0rd!';
-- Pour PostgreSQL:
CREATE USER webapp WITH PASSWORD 'SuperP@ssw0rd!';
🔐 Bonnes pratiques
  • Mot de passe robuste (12+ caractères, chiffres, symboles)
  • Limiter l'hôte d'origine (localhost plutôt que %)
  • Ne jamais partager de compte générique
🧑‍🏫 Vérifier les utilisateurs

MySQL : SELECT user, host FROM mysql.user;
PostgreSQL : \du ou SELECT usename FROM pg_user;

🎖️ 2. Accorder & révoquer les privilèges

Une fois l'utilisateur créé, on lui donne les droits nécessaires (SELECT, INSERT, UPDATE, DELETE, etc.)

-- GRANT : donner des droits
GRANT SELECT, INSERT ON base_entreprise.* TO 'webapp'@'localhost';
GRANT ALL PRIVILEGES ON projet_db.* TO 'admin_local'@'localhost' WITH GRANT OPTION;
-- REVOKE : retirer des droits
REVOKE DELETE ON base_entreprise.stock FROM 'webapp'@'localhost';
-- Appliquer les changements (MySQL)
FLUSH PRIVILEGES;
✅ SELECT (lire)
✏️ INSERT / UPDATE
🗑️ DELETE
🔧 CREATE / ALTER / DROP
👑 ALL PRIVILEGES
⚡ Remarque : Dans PostgreSQL, on utilise également GRANT sur les schémas et tables. Ex: GRANT SELECT ON ALL TABLES IN SCHEMA public TO app_user;

🛡️ 3. Principe du moindre privilège

Un utilisateur ne doit avoir que les droits strictement nécessaires à son travail. Ni plus, ni moins.

❌ Mauvais exemple
GRANT ALL PRIVILEGES ON *.* TO 'appli_web'@'%';

⚠️ L'application peut supprimer des tables critiques, modifier la structure, risquer une injection SQL dévastatrice.

✅ Bonne pratique
GRANT SELECT, INSERT, UPDATE ON blog.* TO 'webapp_front'@'localhost';
REVOKE DELETE, DROP ON blog.* FROM 'webapp_front'@'localhost';

✔️ L'application peut lire, ajouter, modifier des articles mais pas supprimer en masse ni altérer le schéma.

🧠
Astuce métier : Séparez les rôles. Ex: un compte pour les rapports (lecture seule), un autre pour la saisie (INSERT/UPDATE), un super-admin pour la maintenance. En cas de piratage, les dégâts sont limités.

🔎 4. Vérifier & gérer les droits existants

SGBDCommande / Requête
MySQLSHOW GRANTS FOR 'user'@'host';
PostgreSQL\dp nom_table ou SELECT * FROM information_schema.role_table_grants;
SQLite(pas de gestion utilisateur native) → géré au niveau OS

📋 Supprimer un utilisateur

DROP USER 'nom'@'hôte';

N'oubliez pas de révoquer d'abord ses privilèges ou de gérer la dépendance.

🔄 Rôles (MySQL 8+ / PostgreSQL)

CREATE ROLE 'lecture_seule';
GRANT SELECT ON *.* TO 'lecture_seule';
GRANT 'lecture_seule' TO 'jean'@'localhost';

✨ À retenir absolument :

  • Jamais de GRANT ALL sur toutes les bases à une application lambda.
  • Utilisez REVOKE quand un collaborateur quitte l’équipe.
  • Auditez régulièrement les droits (moindre privilège) pour éviter les failles.
  • Préférez les connexions via localhost ou réseau sécurisé.
