🏗️ Terraform : écrire l’infrastructure

Terraform (HashiCorp) permet de décrire ses ressources cloud (VPC, sous-réseaux, VM, bases de données) dans des fichiers .tf. Avantage : reproductibilité, versionnage, collaboration.

# Exemple de fichier main.tf (AWS)
provider "aws" {
  region = "eu-west-3"
}

resource "aws_vpc" "main" {
  cidr_block = "10.0.0.0/16"
  tags = { Name = "projet-cloud" }
}

resource "aws_subnet" "public" {
  vpc_id     = aws_vpc.main.id
  cidr_block = "10.0.1.0/24"
  map_public_ip_on_launch = true
}

resource "aws_db_instance" "app_db" {
  allocated_storage    = 20
  engine               = "postgres"
  instance_class       = "db.t3.micro"
  db_name              = "appdb"
  username             = "admin"
  password             = "securepass123"
  skip_final_snapshot  = true
}

🔁 Workflow Terraform : terraform init → terraform plan → terraform apply → terraform destroy

🐳 Déployer sur conteneurs managés

Après avoir provisionné l’infrastructure, on y déploie notre application. Plusieurs options :

  • AWS ECS (Elastic Container Service) avec Fargate (serverless containers)
  • Azure Container Instances / AKS
  • Google Cloud Run ou GKE Autopilot
  • Alternatives serverless : AWS Lambda, Azure Functions

✨ Exemple de définition d’un service ECS dans Terraform :

resource "aws_ecs_task_definition" "app" {
  family                   = "webapp"
  network_mode             = "awsvpc"
  requires_compatibilities = ["FARGATE"]
  cpu                      = "256"
  memory                   = "512"
  container_definitions = jsonencode([{
    name  = "mon-app"
    image = "monrepo/app:latest"
    portMappings = [{ containerPort = 80 }]
  }])
}

⚙️ Pipeline CI/CD intégré

Un pipeline automatise le build, les tests et le déploiement à chaque push sur GitHub.

GitHub Actions (exemple)

name: Deploy to ECS
on: push
jobs:
  deploy:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v3
      - name: Build Docker image
        run: docker build -t monapp .
      - name: Push to ECR
        run: aws ecr push ...
      - name: Update ECS service
        run: aws ecs update-service ...

AWS CodePipeline

Service natif: CodeCommit → CodeBuild → CodeDeploy. Intégration Terraform via actions personnalisées.

🔄 Le pipeline peut exécuter terraform apply après validation des modifications d’infrastructure.

🧩 Assemblage complet : Terraform + ECS + CI/CD

Étapes résumées pour un projet fonctionnel :

  1. Écrire le code Terraform (réseau, cluster ECS, base de données RDS)
  2. Créer un dépôt GitHub avec l’application (Dockerfile)
  3. Configurer GitHub Actions pour construire l’image et la pousser vers un registre (ECR)
  4. Ajouter une étape d’exécution de terraform apply et mise à jour du service ECS
  5. Vérifier l’accès à l’application via l’URL du load balancer
# Exemple de workflow complet (simplifié)
name: Full Cloud Deployment
on: [push]
jobs:
  deploy-infra-and-app:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - name: Setup Terraform
        uses: hashicorp/setup-terraform@v2
      - name: Terraform Init & Apply
        run: |
          cd infra
          terraform init
          terraform apply -auto-approve
      - name: Configure AWS credentials
        uses: aws-actions/configure-aws-credentials@v2
      - name: Build and Push Docker
        run: |
          docker build -t myapp .
          aws ecr get-login-password | docker login ...
          docker push ...
      - name: Update ECS service
        run: aws ecs update-service --cluster main --service app-service --force-new-deployment

📌 Récapitulatif & astuces

✅ Terraform = IaC (déclaratif)

Permet de gérer l'état, de détruire et recréer l'infrastructure facilement.

✅ Conteneurs + orchestration = standard cloud moderne

ECS, EKS, GKE ou AKS simplifient les mises à jour et la scalabilité.

✅ CI/CD = fiabilité et rapidité

Chaque push déclenche les tests et le déploiement sur l’environnement cible.

⚠️ Attention : Ne jamais hardcoder de secrets dans les fichiers .tf. Utilisez des variables sensibles via AWS Secrets Manager ou des variables d’environnement CI/CD.

🤔 Vérification des acquis

Question :

Quelle est la différence principale entre le provisionnement avec Terraform et la configuration avec Ansible dans un pipeline de déploiement cloud ?

Voir la réponse

Terraform est un outil de provisionnement d'infrastructure (création de VMs, réseaux, bases de données). Ansible est un outil de configuration (installation de logiciels, paramétrage des serveurs). Dans un projet cloud, on utilise Terraform pour créer les ressources, puis Ansible (ou des scripts) pour configurer les machines. Avec les conteneurs, la configuration est souvent embarquée dans l'image Docker.
