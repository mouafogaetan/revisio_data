Exercice 1

🔍 Identifier les services GCP

Ressource · niveau easy

Vous êtes consultant cloud pour une startup camerounaise (EdTech) qui souhaite utiliser GCP. Associez chaque besoin au service le plus adapté parmi : Compute Engine, Cloud Storage, BigQuery, Google Kubernetes Engine (GKE).

1.1) Héberger une base de données MySQL pour 500 utilisateurs simultanés. 💾

1.2) Stocker des vidéos de cours et supports PDF accessibles via URL publique. 📁

1.3) Analyser 10 To de logs d’apprentissage pour détecter les sujets difficiles. 📊

1.4) Orchestrer des microservices conteneurisés avec auto-scaling et mises à jour. 🚢

✔️ Correction :

1.1 Compute Engine (VM pour base de données) ou Cloud SQL (service managé) mais dans liste → Compute Engine.
1.2 Cloud Storage (stockage objet universel).
1.3 BigQuery (data warehouse analytique, requêtes SQL massives).
1.4 Google Kubernetes Engine (GKE) pour la gestion de conteneurs.

Exercice 2

🌍 Architecture multi-régions & latence

Compétence · medium

Un site d’information camerounais (Yaoundé + Douala) attire 20 000 visiteurs/jour. L’équipe veut déployer sur GCP en utilisant plusieurs zones de disponibilité pour la haute disponibilité. Ils hésitent entre la région us-central1 (Iowa) et europe-west1 (Belgique).

📡 Latence estimée depuis Douala : vers Iowa ~180 ms, vers Belgique ~110 ms (fibre atlantique).

2.1) Expliquer pourquoi le choix de la région impacte l’expérience utilisateur. (Justifier avec des arguments concrets)

2.2) Proposer deux services GCP permettant de réduire la latence pour les contenus statiques (images, CSS, JS) quel que soit le choix de région.

2.3) Décrire comment configurer un équilibreur de charge (Cloud Load Balancing) pour répartir le trafic entre deux zones de disponibilité dans la même région.

✅ Éléments de correction :

2.1 La latence affecte le First Contentful Paint (FCP). Une latence élevée (180 ms vs 110 ms) peut dégrader l’interactivité, notamment pour les API dynamiques. Le choix de europe-west1 est préférable pour l’Afrique centrale.
2.2 Cloud CDN (cache aux edge locations, dont Johannesburg) + Cloud Storage pour héberger les statiques.
2.3 Créer un groupe d’instances (Managed Instance Group) avec répartition multi-zones, puis un « External HTTP(S) Load Balancer » qui dirige le trafic vers les instances saines dans les zones concernées.

⚙️ Architecture Big Data & Sécurité sur GCP

Exercice 4 · Compétence - niveau very hard

Une entreprise de télécommunications en Afrique centrale génère chaque jour 3 To de données d’appels. Elle souhaite utiliser GCP pour ingérer ces données, les stocker dans un data lake, les analyser via SQL, et créer des dashboards. Elle exige un contrôle d’accès fin (principe du moindre privilège).

4.1) Proposer une chaîne de services GCP (au moins 3) pour réaliser ce pipeline. Justifiez chaque service.

4.2) Expliquer comment IAM (Identity and Access Management) permet à l’équipe Data Analyste d’interroger BigQuery sans avoir accès aux buckets Cloud Storage bruts.

4.3) Évaluer le risque lié à l’absence de clé de chiffrement gérée par le client (CMEK) pour les données sensibles, et proposer une solution native GCP.

4.1 Cloud Pub/Sub (ingestion), Cloud Storage (data lake), BigQuery (analyse), Looker Studio (visualisation). Cloud Composer (Airflow) pourrait orchestrer.
4.2 IAM permet d'attribuer le rôle « BigQuery Job User » et « BigQuery Data Viewer » sur les datasets sans donner les permissions « storage.objects.get » sur les buckets. Séparation des droits par projet.
4.3 Risque : exposition si le fournisseur est compromis ou si la clé Google par défaut est exposée internement. Solution : utiliser Cloud KMS avec CMEK, où l’entreprise contrôle sa propre clé de chiffrement.

✅ Vrai / Faux : Services et régions GCP

easy
5.1 BigQuery est un service de base de données relationnelle transactionnelle (OLTP).
5.2 Les zones de disponibilité GCP sont des data centers distincts au sein d’une région.
5.3 Compute Engine ne permet que des workloads stateful, pas de conteneurs.
5.4 Cloud Functions est l’offre serverless de Google.

5.1 Faux – BigQuery est un entrepôt analytique (OLAP).
5.2 Vrai – Zones de disponibilité = centres de données indépendants.
5.3 Faux – Compute Engine supporte aussi des conteneurs via les VM container-optimized ou GKE.
5.4 Vrai – Cloud Functions (serverless) et Cloud Run.
