Leçon 21 : IAM & Contrôle d'accès

"Gérer les identités, utilisateurs, rôles et politiques dans le cloud"

👨‍🏫 Classe : Développement Web | Matière : Initiation au Cloud Computing

Cameroun 🇨🇲 | Enseignant : Cours interactif

🎯 Objectifs de la leçon

  • 1 Comprendre les utilisateurs, groupes, rôles et politiques IAM.
  • 2 Appliquer le principe du moindre privilège (ne donner que les accès nécessaires).
  • 3 Utiliser les rôles pour les services (pas de clés statiques) et sécuriser les accès.
💡 Pourquoi c'est essentiel ? Dans le cloud, bien gérer les identités empêche les fuites de données et les accès non autorisés.

🔐 Qu'est-ce que l'IAM ?

IAM = Identity and Access Management (Gestion des identités et des accès). C'est le cœur de la sécurité cloud.

👤 Utilisateurs

Personnes réelles (admin, développeur, stagiaire). Chaque utilisateur a des identifiants uniques.

👥 Groupes

Ensemble d'utilisateurs partageant les mêmes permissions (ex: "Équipe DevOps"). On applique les politiques au groupe.

🎭 Rôles

Attribués à des services AWS/Azure/GCP ou utilisateurs assumant temporairement des permissions.

📜 Politiques (Policy)

Document JSON/YAML qui définit précisément ce qui est autorisé ou interdit (ex: "autoriser la lecture de S3").

⭐ AWS IAM, Azure AD, GCP IAM : tous ces concepts sont similaires.

⚖️ Principe du moindre privilège

"Ne donner à un utilisateur (ou service) que les droits strictement nécessaires à son travail, rien de plus."

❌ Mauvais exemple :

Donner un accès administrateur complet à un stagiaire qui doit juste consulter des logs → Risque énorme !

✅ Bonne pratique :

Créer un rôle "LecteurLogs" qui permet uniquement de lire les fichiers de log, et l'attribuer au stagiaire.

🔒 Appliquer ce principe réduit la surface d'attaque et limite les dégâts en cas de compromission.

🤖 Rôles pour les services : adieu les clés statiques !

Dans le cloud, vos applications (serveurs, fonctions Lambda, containers) ont besoin d'accéder à d'autres services (base de données, stockage).

❌ Mauvaise pratique:

Stockage de clés d'API (Access Key / Secret Key) dans le code ou variables d'environnement → Risque de fuite massive.

✅ Bonne pratique:

Attribuer un rôle IAM à votre service (EC2, ECS, fonction). Le cloud fournit automatiquement des identifiants temporaires et sécurisés.

📌 Exemple concret (AWS) :

Une VM EC2 doit lire des objets dans S3. Au lieu de coder des clés, on crée un rôle "S3ReadOnly" et on l'attache au profil EC2. AWS délivre automatiquement des credentials temporaires via le metadata service. 🔐

📝 Synthèse & conseils pratiques

  • Utilisateurs, Groupes, Rôles : organisez les permissions par métier.
  • Moindre privilège : commencez avec le minimum de droits et augmentez si nécessaire.
  • Rôles pour les applications : évitez les clés d'accès permanentes, utilisez des rôles.
  • Politiques : écrivez des règles précises (Action "s3:GetObject", Resource "mon-bucket/logs/*").
  • Audit régulier : dans le cloud, utilisez des outils comme AWS IAM Access Analyzer, Azure AD reviews.
🌍 Adaptation locale (Cameroun) : Que vous utilisiez AWS, Azure ou Google Cloud, ces principes IAM sont universels. Pour vos projets de start-up ou entreprises, une bonne gestion IAM évite les pertes de données sensibles (factures, infos clients).
