Concevoir son projet React Native

Choisir une idée, dessiner les maquettes, planifier les fonctionnalités et structurer ses données

Projet Maquettes Architecture

Lesson 35 · Chapitre 11 · Projet final

🎯 Objectifs de la leçon

1.

Choisir un projet adapté à vos compétences (météo, e-commerce, réseau social, todo list avancée).

2.

Dessiner les maquettes (wireframes) et définir la navigation entre écrans.

3.

Planifier les fonctionnalités par ordre de priorité (MVP).

4.

Concevoir l'architecture des données (state global, API, stockage local).

💡 À retenir : 80% de la réussite d'un projet se joue en amont : bonne idée, bonne organisation !

🚀 1. Choisir son idée d'application

Voici 4 exemples classiques, du plus simple au plus complexe :

🌤️ Application Météo
  • API météo (OpenWeatherMap)
  • Recherche de ville
  • Affichage des prévisions
  • Niveau : facile
📝 To-do list avancée
  • Catégories, priorités
  • Dates, rappels
  • Stockage local (AsyncStorage)
  • Niveau : moyen
🛍️ E-commerce
  • Liste produits, panier
  • Authentification
  • Passer commande
  • Niveau : difficile
👥 Réseau social
  • Publications, likes
  • Commentaires, profils
  • Upload de photos
  • Niveau : très difficile
✅ Conseil : Commencez par un MVP (Minimum Viable Product) – seulement les fonctionnalités essentielles.

✏️ 2. Maquettes et navigation

Avant de coder, dessinez vos écrans sur papier ou avec un outil gratuit (Figma, Excalidraw).

Wireframe typique d'une app e-commerce

📱 Écrans à prévoir :
  • Écran d'accueil
  • Écran de détail
  • Panier / Profil
  • Connexion
🧭 Navigation choisie :
  • Bottom Tabs (Accueil, Panier, Profil)
  • Stack Navigation (Détail produit)
  • Drawer (menu latéral)

📋 3. Planification des fonctionnalités

Utilisez la méthode MoSCoW : Must have, Should have, Could have, Won't have.

🔴 Must have (indispensable)

Afficher la liste des produits, ajouter au panier, valider commande.

🟡 Should have (important)

Recherche de produits, filtre par catégorie.

🟢 Could have (bonus)

Notation des produits, mode sombre.

📅 Exemple de planning (2 semaines) :
  • J1-J2 : Maquettes + navigation
  • J3-J5 : Écrans statiques
  • J6-J8 : API et state management
  • J9-J10 : Persistance + tests
  • J11-J12 : Bug fixing & polish

🏗️ 4. Architecture des données

Décidez comment vos données circulent :

🗄️ Sources de données
  • ✓ API REST (fetch / Axios)
  • ✓ Stockage local (AsyncStorage / SecureStore)
  • ✓ State global (Context API / Redux)
🔄 Flux typique 
API → Action → Reducer → State → UI

Exemple avec Redux Toolkit

📦 Organisation des dossiers (clean architecture) : 
src/
├── components/     # réutilisables (Button, Card)
├── screens/        # écrans (HomeScreen, DetailScreen)
├── navigation/     # Stack, Tabs, Drawer
├── services/       # Appels API (api.js)
├── store/          # Redux slices
├── hooks/          # useFetch, useAuth
└── utils/          # constantes, helpers

🌦️ Étude de cas : App Météo

1
Fonctionnalités MVP – Rechercher une ville, afficher température, icône météo, prévisions 5 jours.
2
Maquettes – Écran principal (recherche + météo actuelle), écran détail prévisions.
3
Navigation – Stack : Accueil → Détail (avec paramètres).
4
Architecture – API OpenWeatherMap, state local (useState/useEffect), stockage de la dernière ville (AsyncStorage).
// Exemple de service météo
const fetchWeather = async (city) => {
  const response = await fetch(`https://api.openweathermap.org/data/2.5/weather?q=${city}&appid=VOTRE_CLE&units=metric`);
  const data = await response.json();
  return data;
};
