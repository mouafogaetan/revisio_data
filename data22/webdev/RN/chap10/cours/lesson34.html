De l'APK à l'App Store

📘 Chapitre 10 - Leçon 34

🎯 Objectifs de la leçon

1.

Générer un fichier APK / AAB pour Android avec Expo (expo build ou EAS Build).

2.

Préparer une application iOS pour l'App Store (certificats, provisioning, TestFlight).

3.

Soumettre l'application au Google Play Store et à l'App Store (fiche, captures, descriptions).

💡 À retenir : La publication est l'étape finale qui rend votre application accessible au public. Chaque store a ses propres règles et délais de validation.

📦 Les différents types de builds

🧪

Debug build

Pour le développement. Lente, avec logs, non optimisée.

🚀

Release build

Optimisée, minifiée, prête pour la distribution.

📱

APK / AAB

APK (fichier unique) ou AAB (format recommandé par Google).

🔧 Formats selon la plateforme :

  • Android : APK (Android Package Kit) ou AAB (Android App Bundle)
  • iOS : IPA (iOS App Store Package) – généré via Xcode ou EAS Build
  • Expo : EAS Build (recommandé) ou expo build (legacy)

⚙️ EAS Build : Configuration

EAS Build est le service d'Expo pour compiler votre application dans le cloud.

# 1. Installer EAS CLI
npm install -g eas-cli

# 2. Se connecter à son compte Expo
eas login

# 3. Configurer le projet
eas build:configure

# 4. Créer le fichier eas.json (généré automatiquement)

📄 eas.json exemple :

{
  "cli": { "version": ">= 3.0.0" },
  "build": {
    "development": { "developmentClient": true, "distribution": "internal" },
    "preview": { "android": { "buildType": "apk" } },
    "production": {}
  }
}

📱 Générer un APK / AAB

1
Build de preview (APK) – pour tester avant publication
eas build --platform android --profile preview
2
Build de production (AAB) – pour le Play Store
eas build --platform android --profile production
3
Télécharger le fichier – depuis le lien fourni ou via eas build:list

✅ Note : Le premier build peut prendre 10-20 minutes. EAS génère automatiquement les clés de signature si vous n'en avez pas.

🍎 Préparer iOS pour l'App Store

Apple nécessite plusieurs prérequis :

🍎 Compte développeur Apple (99$/an)
📜 Certificat de distribution – créez-le dans l'Apple Developer Console
🔐 App ID et Provisioning Profile – liés à votre bundle identifier
# Build pour iOS avec EAS
eas build --platform ios --profile production

EAS Build vous guidera pour créer les certificats automatiquement.

🧪 Tester avec TestFlight

TestFlight permet de distribuer une version bêta de votre application à des testeurs avant la publication officielle.

📲 Étapes :

  1. Après le build iOS, allez dans App Store Connect
  2. Créez une nouvelle version et téléchargez l'IPA via Transporter ou Xcode
  3. Sous l'onglet TestFlight, invitez des testeurs (jusqu'à 10 000)
  4. Les testeurs reçoivent un email et installent via l'app TestFlight

📢 Soumettre au Google Play Store

🔑 Prérequis

  • Compte développeur Google (25$ unique)
  • AAB signé généré
  • Icône 512x512, captures d'écran
  • Description, catégorie, politique de confidentialité

📤 Publication

  • Connectez-vous à Google Play Console
  • Créez une application → "Toutes les applications" → "Créer"
  • Téléversez l'AAB dans "Version de production"
  • Remplissez la fiche (titre, description, graphismes)
  • Soumettez pour examen (24-48h en général)

📱 Soumettre à l'App Store

Étapes clés :

  • 1. Accédez à App Store Connect → "Mes apps" → "+"
  • 2. Remplissez les informations : nom, description, mots-clés, catégorie
  • 3. Téléchargez l'IPA via Transporter (Mac) ou Xcode
  • 4. Ajoutez des captures d'écran pour chaque taille d'iPhone/iPad
  • 5. Configurez les prix et disponibilités
  • 6. Soumettez pour révision (délai : 1 à 7 jours)
⚠️ Apple est très stricte sur les règles de design, de contenu et de vie privée.

✅ Bonnes pratiques avant publication

✓ Tester sur plusieurs appareils réels (différentes tailles et OS)
✓ Vérifier les permissions (AndroidManifest.xml / Info.plist)
✓ Optimiser les assets (images, polices) pour réduire la taille
✓ Mettre à jour le numéro de version (versionName, versionCode)
✓ Configurer app.json avec icônes, splash screen, orientation
✓ Préparer une politique de confidentialité (obligatoire pour les stores)

🔄 Mises à jour OTA (Over-The-Air)

Avec Expo, vous pouvez publier des mises à jour JavaScript sans repasser par les stores.

# Publier une mise à jour OTA
expo publish

# Avec EAS Update (recommandé)
eas update --branch production --message "Fix bug"

⚡ Avantages : correction rapide de bugs, pas de validation store.
⚠️ Limites : ne peut pas changer le code natif (modules, permissions).
