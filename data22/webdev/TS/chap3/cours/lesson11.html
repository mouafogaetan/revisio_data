💡 TypeScript est intelligent mais parfois il a besoin de votre aide. Les assertions vous permettent de lui dire "fais-moi confiance, je sais ce que je fais".

🧠 L'inférence de type

TypeScript analyse automatiquement la valeur que vous assignez pour en déduire le type. Pas besoin de tout typer manuellement !

let message = "Bonjour";        // TypeScript infère: string
let nombre = 42;                // infère: number
let estVrai = true;             // infère: boolean
let tableau = [1, 2, 3];        // infère: number[]
let personne = { nom: "Dupont", age: 30 };  // infère: { nom: string; age: number }

// ❌ Erreur plus tard si on change de type
message = 123;  // Erreur : Type 'number' n'est pas assignable à type 'string'
✅ L'inférence réduit le code redondant tout en gardant la sécurité des types.

⚠️ Limites de l'inférence

Parfois TypeScript ne peut pas deviner le type, surtout avec des valeurs qui viennent de l'extérieur (API, JSON, DOM).

// Exemple : récupération d'un élément HTML
const element = document.getElementById("monBouton");
// TypeScript infère: HTMLElement | null

// ❌ Erreur : Object is possibly 'null'
element.addEventListener("click", () => {});

// Avec une assertion non-null (nous verrons plus tard)
element!.addEventListener("click", () => {});  // ✅ OK si on est sûr qu'il existe
💡 L'inférence est puissante mais elle ne peut pas savoir si un élément existe vraiment dans le DOM au moment de l'exécution.

🔧 Assertion de type (as)

L'assertion de type dit à TypeScript : "Fais-moi confiance, je connais mieux le type que toi".

// Syntaxe 1 (recommandée) : as Type
const maValeur: unknown = "ceci est une chaîne";
const longueur = (maValeur as string).length;  // ✅ Je sais que c'est une string

// Syntaxe 2 (ancienne, déconseillée avec JSX) : <Type>valeur
const longueur2 = (<string>maValeur).length;

// Exemple avec une API
interface Utilisateur {
  id: number;
  nom: string;
}

const data = JSON.parse(`{"id": 1, "nom": "Alice"}`) as Utilisateur;
console.log(data.nom);  // TypeScript sait maintenant que data a la forme Utilisateur

⚖️ `as Type` vs ``

SyntaxeExempleCompatibilité JSXRecommandation
as Typevaleur as string✅ Fonctionne parfaitement✅ Recommandée (moderne, lisible)
<Type>valeur<string>valeur❌ Ambigu avec les tags JSX⚠️ À éviter dans React/TSX

📌 Règle d'or :

Dans un fichier .tsx (React), utilisez uniquement as pour éviter toute confusion avec les éléments JSX. Dans les fichiers .ts purs, les deux fonctionnent.

❗ L'assertion non-null (!)

L'opérateur ! (postfixe) indique à TypeScript qu'une valeur ne sera jamais null ou undefined.

// Problème : une variable qui peut être null
let email: string | null = getEmailFromAPI();

// ❌ TypeScript refuse : Object is possibly 'null'
console.log(email.toUpperCase());

// ✅ Solution 1 : assertion non-null
console.log(email!.toUpperCase());

// ✅ Solution 2 : vérification explicite (plus sûre)
if (email !== null) {
    console.log(email.toUpperCase());
}

// Exemple avec un élément DOM
const monCanvas = document.getElementById("monCanvas") as HTMLCanvasElement;
// Ou mieux :
const ctx = monCanvas!.getContext("2d");

⚠️ Danger de l'assertion non-null !

L'opérateur ! supprime la vérification de null/undefined mais ne rend pas magiquement la valeur non-nulle à l'exécution.

// ❌ MAUVAIS USAGE
let valeur: string | null = null;
console.log(valeur!.length);  // 💥 Erreur à l'exécution : Cannot read property 'length' of null

// ✅ BON USAGE : quand on a une garantie certaine
function getElement(id: string): HTMLElement | null {
    return document.getElementById(id);
}

// On sait que l'élément existe (créé plus tôt dans le code)
const element = getElement("titrePrincipal")!;
element.textContent = "Nouveau titre";  // safe car on est sûr qu'il existe
🔴 Attention : Une assertion non-null ne crée pas la valeur. Si elle est réellement null/undefined, votre programme plantera. Utilisez-la uniquement quand vous avez une certitude absolue.

📝 Cas pratique : useRef avec non-null

En React, useRef initialisé avec null a le type MutableRefObject<HTMLInputElement | null>.

import { useRef } from "react";

function MonComposant() {
    // ref initialisée à null
    const inputRef = useRef<HTMLInputElement>(null);

    const focusInput = () => {
        // TypeScript pense que inputRef.current peut être null
        // On utilise ! car on sait que le composant est monté
        inputRef.current!.focus();
    };

    return (
        <div>
            <input ref={inputRef} type="text" />
            <button onClick={focusInput}>Focus</button>
        </div>
    );
}

💡 L'assertion non-null est courante avec useRef car l'élément DOM existe bien au moment du clic.

🚫 Double assertion (pratique à éviter)

Parfois, les développeurs enchaînent deux assertions pour forcer n'importe quel type :

// ❌ À éviter absolument
const valeur: string = "hello";
const nombre = valeur as unknown as number;  // Contourne complètement le système de types

// ✅ Plutôt, réfléchir à une meilleure conception
// Si vraiment nécessaire, préférer une vérification à l'exécution

📌 Bonne pratique :

N'utilisez la double assertion que dans des cas très spécifiques (interopérabilité avec du code non typé) et toujours accompagnée d'une vérification à l'exécution.

📝 À retenir

  • Inférence : TypeScript devine automatiquement les types. Préférez-la quand c'est possible.
  • Assertion de type as : utilisez-la quand vous avez plus d'informations que TypeScript (API, JSON, DOM).
  • Assertion non-null ! : pour dire "cette valeur n'est pas null/undefined". Utilisez-la avec précaution.
  • Ne forcez jamais un type sans raison valable. Préférez les vérifications explicites (if (valeur !== null)).
  • En React/TSX, utilisez toujours as plutôt que <Type>.

🎓 Le mantra TypeScript :

"Fais le plus de vérifications à la compilation, mais laisse-moi te dire quand tu as tort." – L'assertion est un outil puissant, mais dangereux.
