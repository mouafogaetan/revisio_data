🗄️ Éviter l’injection SQL avec TypeORM

Problème : concaténer des variables directement dans une requête SQL permet à un attaquant d’exécuter du code malveillant.

// ❌ DANGEREUX - injection possible
const product = await getConnection().query(
  `SELECT * FROM product WHERE name = '${userInput}'`
);

// ✅ SÉCURISÉ - paramètres préparés (TypeORM QueryBuilder)
const product = await this.productRepo
  .createQueryBuilder('p')
  .where('p.name = :name', { name: userInput })
  .getOne();

// ✅ findOne avec objet where (TypeORM aussi sécurisé)
const product = await this.productRepo.findOne({
  where: { name: userInput }
});

💡 TypeORM utilise des paramètres préparés automatiquement avec les méthodes find, QueryBuilder et createQueryBuilder. Évitez query() avec concaténation.

🧼 Nettoyer les entrées (XSS)

Un utilisateur pourrait envoyer du code HTML/JS dans un champ (ex: avis produit). À la réaffichage, ce code s’exécute dans le navigateur des autres clients.

// 1. Installer class-sanitizer
// npm install class-sanitizer class-transformer

// 2. Utiliser @Trim() et @Sanitize() dans les DTO
import { Trim, Sanitize } from 'class-sanitizer';

export class CreateReviewDto {
  @IsString()
  @Trim()
  @Sanitize() // supprime les balises HTML dangereuses
  comment: string;
}

// 3. Alternative : validation manuelle avec sanitize-html
import sanitizeHtml from 'sanitize-html';
const cleanComment = sanitizeHtml(userComment, {
  allowedTags: ['b', 'i', 'em'],
  allowedAttributes: {}
});

📌 Toujours échapper ou nettoyer les données avant de les retourner au frontend ou de les stocker.

🪖 Helmet.js : protection des en-têtes

Helmet définit 11 en-têtes HTTP sécurisés (XSS, clickjacking, sniffing MIME, etc.).

// Installation : npm install helmet

// main.ts
import * as helmet from 'helmet';

async function bootstrap() {
  const app = await NestFactory.create(AppModule);
  app.use(helmet());
  await app.listen(3000);
}

✨ Principaux en-têtes ajoutés :

  • X-DNS-Prefetch-Control : désactive le prefetch DNS
  • X-Frame-Options : empêche le clickjacking (framing)
  • X-Content-Type-Options : évite le MIME type sniffing
  • X-XSS-Protection : active le filtre XSS des navigateurs
  • Strict-Transport-Security : force HTTPS (HSTS)

🔐 Autres bonnes pratiques

  • Validation stricte : utiliser ValidationPipe avec whitelist: true pour rejeter les propriétés inattendues.
  • Rate limiting : protéger contre les attaques par force brute (module @nestjs/throttler).
  • Ne jamais exposer les erreurs détaillées en production (éviter stack trace).
  • Utiliser HTTPS en production (avec app.enable('trust proxy')).
  • Éviter les requêtes SQL brutes : privilégier QueryBuilder ou Repository.
  • Échapper les sorties : côté frontend, utiliser des bibliothèques comme DOMPurify.

⚠️ À retenir :

La sécurité est une couche transversale : entrées (validation/sanitization) + requêtes (paramètres préparés) + sorties (échappement) + infrastructure (Helmet, HTTPS).

📝 Exemple : sécuriser un avis client

// review.controller.ts
@Post()
@UseGuards(JwtAuthGuard)
async create(@Body() dto: CreateReviewDto, @Req() req) {
  // ✅ le DTO applique déjà Trim, Sanitize
  // ✅ TypeORM utilise des paramètres préparés
  const review = this.reviewRepo.create({
    ...dto,
    user: req.user,
    product: { id: dto.productId }
  });
  return this.reviewRepo.save(review);
}

✅ Résultat :

  • Injection SQL impossible grâce aux paramètres préparés.
  • Balises HTML malveillantes supprimées par @Sanitize().
  • Données stockées propres et sans risque.

🧪 Exercice :

Un utilisateur envoie le commentaire suivant : "Super produit ! <script>alert('hack')</script>". L’API NestJS reçoit ce DTO. Que doit-il se passer avant stockage ?

🔍 Voir solution

Le DTO avec @Sanitize() supprime les balises <script>. On peut aussi configurer sanitize-html pour autoriser seulement certaines balises (b, i, etc.). Ensuite, TypeORM insère le commentaire nettoyé : "Super produit ! ". Pas de code exécuté.
