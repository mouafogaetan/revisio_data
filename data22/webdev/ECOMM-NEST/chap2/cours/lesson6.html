📦 Installation des dépendances

npm install @nestjs/typeorm typeorm pg

Configuration du module TypeORM (dans app.module.ts) :

TypeOrmModule.forRoot({
  type: 'postgres',
  host: process.env.DB_HOST,
  port: 5432,
  username: process.env.DB_USER,
  password: process.env.DB_PASSWORD,
  database: process.env.DB_NAME,
  entities: [__dirname + '/**/*.entity{.ts,.js}'],
  synchronize: false, // ← à éviter en production !
  migrations: ['dist/migrations/*.js'],
  cli: { migrationsDir: 'src/migrations' }
})

📌 synchronize: true recrée le schéma automatiquement (dangereux en prod). On préfère les migrations.

🏷️ Exemple d’entité : Product

// product.entity.ts
import { Entity, Column, PrimaryGeneratedColumn, ManyToOne } from 'typeorm';
import { Category } from './category.entity';

@Entity()
export class Product {
  @PrimaryGeneratedColumn() id: number;
  @Column() name: string;
  @Column('decimal') price: number;
  @Column({ default: 0 }) stock: number;
  @ManyToOne(() => Category, cat => cat.products)
  category: Category;
}

💡 Les relations @OneToMany, @ManyToMany s’utilisent de la même manière. TypeORM génère les clés étrangères.

📜 Migrations : gérer l’évolution du schéma

Créer une migration après modification des entités :

npx typeorm migration:create ./src/migrations/AddPriceColumn
# ou avec NestJS :
npm run typeorm -- migration:generate ./src/migrations/UpdateProduct

Exemple de migration générée (up/down) :

export class AddPriceColumn1678901234567 implements MigrationInterface {
  public async up(queryRunner: QueryRunner): Promise<void> {
    await queryRunner.addColumn('product', new Column({ name: 'price', type: 'decimal' }));
  }
  public async down(queryRunner: QueryRunner): Promise<void> {
    await queryRunner.dropColumn('product', 'price');
  }
}

Exécuter les migrations : npx typeorm migration:run ou via script npm.

🌱 Seeds : données initiales

Créer un script seed (ex: src/seeds/product.seed.ts) :

import { DataSource } from 'typeorm';
import { Product } from '../products/product.entity';

export async function seedProducts(dataSource: DataSource) {
  const repo = dataSource.getRepository(Product);
  await repo.save([
    { name: 'Sac à main', price: 15000, stock: 10 },
    { name: 'T-shirt traditionnel', price: 5000, stock: 25 }
  ]);
}

Appeler les seeds via un script CLI ou dans un fichier séparé :

npx ts-node src/seeds/run.ts

📌 Alternative : utiliser typeorm-seeding pour des seeds plus sophistiquées.

✨ Bonnes pratiques

  • ✔️ Toujours utiliser migrations en production, jamais synchronize: true.
  • ✔️ Versionner les dossiers migrations et seeds dans Git.
  • ✔️ Tester les migrations sur une copie de la base avant de les exécuter en production.
  • ✔️ Utiliser .env pour les identifiants de base de données.
  • ✔️ Pour les seeds, prévoir une commande qui supprime/réinsère (évite les doublons).

⚠️ Attention :

Les migrations doivent être idempotentes (exécutables plusieurs fois sans erreur). Utilisez IF NOT EXISTS ou vérifiez les colonnes avant ajout.

🧪 Exercice (mise en situation) :

Vous ajoutez une colonne description (text) à l’entité Product. Décrivez la procédure complète :

  1. Modifier l’entité Product.
  2. Générer une migration.
  3. Exécuter la migration.
  4. Écrire un seed qui insère 2 produits avec description.
🔍 Proposition de solution

1. Ajouter @Column({ type: 'text', nullable: true }) description: string;
2. npx typeorm migration:generate ./src/migrations/AddDescriptionToProduct -d src/data-source.ts
3. npm run typeorm migration:run
4. Dans le script seed : repository.save([{ name: 'Artisanat', description: 'Fait main', price: 7000 }])
