📄 Logs structurés avec Winston

Les logs structurés (format JSON) permettent une exploitation automatique par des outils comme ELK, Datadog ou AWS CloudWatch.

// Installation
npm install winston @nestjs/winston

// app.module.ts
import { WinstonModule } from 'nest-winston';
import * as winston from 'winston';

const winstonConfig = WinstonModule.createLogger({
  transports: [
    new winston.transports.Console({
      format: winston.format.combine(
        winston.format.timestamp(),
        winston.format.json()
      )
    }),
    new winston.transports.File({
      filename: 'logs/error.log',
      level: 'error',
      format: winston.format.json()
    })
  ]
});

@Module({
  imports: [],
  providers: [],
})
export class AppModule {}
✨ Résultat : chaque ligne de log est un objet JSON avec timestamp, niveau, message, contexte.

🔍 Exemple de log structuré

{
  "timestamp": "2025-06-15T10:32:15.123Z",
  "level": "error",
  "message": "Échec de paiement Stripe",
  "context": "PaymentService",
  "userId": "usr_123",
  "orderId": "ord_456",
  "error": "card_declined",
  "stack": "..."
}
✅ Avantage : Filtrage par champ, agrégation automatique, corrélation avec d'autres services.
📌 Bonne pratique : Ajouter toujours un correlationId pour tracer une requête de bout en bout.

🚨 Sentry : capture d'erreurs

Sentry centralise les erreurs en production, remonte les stack traces et le contexte utilisateur.

// Installation
npm install @sentry/node @sentry/profiling-node

// main.ts
import * as Sentry from '@sentry/node';

Sentry.init({
  dsn: 'https://votre-dsn@sentry.io/project-id',
  environment: process.env.NODE_ENV,
  tracesSampleRate: 0.5,
  integrations: [new Sentry.Integrations.Http({ tracing: true })],
});

// Dans un service
try {
  await stripe.paymentIntent.create(...);
} catch (err) {
  Sentry.captureException(err, {
    tags: { service: 'payment' },
    extra: { userId, orderId }
  });
  throw new InternalServerErrorException('Paiement échoué');
}

👤 Associer un utilisateur à une erreur Sentry

Facilite le support : savoir quel client a rencontré l'erreur.

// Dans un guard ou middleware
Sentry.setUser({
  id: user.id,
  email: user.email,
  ip_address: req.ip
});

// Lors d'une erreur critique
Sentry.captureMessage('Panier abandonné après 3 échecs paiement', {
  level: 'warning',
  user: { id: userId }
});
💡 Sentry regroupe automatiquement les erreurs similaires (fingerprinting) et vous alerte par email/webhook.

📊 PM2 : Process Manager pour Node.js

PM2 permet de garder l'application vivante, de redémarrer automatiquement et d'exposer des métriques.

# Démarrer l'application
pm2 start dist/main.js --name ecommerce-api

# Surveiller les logs en temps réel
pm2 logs ecommerce-api

# Afficher les métriques (CPU, mémoire, uptime)
pm2 monit

# Redémarrer après crash automatiquement
pm2 startup

# Générer un rapport de performance
pm2 report
🔁 PM2 peut aussi faire du load balancing (mode cluster) avec pm2 start main.js -i max.

📈 New Relic : Application Performance Monitoring

New Relic offre des fonctionnalités avancées : analyse des temps de réponse, bases de données, vues lentes, alertes.

// Installation
npm install newrelic

// newrelic.js (à la racine)
exports.config = {
  app_name: ['Ecommerce API'],
  license_key: process.env.NEW_RELIC_LICENSE_KEY,
  logging: { level: 'info' },
  allow_all_headers: true,
  attributes: { include: ['request.headers.*', 'user.id'] }
};

// Dans main.ts, TOUT EN PREMIER
import 'newrelic';
import { NestFactory } from '@nestjs/core';
📌 Dashboard : transactions lentes, erreurs par endpoint, top des routes, base de données lentes.

Comparaison rapide

OutilFonction principaleIdéal pour
PM2Process management, logs basiques, redémarrage autoPetites structures, VPS simple
SentryErreurs, stack traces, alertesDétection rapide des bugs en production
New RelicAPM, base de données, analyse de performancesApplications critiques, scaling horizontal
Winston/PinoLogs structurés (JSON)Centralisation et analyse (ELK, Datadog)

✅ Bonnes pratiques

  • Ne jamais logger de données sensibles (passwords, tokens, cartes bancaires)
  • Utiliser des niveaux de log : error, warn, info, debug
  • Mettre en place une alerte Sentry pour les erreurs 5xx en production
  • Surveiller la mémoire et le CPU avec PM2 ou New Relic
  • Ajouter un health check (ex: /health) pour le load balancer
  • Corréler les logs et les traces avec un ID unique (correlationId)
🔍 “Ce qui ne se mesure pas ne s’améliore pas.” – Appliquez le monitoring dès le premier jour en prod.
