🏗️ Docker multi-stage : principe

Un Dockerfile multi-stage utilise plusieurs phases (FROM ... AS ...) pour ne garder que le strict nécessaire dans l'image finale.

📦 Phase de build

Installation des dépendances, compilation TypeScript, génération du dossier dist/.

🚀 Phase production

Copie uniquement le build + node_modules --production. Image légère et sécurisée.

📄 Dockerfile multi-stage (NestJS)

Voici un Dockerfile professionnel pour une API NestJS avec TypeORM, utilisant l'image officielle Node et l'alpine pour la taille réduite.

# Étape 1: build
FROM node:18 AS builder
WORKDIR /app
COPY package*.json ./
RUN npm ci --only=production && npm cache clean --force
COPY . .
RUN npm run build

# Étape 2: production (alpine)
FROM node:18-alpine AS production
WORKDIR /app
RUN addgroup -g 1001 -S nodejs && adduser -S nestjs -u 1001
COPY --from=builder --chown=nestjs:nodejs /app/dist ./dist
COPY --from=builder --chown=nestjs:nodejs /app/node_modules ./node_modules
COPY package*.json ./
USER nestjs
EXPOSE 3000
CMD ["node", "dist/main.js"]

🔐 Sécurité : on crée un utilisateur non-root (nestjs) pour exécuter l'application, limitant les risques.

⚙️ docker-compose.yml pour l'e-commerce

Orchestrer l'API, PostgreSQL et Redis en un seul fichier. Simplifie les déploiements locaux et en production.

version: '3.8'
services:
  postgres:
    image: postgres:15-alpine
    environment:
      POSTGRES_USER: ecommerce
      POSTGRES_PASSWORD: secret
      POSTGRES_DB: ecommerce_db
    volumes:
      - pgdata:/var/lib/postgresql/data
    ports:
      - "5432:5432"

  redis:
    image: redis:7-alpine
    ports:
      - "6379:6379"

  api:
    build: .
    ports:
      - "3000:3000"
    depends_on:
      - postgres
      - redis
    environment:
      DATABASE_URL: postgresql://ecommerce:secret@postgres:5432/ecommerce_db
      REDIS_HOST: redis
    env_file:
      - .env.production

volumes:
  pgdata:

🔐 Variables d'environnement sécurisées

Ne jamais hardcoder les secrets dans le Dockerfile. Utilisez env_file ou environment dans docker-compose.

.env.production

JWT_SECRET=supersecurekey
DB_PASSWORD=strongpass
STRIPE_KEY=sk_live_xxx

✅ Bonnes pratiques :

  • Ne pas versionner .env*
  • Utiliser Docker secrets en mode swarm
  • Injecter via CI/CD (GitHub Actions)
📌 Rappel : process.env.DB_PASSWORD dans NestJS lira la variable depuis le conteneur.

🚀 Mise en pratique

Commandes essentielles pour builder et lancer votre API containerisée :

# Construire l'image multi-stage
docker build -t ecommerce-api:latest .

# Lancer tous les services (API + DB + Redis)
docker-compose up -d

# Voir les logs
docker-compose logs -f api

# Arrêter et supprimer les conteneurs
docker-compose down -v

🏁 Défi final :

Améliorez le Dockerfile en ajoutant une étape de test (npm run test) avant le build, et optimisez le cache des dépendances pour accélérer les reconstructions.
