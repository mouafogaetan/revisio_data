🔄 Mise à jour du statut d’une commande

Transition autorisée : pending → paid → shipped → delivered (ou cancelled à tout moment si paiement non capturé).

// order.service.ts
async updateOrderStatus(orderId: number, newStatus: OrderStatus, userId: number, userRoles: string[]) {
  const order = await this.orderRepo.findOne({ where: { id: orderId }, relations: ['items', 'items.product'] });
  if (!order) throw new NotFoundException('Commande introuvable');

  // Vérifier permission : admin ou seller propriétaire du produit
  if (!userRoles.includes('admin')) {
    const isSeller = order.items.some(item => item.product.sellerId === userId);
    if (!isSeller) throw new ForbiddenException('Vous ne pouvez modifier que vos propres commandes');
  }

  // Logique de transition
  const allowedTransitions = { pending: ['paid', 'cancelled'], paid: ['shipped', 'cancelled'], shipped: ['delivered', 'cancelled'] };
  if (!allowedTransitions[order.status]?.includes(newStatus)) {
    throw new BadRequestException(`Transition de ${order.status} à ${newStatus} impossible`);
  }

  order.status = newStatus;
  if (newStatus === 'cancelled') await this.refundStock(order); // libérer stock
  await this.orderRepo.save(order);
  await this.logActivity(order, newStatus); // historique
  return order;
}
📌 On conserve un historique des statuts (table OrderHistory) pour la traçabilité.

🧾 Génération de factures PDF

Librairies possibles : pdfmake, PDFKit ou puppeteer. Ici on utilise PDFKit pour un rendu serveur.

// invoice.service.ts
import * as PDFDocument from 'pdfkit';
import { Response } from 'express';

async generateInvoice(order: Order, res: Response) {
  const doc = new PDFDocument({ margin: 50 });
  res.setHeader('Content-Type', 'application/pdf');
  res.setHeader('Content-Disposition', `attachment; filename=facture-${order.id}.pdf`);
  doc.pipe(res);

  doc.fontSize(20).text('FACTURE', { align: 'center' });
  doc.moveDown();
  doc.fontSize(12).text(`N° commande : ${order.id}`);
  doc.text(`Date : ${order.createdAt.toLocaleDateString()}`);
  doc.text(`Client : ${order.user.firstName} ${order.user.lastName}`);
  doc.moveDown();
  doc.text('Détails des articles :', { underline: true });
  order.items.forEach(item => {
    doc.text(`${item.productName} x ${item.quantity} = ${item.price * item.quantity} FCFA`);
  });
  doc.moveDown();
  doc.fontSize(14).text(`Total TTC : ${order.total} FCFA`, { align: 'right' });
  doc.end();
}

Endpoint : GET /orders/:id/invoice (admin/seller/acheteur).

👥 Différence admin / seller dans le dashboard

  • Admin : voit toutes les commandes, peut modifier n'importe quel statut, accède aux rapports globaux.
  • Seller (vendeur) : ne voit que les commandes contenant ses propres produits. Ne peut pas voir les détails des commandes des autres vendeurs.
  • Customer : voit seulement ses propres commandes (déjà implémenté dans lesson 21).

🛡️ Bonnes pratiques – Dashboard & Factures

  • ✅ Toujours vérifier les droits (admin / seller / propriétaire) avant d'afficher ou modifier une commande.
  • ✅ Ajouter un rate limiting sur les endpoints de mise à jour de statut.
  • ✅ Générer les factures en async (queue Bull) pour ne pas bloquer la réponse API.
  • ✅ Stocker les factures générées sur un CDN/S3 pour éviter de les regénérer à chaque demande.
  • ✅ Inclure un numéro unique de facture et un QR code pour vérification.
💡 Les factures doivent respecter la législation fiscale locale (Cameroun : mentions obligatoires, timbre fiscal électronique si seuil dépassé).

🏁 Ce qu'il faut retenir

  • ✔️ Dashboard admin/seller : agrégats, filtres, pagination
  • ✔️ Mise à jour du statut : validation des transitions + permissions
  • ✔️ Génération de facture PDF (PDFKit / Puppeteer) via endpoint sécurisé
  • ✔️ Différence de périmètre entre admin (global) et seller (ses produits)
  • ✔️ Traçabilité des actions (OrderHistory) pour audit
