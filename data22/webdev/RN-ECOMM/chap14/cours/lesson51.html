⚙️ Configuration d'EAS Build

EAS Build est le service d'Expo pour compiler des applications natives dans le cloud.

# Installation
npm install -g eas-cli

# Connexion à Expo
eas login

# Configuration du projet
eas build:configure

# Fichier eas.json généré
{
  "cli": { "version": ">= 3.0.0" },
  "build": {
    "development": { "developmentClient": true, "distribution": "internal" },
    "preview": { "distribution": "internal" },
    "production": { "distribution": "store" }
  },
  "submit": {
    "production": { "ios": { "appleId": "votre@email.com" } }
  }
}
# Lancer un build manuel
eas build --platform ios --profile production
eas build --platform android --profile production

# Soumettre manuellement aux stores
eas submit --platform ios --profile production
eas submit --platform android --profile production

🤖 Workflow GitHub Actions

Automatisation complète : tests → build → soumission aux stores.

# .github/workflows/deploy.yml
name: Deploy to Stores

on:
  push:
    branches: [main]
  pull_request:
    branches: [main]

jobs:
  test:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with: { node-version: 18 }
      - run: npm ci
      - run: npm test

  eas-build:
    needs: test
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: expo/expo-github-action@v8
        with:
          expo-version: latest
          eas-version: latest
          token: ${{ secrets.EXPO_TOKEN }}
      - run: npm ci
      - name: Build iOS
        run: eas build --platform ios --profile production --non-interactive
      - name: Build Android
        run: eas build --platform android --profile production --non-interactive

📱 Soumission automatique aux stores

Après le build, soumettre automatiquement aux stores avec eas submit.

jobs:
  submit:
    needs: eas-build
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: expo/expo-github-action@v8
        with:
          expo-version: latest
          eas-version: latest
          token: ${{ secrets.EXPO_TOKEN }}
      
      - name: Submit to iOS App Store
        run: eas submit --platform ios --profile production --non-interactive
        env:
          EXPO_APPLE_ID: ${{ secrets.APPLE_ID }}
          EXPO_APPLE_APP_SPECIFIC_PASSWORD: ${{ secrets.APP_SPECIFIC_PASSWORD }}
          EXPO_APPLE_TEAM_ID: ${{ secrets.APPLE_TEAM_ID }}
      
      - name: Submit to Google Play
        run: eas submit --platform android --profile production --non-interactive
        env:
          EXPO_ANDROID_KEYSTORE_PASSWORD: ${{ secrets.ANDROID_KEYSTORE_PASSWORD }}
          EXPO_ANDROID_KEY_PASSWORD: ${{ secrets.ANDROID_KEY_PASSWORD }}

⚠️ Important : Les secrets doivent être configurés dans GitHub (Settings → Secrets and variables → Actions).

📲 Mises à jour OTA avec EAS Update

Modifiez le code JavaScript/TS de votre application sans passer par l'approbation des stores.

# Installation d'EAS Update
eas update:configure

# Publication d'une mise à jour
eas update --branch production --message "Correction bug panier"

# Configuration dans app.json
{
  "expo": {
    "updates": {
      "url": "https://u.expo.dev/votre-projet-id",
      "fallbackToCacheTimeout": 0
    },
    "runtimeVersion": {
      "policy": "sdkVersion"
    }
  }
}

# Dans le code : forcer la mise à jour
import * as Updates from 'expo-updates';

const checkForUpdates = async () => {
  try {
    const update = await Updates.checkForUpdateAsync();
    if (update.isAvailable) {
      await Updates.fetchUpdateAsync();
      await Updates.reloadAsync(); // Redémarre l'app
    }
  } catch (error) {
    console.log('Erreur OTA:', error);
  }
};

✅ Avantage : Les mises à jour OTA sont déployées en quelques minutes, sans validation Apple/Google.

📝 Exercices d'application

Exercice 1 (compétence – medium)

Créez un workflow GitHub Actions complet qui :

  • Se déclenche sur push sur la branche main
  • Installe les dépendances
  • Lance les tests unitaires
  • Build l'application Android et iOS via EAS Build
  • Soumet la version Android au Google Play (track alpha)
🔍 Voir corrigé 
name: CI/CD EAS
on: push
jobs:
  test-and-build:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: expo/expo-github-action@v8
        with: { token: ${{ secrets.EXPO_TOKEN }} }
      - run: npm ci && npm test
      - run: eas build --platform android --profile preview --non-interactive
      - run: eas submit --platform android --profile alpha --non-interactive

Exercice 2 (ressource – hard)

Vous devez déployer une mise à jour OTA pour corriger un bug sur l'écran de paiement. Expliquez la procédure complète et les précautions à prendre.

🔍 Voir corrigé

1. Fixer le bug dans le code
2. Tester localement avec `expo start`
3. Lancer `eas update --branch production --message "Fix checkout bug"`
4. Vérifier sur un appareil de test via Expo Go ou build de développement
5. Les utilisateurs recevront la mise à jour au prochain lancement de l'application
6. Précaution : versionner le runtime (app.json) pour éviter les incompatibilités natives
7. Backup : toujours tester d'abord sur un canal 'staging' avant 'production'
8. Utiliser `Updates.reloadAsync()` uniquement pour les mises à jour critiques

Exercice 3 (compétence – easy)

Écrivez la commande EAS pour créer un build de développement (development client) sur Android.

🔍 Voir corrigé 
eas build --platform android --profile development

Le fichier eas.json doit contenir une section `"development": { "developmentClient": true }`

✅ Bonnes pratiques & récapitulatif

  • Utilisez des branches séparées (dev, staging, prod) avec des profils EAS différents
  • Protégez tous les secrets GitHub (EXPO_TOKEN, APPLE_ID, clés de signature)
  • Testez toujours les mises à jour OTA en staging avant la production
  • Versionnez vos mises à jour OTA avec des tags Git
  • Pour les applications critiques, combinez OTA + store updates
  • Surveillez les échecs de build via les notifications GitHub
  • Utilisez `expo-doctor` pour vérifier la compatibilité avant build
