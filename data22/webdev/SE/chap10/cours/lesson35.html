🧩 Implémentation : SOLID & Patterns

Le code doit être flexible, maintenable et extensible. Appliquez ces piliers :

SRP – Une classe = une responsabilité.
OCP – Ouvert à l'extension, fermé à la modification.
LSP – Sous-types substituables.
ISP – Interfaces spécifiques plutôt que générale.
DIP – Dépendre d'abstractions, pas de concrétions.

🛠️ Patrons courants : Strategy, Factory, Observer, Repository… améliorent la structure.
Exemple : injection de dépendances pour découpler la couche métier de la persistance.

🧪 Tests : Unitaire, Intégration, E2E

Pour garantir la fiabilité, la pyramide des tests est indispensable :

📐 Tests unitaires

Vérifient chaque fonction/classe isolément. Outils: JUnit, pytest, Jest.

assertEqual(calcul(2,3), 5)
🔗 Tests d'intégration

Vérifient les interactions entre modules (base de données, API).

🌐 Tests E2E

Simulent le parcours utilisateur complet (Cypress, Selenium).

✅ TDD (Test Driven Development) :

1. Écrire un test rouge → 2. Implémentation minimale → 3. Refactorisation (Green).

🌿 Git & Stratégies de branchement

Une bonne stratégie Git assure la collaboration et la qualité. Deux modèles populaires :

Git Flow

  • Branches : main, develop, feature/, release/, hotfix/
  • Idéal pour releases planifiées

GitHub Flow

  • Branche principale 'main' + branches features
  • Déploiement continu, revues via pull requests
$ git checkout -b feature/user-auth
# développement + commits
$ git push origin feature/user-auth
# création de Pull Request → revue → merge → suppression branche

🏷️ Version sémantique (major.minor.patch) et tags pour chaque livraison.

⚙️ CI/CD : Pipeline automatisé

CI (Intégration Continue) → chaque push déclenche tests & vérifications.
CD (Livraison / Déploiement Continu) → déploiement automatique après validation.

🚀 Atelier pratique : Mise en œuvre

Imaginons une petite API de gestion de tâches :

  1. Dépôt Git avec branche main et develop.
  2. Implémentation de la couche domain (SOLID). Ex: pattern Repository, injection dépendance.
  3. Tests unitaires avec Jest/Mocha (coverage > 80%).
  4. Tests d'intégration avec base de données en mémoire.
  5. Configuration GitHub Actions : à chaque pull request → lancement des tests, lint, build.
  6. Déploiement automatique sur une plateforme (Render, Heroku) après fusion sur main.
✅ Code review obligatoire 📈 Qualité avec SonarQube 🧪 TDD pour chaque fonctionnalité
