🔄 Qu'est-ce que la CI ?

L'intégration Continue (Continuous Integration) est une pratique de développement où les développeurs fusionnent leurs changements dans un dépôt partagé plusieurs fois par jour. Chaque fusion déclenche automatiquement une construction et des tests.

📌 Principes fondamentaux :

  • ✅ Maintenir un dépôt de code unique et versionné (Git).
  • ✅ Automatiser la construction (build).
  • ✅ Automatiser les tests (unitaires, intégration).
  • ✅ Tout développeur doit intégrer au moins une fois par jour.
  • ✅ Chaque commit doit être vérifié par le serveur CI.
  • ✅ Résoudre rapidement les échecs (“fail fast”).

🔁 Flux simplifié : push → déclenchement CI → exécution des tests

🌟 Pourquoi utiliser la CI ?

🚀 Détection précoce des bugs

Les erreurs sont trouvées juste après le commit, pas en production.

🤝 Réduction des conflits

Intégrations fréquentes = moins de conflits de fusion.

⏱️ Feedback immédiat

Le développeur sait rapidement si son code casse quelque chose.

📈 Métriques qualité

On peut vérifier couverture de tests, style de code, sécurité.

"Intégrer tôt, intégrer souvent" — L'un des mantras agiles

🛠️ Outils d'intégration continue

Voici les plus utilisés par les équipes de développement Web et logiciel :

GitHub ActionsIntégré à GitHub, YAML comme code, très simple pour les dépôts GitHub. Écosystème riche.
GitLab CIPipeline as code, excellent pour DevOps, tout-en-un (CI/CD, registry, monitoring).
JenkinsTrès flexible, extensible via plugins, historique mature, auto-hébergé.
CircleCI / TravisAlternatives cloud populaires, configuration simple.

💡 Dans ce cours, nous nous concentrons sur GitHub Actions et GitLab CI, très accessibles.

⚙️ Exemple de pipeline CI avec GitHub Actions

Fichier .github/workflows/ci.yml

name: CI Pipeline
on: [push, pull_request]
jobs:
  test:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - name: Setup Node.js
        uses: actions/setup-node@v4
        with:
          node-version: '18'
      - run: npm install
      - run: npm test
      - name: Lint code
        run: npm run lint
  • 📌 Déclencheurs : push sur n'importe quelle branche et pull request.
  • 📌 Étapes : checkout, installation dépendances, exécution tests, linting.
  • 📌 Résultat visible directement dans l'interface GitHub.

🧪 Automatisation des tests & qualité

Un bon pipeline CI intègre plusieurs vérifications automatiques :

🔬 Tests unitaires : validation de chaque fonction/classe (Jest, JUnit, pytest).
🔗 Tests d'intégration : interaction entre modules, base de données, API.
📏 Analyse statique : linters (ESLint, Pylint), formatage (Prettier, Black).
🧹 Couverture de code : seuil minimal défini (ex: 80%).
🔒 Scan de sécurité : détection de vulnérabilités (Snyk, dependabot).

✨ Astuce : La CI peut générer des badges de statut que vous affichez dans votre README (ex: "passing", "failing").

✅ Bonnes pratiques CI

  • ✔️ Garder le pipeline rapide : moins de 10 minutes pour les tests unitaires.
  • ✔️ Ne pas intégrer sur la branche principale si les tests échouent -> protéger `main`.
  • ✔️ Exécuter la CI sur les branches de fonctionnalité avant la fusion.
  • ✔️ Utiliser des caches (dépendances) pour accélérer les builds.
  • ✔️ Notifier l'équipe (Slack, email) en cas d'échec critique.
  • ✔️ Faire évoluer le pipeline : ajouter progressivement des checks (qualité, perf).

📢 Rappel : La CI n'est pas une fin en soi, c'est un outil pour améliorer la confiance dans le code et faciliter le travail d'équipe.
