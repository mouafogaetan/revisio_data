📐 La pyramide des tests

Idée fondamentale proposée par Mike Cohn : privilégier beaucoup de tests rapides et isolés (unitaires), moins de tests d'intégration, et encore moins de tests de bout en bout (lents et fragiles).

⚛️ Unitaires

Vérifient une fonction/classe isolée. Rapides, nombreux.

🔗 Intégration

Testent les interactions entre modules / base de données.

🌐 E2E / UI

Parcours complets utilisateur, lents mais critiques.

🖱️ vs 🤖 Tests manuels & automatisés

👤 Manuel
  • Exécutés par un humain, sans scripts
  • Idéal pour l’exploratoire, l’UX, les cas rares
  • Répétitif → erreurs humaines possibles
⚙️ Automatisé
  • Scripts (JUnit, pytest, Jest) exécutables à tout moment
  • Idéal pour régression, intégration continue
  • Coût initial mais rentable sur le long terme
💎 Dans un projet professionnel, on combine les deux : automates pour les régressions, manuel pour les tests d’acceptation et d’exploration.

📊 Les 4 niveaux essentiels de test

Unitaire
Vérifie chaque composant isolément
Ex: tester une fonction de calcul, une classe Java.
Intégration
Teste les interactions entre modules
Ex: vérifier qu’une API dialogue bien avec la base de données.
Système
Test complet de l’application intégrée
Valide les exigences fonctionnelles et non fonctionnelles.
Acceptation
Validation par le client / utilisateur final
Confirme que le livrable correspond au besoin métier.
🔁 Chaque niveau augmente en portée et en coût, mais diminue en fréquence d’exécution.

🧠 Pourquoi une bonne stratégie de test ?

Une stratégie bien pensée permet de :

✔️ Détecter les bugs tôt (coût moindre).
✔️ Accélérer les mises en production (CI/CD).
✔️ Fournir des indicateurs de qualité.
✔️ Éviter les régressions lors des évolutions.

🎓 En résumé (à retenir)

  • La pyramide des tests favorise des tests unitaires nombreux et rapides.
  • Il faut équilibrer automatisation et tests manuels selon le contexte.
  • Les niveaux de test : unitaire → intégration → système → acceptation.
  • Une bonne stratégie réduit les bugs, la dette technique et rassure l’équipe.
🧪 « Les tests ne garantissent pas l'absence de bugs, mais ils sont notre meilleur rempart contre les régressions. »
— Agile Testing
