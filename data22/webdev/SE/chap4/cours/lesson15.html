💡 Pourquoi ? Le diagramme des cas d'utilisation est l'outil clé pour dialoguer avec le client et capturer les besoins.

🧩 Qu'est-ce qu'un diagramme de cas d'utilisation ?

Un diagramme de cas d'utilisation (Use Case Diagram) est un type de diagramme comportemental UML. Il représente les fonctionnalités offertes par un système du point de vue de l'utilisateur, sans détailler l'implémentation technique.

🔍 À retenir :

Il répond à la question "Qui fait quoi dans le système ?" C'est un outil parfait pour recueillir les exigences fonctionnelles.

👥 Acteurs et cas d'utilisation

👤 Acteur

Rôle joué par un utilisateur ou un système externe qui interagit avec le système. Ex: Client, Administrateur, Passerelle de paiement.

⚙️ Cas d'utilisation

Ensemble d'actions réalisées par le système qui produit un résultat observable pour un acteur. Ex: "Passer commande", "S'authentifier".

📌 Exemple concret : Dans un distributeur de billets, l'acteur "Client" a le cas d'utilisation "Retirer argent".

🔗 Relations importantes

« include » : Un cas d’usage appelle obligatoirement un autre (sous-fonction réutilisable). Ex: "Retirer argent" inclut "Vérifier solde".
« extend » : Un cas d’usage optionnel qui étend le comportement de base sous certaines conditions. Ex: "Payer" peut étendre "Ajouter réduction si code promo".
Généralisation / héritage : Un acteur ou cas d’usage enfant hérite du comportement du parent. Ex: "Utilisateur Premium" hérite de "Utilisateur Standard".

📄 Rédiger la description textuelle

Un diagramme ne suffit pas, il faut décrire chaque cas d’utilisation en texte structuré : acteurs, préconditions, scénario nominal, exceptions.

📌 Exemple : "Passer commande"

  • Acteur principal : Client
  • Précondition : Client connecté et panier non vide
  • Scénario nominal : 1. Client valide son panier → 2. Système affiche récapitulatif → 3. Client choisit livraison/paiement → 4. Système enregistre commande
  • Extensions : 3a. Paiement refusé → afficher erreur
  • Postcondition : Commande créée et notifiée

⭐ Cette fiche texte est souvent appelée "specification du cas d'utilisation" (UC spécification).

✅ Bonnes pratiques & erreurs à éviter

👍 À faire :

  • Nommer les cas avec un verbe à l'infinitif
  • Se concentrer sur le "quoi" pas le "comment"
  • Limiter la complexité (max 15-20 UC par diagramme)

⚠️ Éviter :

  • Détailler l'ordre des opérations
  • Oublier les relations entre acteurs
  • Mélanger cas techniques avec fonctionnels

📖 Astuce : Pour chaque cas d'utilisation, rédigez un mini-scénario comme une petite histoire que le système réalise avec l'acteur.

✏️ Petit exercice (en classe / maison)

🚀 Pour un site de réservation de cours en ligne (comme une plateforme de soutien scolaire) :

  1. Identifiez au moins 2 acteurs différents.
  2. Citez 3 cas d’utilisation.
  3. Donnez un exemple de relation « extend » possible.
  4. Rédigez la description textuelle d’un cas "Consulter planning".

📌 Proposez vos réponses en groupe ou réfléchissez individuellement. Le diagramme des cas d'usage est vivant !

Résumons !

Le diagramme de cas d'utilisation permet de modéliser les besoins fonctionnels et de dialoguer avec les parties prenantes. Il repose sur acteurs, cas d'utilisation, relations et une description textuelle rigoureuse.
