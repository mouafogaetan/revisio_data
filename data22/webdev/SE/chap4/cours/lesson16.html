Partie 1 : Principes SOLID (14 pts)

Question 1 – Single Responsibility & Open/Closed (7 pts)

On vous donne la classe suivante en pseudo-code :

class Rapport {
    void genererPDF() { /* génération PDF */ }
    void envoyerParEmail() { /* envoi email */ }
    void sauvegarderFichier() { /* sauvegarde sur disque */ }
}

Cette classe viole deux principes SOLID. Lesquels ? Expliquez et proposez un refactoring (diagramme de classes ou code).

Question 2 – Liskov Substitution & Dependency Inversion (7 pts)

Une classe Carre hérite de Rectangle. Le code suivant pose un problème :

Rectangle r = new Carre();
r.setLargeur(5);
r.setHauteur(10);
// Quelle est la surface attendue ?

Pourquoi cela viole-t-il LSP ? Quel pattern peut-on utiliser pour éviter cette hiérarchie problématique ? Expliquez également comment appliquer DIP avec une interface.

Partie 2 : Patrons de conception (16 pts)

Question 3 – Singleton & Factory (8 pts)

Une application de gestion de configuration doit charger un fichier de paramètres. On souhaite qu'une seule instance de ConfigManager existe dans toute l'application. Par ailleurs, la création d'objets Logger (console, fichier, distant) doit être déléguée à une fabrique.

  • Dessinez le diagramme de classes UML du pattern Singleton appliqué à ConfigManager.
  • Donnez un exemple de code pour une Factory Method créant des ILogger.

Question 4 – Observer & Strategy (8 pts)

Dans un système de panier d'achat, la vue doit être mise à jour automatiquement quand la quantité d'un article change (Observer). Par ailleurs, le calcul des frais de livraison doit être interchangeable selon la méthode (Standard, Express, Relais) – Strategy.

Décrivez brièvement comment implémenter ces deux patterns en pseudo-code ou UML. Expliquez l'avantage par rapport à une approche monolithique.
