📊 Kanban : visualiser et optimiser le flux

Le Kanban est une méthode agile qui se concentre sur la visualisation du travail, la limitation des en-cours (WIP) et l'optimisation du flux.

🎯 Les 4 principes fondamentaux :

  1. Commencer avec ce que vous faites actuellement
  2. Accepter le changement incrémental et évolutif
  3. Respecter les rôles et responsabilités actuels
  4. Encourager le leadership à tous les niveaux

📈 Les 6 pratiques essentielles :

  • Visualiser le flux de travail (tableau Kanban)
  • Limiter le travail en cours (WIP)
  • Gérer le flux (lead time, cycle time)
  • Rendre les règles explicites
  • Mettre en place des boucles de rétroaction
  • Améliorer collaborativement, évoluer expérimentalement

⚡ Extreme Programming (XP) : l'excellence technique

Extreme Programming (XP) met l'accent sur les pratiques d'ingénierie logicielle pour produire un code de haute qualité.

🔧 Pratiques clés :

  • Test-Driven Development (TDD)
  • Pair programming
  • Refactoring continu
  • Intégration continue (CI)
  • Petites versions (small releases)
  • Planning game

🎯 Valeurs XP :

  • Communication
  • Simplicité (simplicity)
  • Feedback
  • Respect
  • Courage

🧪 TDD (Test-Driven Development) : cycle Red-Green-Refactor

🔴 RED → 🟢 GREEN → ♻️ REFACTOR

Écrire un test qui échoue → coder le minimum pour le faire passer → améliorer la structure du code.

🔄 Scrum vs Kanban vs Scrumban

CritèreScrumKanbanScrumban
ItérationsSprints fixes (1-4 semaines)Flux continuSprints optionnels
RôlesPO, SM, Dev TeamAucun rôle spécifiqueRôles Scrum allégés
MétriquesVélocité, burndownLead time, WIP, cycle timeMix des deux
ChangementsÉvités en cours de sprintÀ tout momentFlexibles

💡 Quand choisir ?

  • Scrum : projets avec priorités stables, équipe dédiée, livraisons régulières.
  • Kanban : support/maintenance, demandes imprévisibles, optimisation du flux.
  • Scrumban : transition entre Scrum et Kanban, équipes qui souhaitent de la flexibilité avec un peu de structure.
