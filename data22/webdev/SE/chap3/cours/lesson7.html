🎯 Objectifs de la leçon

1

Comprendre le principe de responsabilité unique (SRP) – Chaque classe doit avoir une seule raison de changer.

2

Maîtriser Ouvert/Fermé (OCP) – Entités logicielles ouvertes à l'extension, fermées à la modification.

3

Comprendre le principe de substitution de Liskov (LSP) – Les sous-classes doivent pouvoir remplacer leur classe de base sans altérer le bon fonctionnement.

4

Savoir identifier des violations simples et appliquer ces principes dans du code.

✨ Ces principes sont essentiels pour écrire du code propre et évolutif.

🧩 Qu'est-ce que SOLID ?

SOLID est un acronyme qui regroupe 5 principes de conception en programmation orientée objet. Ils ont été popularisés par Robert C. Martin (Uncle Bob).

S – Single Responsibility
O – Open/Closed
L – Liskov Substitution
I – Interface Segregation (leçon suivante)
D – Dependency Inversion (prochaine leçon)

✅ Cette leçon couvre S, O et L.

Ces principes permettent de réduire la dette technique et faciliter la maintenance.

📍 SRP : Responsabilité Unique

« Une classe ne devrait avoir qu'une seule raison de changer. »

🎯 Exemple concret :

Une classe Employe qui calcule le salaire et génère un rapport PDF. → 2 responsabilités = violation SRP.

✔️ Solution : séparer la logique métier (calcul salaire) et la gestion de rapport (export PDF).

💡 Pourquoi c’est important ?

  • Plus facile à tester, lire et maintenir.
  • Les modifications d’une fonctionnalité n’impactent pas les autres.

📝 Exemple de code (avant/après SRP)

❌ Violation SRP :

class User {
  constructor(name, email) { ... }
  saveToDatabase() { /* logique DB */ }
  sendEmail() { /* envoie email */ }
}

✅ Refactorisation SRP :

class User { constructor(name, email) {...} }
class UserRepository { save(user) {...} }
class EmailService { sendWelcome(user) {...} }

Chaque classe a une raison unique de changer : gestion des données vs métier vs communication externe.

🔓 OCP : Ouvert/Fermé

« Les entités (classes, modules) doivent être ouvertes à l’extension mais fermées à la modification. »

🚫 Mauvaise pratique

Modifier une classe existante pour ajouter un nouveau type de paiement (ajout de if/else). Risque d’effets de bord.

✅ Bonne pratique

Utiliser l’abstraction (interfaces / classes abstraites) et injecter de nouveaux comportements via des sous-classes ou stratégies.

📌 Exemple : Pattern Strategy permet d’ajouter des algorithmes sans changer la classe qui les utilise.

🧪 Illustration OCP – Système de remises

// Avant OCP (problème)
class Discount {
  calculate(price, type) {
    if (type === "VIP") return price * 0.8;
    if (type === "NEW") return price * 0.95;
    // Nouveau type ? On modifie cette classe !
  }
}

// Après OCP avec polymorphisme
interface DiscountStrategy {
  apply(price: number): number;
}
class VipDiscount implements DiscountStrategy {
  apply(price) { return price * 0.8; }
}
class NewCustomerDiscount implements DiscountStrategy {
  apply(price) { return price * 0.95; }
}
// Nouvelle remande : ajouter SeasonalDiscount SANS toucher le code existant.

✨ Résultat : la classe qui utilise DiscountStrategy reste stable, on étend via de nouvelles classes.

🔄 LSP : Substitution de Liskov

« Si S est un sous-type de T, alors des objets de type T peuvent être remplacés par des objets de type S sans altérer les propriétés du programme. »

🚨 Exemple classique violant LSP :

Classe Rectangle avec méthode setLargeur() / setHauteur() et sous-classe Carre qui modifie les deux dimensions à la fois. Cela peut casser le comportement attendu.

✔️ Correction : ne pas hériter si le comportement n'est pas substituable, utiliser composition ou une abstraction mieux conçue.

🐧 Exemple simple & compréhensible

Imaginons une classe Oiseau avec méthode voler(). Si on crée une sous-classe Pingouin qui ne vole pas, alors remplacer Oiseau par Pingouin peut générer une erreur.

function faireVoler(oiseau: Oiseau) { oiseau.voler(); }

Avec un pingouin → comportement inattendu. Violation LSP.

💡 Solution: mieux vaut privilégier des interfaces spécifiques (Volant et NonVolant).

LSP garantit que notre code est robuste et fiable lors de l'utilisation du polymorphisme.

📌 Récapitulons les 3 principes

✔️ SRP – Une seule responsabilité par classe → code plus clair et moins de bugs.
✔️ OCP – Extensible sans modification → design évolutif, on ajoute des fonctionnalités par nouvelles classes.
✔️ LSP – Sous-types substituables sans effets secondaires → héritage bien utilisé.
🧠 À retenir : SOLID n’est pas une règle absolue mais un guide puissant pour éviter un code rigide et cassant.
