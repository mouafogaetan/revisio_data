Sécuriser son API avec Helmet, CORS et Rate Limiting

Protégez votre application Node.js/Express contre les attaques courantes du web

🎯 Objectifs de la leçon

✓ Utiliser Helmet pour sécuriser les en-têtes HTTP et prévenir certaines vulnérabilités (XSS, clickjacking...).
✓ Configurer CORS afin d’autoriser ou restreindre les origines qui peuvent accéder à votre API.
✓ Implémenter un rate limiting (express-rate-limit) pour contrer les attaques DDoS et abus.
✓ Comprendre les bonnes pratiques de sécurité pour une API Node.js en production.
“Un backend sécurisé, c'est la base de la confiance utilisateur.”

🔐 Pourquoi renforcer la sécurité d'une API ?

Les API sont exposées sur internet et peuvent subir plusieurs types d'attaques :

  • Cross-Site Scripting (XSS) : injection de scripts malveillants.
  • Clickjacking : tromper l'utilisateur pour qu'il clique sur des éléments cachés.
  • Requêtes malveillantes depuis d'autres domaines (CSRF, vols de données).
  • Attaques par déni de service (DDoS) : surcharge du serveur.

💡 Node.js + Express propose des middlewares simples à intégrer pour mitiger ces risques : Helmet, CORS et express-rate-limit.

Les middlewares de sécurité agissent avant vos routes

🪖 Helmet : Protéger les en-têtes HTTP

Helmet définit plusieurs en-têtes pour bloquer les attaques courantes : X-Content-Type-Options, X-Frame-Options, X-XSS-Protection, etc.

// Installation
npm install helmet

// Dans votre app Express
const helmet = require('helmet');
app.use(helmet()); // Active toutes les protections par défaut

// Configuration personnalisée (exemple)
app.use(helmet({
    frameguard: { action: 'deny' },   // anti-clickjacking
    contentSecurityPolicy: {
        directives: {
            defaultSrc: ["'self'"],
        }
    }
}));
✅ Helmet réduit la surface d'attaque, indispensable en production.

🌐 CORS : Qui peut appeler votre API ?

Par défaut, les navigateurs bloquent les requêtes AJAX vers un domaine différent. Avec CORS, vous autorisez des origines spécifiques.

npm install cors

const cors = require('cors');

// Autoriser toutes les origines (⚠️ à éviter en production)
app.use(cors());

// Configuration sécurisée
app.use(cors({
    origin: 'https://monfrontend.com', // domaine autorisé
    methods: ['GET', 'POST', 'PUT', 'DELETE'],
    credentials: true, // autoriser cookies / headers auth
}));

// Ou utiliser dynamiquement
const corsOptions = {
    origin: (origin, callback) => {
        if (process.env.WHITELIST.split(',').includes(origin)) callback(null, true);
        else callback(new Error('Non autorisé par CORS'));
    }
}

💡 Pour le développement, on utilise cors({ origin: '*' }), mais en production, restreignez les origines.

⏱️ Rate Limiting : Limiter les requêtes abusives

Empêche une même IP d'envoyer trop de requêtes en peu de temps (brute force, DDoS).

npm install express-rate-limit

const rateLimit = require('express-rate-limit');

// Limiteur global
const limiter = rateLimit({
    windowMs: 15 * 60 * 1000, // 15 minutes
    max: 100, // maximum 100 requêtes par IP
    message: 'Trop de requêtes, veuillez réessayer plus tard.',
    standardHeaders: true,
    legacyHeaders: false,
});

app.use(limiter);

// Limiteur plus strict pour une route sensible (login)
const loginLimiter = rateLimit({
    windowMs: 5 * 60 * 1000,
    max: 10,
    skipSuccessfulRequests: true, // éviter de bloquer les succès
});
app.post('/api/login', loginLimiter, (req, res) => { ... });
📌 Astuce : Combine rate limit + Redis en production pour du partagé entre plusieurs instances.

🛠️ Mise en pratique : Exemple complet d'index.js

const express = require('express');
const helmet = require('helmet');
const cors = require('cors');
const rateLimit = require('express-rate-limit');

const app = express();

// 1. Sécurité des en-têtes
app.use(helmet());

// 2. CORS pour un frontend spécifique
app.use(cors({ origin: 'https://monapp-cameroun.com' }));

// 3. Rate limit global
const globalLimiter = rateLimit({
    windowMs: 10 * 60 * 1000,
    max: 200,
});
app.use(globalLimiter);

// 4. Middleware JSON
app.use(express.json());

// 5. Routes protégées
app.get('/api/public', (req, res) => {
    res.json({ message: 'Accès autorisé avec sécurité' });
});

app.listen(3000, () => console.log('API sécurisée en ligne'));
🔁 Ordre des middlewares important : helmet → cors → rateLimit → body parsers → routes.

📌 Meilleures pratiques et pièges à éviter

✅ À faire
  • Utiliser Helmet par défaut
  • Limiter les origines CORS strictement
  • Mettre des rate limits différents selon les endpoints (login, inscription)
  • Utiliser des variables d'environnement pour la liste blanche
❌ À éviter
  • cors({ origin: '*' }) en production
  • Oublier Helmet sur les API sensibles
  • Rate limit trop bas qui bloque les utilisateurs légitimes
  • Ne pas gérer les erreurs CORS correctement

"La sécurité est un processus, pas un état final — testez toujours votre API avec des outils comme Postman en simulant des origines différentes."

🧪 Exercice pratique (à réaliser chez vous)

Objectif : Sécuriser un mini serveur Express

  1. Créez un projet Node.js, installez express, helmet, cors, express-rate-limit.
  2. Configurez Helmet, CORS (autoriser uniquement http://localhost:3001).
  3. Ajoutez un limiteur : max 50 requêtes toutes les 5 minutes.
  4. Créez une route GET /api/securite qui retourne "API protégée ✅".
  5. Testez avec un script ou via un navigateur : dépasser la limite doit retourner 429 Too Many Requests.
💡 Pour vérifier les en-têtes Helmet, ouvrez l'onglet réseau dans les outils développeur.
👨‍🏫 Une fois terminé, vous pourrez impressionner vos collègues avec une API robuste !
