🔍 Debugging 📊 Monitoring ⚙️ Winston/Pino

Objectifs de la leçon

  • ✓ Installer et configurer Winston ou Pino pour la journalisation.
  • ✓ Différencier les niveaux de logs (info, warn, error, debug).
  • ✓ Utiliser les logs pour le débogage et la surveillance applicative.
  • ✓ Savoir transporter les logs vers la console, fichiers ou services externes.
💡 Pourquoi logger ? Un bon logging vous aide à trouver les bugs plus vite et comprendre le comportement de l'API en production.

📜 Journalisation : bien plus que console.log

En production, console.log() est insuffisant : pas de niveaux, pas de persistance, difficile à analyser. Les bibliothèques comme Winston et Pino apportent :

✔️ Niveaux (error, warn, info, debug)
✔️ Transports (fichiers, bases de données, cloud)
✔️ Format JSON (exploitable par des outils)
✔️ Performance (Pino est ultra-rapide)
// ❌ Mauvaise pratique
console.log("Utilisateur connecté");
// ✅ Avec Winston
logger.info("Nouvelle connexion", { userId: 42, ip: "192.168.1.1" });

📦 Winston – Le couteau suisse des logs

npm install winston

Configuration basique (logger.js) :

const winston = require('winston');

const logger = winston.createLogger({
  level: 'info',
  format: winston.format.json(),
  transports: [
    new winston.transports.Console({ format: winston.format.simple() }),
    new winston.transports.File({ filename: 'error.log', level: 'error' }),
    new winston.transports.File({ filename: 'combined.log' })
  ]
});

module.exports = logger;
🔍 Niveaux par défaut : error, warn, info, http, verbose, debug, silly. On peut définir un seuil.

⚡ Pino – Logging ultra-rapide

Idéal pour les applications à fort trafic, jusqu'à 5x plus rapide que Winston.

npm install pino pino-pretty
const pino = require('pino');

const logger = pino({
  level: process.env.LOG_LEVEL || 'info',
  transport: {
    target: 'pino-pretty',
    options: { colorize: true, translateTime: 'SYS:standard' }
  }
});

// Utilisation
logger.info({ user: 'Mboua' }, 'Requête API reçue');
logger.error(new Error('Connexion DB perdue'));
✅Pino génère du JSON compact, facile à intégrer avec des systèmes comme Datadog ou ELK.

🎚️ Niveaux de logs & conseils

error → Erreurs fatales, exceptions non gérées, plantages.
warn → Comportement suspect, ressource faible, dépréciation.
info → Événements normaux: serveur démarré, utilisateur créé.
debug → Détails pour développement uniquement.
📌 Règle d'or : Ne jamais logger des informations sensibles (mots de passe, tokens). Utilisez des champs masqués.

🖥️ Intégration dans une API Express (middleware)

const express = require('express');
const winston = require('winston');
const app = express();

// Middleware de logging avec Winston
app.use((req, res, next) => {
  const start = Date.now();
  res.on('finish', () => {
    const duration = Date.now() - start;
    winston.info(`${req.method} ${req.originalUrl} ${res.statusCode} - ${duration}ms`);
  });
  next();
});

// Route exemple
app.get('/api/users', (req, res) => {
  winston.debug('Fetch users avec query', req.query);
  res.json([{ id: 1, name: "Claude" }]);
});

✨ Cela produit des traces exploitables pour surveiller les performances et les erreurs.

🔎 Déboguer avec les logs

En développement, activez le niveau debug. En production, utilisez info ou warn pour éviter la surcharge.

$ NODE_ENV=development node server.js // logs debug visibles
$ NODE_ENV=production node server.js // logs info+
🚀 Surveillance: On peut envoyer les logs vers Papertrail, Logtail ou un fichier centralisé pour analyser les erreurs fréquentes.
📈 Logs → agrégation → alertes (Ex: Plus de 10 erreurs/min → email à l'admin)

📌 Ce qu'il faut retenir

  • Winston → polyvalent, transports multiples, parfait pour la plupart des projets.
  • Pino → performance extrême, faible overhead pour microservices.
  • Choisir un niveau adapté selon l'environnement (production vs dev).
  • Logs structurés (JSON) facilitent l'analyse automatique.
  • Ne jamais logger les secrets, mais ajouter requestId pour tracer une requête.

🧪 Exercice pratique :

Créez un petit serveur Express, ajoutez Winston avec deux transports (console + fichier app.log). Logguez chaque requête avec méthode, URL et temps de réponse. Testez en générant des erreurs 404 et observez le fichier de logs.
