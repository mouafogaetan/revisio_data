Middleware d'erreur : signature spéciale

Dans Express, un middleware classique reçoit (req, res, next). Pour gérer les erreurs, on utilise quatre paramètres : err, req, res, next. Express reconnaît automatiquement ce middleware comme gestionnaire d'erreurs.

// middleware d'erreur basique
app.use((err, req, res, next) => {
  console.error(err.stack);
  res.status(500).json({ message: "Quelque chose s'est mal passé !" });
});

💡 Important :

Placez ce middleware après toutes vos routes et vos app.use classiques pour capturer les erreurs propagées via next(err).

⚡ Erreurs synchrones & asynchrones

Les erreurs synchrones (dans les routes) sont automatiquement capturées par Express. En revanche, pour les opérations asynchrones (promesses, callbacks), il faut appeler explicitement next(err).

✅ Synchrone 
app.get('/sync', (req, res) => {
  throw new Error('Boom!'); // Express catch automatiquement
});
🔄 Asynchrone (promesse) 
app.get('/async', async (req, res, next) => {
  try {
    await operationRisquee();
  } catch (err) {
    next(err); // propagation
  }
});

📦 Utilisation de express-async-errors :

Il suffit d'importer le module au démarrage, plus besoin de try/catch dans les fonctions async — les erreurs seront automatiquement transmises au middleware d'erreur.

require('express-async-errors');
app.get('/safe', async (req, res) => {
  // une erreur async ici ira directement au middleware d'erreur
});

📟 Réponses d'erreur adaptées

Pour une API claire, utilisez les bons codes HTTP :

🎯 Bonne pratique : Ne jamais envoyer la stack d'erreur en production (utiliser NODE_ENV === 'production' pour masquer les détails internes).

💻 Mise en pratique – Gestionnaire unifié

On peut créer des classes d'erreur personnalisées et un middleware centralisé.

// AppError personnalisé
class AppError extends Error {
  constructor(message, statusCode) {
    super(message);
    this.statusCode = statusCode;
    this.isOperational = true;
  }
}

// Route
app.get('/user/:id', async (req, res, next) => {
  const user = await findUser(req.params.id);
  if (!user) return next(new AppError('Utilisateur non trouvé', 404));
  res.json(user);
});

// Middleware final
app.use((err, req, res, next) => {
  const status = err.statusCode || 500;
  const message = err.message || 'Erreur interne';
  if (status === 500) console.error(err); // log
  res.status(status).json({ 
    error: message, 
    status 
  });
});

Flux d'une erreur propagée par next(err)

🧪 Simplifier les erreurs async

Sans wrapper : chaque route async doit être enrobée de try/catch + next().
Avec express-async-errors : écrivez vos async sans try/catch, les erreurs sont propagées automatiquement.

// Après installation et require('express-async-errors')
app.get('/products', async (req, res) => {
  const products = await db.fetchAll(); // si erreur de DB -> middleware erreur capturé
  res.json(products);
});

✅ Le code est plus propre, moins de next explicite.

⚠️ Note :

Les erreurs dans les événements ou callbacks non promisifiés nécessitent toujours un next(err) manuel.

📌

Bonnes pratiques à retenir

  • Toujours un middleware d'erreur unique en fin de chaîne.
  • Ne pas exposer les détails techniques en production (utiliser process.env.NODE_ENV).
  • Pour les opérations asynchrones, toujours capturer les erreurs et appeler next() avec l'erreur.
  • Utiliser des classes d'erreur personnalisées (400, 404..) pour uniformiser les réponses.
  • Ajouter un logger (Winston, morgan) dans le middleware d'erreur pour le monitoring.
🚀 Défi : Intégrez ce système d'erreurs dans votre API Express et testez les différents scénarios (404, 500, validation).
