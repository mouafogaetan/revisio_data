Leçon 38 – API complète de A à Z

De l'idée à la mise en ligne : tester, déployer et présenter son API professionnelle

🚀 Production 📦 Heroku/Railway 🖥️ VPS + Nginx

Objectifs de la leçon

  • ✓ Tester l'API complète avec Postman/Insomnia (manuellement).
  • ✓ Déployer l'API en production sur Heroku, Railway ou un VPS.
  • ✓ Configurer les variables d'environnement et la base de données distante.
  • ✓ Présenter le projet : documentation, démo live, explications techniques.
  • ✓ Savoir débugger une API déployée (logs, erreurs courantes).
💡 À la fin de ce cours, vous serez capable de livrer un projet backend prêt pour le monde professionnel.

🛠️ Tester son API avant déploiement

Avant la mise en ligne, validez chaque endpoint :

GET /api/users → liste utilisateurs
POST /api/auth/login → authentification JWT
PUT /api/products/:id → modification produit
DELETE /api/orders/:id → suppression
// Exemple de collection Postman à importer
{
  "info": { "name": "API Ecommerce" },
  "item": [
    { "name": "Login", "request": { "method": "POST", "url": "{{baseUrl}}/auth/login" } }
  ]
}

✅ Testez aussi les erreurs (404, 401, 500) et les validations.

☁️ Déployer sur Heroku ou Railway

Étapes clés (Railway / Heroku) :

  • 1. Avoir un package.json avec un script "start": "node server.js".
  • 2. Utiliser process.env.PORT pour le port d'écoute.
  • 3. Définir les variables d'environnement (DATABASE_URL, JWT_SECRET) dans l'interface PaaS.
  • 4. Connecter le dépôt GitHub et activer les déploiements automatiques.
// Exemple server.js (extrait)
const PORT = process.env.PORT || 3000;
app.listen(PORT, () => console.log(`API running on port ${PORT}`));
🚀 Railway : "railway up" ou via GitHub ⚡ Heroku : git push heroku main
[ Git push → Build → Release → Live URL ]

🐧 VPS (Ubuntu) + Nginx + PM2

Pour plus de contrôle, déployez sur un serveur privé virtuel.

# 1. Transférer le code (git clone / rsync)
# 2. Installer les dépendances
npm install --production
# 3. Démarrer avec PM2
pm2 start server.js --name api-prod
pm2 save && pm2 startup

# 4. Configurer Nginx (reverse proxy)
server {
    listen 80;
    server_name api.monsite.com;
    location / {
        proxy_pass http://localhost:3000;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
    }
}

🔒 Pensez à sécuriser avec Certbot (HTTPS) et à ouvrir les bons ports.

⚠️ N'oubliez pas de configurer la base de données distante (MongoDB Atlas, PostgreSQL sur le VPS).

🔐 Variables d'environnement sécurisées

Ne jamais versionner .env. Utilisez les outils des plateformes :

  • Heroku : heroku config:set JWT_SECRET=monsecret
  • Railway : onglet "Variables" dans le dashboard
  • VPS : créer un fichier .env sur le serveur ou les exporter dans /etc/environment
// .env.example (à partager dans le README)
PORT=5000
MONGO_URI=mongodb+srv://...
JWT_SECRET=change_this_in_production
NODE_ENV=production
⚠️ Le mot de passe de la base de données ne doit jamais apparaître dans le code !

🐛 Déboguer l'API déployée

Outils indispensables pour comprendre les erreurs en production :

  • Logs de la plateforme : heroku logs --tail ; Railway -> onglet Logs ; VPS -> pm2 logs ou journaux Nginx.
  • Middleware d'erreur express : renvoyer un JSON explicite sans exposer la stack trace.
  • Surveillance : utiliser Winston/Pino avec transport fichier ou service externe (Logtail).
// Middleware d'erreur adapté production
app.use((err, req, res, next) => {
  const status = err.status || 500;
  res.status(status).json({
    message: process.env.NODE_ENV === 'production' ? 'Erreur serveur' : err.message,
  });
});

✅ Tester avec curl ou Postman en ligne de commande pour vérifier les headers.
