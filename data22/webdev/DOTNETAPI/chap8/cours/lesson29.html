🧠 1. CORS c’est quoi ?

CORS = Cross-Origin Resource Sharing. C'est un mécanisme de sécurité des navigateurs qui limite les requêtes HTTP entre différents domaines (origines).

⚠️ Pourquoi ce problème ?

Par défaut, une page web sur https://mon-site.com ne peut pas appeler une API sur https://api-externe.com sauf si l’API l’autorise explicitement via CORS.

🌐

Origine A : frontend: localhost:3000

→ 🚫 blocage ←

Origine B : API: localhost:5000 (si non CORS)

✅

Avec en-têtes CORS :

Access-Control-Allow-Origin: * ou http://localhost:3000

✔️ La requête devient autorisée

📌 Sans CORS, le navigateur bloque les appels AJAX/fetch entre origines différentes. Votre API doit envoyer les bons en-têtes.

⚙️ 2. Configurer CORS dans .NET

Dans une API ASP.NET Core, on configure CORS via deux étapes essentielles :

// 1. Ajouter le service CORS (Program.cs)
builder.Services.AddCors(options =>
{
    options.AddPolicy("MaPolitiqueCORS", policy =>
    {
        policy.WithOrigins("https://monfrontend.com", "http://localhost:4200")
              .AllowAnyMethod()
              .AllowAnyHeader();
    });
});

// 2. Utiliser le middleware CORS (après routing, avant auth)
app.UseCors("MaPolitiqueCORS");

📌 Ordre important :

✅ app.UseRouting() → app.UseCors() → app.UseAuthentication() / UseAuthorization() → app.MapControllers()

🔐 3. Politiques granulaires

Une API professionnelle doit restreindre les accès :

Origines autorisées 
.WithOrigins("https://app.monclient.com", "https://admin.dashboard.com")

✔️ uniquement ces domaines peuvent appeler l’API

Méthodes HTTP autorisées 
.WithMethods("GET", "POST", "PUT")

🚫 Interdire DELETE / PATCH si nécessaire

En-têtes autorisés 
.WithHeaders("Authorization", "Content-Type", "X-Custom-Header")

📎 pour les tokens JWT par ex.

Autoriser les credentials 
.AllowCredentials()

🍪 cookies / headers d’authentification

⚠️ Attention : .AllowAnyOrigin() et .AllowCredentials() ne sont pas compatibles. Pour la sécurité, privilégiez des origines précises.

💡 5. Bonnes pratiques CORS

  • ✅ En développement : politique permissive AllowAnyOrigin() mais jamais en production.
  • ✅ En production : lister les origines exactes (pas de * avec credentials).
  • ✅ Utiliser des variables d’environnement : charger les origines autorisées depuis appsettings.json.
  • ✅ Tester avec des outils comme Postman (CORS n’est pas bloqué par Postman car c'est un mécanisme navigateur !).
  • ✅ Séparer plusieurs politiques : par exemple une politique "Public" et une "Admin".

📁 Exemple d’appsettings.json :

{
  "CorsSettings": {
    "AllowedOrigins": [ "https://client.com", "https://admin.client.com" ]
  }
}

Et injection via IConfiguration pour construire la politique dynamiquement.
