📦 Migrations avec Entity Framework Core

Gérer l’évolution de la base de données sans perdre vos données

Appuyez sur ➡️ pour commencer

🎯 Objectifs de la leçon

  • ✓ Installer les outils dotnet-ef globalement
  • ✓ Créer une migration initiale avec dotnet ef migrations add
  • ✓ Appliquer la migration à la base de données avec dotnet ef database update
  • ✓ Comprendre le principe Code-First et le versionnement du schéma
💡 Les migrations permettent de synchroniser vos modèles C# avec votre base SQL.

🔧 Pourquoi les migrations ?

Quand vous développez une application, vos classes (entités) changent souvent : nouvelles propriétés, relations, tables. Les migrations EF Core transforment ces changements en commandes SQL, sans perdre les données existantes.

✅ Avantages :

  • Historique des modifications (fichiers .cs dans Migrations/)
  • Mise à jour simple en production
  • Collaboration facile (Git, partage)

📌 Code First : vous codez vos modèles C#, EF Core génère la base → les migrations sont le pont.

🛠️ Étape 1 : Installer l'outil CLI

dotnet tool install --global dotnet-ef

Cette commande installe dotnet ef globalement. Vérifiez l’installation avec :

dotnet ef

⚠️ Attention : Dans un projet Web API, il faut aussi les packages : Microsoft.EntityFrameworkCore.Design (installé via NuGet).

📝 Étape 2 : Créer une migration

Après avoir défini votre DbContext et vos entités, ouvrez un terminal dans le dossier du projet :

dotnet ef migrations add NomDeMaMigration

Par exemple : dotnet ef migrations add InitialCreate

📁 Que se passe-t-il ?

Un dossier Migrations est créé contenant :

  • Un fichier horodaté avec les instructions Up() et Down()
  • Un snapshot de votre modèle (DbContextModelSnapshot)

⚙️ Étape 3 : Mettre à jour la base de données

dotnet ef database update

Cette commande exécute toutes les migrations qui n’ont pas encore été appliquées à la base cible (définie dans appsettings.json).

✅ Résultat :

Tables créées / modifiées selon vos entités.

🔄 Annuler / Revenir :

dotnet ef database update NomMigrationPrecedente permet de reculer.

📊 Schéma visuel du cycle migrations

Le modèle C# → génère un script de migration → appliqué sur la base réelle.

🧠 Commandes supplémentaires & astuces

dotnet ef migrations remove

Supprime la dernière migration (avant de l’appliquer à la base). Utile pour corriger.

dotnet ef migrations script

Génère un script SQL complet sans l’exécuter. Pratique pour déploiement manuel.

💡 Bonne pratique :

Toujours tester les migrations en environnement de développement avant production. Utiliser le contrôle de version (Git) pour partager le dossier Migrations avec l’équipe.

📄 Exemple rapide dans un projet

Admettons que vous ayez un modèle Produit et un AppDbContext, voici le workflow :

# 1. Créer la migration "AjoutProduits"
dotnet ef migrations add AjoutProduits

# 2. Appliquer la migration (création table Produits)
dotnet ef database update

# 3. Si une nouvelle propriété 'Prix' est ajoutée dans la classe :
dotnet ef migrations add AjoutPrixProduit
dotnet ef database update

Les migrations détectent automatiquement les changements et proposent le code SQL approprié.

🔍 Vérifier l’état des migrations

dotnet ef migrations list

Affiche toutes les migrations avec indication de celle appliquée à la base.

🏁 En résumé pour vos projets au Cameroun :

  • Installer dotnet-ef une seule fois sur votre machine
  • Pour tout changement de schéma => migration + update
  • Poussez le dossier Migrations sur GitHub pour l'équipe

✏️ Mini exercice pratique

Scénario :

Vous développez une API pour la gestion d'étudiants. Vous avez déjà un DbContext et une entité Etudiant (Id, Nom). Vous venez d’ajouter une propriété Email. Écrivez les deux commandes nécessaires pour mettre à jour la base de données sans supprimer les étudiants existants.

Indice : il faut d’abord ajouter une migration, puis l’appliquer.
