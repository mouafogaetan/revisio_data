🛡️ Filtres d'exception & ILogger

Gérer proprement les erreurs, logger chaque incident et retourner des réponses élégantes dans ASP.NET Core

Qu'est-ce qu'un filtre d'exception ?

Les filtres d'exception (IExceptionFilter ou IAsyncExceptionFilter) interceptent toute exception non gérée dans l'action d'un contrôleur. Ils permettent de :

  • Logger l'erreur (stack trace, paramètres, utilisateur)
  • Retourner une réponse personnalisée (ex: ProblemDetails)
  • Éviter de polluer l'API avec des détails techniques exposés au client
⚙️ Middleware vs Filtres : Les middlewares capturent les erreurs avant même le routing, tandis que les filtres s'exécutent dans le contexte MVC (accès aux modèles, aux données de l'action).

📝 Créer un filtre personnalisé

Héritez de ExceptionFilterAttribute et surchargez OnException :

public class GlobalExceptionFilter : IExceptionFilter
{
    private readonly ILogger<GlobalExceptionFilter> _logger;
    public GlobalExceptionFilter(ILogger<GlobalExceptionFilter> logger)
        => _logger = logger;

    public void OnException(ExceptionContext context)
    {
        _logger.LogError(context.Exception, 
            "Erreur sur {Action} - {Message}", 
            context.ActionDescriptor.DisplayName,
            context.Exception.Message);

        var details = new ProblemDetails
        {
            Status = StatusCodes.Status500InternalServerError,
            Title = "Une erreur interne est survenue",
            Detail = "Veuillez contacter l'administrateur",
            Instance = context.HttpContext.Request.Path
        };
        context.Result = new ObjectResult(details) 
            { StatusCode = details.Status };
        context.ExceptionHandled = true;
    }
}

✅ ExceptionHandled = true empêche la propagation de l'erreur.

📦 Enregistrer le filtre & Utiliser ILogger

Dans Program.cs, ajoutez le filtre globalement :

builder.Services.AddControllers(options => 
{
    options.Filters.Add<GlobalExceptionFilter>();
});
// Le logger est automatiquement injecté par DI

🔍 ILogger<T> : meilleures pratiques

  • LogError() : exceptions critiques
  • LogWarning() : comportement anormal mais récupérable
  • LogInformation() : traces de fonctionnement normal
  • Utilisez des paramètres structurés ({Action}) pour faciliter le querying
📌 ILogger<WeatherForecastController> → nom de catégorie = nom complet du contrôleur.

🧪 Test & bonnes pratiques

Un contrôleur qui lève une exception volontaire :

[ApiController]
[Route("api/[controller]")]
public class DemoController : ControllerBase
{
    private readonly ILogger<DemoController> _logger;
    public DemoController(ILogger<DemoController> logger) => _logger = logger;

    [HttpGet("crash")]
    public IActionResult Crash()
    {
        _logger.LogInformation("Tentative de crash simulé");
        throw new InvalidOperationException("Base de données injoignable");
    }
}

✔️ Le filtre capture l'exception → log dans la console/fichier → retour JSON ProblemDetails (status 500).

✅ Recommandations
- Toujours logger avec le niveau approprié
- Ne pas exposer la stack trace en production
- Utiliser IExceptionFilter pour la cohérence REST
⚠️ À éviter
- Afficher les détails techniques aux clients
- Oublier de marquer ExceptionHandled = true
- Logger des données sensibles
🏁
