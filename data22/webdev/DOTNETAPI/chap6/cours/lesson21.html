🔐 Hachage sécurisé des mots de passe

BCrypt.Net-Next dans ASP.NET Core

Protéger les identifiants des utilisateurs, une obligation éthique et légale

Appuyez sur ➡️ pour commencer

🎯 Objectifs de la leçon

  • ✓ Installer et utiliser BCrypt.Net-Next dans un projet .NET
  • ✓ Hacher un mot de passe lors de l’inscription (enregistrement utilisateur)
  • ✓ Vérifier le mot de passe saisi par rapport au hash stocké à la connexion
  • ✓ Comprendre pourquoi le hachage salé (BCrypt) est résistant aux attaques
💡 Rappel : On ne stocke JAMAIS les mots de passe en clair. BCrypt ajoute un "salt" aléatoire et est lent par conception (résistance aux attaques brute force).

🧂 Pourquoi BCrypt ?

⚡ Limites du hachage simple (MD5, SHA)

Les algorithmes rapides comme SHA256 sont vulnérables aux attaques par force brute ou tables rainbow. Un attaquant peut calculer des milliards de hash par seconde.

🛡️ Forces de BCrypt

Basé sur l'algorithme Blowfish, il intègre un salt (32 octets) et un facteur de coût (work factor). Le coût rend le hachage volontairement lent (ex: 100ms) pour décourager les attaques.

// Exemple de hash BCrypt (format) : 
$2a$10$N9qo8uLOickgx2ZMRZoMy.Mr/.cZJgWx9YpRjYwF.t3UqVvlM5O7O
 │  │  │      │                         │
 │  │  │      └── salt (22 caractères)  └── hash final (31 caractères)
 │  │  └── coût (10 = 2^10 itérations)
 │  └── version (2a)
 └── identifiant

📦 Installation du package BCrypt

dotnet add package BCrypt.Net-Next

Ou via NuGet Package Manager : BCrypt.Net-Next (version récente, compatible .NET Core / .NET 5+).

📌 À retenir :

  • Pas besoin de configurer de salt manuellement – BCrypt le génère automatiquement.
  • La méthode HashPassword inclut le salt dans le résultat.
  • Le facteur de coût par défaut est 10 (recommandé pour la plupart des apps).

✍️ Étape 1 : Hacher lors de l'inscription

using BCryptNet = BCrypt.Net.BCrypt;

// Dans votre service ou contrôleur (RegisterDto)
public async Task<IActionResult> Register(RegisterRequest request)
{
    // Vérifier si l'utilisateur n'existe pas déjà...
    var hashedPassword = BCryptNet.HashPassword(request.Password);
    
    var user = new User
    {
        Email = request.Email,
        PasswordHash = hashedPassword,
        // autres propriétés
    };
    
    _context.Users.Add(user);
    await _context.SaveChangesAsync();
    return Ok(new { message = "Inscription réussie" });
}
🔒 Résultat : La base de données ne contient que le hash, jamais le mot de passe original.

🔑 Étape 2 : Vérification à la connexion

public async Task<IActionResult> Login(LoginRequest request)
{
    var user = await _context.Users
        .FirstOrDefaultAsync(u => u.Email == request.Email);
    
    if (user == null)
        return Unauthorized("Email ou mot de passe incorrect");
    
    bool isValid = BCryptNet.Verify(request.Password, user.PasswordHash);
    
    if (!isValid)
        return Unauthorized("Email ou mot de passe incorrect");
    
    // Générer token JWT ou session...
    return Ok(new { token = GenerateJwtToken(user) });
}
⚠️ Important : Verify() compare le mot de passe fourni + le salt extrait du hash stocké. Ne jamais tenter de recréer manuellement le salt.

⚙️ Personnaliser le coût (work factor)

BCrypt permet d'augmenter la difficulté de calcul (itérations = 2^workFactor). Plus le coût est élevé, plus le hachage est lent (sécurisé).

// Coût 12 (2^12 = 4096 itérations) – plus sécurisé, mais plus lent
string hash = BCryptNet.HashPassword(password, workFactor: 12);

// Vérification automatique (le work factor est stocké dans le hash)
bool valid = BCryptNet.Verify(password, hashFromDb);
📈 Recommandation : utiliser coût 10 pour développement, 12-13 pour production selon les performances.

✅ Bonnes pratiques

  • ✔️ Toujours hacher côté serveur, jamais côté client.
  • ✔️ Utiliser un work factor suffisamment élevé (10 minimum, 12 recommandé).
  • ✔️ Stocker le hash dans une colonne de type texte (nvarchar(255)).
  • ❌ Ne jamais utiliser MD5, SHA1, ou un algorithme rapide seul pour les mots de passe.
  • ❌ Ne jamais écrire son propre algorithme de hachage.
🧠 Rappel de sécurité : BCrypt + HTTPS + validation forte = première ligne de défense

📝 Mini synthèse / code complet

UserService.cs (extrait)
public class AuthService {
    private readonly AppDbContext _db;
    public AuthService(AppDbContext db) => _db = db;
    
    public async Task<bool> RegisterUser(string email, string password) {
        var existing = await _db.Users.AnyAsync(u => u.Email == email);
        if(existing) return false;
        var hash = BCrypt.Net.BCrypt.HashPassword(password, 12);
        _db.Users.Add(new User{ Email = email, PasswordHash = hash });
        await _db.SaveChangesAsync();
        return true;
    }
    
    public async Task<User> Authenticate(string email, string password) {
        var user = await _db.Users.FirstOrDefaultAsync(u => u.Email == email);
        if(user == null) return null;
        bool verified = BCrypt.Net.BCrypt.Verify(password, user.PasswordHash);
        return verified ? user : null;
    }
}

💡 Cette approche est compatible avec l'injection de dépendances et les contrôleurs d'API .NET Core.
