🔒 Qu'est-ce que [Authorize] ?

L’attribut [Authorize] empêche l’accès à un contrôleur ou une action si l’utilisateur n’est pas authentifié. Il s’utilise dans les API REST sécurisées par token JWT.

// Exemple simple : tout le contrôleur nécessite authentification
[ApiController]
[Route("api/[controller]")]
[Authorize]    // 👈 protège toutes les actions
public class CommandesController : ControllerBase
{
    [HttpGet]
    public IActionResult GetCommandes() { ... }
}

💡 Important : Sans token valide dans le header Authorization: Bearer <token>, l’API retourne 401 Unauthorized.

👑 [Authorize(Roles = "Admin")]

Permet de limiter l’accès aux utilisateurs possédant un rôle spécifique. Les rôles sont des claims inclus dans le token JWT.

// Seuls les Admins peuvent créer/supprimer des produits
[Authorize(Roles = "Admin")]
[HttpPost]
public IActionResult CreateProduit(ProduitDto dto) { ... }

// Plusieurs rôles : Admin ou Gestionnaire
[Authorize(Roles = "Admin, Gestionnaire")]
[HttpPut("{id}")]
public IActionResult UpdateProduit(int id, ...) { ... }
✅ Bonne pratique

Attribuer des rôles aux claims lors de la génération JWT (ex: new Claim(ClaimTypes.Role, "Admin")).

🔧 Vérification

Le middleware d’authentification valide automatiquement les rôles à chaque requête.

📝 Exemple complet - Login avec rôle

Lors de la création du token JWT, ajoutez le(s) rôle(s) dans les claims.

// Extrait du endpoint login
var claims = new List<Claim>
{
    new Claim(JwtRegisteredClaimNames.Sub, user.Id),
    new Claim(ClaimTypes.Email, user.Email),
    new Claim(ClaimTypes.Role, user.Role) // "Admin" ou "User"
};
var token = new JwtSecurityToken(issuer, audience, claims,
    expires: DateTime.Now.AddHours(1),
    signingCredentials: creds);
return Ok(new { token = new JwtSecurityTokenHandler().WriteToken(token) });

🔐 Contrôleur avec autorisation par rôle :

[Authorize(Roles = "Admin")]
[HttpGet("admin/stats")]
public IActionResult GetStats() => Ok("Données admin sensibles");

⚙️ Politiques d’autorisation sur mesure

Si les rôles simples ne suffisent pas, créez des politiques basées sur l’âge, un claim spécifique ou une exigence métier.

Program.cs :

builder.Services.AddAuthorization(options =>
{
    options.AddPolicy("MinimumAge18", policy =>
        policy.RequireClaim("Age", "18", "19", "20+"));
    options.AddPolicy("RequireAdminOrManager", policy =>
        policy.RequireRole("Admin", "Manager"));
});

Utilisation dans un contrôleur :

[Authorize(Policy = "MinimumAge18")]
[HttpGet("adult-content")]
public IActionResult AdultContent() => Ok("Accès réservé aux +18");
🧩 Avantage : les politiques centralisent les règles complexes et évitent la duplication.

🧠 Bonnes pratiques & astuces

  • Placer [Authorize] au niveau contrôleur, puis [AllowAnonymous] sur certaines actions (login, register).
  • Ne jamais exposer d’endpoint sensible sans autorisation.
  • Combiner rôles et politiques : [Authorize(Roles = "Admin", Policy = "Premium")] → toutes conditions doivent être remplies.
  • Utiliser [Authorize(Roles = "User")] sur des endpoints de base et [Authorize(Roles = "Admin")] pour l’administration.
  • Vérifier les rôles dans les tests : simulez un utilisateur avec ClaimsPrincipal.
⚠️

Attention : Par défaut, [Authorize] ne valide que l’authentification. Si aucun rôle n’est précisé, tout utilisateur authentifié (même sans rôle) a accès.

🧪 Testons notre compréhension

Scénario : API de blog avec rôles "Auteur" et "Lecteur".

Endpoint : DELETE /api/articles/{id}

Seul un "Admin" ou l’Auteur de l’article peut supprimer. Comment faire ?

Réponse : implémenter une politique personnalisée vérifiant le claim "UserId" et le rôle Admin.

[Authorize(Policy = "CanDeleteArticle")]
public IActionResult DeleteArticle(int id) { ... }

✨ À retenir :

L’attribut [Authorize] protège votre API, et Roles affine la sécurité. Les politiques personnalisées apportent une flexibilité ultime.
