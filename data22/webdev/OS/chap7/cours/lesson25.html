Pourquoi c'est important ?

Les conteneurs sont au cœur du DevOps moderne et du cloud computing. Docker, Kubernetes, Podman... tous reposent sur ces concepts !

🖥️ Qu'est-ce qu'une Machine Virtuelle ?

Une machine virtuelle (VM) est une émulation logicielle d'un ordinateur physique. Elle exécute un système d'exploitation complet (invité) sur un hyperviseur.

Matériel physique
↑
Hyperviseur (Type 1/2)
↑
VM1 (Windows)
VM2 (Linux)
VM3 (macOS)
  • ✅ Isolation complète (chacun son OS)
  • ✅ Compatibilité multiplateforme
  • ❌ Plus lourdes (Go de RAM, temps de démarrage long)

📦 Qu'est-ce qu'un Conteneur ?

Un conteneur est une virtualisation au niveau du système d'exploitation. Il partage le noyau de l'hôte mais isole les processus, le réseau et le système de fichiers.

Matériel physique
↑
Noyau Linux (hôte)
↑
Conteneur A
Conteneur B
Conteneur C
  • ✅ Légers (quelques Mo), démarrage instantané
  • ✅ Haute densité (plusieurs conteneurs sur un serveur)
  • ⚠️ Partage le noyau hôte (moins d'isolation)

🔒 Namespaces : l'isolation des processus

Les namespaces sont une fonctionnalité du noyau Linux qui permet d'isoler les processus conteneurisés.

PID namespace : chaque conteneur voit ses propres processus (PID 1)
NET namespace : pile réseau isolée (IP, routes)
MNT namespace : système de fichiers isolé
USER namespace : isolation des utilisateurs (root dans conteneur ≠ root hôte)
UTS namespace : hostname isolé
IPC namespace : isolation des communications inter-processus
Un conteneur est un ensemble de processus qui partagent les mêmes namespaces.

⚙️ cgroups : contrôle des ressources

Les cgroups (control groups) permettent de limiter, prioriser et surveiller l'utilisation des ressources par un groupe de processus.

CPU : limite le temps processeur
Mémoire : limite RAM (hard limit)
Bloc I/O : bande passante disque
pids : limite nombre de processus
# Exemple : limiter un conteneur Docker à 512 Mo de RAM
docker run -m 512m --cpus="1.5" mon_image

# Voir les cgroups d'un conteneur
cat /sys/fs/cgroup/memory/docker/<id>/memory.limit_in_bytes

🐳 Architecture de Docker

Docker est la solution de conteneurisation la plus populaire. Son architecture repose sur plusieurs composants :


Client (docker CLI)

Docker Daemon (dockerd)

Registry (Docker Hub)
  • Docker Daemon : service qui gère les conteneurs, images, volumes
  • Client : commande docker qui communique avec le daemon (API REST)
  • Registry : dépôt d'images (Docker Hub, privé)
  • Images : modèles en lecture seule (ex: Ubuntu, Node.js)
  • Conteneurs : instances exécutables d'une image

🤔 Quel outil choisir ?

Privilégier les Conteneurs

  • Applications cloud natives (microservices)
  • Environnements de développement reproductibles
  • CI/CD et déploiement rapide
  • Optimisation des ressources (plus de services sur même serveur)

Privilégier les VMs

  • Nécessité d'un OS invité différent (ex: Windows sur Linux)
  • Isolation de sécurité maximale
  • Applications legacy nécessitant un noyau spécifique
  • Virtualisation de bureaux (VDI)
Pratique courante : VMs sur les serveurs physiques, conteneurs à l'intérieur des VMs !

🏁 Récapitulatif

Machine Virtuelle : virtualisation matérielle, OS complet, lourde mais très isolée
Conteneur : virtualisation au niveau OS, partage le noyau, léger et rapide
Namespaces : isolation des processus (PID, réseau, fichiers...)
cgroups : limitation des ressources CPU, mémoire, I/O
Docker : architecture client/daemon, images immuables, registries

Félicitations ! Vous maîtrisez les bases de la conteneurisation.
