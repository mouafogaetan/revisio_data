⏰ Introduction à cron

Cron est un démon (processus en arrière-plan) présent sur les systèmes UNIX/Linux. Il permet d’exécuter des commandes ou scripts à des moments précis, répétés dans le temps.

📌 Commande principale : crontab -e pour éditer ses tâches personnelles.

Structure d’une ligne cron :

* * * * * commande_à_exécuter
│ │ │ │ │
│ │ │ │ └─── jour semaine (0-7, 0=dim)
│ │ │ └───── mois (1-12)
│ │ └─────── jour du mois (1-31)
│ └───────── heure (0-23)
└─────────── minute (0-59)

Exemple : 30 2 * * * /home/user/backup.sh → tous les jours à 2h30.

🧩 Syntaxe détaillée & exemples

*/15 * * * *

Toutes les 15 minutes

0 9 * * 1-5

9h du matin du lundi au vendredi

0 0 1 * *

Le premier jour de chaque mois à minuit

30 22 15 * *

Le 15 de chaque mois à 22h30

📁 Commandes pratiques :

  • crontab -l : lister mes tâches
  • crontab -r : supprimer toutes les tâches
  • crontab -e : éditer

💡 Astuce Cameroun : planifiez des sauvegardes automatiques la nuit pour ne pas impacter les utilisateurs.

🔄 systemd timers : l’alternative moderne

Sur les distributions récentes (Ubuntu 16.04+, Debian 8+, CentOS 7+), systemd remplace les scripts d'init. Les timers offrent plus de flexibilité (monotonic, calendrier, journalisation, persistance après boot).

# Créer deux fichiers :

/etc/systemd/system/mon-service.timer

/etc/systemd/system/mon-service.service

✅ Avantages :

  • Dépendances entre timers et services.
  • Exécution après le démarrage réseau, etc.
  • Meilleure gestion des échecs (journald).

⚙️ Exemple d’un timer systemd

Nous créons un script qui affiche "Bonjour Cameroun" tous les jours à 14h30.

📄 /etc/systemd/system/hello.service

[Unit]
Description=Script de test

[Service]
Type=oneshot
ExecStart=/usr/bin/bash -c 'echo "Bonjour Cameroun - $(date)" >> /tmp/hello.log'

⏲️ /etc/systemd/system/hello.timer

[Unit]
Description=Timer pour hello.service

[Timer]
OnCalendar=*-*-* 14:30:00
Persistent=true

[Install]
WantedBy=timers.target

Commande : sudo systemctl enable hello.timer && sudo systemctl start hello.timer puis systemctl list-timers

⚖️ Cron vs systemd timers : quand utiliser ?

CritèreCronsystemd timers
Simplicité✅ Très simple⚠️ Courbe d’apprentissage
JournalisationPar syslog✅ journalctl intégré
Rattrapage tâches manquéesNon natif✅ (Persistent=true)
Environnement utilisateurFacile (crontab -e)Timers utilisateur possible

✏️ En entreprise / serveurs modernes : systemd timers recommandé pour les services critiques. Cron reste parfait pour des scripts perso simples.

🧠 Bonnes pratiques pour la planification

  • Toujours rediriger la sortie vers un fichier ou /dev/null pour éviter de remplir la boîte mail locale.
  • Exemple : 0 3 * * * /usr/bin/backup.sh > /var/log/backup.log 2>&1
  • Vérifier que votre script a les droits d’exécution (chmod +x).
  • Utiliser des chemins absolus dans cron, car l’environnement est minimal.
  • Pour systemd timer : consulter les logs avec journalctl -u nom.service.
🌍 Exemple concret Cameroun : Planifier l’envoi d’un rapport d’activité du site web chaque lundi à 8h, avec cron : 0 8 * * 1 /home/dev/send_report.sh.
