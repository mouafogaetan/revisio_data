🧩 Processus lourd vs Thread (léger)

⚙️ Processus lourd
  • Possède son propre espace mémoire (code, données, pile).
  • Changement de contexte coûteux (sauvegarde de tous les registres, mapping mémoire).
  • Communication inter-processus (IPC) complexe.
  • Isolation forte : un plantage n'affecte pas les autres processus.
🧵 Thread (processus léger)
  • Unité d'exécution au sein d'un même processus.
  • Partage la mémoire (code, données, fichiers ouverts) avec les autres threads du processus.
  • Création et commutation plus rapides.
  • Nécessite une synchronisation (mutex, sémaphores).
💡 À retenir : Un processus contient au moins un thread (le thread principal). Les threads sont comme des "sous-tâches" qui s'exécutent en parallèle et partagent les ressources.

         ↑
    ┌────┴────┐
    │Processus│ (mémoire propre)
    │  ┌────┐ │
    │  │Thread1 │ ──┐
    │  ├────┤   │   partage
    │  │Thread2 │ ◄─┘ mémoire
    │  └────┘   │
    └───────────┘

🔗 Modèles de mapping threads – noyau

Comment les threads utilisateurs sont liés aux threads noyau ?

📌 Modèle 1:1 (Linux, Windows)

Chaque thread utilisateur correspond à un thread noyau. ✅ Bon parallélisme réel, mais coût de création modéré.

📌 Modèle N:1 (ex: anciens systèmes / bibliothèques green threads)

Plusieurs threads utilisateurs mappés sur un seul thread noyau. ❌ Pas de vrai parallélisme (un seul peut s’exécuter à la fois).

📌 Modèle M:N (mixte, ex: Scheduler Activations)

M threads utilisateurs sur N threads noyau (M > N). Flexible, mais complexe à implémenter.

🧠 Résumé: Modèle 1:1 est le plus répandu aujourd'hui (pthreads, threads Java natifs).

✅ Avantages & ⚠️ Problèmes des threads

Avantages

  • 🚀 Création rapide (pas d'allocation mémoire lourde).
  • 💾 Partage efficace de la mémoire (évite copies).
  • ⚡ Communication simple via variables partagées.
  • 📈 Meilleure réactivité (blocage d'un thread ne bloque pas tout le processus).

Problèmes / défis

  • 🔐 Synchronisation nécessaire (mutex, sémaphores) pour éviter les accès concurrents.
  • 💣 Risque d’interblocage (deadlock) entre threads.
  • 🐞 Debugging plus complexe.
  • ⚠️ Un thread qui plante peut corrompre les données partagées → tout le processus impacté.
Exemple partage mémoire :
Thread A ✍️ écrit
📦 Variable globale x
Thread B 📖 lit
→ nécessite un verrou (mutex) pour l'accès simultané

🏗️ Création thread vs processus – Analogie

Processus = construire une nouvelle maison (terrain, fondations, murs, etc.). Coûteux et lent.

Thread = ajouter un nouvel ouvrier dans une maison déjà construite, ils partagent les outils et la cuisine. Rapide et efficace.

📌 En programmation C (pthread) : création thread vs fork()

// Thread : création légère
pthread_create(&thread, NULL, fonction, arg);

// Processus (fork) : lourd
pid_t pid = fork();  // copie tout l'espace mémoire

🎯 À savoir: Dans un navigateur web, chaque onglet peut être un processus (isolation) tandis que les threads gèrent les animations internes.

❓ Question : Lequel des avantages suivants est UNIQUEMENT associé aux threads (par rapport aux processus lourds) ?

💡 Rappel : Les threads partagent le même espace d'adressage, d'où une grande rapidité de communication mais nécessite synchronisation.
