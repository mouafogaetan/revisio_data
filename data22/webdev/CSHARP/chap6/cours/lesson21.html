📐 Qu'est-ce qu'une interface ?

Une interface définit un contrat. Toute classe qui implémente une interface s'engage à fournir le comportement demandé (méthodes, propriétés, événements...).

// Syntaxe de base d'une interface
public interface IVolant
{
    void Decoller();
    void Atterrir();
    int Altitude { get; }
}

// Implémentation par une classe
public class Avion : IVolant
{
    public int Altitude { get; private set; }
    public void Decoller() { Altitude = 100; Console.WriteLine("Décollage!"); }
    public void Atterrir() { Altitude = 0; Console.WriteLine("Atterrissage!"); }
}

💡 Règles essentielles : Une interface ne contient PAS d'implémentation (avant C# 8 par défaut, mais on reste sur les bases). Tous les membres sont publics par nature.

🔧 Implémentation explicite

Quand une classe implémente plusieurs interfaces ayant des membres identiques, ou pour masquer un membre, on utilise la syntaxe explicite.

interface IActionHumaine { void Marcher(); }
interface IAnimal { void Marcher(); }

class Homme : IActionHumaine, IAnimal
{
    // Implémentation explicite pour IAnimal
    void IAnimal.Marcher() => Console.WriteLine("L'homme marche comme un animal?");
    
    // Implémentation normale pour IActionHumaine
    public void Marcher() => Console.WriteLine("L'homme marche debout.");
}

// Appel :
Homme homme = new Homme();
homme.Marcher();              // appelle IActionHumaine.Marcher
((IAnimal)homme).Marcher();   // appelle IAnimal.Marcher

⚠️ Les membres implémentés explicitement ne sont pas accessibles directement via l'instance de la classe, seulement via la référence de l'interface.

🔄 Polymorphisme via interfaces

Une interface permet de manipuler des objets de classes différentes de manière uniforme.

Exemple concret :

public interface IPaiement
{
    bool EffectuerPaiement(decimal montant);
}

public class CarteCredit : IPaiement { ... }
public class PayPal : IPaiement { ... }

// Utilisation polymorphique
IPaiement paiement = new CarteCredit();
paiement.EffectuerPaiement(150);

🎯 Avantages :

  • Découplage du code
  • Test unitaire facilité (mock)
  • Évolutivité : ajouter une nouvelle classe sans modifier l'existant
Les interfaces sont au cœur des bonnes pratiques comme l'injection de dépendances.

⚖️ Interface vs Classe abstraite

CritèreInterfaceClasse abstraite
ImplémentationAucune (sauf membres par défaut C# 8+)Peut contenir du code concret
Mots-clésinterfaceabstract class
Héritage multiple✅ Oui (une classe peut implémenter plusieurs interfaces)❌ Non (pas d'héritage multiple de classes)
ChampsNon (uniquement propriétés)Oui
ConstructeurNonOui

✏️ Règle pratique : Utilisez une interface quand vous voulez définir un contrat commun à des classes sans hiérarchie. Préférez une classe abstraite quand vous partagez du code commun ou un état (champs).

🏗️ Cas concrets et bonnes pratiques

✅
Nommer avec préfixe "I" (ex: IRepository, ILogger) pour les distinguer.
✅
Principe de ségrégation (ISP) : Préférez plusieurs petites interfaces qu'une seule grosse (interface "fourre-tout").
✅
Interfaces explicites pour éviter les collisions lorsque des noms de méthodes coïncident.

📌 Exemple métier (API de gestion d'étudiants) :

public interface IEtudiantService
{
    IEnumerable<Etudiant> ObtenirTous();
    Etudiant ObtenirParId(int id);
    void Ajouter(Etudiant e);
}

// Implémentation réelle avec base de données
public class EtudiantService : IEtudiantService { ... }

👉 Cela permet de changer facilement d'implémentation (mock pour tests, version fichier, version SQL...)
