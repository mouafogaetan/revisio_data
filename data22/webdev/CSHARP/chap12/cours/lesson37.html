🏗️ Structure d'un contrôleur API

Dans ASP.NET Core, un contrôleur d’API est une classe qui hérite de ControllerBase. On utilise deux attributs essentiels :

using Microsoft.AspNetCore.Mvc;

[ApiController]           // Active des comportements automatiques (validation, etc.)
[Route("api/[controller]")] // Exemple: /api/products
public class ProductsController : ControllerBase
{
    // Actions (méthodes HTTP)
}

📌 [Route] : définit le modèle d'URL. [controller] est remplacé par le nom de la classe sans "Controller".

📌 [ApiController] : rend les erreurs de modèle automatiques, lie les paramètres depuis le corps de la requête, etc.

🌐 Requête HTTP → https://monapi.com/api/products/5 →
Controller: ProductsController → Action: GetProduct(int id)

📡 Les verbes HTTP essentiels

VerbeAttributObjectif
GET[HttpGet]Récupérer des ressources (lecture)
POST[HttpPost]Créer une nouvelle ressource
PUT[HttpPut]Mettre à jour complètement une ressource
DELETE[HttpDelete]Supprimer une ressource

✨ Exemple typique CRUD sur /api/products :

  • GET /api/products → récupère tous les produits
  • GET /api/products/42 → récupère le produit id=42
  • POST /api/products → crée un nouveau produit (données dans le body)
  • PUT /api/products/42 → remplace le produit 42
  • DELETE /api/products/42 → supprime le produit 42

🎯 Paramètres : FromBody, FromQuery, FromRoute

Pour récupérer les données envoyées par le client, on utilise des attributs de liaison :

[FromRoute]

Capture une valeur depuis le template de l'URL.

[HttpGet("{id}")]
public IActionResult GetById([FromRoute] int id) { ... }
[FromQuery]

Lit les paramètres de la chaîne de requête (?page=2&size=10).

[HttpGet]
public IActionResult List([FromQuery] int page = 1, [FromQuery] string search = "") { ... }
[FromBody]

Lit les données du corps de la requête (JSON, XML) pour les verbes POST/PUT.

[HttpPost]
public IActionResult Create([FromBody] ProductDto product) { ... }
⚠️ Pour les API modernes, [ApiController] infère automatiquement [FromBody] pour les types complexes en POST, mais expliciter est toujours plus clair.

💻 Exemple complet d’un contrôleur produit

[ApiController]
[Route("api/products")]
public class ProductsController : ControllerBase
{
    private static List<Product> _products = new();
    private static int _nextId = 1;

    // GET: api/products
    [HttpGet]
    public IActionResult GetAll() => Ok(_products);

    // GET: api/products/5
    [HttpGet("{id}")]
    public IActionResult GetById([FromRoute] int id)
    {
        var product = _products.FirstOrDefault(p => p.Id == id);
        return product is null ? NotFound() : Ok(product);
    }

    // POST: api/products
    [HttpPost]
    public IActionResult Create([FromBody] ProductCreateDto dto)
    {
        var product = new Product { Id = _nextId++, Name = dto.Name, Price = dto.Price };
        _products.Add(product);
        return CreatedAtAction(nameof(GetById), new { id = product.Id }, product);
    }

    // PUT: api/products/5
    [HttpPut("{id}")]
    public IActionResult Update([FromRoute] int id, [FromBody] ProductUpdateDto dto)
    {
        var product = _products.FirstOrDefault(p => p.Id == id);
        if (product is null) return NotFound();
        product.Name = dto.Name;
        product.Price = dto.Price;
        return NoContent();
    }

    // DELETE: api/products/5
    [HttpDelete("{id}")]
    public IActionResult Delete([FromRoute] int id)
    {
        var removed = _products.RemoveAll(p => p.Id == id);
        return removed > 0 ? NoContent() : NotFound();
    }
}

✨ Résultat : une API REST complète sans base de données (simulation mémoire).

🧪 Tester facilement avec Swagger

ASP.NET Core intègre Swagger (OpenAPI) qui génère automatiquement une documentation interactive. Ajoutez simplement le package Swashbuckle et tout devient testable dans le navigateur.

📄 Swagger UI - Products API
GET /api/products
→ Essayez : [Execute] → Réponse 200 avec liste JSON
POST /api/products
Body : { "name": "Livre C#", "price": 29.99 } → 201 Created
DELETE /api/products/{id}

📌 En production, on peut aussi utiliser Postman, Insomnia ou nos propres tests. Mais Swagger facilite la compréhension pour toute l'équipe.

✅ Bonnes pratiques & pièges à éviter

  • ✔️ Toujours utiliser [ApiController] sur vos contrôleurs d’API.
  • ✔️ Nommer vos actions avec des noms explicites ou utiliser les attributs [HttpGet(Name = "...")].
  • ✔️ Pour les paramètres optionnels, privilégier [FromQuery] et donner des valeurs par défaut.
  • ❌ Ne pas confondre [FromBody] avec plusieurs paramètres complexes (un seul FromBody par action).
  • ❌ Éviter d'envoyer des données sensibles en [FromQuery] (préférer le body ou les en-têtes).
  • 📌 Pour les routes personnalisées : [HttpGet("category/{cat}")] surcharge la route de base.
💡 Rappel : Les contrôleurs web API n’héritent pas de Controller (qui est pour les vues MVC) mais de ControllerBase.

📚 Exercice pratique

Objectif : Créez un mini contrôleur "TodoItems".

1. Définissez un modèle Todo (Id, Titre, Terminé).
2. Implémentez les endpoints suivants :

  • GET /api/todos → retourne tous les todos
  • GET /api/todos/{id} → todo par id
  • POST /api/todos → crée un todo (FromBody)
  • PUT /api/todos/{id} → modifie un todo existant (FromRoute, FromBody)
  • DELETE /api/todos/{id} → supprime
🔧 Pour tester, utilisez Swagger ou un outil comme curl. Envoyez un JSON valide.
