💡 Pourquoi Code First ? Vous écrivez vos classes C# d'abord, EF Core génère automatiquement la base de données. Idéal pour développer rapidement.

🔧 Qu'est-ce qu'Entity Framework Core ?

EF Core est un ORM (Object-Relational Mapping) qui permet de manipuler une base de données avec du C#. Fini les requêtes SQL écrites à la main !

📦 Installation (dans un projet .NET) :

dotnet add package Microsoft.EntityFrameworkCore
dotnet add package Microsoft.EntityFrameworkCore.SqlServer
dotnet add package Microsoft.EntityFrameworkCore.Tools
# Pour SQLite (léger) :
dotnet add package Microsoft.EntityFrameworkCore.Sqlite

💡 Pour les tests, SQLite est parfait. En production, SQL Server / PostgreSQL.

📦 Étape 1 : Créer nos modèles

Imaginons une application de blog. Nous écrivons les classes C# classiques :

// Modèle Article
public class Article
{
   public int Id { get; set; }
   public string Titre { get; set; }
   public string Contenu { get; set; }
   public DateTime DatePublication { get; set; }
   public int AuteurId { get; set; }
   public Auteur Auteur { get; set; }
}

// Modèle Auteur
public class Auteur
{
   public int Id { get; set; }
   public string Nom { get; set; }
   public List<Article> Articles { get; set; }
}

✨ La clé primaire par convention est la propriété Id ou ArticleId. Les relations sont déduites par EF Core.

🔌 DbContext : le cœur de la communication

La classe DbContext représente la session avec la base de données.

using Microsoft.EntityFrameworkCore;

public class BlogDbContext : DbContext
{
   public DbSet<Article> Articles { get; set; }
   public DbSet<Auteur> Auteurs { get; set; }

   protected override void OnConfiguring(DbContextOptionsBuilder options)
   {
      options.UseSqlite("Data Source=blog.db");
   }
}
DbSet<T> → représente une table
OnConfiguring → chaîne de connexion

🔄 Migrations : versionner le schéma

Les migrations permettent de mettre à jour la base de données à partir des modifications de vos classes.

# 1. Créer une migration (basée sur les changements)
dotnet ef migrations add InitialCreate

# 2. Appliquer la migration à la base de données
dotnet ef database update

# 3. (Optionnel) Annuler / modifier
dotnet ef migrations remove

💡 Bon à savoir : Chaque migration est un fichier C# contenant les instructions Up() et Down(). Vous pouvez retrouver l'historique dans la table __EFMigrationsHistory.

💻 Code First en action

Une fois les migrations appliquées, on peut interagir avec la base :

using (var context = new BlogDbContext())
{
   // Ajouter un auteur
   var auteur = new Auteur { Nom = "Alice" };
   context.Auteurs.Add(auteur);
   context.SaveChanges();

   // Lire des articles
   var articles = context.Articles
      .Where(a => a.Titre.Contains("C#"))
      .ToList();
}

✨ Nuance importante : Code First n'exige pas d'écrire manuellement le SQL. EF Core traduit vos requêtes LINQ en commandes SQL optimales.

📌 Récapitulons la démarche Code First

  1. ✔️ Création des classes d'entités (POCO).
  2. ✔️ Création d'un DbContext avec les DbSet<T>.
  3. ✔️ Configuration de la chaîne de connexion (appsettings.json ou OnConfiguring).
  4. ✔️ Ajout d'une migration via dotnet ef migrations add NomMigration.
  5. ✔️ Mise à jour de la base avec dotnet ef database update.
  6. ✔️ La base est synchronisée, on peut requêter/modifier !

⚠️ Pièges à éviter : oublier d'installer les outils EF, ne pas propager les migrations en équipe, ne pas gérer les conflits de migrations.

💡 Indice : Pour la relation many-to-many, EF Core 5+ permet de créer une classe de jointure ou de laisser EF l’inférer.

// Dans DbContext :
modelBuilder.Entity<Etudiant>()
   .HasMany(e => e.Cours)
   .WithMany(c => c.Etudiants);
// Puis migration Add-Migration ManyToMany
