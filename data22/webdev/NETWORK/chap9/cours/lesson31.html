Initiation aux Réseaux
Lesson 31 : Outils de diagnostic réseau

🛠️ Outils de diagnostic réseau

netstat, ipconfig, ifconfig, nslookup, dig

Les commandes essentielles pour analyser, diagnostiquer et dépanner un réseau

Connexions réseau Interfaces IP Résolution DNS

🎯 Objectifs de la leçon

Utiliser netstat pour afficher les connexions et ports
Utiliser ipconfig (Windows) / ifconfig (Linux) pour les interfaces réseau
Utiliser nslookup et dig pour interroger le DNS
Interpréter les résultats pour diagnostiquer des pannes
💡 Ces commandes sont disponibles sur la plupart des systèmes d'exploitation (Windows, Linux, macOS).

🔌 netstat : afficher les connexions réseau

netstat (Network Statistics) affiche les connexions actives, ports d'écoute et statistiques.

# Connexions actives (simplifié) netstat -an # Connexions avec résolution de noms (plus lent) netstat -a # Afficher uniquement les ports d'écoute netstat -an | findstr LISTENING (Windows) netstat -an | grep LISTEN (Linux) # Statistiques par protocole netstat -s # Afficher le PID et le programme associé (Windows) netstat -ano # Afficher la table de routage netstat -r
Interprétation : LISTENING = port ouvert en attente, ESTABLISHED = connexion active, TIME_WAIT = connexion fermée en attente.

🖥️ ipconfig (Windows) / ifconfig (Linux/macOS)

Affiche la configuration des interfaces réseau (adresses IP, masque, passerelle, DNS).

Windows (ipconfig)
ipconfig # configuration de base ipconfig /all # configuration détaillée (MAC, DHCP, DNS) ipconfig /release # libérer l'adresse DHCP ipconfig /renew # renouveler l'adresse DHCP ipconfig /flushdns # vider le cache DNS ipconfig /displaydns # afficher le cache DNS
Linux/macOS (ifconfig)
ifconfig # interfaces actives ifconfig -a # toutes les interfaces ifconfig eth0 # interface spécifique sudo ifconfig eth0 up # activer sudo ifconfig eth0 down # désactiver ip addr show # alternative moderne (iproute2)
Sur Linux moderne, la commande ip addr remplace avantageusement ifconfig.

📊 Exemple d'interprétation

Carte Ethernet Ethernet0 : Adresse IPv4. . . . . . . . . . : 192.168.1.25 Masque de sous-réseau. . . . . : 255.255.255.0 Passerelle par défaut . . . . . : 192.168.1.1 Serveur DHCP . . . . . . . . . : 192.168.1.1 Serveurs DNS . . . . . . . . . : 8.8.8.8 1.1.1.1

🔹 Adresse IP : 192.168.1.25 → Réseau local classe C

🔹 Masque : 255.255.255.0 (/24) → réseau de 254 hôtes max

🔹 Passerelle : 192.168.1.1 → accès vers Internet

🔹 DNS : 8.8.8.8 (Google) → résolution de noms

🔹 Si l'adresse commence par 169.254.x.x → échec DHCP (APIPA)

🔹 Si pas de passerelle → pas d'accès Internet

🔍 nslookup : requêtes DNS

nslookup (Name Server Lookup) interroge les serveurs DNS pour résoudre noms de domaine.

# Résolution simple nslookup google.com # Résolution inversée (IP → nom) nslookup 8.8.8.8 # Changer de serveur DNS nslookup google.com 1.1.1.1 # Mode interactif nslookup > set type=MX > gmail.com > exit # Afficher l'enregistrement A (IPv4) nslookup -type=A google.com
nslookup ignore le cache local et interroge directement le DNS.

🕵️ dig : requêtes DNS détaillées

dig (Domain Information Groper) est plus puissant que nslookup, disponible sur Linux/macOS.

# Requête standard dig google.com # Uniquement l'adresse IP (short answer) dig google.com +short # Enregistrement MX (mail exchange) dig gmail.com MX # Enregistrement TXT (SPF, DKIM) dig google.com TXT # Requête inversée dig -x 8.8.8.8 # Suivre les redirections CNAME dig www.facebook.com +trace
dig affiche le temps de réponse (Query time), le serveur utilisé, et les sections complètes (QUESTION, ANSWER, AUTHORITY, ADDITIONAL).

📖 Lecture d'une réponse dig

;; QUESTION SECTION: ;google.com. IN A ;; ANSWER SECTION: google.com. 300 IN A 142.250.185.46 ;; Query time: 23 msec ;; SERVER: 192.168.1.1#53(192.168.1.1)

🔹 QUESTION SECTION : la requête posée (google.com, type A)

🔹 ANSWER SECTION : la réponse (IP 142.250.185.46, TTL 300 secondes)

🔹 Query time : temps de réponse (23 ms)

🔹 SERVER : serveur DNS interrogé

🔹 TTL (Time To Live) : durée de mise en cache de la réponse

🔧 Cas pratique : diagnostiquer une panne

Un utilisateur n'arrive pas à accéder à www.exemple.com. Voici la marche à suivre :

  1. ipconfig /all → vérifier IP, masque, passerelle, DNS
  2. ping 8.8.8.8 → tester connectivité Internet (DNS pas encore testé)
  3. ping www.exemple.com → tester résolution de nom
  4. nslookup www.exemple.com → vérifier réponse DNS
  5. tracert www.exemple.com → tracer le chemin
  6. netstat -an → vérifier connexions actives
Si ping IP fonctionne mais pas ping nom → problème DNS. Si ping passerelle fonctionne mais pas ping 8.8.8.8 → problème routeur/accès Internet.

📌 Récapitulatif

netstat : affiche connexions, ports ouverts, table de routage
ipconfig/ifconfig : configuration IP, MAC, DHCP, DNS
nslookup : requêtes DNS simples (interactif possible)
dig : requêtes DNS détaillées (Linux/macOS)
Dépannage systématique : ipconfig → ping → nslookup → tracert
