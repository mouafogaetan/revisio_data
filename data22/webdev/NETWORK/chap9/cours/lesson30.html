🎯 Objectifs pédagogiques

  • ✓ Utiliser ping pour tester la connectivité (ICMP echo)
  • ✓ Utiliser traceroute (tracert) pour tracer le chemin
  • ✓ Interpréter les résultats (RTT, pertes de paquets, * * *)

💡 ping et traceroute sont les couteaux suisses du technicien réseau. Ils permettent de diagnostiquer rapidement une panne.

📡 Le protocole ICMP

ICMP (Internet Control Message Protocol) est un protocole de la couche réseau (niveau 3) utilisé pour les messages d'erreur et de diagnostic.

Types de messages ICMP courants :

  • Type 8 / Code 0 : Echo Request (requête ping)
  • Type 0 / Code 0 : Echo Reply (réponse ping)
  • Type 3 / Code 3 : Destination unreachable (port inaccessible)
  • Type 11 / Code 0 : Time Exceeded (TTL expiré)
$ ping 8.8.8.8
PING 8.8.8.8 (8.8.8.8) 56(84) bytes of data.
64 bytes from 8.8.8.8: icmp_seq=1 ttl=117 time=12.5 ms
64 bytes from 8.8.8.8: icmp_seq=2 ttl=117 time=11.8 ms
--- 8.8.8.8 ping statistics ---
2 packets transmitted, 2 received, 0% packet loss
rtt min/avg/max/mdev = 11.8/12.15/12.5/0.35 ms

🏓 ping : tester la connectivité

Syntaxe : ping [options] destination

📌 Options utiles (Windows/Linux)

  • -c 4 (Linux) : 4 requêtes
  • -n 4 (Windows) : 4 requêtes
  • -t (Windows) : ping en continu (Ctrl+C arrête)
  • -s 1400 : taille du paquet
  • -i 2 : intervalle 2 secondes

📊 Ce qu'il faut regarder

  • 0% loss : tout va bien
  • Pertes partielles : saturation ou liens instables
  • 100% loss : coupure ou pare-feu bloque ICMP
  • RTT élevé > 200 ms : latence (satellite, congestion)
⚠️ Certains administrateurs bloquent ICMP sur leur pare-feu. Dans ce cas, ping échoue même si le serveur est accessible (HTTP, SSH fonctionnent).

📈 Analyser un ping

Message / résultatInterprétation
Reply from ... time=10ms✅ Connexion OK, latence normale (réseau local/câblé)
time=150-300ms⚠️ Latence élevée (connexion satellite, encombrement, distance)
Request timed out❌ Pas de réponse (panne, pare-feu, hôte éteint)
Destination unreachable❌ Routeur ne sait pas joindre la destination (problème routage)
TTL expired in transit❌ Boucle de routage ou TTL initial trop petit
Packet loss 20%⚠️ Pertes intermittentes (WiFi bruité, saturation, câble défectueux)

TTL (Time To Live) : nombre maximal de sauts. Sa valeur initiale dépend de l'OS (Linux 64, Windows 128). À chaque routeur, TTL décrémenté de 1.

🗺️ traceroute : cartographier le chemin

traceroute (Linux/macOS) ou tracert (Windows) affiche chaque routeur traversé entre la source et la destination.

$ traceroute google.cm
1 192.168.1.1 (192.168.1.1) 2.5 ms 1.8 ms 2.1 ms
2 10.0.0.1 (10.0.0.1) 15.2 ms 14.9 ms 15.5 ms
3 41.204.xxx.xxx 32.4 ms 31.8 ms 32.1 ms
4 72.14.xxx.xxx (Google) 45.2 ms 44.8 ms 45.0 ms
5 google.cm (216.58.xxx.xxx) 46.1 ms 45.5 ms 45.9 ms

Principe : envoie des paquets avec TTL=1, puis TTL=2, etc. Chaque routeur répond "Time Exceeded" quand TTL expire.

🔍 Interpréter un traceroute

✅ Sauts normaux : temps de réponse cohérents (augmentation progressive).
⚠️ * * * (timeout) : routeur ne répond pas aux probes ICMP (bloqué par pare-feu). Cela ne signifie pas forcément une panne.
❌ Boucle de routage : mêmes adresses IP apparaissent plusieurs fois en boucle. Le paquet ne parvient jamais à destination.
📈 Latence qui augmente brutalement : goulot d'étranglement (liaison satellite ou transcontinentale).

Astuce :

Pour identifier le propriétaire d'une IP intermédiaire, utilisez whois ou des sites comme bgp.he.net.

🏥 Étude de cas : panne Internet

Un utilisateur à Douala n'arrive plus à accéder à www.camwater.cm. Voici le traceroute :

1 192.168.1.1 2ms
2 10.20.1.1 15ms
3 * * *
4 * * *
5 196.xxx.xxx.1 150ms
6 ...
... temps de réponse normaux jusqu'à destination

Diagnostic :

Les sauts 3 et 4 ne répondent pas aux probes ICMP (pare-feu). Mais le saut 5 répond. La panne n'est pas au niveau de ces routeurs ; le problème est ailleurs (peut-être DNS ou serveur web).

💡 Un traceroute avec des "*" au milieu n'indique pas forcément une panne. Il faut regarder si les sauts suivants répondent.

🧪 Exercice pratique

  1. Ouvrez un terminal (cmd sur Windows, bash sur Linux/macOS).
  2. Pingez votre passerelle par défaut (ex: 192.168.1.1). Notez le RTT et le taux de perte.
  3. Pingez 8.8.8.8 (Google DNS). Comparez les RTT.
  4. Lancez tracert google.cm (ou traceroute). Combien de sauts ?
  5. Testez avec ping -c 10 google.cm (Linux) ou ping -n 10 google.cm (Windows).
📌 Question réflexe : Si ping 8.8.8.8 réussit mais ping google.com échoue, quel est le problème probable ?

📝 À retenir

  • ping : test de base de connectivité (ICMP Echo). Répond "Réponse de ..." ou "Délai d'attente dépassé".
  • traceroute/tracert : liste les routeurs intermédiaires grâce au champ TTL.
  • Des pertes de paquets ou des "*" ne sont pas toujours synonymes de panne (pare-feu ICMP).
  • Une latence élevée peut venir de la distance géographique, d'une congestion ou d'une liaison satellite.
  • Les outils en ligne (mtr, WinMTR, PingPlotter) combinent ping + traceroute en continu.
