Terraform : Infrastructure as Code (IaC)

Automatisez et versionnez votre infrastructure cloud comme vous le faites pour votre code applicatif.

📘 Module: Automatisation du provisionnement · 🎓 Niveau: Intermédiaire

🎯 Objectifs de la leçon

  • ✓ Installer Terraform et comprendre le workflow de base (init, plan, apply, destroy).
  • ✓ Écrire des fichiers .tf avec providers, ressources et variables.
  • ✓ Gérer l'état (terraform.tfstate) et utiliser des backends distants (local, S3, Azure).
  • ✓ Appliquer les bonnes pratiques pour une infrastructure reproductible.

🏗️ Infrastructure as Code (IaC)

Concept clé

L'IaC consiste à gérer et provisionner l'infrastructure (serveurs, bases de données, réseaux, etc.) à l'aide de fichiers de configuration lisibles, plutôt qu'avec des clics manuels dans une interface.

✅ Bénéfices

  • Reproductibilité absolue
  • Versionnage dans Git
  • Réduction des erreurs humaines
  • Déploiement rapide et fiable

⚖️ Déclaratif vs Impératif

Terraform (déclaratif) : on décrit l’état final, il trouve le chemin.

Ansible (impératif) : on décrit les étapes. Terraform excelle pour la création de ressources cloud.

📥 Installation + Workflow de base

# Linux / Mac / WSL : Télécharger depuis https://terraform.io/downloads

wget https://releases.hashicorp.com/terraform/1.7.0/terraform_1.7.0_linux_amd64.zip
unzip terraform_1.7.0_linux_amd64.zip
sudo mv terraform /usr/local/bin/

Vérifier : terraform --version

⚙️ Workflow Terraform (4 commandes essentielles)

  1. terraform init → initialise le répertoire, télécharge les providers
  2. terraform plan → montre les changements à appliquer (dry-run)
  3. terraform apply → applique la configuration pour créer/modifier les ressources
  4. terraform destroy → supprime toutes les ressources gérées

📄 Écrire un fichier Terraform

Exemple main.tf pour créer un fichier local (démonstration simple) :

# main.tf
provider "local" {
  # pas de configuration requise
}

resource "local_file" "example" {
  filename = "${path.module}/hello_terraform.txt"
  content  = "Bienvenue dans l'IaC !"
}

# Avec variables
variable "file_name" {
  default = "exemple.txt"
}

output "file_path" {
  value = local_file.example.filename
}
💡 À retenir : providers (AWS, Azure, GCP, local...), resources (serveurs, stockage), variables et outputs rendent la config réutilisable.

définit la source
🔧 Resource

📌 Variable

📢 Output

🗄️ Gestion d'état (terraform.tfstate)

Le fichier terraform.tfstate contient l'état réel de l'infrastructure. Il est crucial : Terraform s'en sert pour détecter les différences.

⚠️ State local (par défaut)

Fichier dans le répertoire local. Risque de conflit en équipe.

☁️ Backend distant (recommandé)

Stockage dans AWS S3, Azure Storage, Terraform Cloud, etc. Sécurisé et partagé.

# backend.tf (exemple AWS S3)
terraform {
  backend "s3" {
    bucket = "mon-terraform-state"
    key    = "devops/terraform.tfstate"
    region = "us-east-1"
  }
}
🔐 Le state peut contenir des données sensibles; ne pas le versionner dans Git sans backend distant sécurisé !

✨ Astuces & Bonnes pratiques

  • ✔️ Utilisez terraform fmt pour formater vos fichiers .tf proprement.
  • ✔️ Validez avec terraform validate avant de planifier.
  • ✔️ Versionnez vos fichiers .tf dans Git, mais ignorez .terraform/ et .tfstate* (sauf si backend distant).
  • ✔️ Organisez les variables avec des fichiers .tfvars pour les environnements (dev, prod).
  • ✔️ Utilisez des modules pour réutiliser du code Terraform.

📌 Rappel commandes essentielles :
