📄 Qu'est‑ce qu'un Dockerfile ?

Un Dockerfile est un fichier texte contenant une série d’instructions pour construire automatiquement une image Docker. C'est la recette de cuisine de votre application !

Avantages :
  • ✔️ Reproductibilité : l'image est construite de la même manière partout
  • ✔️ Versionnable (on peut mettre le Dockerfile dans Git)
  • ✔️ Automatisation CI/CD : construction d’image sans intervention manuelle
# Exemple très simple de Dockerfile pour une app Node.js
FROM node:18-alpine # image de base
WORKDIR /app # répertoire de travail
COPY package*.json ./ # copier les fichiers de dépendances
RUN npm install # installer les dépendances
COPY . . # copier le code source
EXPOSE 3000 # documenter le port
CMD ["node", "app.js"] # commande par défaut

🧱 Instructions essentielles (1/2)

FROM

Définit l'image de base (ex: alpine, ubuntu, node:18). Toujours en première instruction.

WORKDIR

Définit le répertoire courant pour les instructions suivantes (équivalent de cd).

COPY

Copie des fichiers de l'hôte vers l'image (COPY src/ /app/src/).

ADD

Similaire à COPY mais peut décompresser des archives .tar et accepter des URLs. Préférez COPY par sécurité.

🚀 Bonne pratique : Utiliser COPY plutôt que ADD sauf besoin explicite.

⚙️ Instructions essentielles (2/2)

RUN – Exécute une commande lors de la construction de l'image (ex: RUN apt-get update && apt-get install -y python3).
CMD – Commande par défaut quand on lance le conteneur (peut être écrasée). Un seul CMD par Dockerfile.
ENTRYPOINT – Commande qui ne peut pas être écrasée facilement. Combine souvent avec CMD pour les paramètres.
EXPOSE – Indique le port sur lequel l’application écoute (simple documentation, ne publie pas réellement le port).
ENV / ARG – Définit des variables d’environnement (ENV NODE_ENV=production). ARG est temporaire lors du build.

⚡ Optimisation : Multi‑stage builds

Construire une image finale légère en séparant l’environnement de compilation de l’environnement d’exécution.

# Étape 1 : construction (golang / npm / maven)
FROM golang:1.20 AS builder
WORKDIR /src
COPY . .
RUN go build -o myapp

# Étape 2 : image finale ultra légère
FROM alpine:latest
COPY --from=builder /src/myapp /usr/local/bin/myapp
CMD ["myapp"]
Réduction de taille (Go de 500MB → 20MB) Moins de vulnérabilités
Le multi‑stage est indispensable pour les languages compilés (Go, Rust, Java, .NET)

📦 .dockerignore & bonnes pratiques

# Fichier .dockerignore - à placer à côté du Dockerfile
node_modules/
.git/
.env
*.log
Dockerfile
README.md

Le .dockerignore empêche l’envoi de fichiers inutiles au contexte de build → builds plus rapides et image plus propre.

Chaque instruction RUN doit être regroupée (ex: RUN apt update && apt install -y ...)
Éviter de lancer des processus en arrière‑plan dans le Dockerfile
Utiliser des images officielles *-alpine pour alléger
Ordonner les couches : du moins changeant au plus changeant (bénéficier du cache)

🏁 Récapitulatif & commandes utiles

docker build -t monapp:v1 . # Construire l’image à partir du Dockerfile
docker images # Lister les images
docker run -p 3000:3000 monapp:v1 # Lancer le conteneur
docker history monapp:v1 # Voir les couches de l’image

Félicitations ! Vous savez désormais écrire des Dockerfiles professionnels.

Pratiquez en containerisant vos propres projets (API Node.js, Flask, ou même un script bash).
