💡 Pourquoi centraliser les logs ?
Dans une architecture distribuée (microservices, conteneurs), il devient impossible de consulter les logs un par un. La centralisation permet : recherche rapide, corrélation d'événements, alerting et conformité.

🔍 Qu'est-ce que la stack ELK ?

ELK est l'acronyme de trois projets open source développés par Elastic :

🗄️

Elasticsearch

Moteur de recherche et d'analyse distribué. Stocke et indexe les logs pour des recherches ultra-rapides.

🔄

Logstash

Pipeline de traitement de données. Collecte, transforme et enrichit les logs avant de les envoyer à Elasticsearch.

📈

Kibana

Interface de visualisation. Permet d'explorer, analyser et créer des tableaux de bord à partir des logs.

📌 On parle aussi de Elastic Stack (ELK + Beats)

Beats sont des agents légers (Filebeat, Metricbeat) qui envoient les données à Logstash ou directement à Elasticsearch.

🏗️ Architecture de la stack ELK

Voici comment les données circulent dans une infrastructure typique :

📱 Application 1
🗄️ Base de données
🐳 Conteneur
↓ (logs)
🦐 Filebeat / Logstash (collecte)
↓ (traitement & envoi)
🗄️ Elasticsearch (stockage + indexation)
↓ (requêtes)
📊 Kibana (visualisation)
↓
👨‍💻 Ingénieur DevOps / Admin

Flux de travail :

  1. Les applications génèrent des logs (fichiers, stdout).
  2. Filebeat (agent léger) lit ces logs et les envoie à Logstash.
  3. Logstash transforme, filtre et enrichit les données.
  4. Elasticsearch indexe et stocke les logs.
  5. Kibana interroge Elasticsearch et affiche les résultats.

⚙️ Exemple : configurer Filebeat

Voici un fichier filebeat.yml typique pour envoyer des logs d'une application Node.js :

# filebeat.yml
filebeat.inputs:
- type: log
  enabled: true
  paths:
    - /var/log/myapp/*.log

output.logstash:
  hosts: ["logstash-server:5044"]

setup.kibana:
  host: "kibana-server:5601"

Ensuite, on lance Filebeat :

sudo systemctl start filebeat

✅ Alternative directe :

Filebeat peut envoyer directement à Elasticsearch si Logstash n'est pas nécessaire.

🔧 Traitement des logs avec Logstash

Logstash utilise une configuration avec trois sections : input → filter → output.

input {
  beats { port => 5044 }
}

filter {
  grok {
    match => { "message" => "%{COMBINEDAPACHELOG}" }
  }
  date {
    match => [ "timestamp", "dd/MMM/yyyy:HH:mm:ss Z" ]
  }
}

output {
  elasticsearch {
    hosts => ["localhost:9200"]
    index => "app-logs-%{+YYYY.MM.dd}"
  }
  stdout { codec => rubydebug }
}

📌 Le filtre grok permet d'extraire des champs structurés à partir de logs bruts (exemple : logs Apache).

📊 Kibana : le tableau de bord des logs

Kibana offre plusieurs fonctionnalités puissantes :

🔎 Discover

Recherche interactive, filtres, affichage des logs bruts ou structurés.

📈 Dashboard

Créer des visualisations (graphiques, camemberts, jauges) et les regrouper.

⚡ Alertes

Définir des conditions (ex: 500 erreurs > 10 par minute) et déclencher des notifications.

📋 Maps

Géolocaliser des adresses IP extraites des logs.

Exemple de requête Kibana (Lucene) :
status:500 AND service:"api-gateway" → Affiche toutes les erreurs 500 du service API.

🛠️ Mise en pratique simplifiée

Pour tester ELK en local, vous pouvez utiliser Docker Compose :

version: '3'
services:
  elasticsearch:
    image: docker.elastic.co/elasticsearch/elasticsearch:8.10.0
    environment:
      - discovery.type=single-node
    ports:
      - "9200:9200"
  logstash:
    image: docker.elastic.co/logstash/logstash:8.10.0
    ports:
      - "5044:5044"
  kibana:
    image: docker.elastic.co/kibana/kibana:8.10.0
    ports:
      - "5601:5601"
    environment:
      - ELASTICSEARCH_HOSTS=http://elasticsearch:9200

📌 Commandes utiles :
- docker-compose up -d → Lance les 3 services
- Accéder à Kibana : http://localhost:5601
- Envoyer des logs de test : curl -X POST localhost:9200/logs/_doc -H 'Content-Type: application/json' -d '{"message":"test log"}'

📝 Résumé & bonnes pratiques

  • Centraliser tous les logs (applications, système, bases de données, conteneurs).
  • Structurer les logs (format JSON) pour faciliter l'indexation et la recherche.
  • Utiliser des index rotatifs (par jour) pour la gestion de la rétention.
  • Mettre en place des alertes sur les erreurs critiques ou les anomalies.
  • Sécuriser ELK : authentification, TLS, contrôle d'accès (RBAC).
  • Surveiller les performances d'Elasticsearch (heap memory, node stats).

🎯 À retenir :

ELK transforme une masse de logs illisible en données exploitables pour le débogage, la sécurité et l'optimisation des performances.
