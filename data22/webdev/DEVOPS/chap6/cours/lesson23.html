📦 Volumes et persistance des données dans Kubernetes

Gérer les données durables dans un cluster éphémère

🗂️ Leçon 23 ⏱️ Durée : 25 min 🎯 Niveau : Intermédiaire

🎯 Objectifs de la leçon

  • ✓ Comprendre la problématique de la persistance : pourquoi les données des pods sont éphémères.
  • ✓ Distinguer les types de volumes : emptyDir, hostPath, et volumes persistants.
  • ✓ Maîtriser les concepts clés : PersistentVolume (PV), PersistentVolumeClaim (PVC), StorageClass.
  • ✓ Savoir monter un volume persistant dans un pod et comprendre le provisionnement dynamique.
  • ✓ Connaître les bonnes pratiques pour la gestion des données en production.

📌 Problématique : Pods éphémères

Dans Kubernetes, un pod peut être recréé à tout moment (mise à jour, scaling, panne d’un nœud). Les données stockées localement dans le conteneur sont alors perdues.

⚠️ Exemple concret : Une base de données MySQL ou des fichiers uploadés par les utilisateurs ne doivent pas disparaître après un redémarrage. D’où la nécessité des volumes persistants.

# Un pod sans volume persistant — les données sont perdues si le pod est supprimé
apiVersion: v1
kind: Pod
metadata:
  name: mon-app
spec:
  containers:
  - name: app
    image: nginx
    # Pas de volume → le stockage est éphémère

🧩 Types de volumes standards

📁 emptyDir

Volume temporaire partagé entre les conteneurs d’un même pod. Les données sont supprimées quand le pod est détruit. Idéal pour des caches ou traitements temporaires.

💾 hostPath

Monte un répertoire du nœud dans le pod. Peu recommandé en production (dépend du nœud, problèmes de scheduling). Utile pour du développement ou accès à des fichiers système.

# Exemple emptyDir
volumes:
- name: cache-vol
  emptyDir: {}
# Exemple hostPath
volumes:
- name: host-data
  hostPath:
    path: /data/host

🔐 PersistentVolume (PV) et PersistentVolumeClaim (PVC)

PV (PersistentVolume) : Ressource de stockage provisionnée par l’administrateur du cluster (disque réseau, NFS, EBS, etc.). Indépendant des pods.

PVC (PersistentVolumeClaim) : Demande de stockage par un utilisateur ou un pod (taille, mode d’accès). Kubernetes lie le PVC à un PV compatible.

apiVersion: v1
kind: PersistentVolume
metadata:
  name: pv-storage-1
spec:
  capacity: { storage: 10Gi }
  accessModes: [ "ReadWriteOnce" ]
  hostPath: { path: "/mnt/data" }   # (pour test, en production utiliser un stockage réseau)
apiVersion: v1
kind: PersistentVolumeClaim
metadata:
  name: mon-claim
spec:
  accessModes: [ "ReadWriteOnce" ]
  resources: { requests: { storage: 5Gi } }

⚙️ Monter un PVC dans un Pod

On référence le PVC dans la section volumes du Pod. Kubernetes monte automatiquement le volume correspondant.

apiVersion: v1
kind: Pod
metadata:
  name: app-avec-stockage
spec:
  containers:
  - name: app
    image: nginx
    volumeMounts:
    - mountPath: "/usr/share/nginx/html"
      name: data-vol
  volumes:
  - name: data-vol
    persistentVolumeClaim:
      claimName: mon-claim   # Nom du PVC
✅ Avantage : Le Pod n’a pas besoin de connaître les détails du stockage (type, IP, chemin). La persistance est garantie même si le pod est recréé.

⚡ StorageClass & Provisionnement dynamique

Une StorageClass permet de créer automatiquement des PV à la demande lorsqu’un PVC est soumis. Le cloud (AWS EBS, Azure Disk, GCE PD) ou des solutions comme NFS peuvent être utilisés.

apiVersion: storage.k8s.io/v1
kind: StorageClass
metadata:
  name: fast-storage
provisioner: kubernetes.io/aws-ebs
parameters:
  type: gp3

Ainsi, sans créer manuellement de PV, un PVC avec storageClassName: fast-storage déclenche la création automatique d’un volume.

💡 Production tip : Pour MySQL, MongoDB, ou toute application avec état, utilisez toujours des PVC + StorageClass pour garantir la persistance et la portabilité.

📝 Bonnes pratiques & Récapitulatif

  • Ne stockez jamais de données d’état dans le conteneur lui-même → utilisez toujours un volume.
  • Préférez les PVC et StorageClass aux PV manuels pour l’agilité.
  • Pour les bases de données en production, utilisez des StatefulSets avec des PVC dédiés.
  • Surveillez l’utilisation des volumes (Prometheus + kubelet metrics).
  • Pensez aux sauvegardes des volumes persistants (snapshots, Velero).

🧠 À retenir :

Un Pod sans volume persistant = perte de données assurée après redémarrage. Un PVC + StorageClass = solution professionnelle pour les applications dites “stateful”.

🎉 Félicitations !

Vous maîtrisez maintenant les volumes persistants dans Kubernetes.

✔ PersistentVolume (PV) : stockage physique du cluster.

✔ PersistentVolumeClaim (PVC) : demande de ressources par l’application.

✔ StorageClass : provisionnement automatique à la demande.
