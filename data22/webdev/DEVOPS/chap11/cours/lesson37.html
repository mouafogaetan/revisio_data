Étape 1 — Contexte

📌 1. Choisir une application à déployer

Avant toute automatisation, on sélectionne l'application cible. Idéalement une app web simple avec base de données.

✨ Exemples d'applications adaptées :

Node.js + Express Python Flask / Django .NET Core API Java Spring Boot

💡 L'important : l'application doit pouvoir être containerisée (Docker) et versionnée dans Git.

✅ Critères de choix
  • Stack légère pour les démos
  • Code source ouvert ou exemple pédagogique
  • Présence de tests unitaires (intégration CI)
  • Utilisation de variables d'environnement
📦 Notre choix pour le projet final

Application “Task Manager” en Node.js avec base de données MongoDB. Code hébergé sur GitHub, idéal pour démonstration CI/CD.

Architecture DevOps

🏗️ 2. Définir l'architecture (Git, CI/CD, Docker, K8s, monitoring)

Un schéma simple mais complet des briques essentielles.

+------------+      +------------------+      +-----------------+
|            |      |  CI/CD Pipeline   |      |   Container     |
|   Git      | ---> | GitHub Actions    | ---> |   Registry      |
| (GitHub)   |      | ou GitLab CI      |      |  (Docker Hub)   |
+------------+      +------------------+      +-----------------+
                                   |                     |
                                   v                     v
                          +------------------+    +-----------------+
                          |   Build & Test    |    |  Orchestration  |
                          |  (npm test,       |    |   Kubernetes    |
                          |   SonarQube)      |    | (Minikube/Cloud)|
                          +------------------+    +-----------------+
                                                            |
                                                            v
                                               +------------------------+
                                               |  Déploiement + Ingress |
                                               |  (Service, LoadBalancer)|
                                               +------------------------+
                                                            |
                                                            v
                                              +----------------------------+
                                              |  Monitoring & Logs         |
                                              |  Prometheus + Grafana + ELK|
                                              +----------------------------+

🔁 Flux détaillé :

  1. Développeur push sur branche principale → déclenche CI (tests, build image Docker).
  2. Image poussée vers un registry (Docker Hub / GHCR).
  3. CD déploie automatiquement l'image sur Kubernetes (via kubectl ou helm).
  4. Ingress expose l'application, Prometheus collecte les métriques, Grafana visualise.
☁️ Infrastructure as Code

3. Planifier l'infrastructure cloud avec Terraform

On définit de manière déclarative nos ressources (VM, VPC, Kubernetes cluster, base de données).

🌱 Exemple de plan Terraform (AWS)
provider "aws" {
  region = "eu-west-3"
}

resource "aws_vpc" "devops_vpc" {
  cidr_block = "10.0.0.0/16"
}

resource "aws_eks_cluster" "devops_cluster" {
  name     = "devops-cluster"
  role_arn = aws_iam_role.eks_role.arn
  vpc_config {
    subnet_ids = aws_subnet.public[*].id
  }
}
// etc...

(fichier .tf) versionné dans Git

🧩 Avantages de Terraform ici :

  • Reproductibilité à l'infini
  • Planification "terraform plan" avant tout changement
  • Gestion d'état (state) pour suivre les ressources
  • Modules réutilisables (cluster K8s, bases de données)
➕ On combine Terraform + Ansible pour la configuration.

📌 À retenir : La planification inclut le choix du cloud provider (AWS, Azure, GCP) et la définition des ressources réseau (VPC, sous-réseaux, groupes de sécurité). On prépare aussi le stockage des état Terraform (backend S3).
