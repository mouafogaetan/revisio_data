Build & Variables d'environnement

Configurer les builds (dev, staging, prod), variables d'environnement (.env) et optimisations (minification, tree-shaking)

🚀 Build optimisé 🔐 Variables sécurisées 🌳 Tree shaking

  • ✓ Configurer différents environnements (development, staging, production) avec Vite

🌍

Environnements : dev / staging / prod

On distingue généralement trois environnements :

  • development : travail local, hot-reload, logs détaillés
  • staging : pré-production, tests, similaire à la prod
  • production : build optimisé, minifié, sans outils de debug
// Dans package.json
"scripts": {
  "dev": "vite",
  "build:staging": "vite build --mode staging",
  "build": "vite build",        // production par défaut
  "preview": "vite preview"
}
📌 Avec Vite, le mode est défini par --mode. Chaque mode charge son fichier .env.[mode].
🔐

Fichiers .env et variables

Créer des fichiers à la racine :

.env                # chargé partout
.env.development    # chargé en mode dev
.env.staging        # mode staging
.env.production     # mode production

Dans ces fichiers :

VITE_API_URL=https://api.exemple.com
VITE_APP_TITLE="Mon E-commerce Cameroun"
VITE_GOOGLE_CLIENT_ID=xxxxx.apps.googleusercontent.com

# Accessible dans le code via import.meta.env.VITE_API_URL
⚠️ Ne jamais stocker de secrets (clés privées) côté frontend ! Les variables VITE_* sont exposées dans le bundle. Pour les clés API sensibles, utilisez un backend.
✂️

Optimisations : minification & tree shaking

Minification : suppression des espaces, renommage des variables, code compact.

Tree shaking : élimination du code mort (non utilisé) lors du build.

// Vite applique automatiquement ces optimisations en production
// On peut personnaliser dans vite.config.js
export default defineConfig({
  build: {
    minify: 'terser', // ou 'esbuild'
    target: 'modules',
    sourcemap: false,
    rollupOptions: {
      output: {
        manualChunks: { // code splitting
          vendor: ['react', 'react-dom'],
        }
      }
    }
  }
})
🌳 Tree shaking : import { useMemo } from 'react' ne garde que useMemo
📦 Bundle splitting : sépare les librairies tierces
📁

Exemple : configuration multi-environnement

Création de .env.development :

VITE_API_URL=http://localhost:3000/api
VITE_DEBUG=true

.env.production :

VITE_API_URL=https://api.maboutique.cm/api
VITE_DEBUG=false

Utilisation dans le code :

const apiUrl = import.meta.env.VITE_API_URL;
const isDebug = import.meta.env.VITE_DEBUG === 'true';

if (isDebug) console.log('Mode développement');
✨ Astuce : TypeScript peut bénéficier de vite/client pour typer import.meta.env.
📊

Analyser et réduire la taille du bundle

Outils utiles : vite-bundle-visualizer

npm install --save-dev rollup-plugin-visualizer
// vite.config.js
import { visualizer } from 'rollup-plugin-visualizer';
export default {
  plugins: [visualizer({ open: true })],
}
  • Éviter les imports lourds inutiles (ex: import * as lodash → préférer import debounce from 'lodash/debounce')
  • Utiliser le lazy loading pour les routes
  • Compresser les images avant intégration

📌 Synthèse & commandes essentielles

  • Build dev : npm run dev (non optimisé, hot reload)
  • Build production : npm run build (minifié, tree-shaken)
  • Variables d'env : préfixées par VITE_ et accessibles via import.meta.env
  • Mode spécifique : vite build --mode staging charge .env.staging
  • Analyse bundle : plugin visualizer ou npx vite-bundle-visualizer
