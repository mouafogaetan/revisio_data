🎯 Objectifs de la leçon

✔️ Exécuter les tests avant déploiement

Automatiser la vérification du code avec npm test dans GitHub Actions.

✔️ Déployer sur Vercel/Netlify

Utiliser les intégrations officielles ou actions personnalisées.

✔️ Preview deployments (PR)

Générer un environnement de test unique pour chaque pull request.

✔️ Variables d’environnement

Gérer secrets et configuration selon l’environnement (staging/prod).

💡 Pourquoi CI/CD ? Réduire les erreurs, livrer plus vite et en confiance.

🧪 Étape 1 : Tester automatiquement avant déploiement

Dans votre workflow GitHub Actions, on ajoute un job de test qui échoue si un test casse la build.

# .github/workflows/deploy.yml
name: CI/CD Pipeline

on:
  push:
    branches: [ main ]
  pull_request:
    branches: [ main ]

jobs:
  test:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          node-version: 20
      - run: npm ci
      - run: npm run test           # exécute les tests unitaires & intégration
      - run: npm run build          # vérifie que la build passe

  deploy:
    needs: test                     # attend que les tests réussissent
    if: github.ref == 'refs/heads/main'
    runs-on: ubuntu-latest
    steps:
      - name: Déploiement vers Vercel
        run: npx vercel --prod --token=${{ secrets.VERCEL_TOKEN }}
⚠️ Important : Ne déployez jamais une branche qui ne passe pas les tests.

🚀 Étape 2 : Déployer sur Vercel (via GitHub Actions)

Vercel propose une action officielle. Alternative : utiliser l’intégration native Vercel + GitHub (sans écrire de YAML) mais l’action donne plus de contrôle.

# Exemple d'action Vercel
- name: Deploy to Vercel
  uses: amondnet/vercel-action@v25
  with:
    vercel-token: ${{ secrets.VERCEL_TOKEN }}
    vercel-org-id: ${{ secrets.ORG_ID}}
    vercel-project-id: ${{ secrets.PROJECT_ID }}
    vercel-args: '--prod'

🔐 Les secrets (VERCEL_TOKEN, ORG_ID) se paramètrent dans Settings → Secrets and variables → Actions.

✨ Résultat : à chaque push sur main, le site est mis en ligne automatiquement.

👀 Preview deployments (environnements de test)

Pour chaque Pull Request, on génère une URL unique afin de tester la fonctionnalité sans toucher à la production.

  • Netlify : active les "Deploy Previews" automatiquement si le repo est lié.
  • Vercel : idem, chaque PR reçoit un préfixe automatique comme pr-42-mon-site.vercel.app.
  • GitHub Actions personnalisé : on peut utiliser peaceiris/actions-gh-pages ou déployer sur un bucket S3 avec une URL temporaire.
# Exemple de commentaire automatique sur la PR
- name: Comment preview URL
  uses: actions/github-script@v7
  with:
    script: |
      github.rest.issues.createComment({
        issue_number: context.issue.number,
        owner: context.repo.owner,
        repo: context.repo.repo,
        body: '🚀 Preview disponible : https://preview-${{ github.sha }}.netlify.app'
      })

✅ Cela facilite la relecture et les tests par l’équipe.

🔧 Gérer les variables d’environnement

Votre application utilise .env.production ou VITE_API_URL. En CI, injectez-les via les secrets GitHub.

# Dans votre workflow
- name: Build with env
  run: npm run build
  env:
    VITE_API_URL: ${{ secrets.VITE_API_URL }}
    VITE_STRIPE_KEY: ${{ secrets.VITE_STRIPE_KEY }}

# Vercel / Netlify : définissez les variables dans le dashboard du projet
VITE_API_URL=https://api.staging.example.com
Utilisé en staging/preview
VITE_API_URL=https://api.prod.example.com
En production

⚠️ Ne jamais versionner les fichiers .env contenant des secrets.

📝 Exercices d'application

Exercice 1 (compétence – medium)

Écrivez un workflow GitHub Actions qui :

  • Se déclenche sur push sur main et sur pull request
  • Installe les dépendances, lance les tests, puis construit l’application
  • Si la branche est main, déploie sur Netlify en production
🔍 Voir corrigé 
name: CI/CD Netlify
on: [push, pull_request]
jobs:
  test:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - run: npm ci && npm test && npm run build
  deploy:
    needs: test
    if: github.ref == 'refs/heads/main'
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: nwtgck/actions-netlify@v3
        with:
          publish-dir: './dist'
          production-branch: main
          deploy-message: "Deploy from GitHub Actions"
        env:
          NETLIFY_AUTH_TOKEN: ${{ secrets.NETLIFY_TOKEN }}
          NETLIFY_SITE_ID: ${{ secrets.NETLIFY_SITE_ID }}

Exercice 2 (ressource – hard)

Sur Vercel, configurez un preview deployment qui utilise une base de données de test différente. Expliquez les variables nécessaires.

🔍 Voir corrigé

Dans Vercel : Settings → Environment Variables → créer DATABASE_URL pour les environnements Preview, Production et Development. Dans GitHub Actions, lors du déploiement preview, passer --environment=preview et Vercel utilisera automatiquement les bonnes variables. On peut aussi générer une URL dynamique via VERCEL_GIT_COMMIT_REF.

✅ Récapitulatif & bonnes pratiques

  • Séparez les jobs test et deploy pour une exécution claire.
  • Utilisez des secrets pour les tokens d’accès aux plateformes.
  • Activez les preview deployments pour chaque PR et commentez l’URL dans la PR.
  • Ne pas déployer en production si les tests échouent.
  • Surveillez les logs d’actions pour déboguer.
