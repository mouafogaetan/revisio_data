💡 Pourquoi c'est crucial ?
Un catalogue e-commerce peut afficher des centaines de produits. Sans optimisation, chaque recherche ou filtre déclenche des re-renders massifs, rendant l'application lente et peu réactive.

⚠️ Le problème : des re-renders en cascade

Dans React, quand un parent se met à jour, tous ses enfants re-rendent (même si leurs props n'ont pas changé).

function Parent() {
  const [count, setCount] = useState(0);
  return (
    <div>
      <button onClick={() => setCount(c => c+1)}>Clic: {count}</button>
      <ExpensiveChild />  {/* ← Re-rend à chaque clic ! */}
    </div>
  );
}

Conséquence :

  • Perte de performance sur les composants lourds
  • Calculs inutiles (filtres, tris, transformations)
  • Expérience utilisateur dégradée

🛡️ Solution 1 : React.memo

React.memo est un HOC (Higher Order Component) qui empêche un re-render si les props n'ont pas changé (comparaison superficielle).

// Avant
const ProductCard = ({ name, price }) => { ... }

// Après - mémorisé
const ProductCard = React.memo(({ name, price }) => {
  return <div>{name} - {price} FCFA</div>
});

📌 À utiliser : Composants qui reçoivent les mêmes props souvent (liste de produits, lignes de tableau, cartes).

⚠️ Attention : Si vous passez des fonctions ou objets créés à chaque render, la comparaison échouera.

🧠 useMemo : mémoriser des valeurs calculées

useMemo recalcule une valeur uniquement quand ses dépendances changent.

// Sans useMemo : recalcul à chaque render
const filteredProducts = products.filter(p => 
  p.price <= maxPrice && p.category === category
);

// Avec useMemo : recalcul seulement si products, maxPrice ou category changent
const filteredProducts = useMemo(() => {
  return products.filter(p => 
    p.price <= maxPrice && p.category === category
  );
}, [products, maxPrice, category]);

✨ Cas d'usage typique : Filtrage, tri, calculs coûteux (ex: total du panier avec taxes).

🔄 useCallback : stabiliser les fonctions

useCallback retourne la même référence de fonction tant que les dépendances n'ont pas changé. Indispensable avec React.memo.

// Sans useCallback : nouvelle fonction à chaque render
const handleClick = () => { doSomething(product.id); };

// Avec useCallback : même fonction stable
const handleClick = useCallback(() => {
  doSomething(product.id);
}, [product.id]);

📌 Pourquoi c'est important ?
Sans cela, React.memo ne fonctionnera pas car la prop onChange sera différente à chaque render, forçant le re-render du composant enfant.

📊 Visualiser les re-renders

Activez le React DevTools Profiler pour identifier les re-renders inutiles :

⚡ React DevTools → Profiler → Enregistrer → Interagir → Analyser

  • Les composants colorés en jaune/vert ont re-rendu
  • Les composants gris ont été optimisés (React.memo)
  • Vérifiez les props qui changent inutilement

💡 Extension : "Why Did You Render" pour debug avancé.

📜 Virtualisation avec react-window

Pour les listes de centaines/milliers d'éléments, afficher tous les DOM nodes tue les performances. La virtualisation ne rend que les éléments visibles.

// Installation
npm install react-window

// Utilisation
import { FixedSizeList as List } from 'react-window';

const Row = ({ index, style, data }) => (
  <div style={style}>
    {data[index].name} - {data[index].price} FCFA
  </div>
);

function ProductList({ products }) {
  return (
    <List
      height={600}
      itemCount={products.length}
      itemSize={80}
      width="100%"
      itemData={products}
    >
      {Row}
    </List>
  );
}

📝 Récapitulatif des bonnes pratiques

✅ À faire

  • Utiliser React.memo sur les composants "feuille"
  • Mémoriser les calculs coûteux (useMemo)
  • Stabiliser les fonctions (useCallback)
  • Virtualiser les longues listes
  • Déplacer l'état au plus bas possible

❌ À éviter

  • Optimiser prématurément (d'abord mesurer)
  • Mémoriser des composants simples
  • Créer des objets/fonctions dans le render
  • Utiliser index comme key dans les listes

🎯 Règle d'or : Optimisez uniquement ce qui est lent. Un code simple et lisible est préférable à un code sur-optimisé.
