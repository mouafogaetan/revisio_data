🖱️ Tester les interactions utilisateur

React Testing Library (RTL) encourage à tester le comportement visible plutôt que les détails d’implémentation.

// Exemple : tester un bouton "Ajouter au panier"
import { render, screen, fireEvent } from '@testing-library/react';
import ProductCard from './ProductCard';

test('ajoute un produit au panier quand on clique', () => {
  const mockAddToCart = jest.fn();
  render(<ProductCard onAdd={mockAddToCart} name="T-shirt" />);
  
  const button = screen.getByRole('button', { name: /ajouter/i });
  fireEvent.click(button);
  
  expect(mockAddToCart).toHaveBeenCalledTimes(1);
});

✏️ À retenir : On utilise screen.getByRole ou getByText pour trouver les éléments, puis fireEvent ou userEvent (simule mieux le comportement réel).

🔄 Mock des API avec MSW

MSW intercepte les requêtes réseau au niveau du service worker. Aucun code à changer dans vos composants !

// src/mocks/handlers.js
import { http, HttpResponse } from 'msw';

export const handlers = [
  http.get('/api/products', () => {
    return HttpResponse.json([
      { id: 1, name: "Sac à main", price: 15000 },
      { id: 2, name: "Chaussures", price: 25000 }
    ]);
  }),
  http.post('/api/cart', async ({ request }) => {
    const newItem = await request.json();
    return HttpResponse.json({ success: true, item: newItem });
  })
];

// src/setupTests.js
import { setupServer } from 'msw/node';
import { handlers } from './mocks/handlers';

const server = setupServer(...handlers);
beforeAll(() => server.listen());
afterEach(() => server.resetHandlers());
afterAll(() => server.close());

✅ Avantage : vos tests restent rapides, sans dépendance réseau, et vous pouvez simuler des erreurs 500 facilement.

🧪 Tester un flux métier complet

Imaginons un scénario :

  1. L'utilisateur se connecte (email/password mocké)
  2. Il ajoute un produit au panier
  3. Il valide la commande et reçoit un numéro de commande
test('flux complet achat utilisateur connecté', async () => {
  render(<App />);
  
  // 1 - Connexion
  await userEvent.type(screen.getByLabelText(/email/i), 'client@example.com');
  await userEvent.type(screen.getByLabelText(/mot de passe/i), 'pass123');
  await userEvent.click(screen.getByRole('button', { name: /se connecter/i }));
  
  // Attendre que le nom apparaisse
  expect(await screen.findByText('Bienvenue Jean')).toBeInTheDocument();
  
  // 2 - Ajout produit
  await userEvent.click(screen.getByTestId('add-to-cart-1'));
  expect(screen.getByText(/1 article/i)).toBeInTheDocument();
  
  // 3 - Aller au panier puis checkout
  await userEvent.click(screen.getByRole('link', { name: /panier/i }));
  await userEvent.click(screen.getByRole('button', { name: /valider la commande/i }));
  
  expect(await screen.findByText(/commande #CMD-123/i)).toBeInTheDocument();
});

Remarque : on utilise findBy* pour les éléments asynchrones et userEvent au lieu de fireEvent pour plus de réalisme.

📝 Exercices d’application

Exercice 1 (compétence - niveau medium)

Créez un test qui vérifie qu’un utilisateur non authentifié ne peut pas accéder à la page de paiement. Utilisez MSW pour simuler un endpoint /api/user renvoyant 401.

🔍 Voir corrigé 
test('redirige vers login si non authentifié', async () => {
  server.use(http.get('/api/user', () => HttpResponse.json(null, { status: 401 })));
  render(<App />);
  await userEvent.click(screen.getByRole('link', { name: /paiement/i }));
  expect(await screen.findByText(/connexion/i)).toBeInTheDocument();
});

Exercice 2 (ressource - niveau hard)

Testez un formulaire d'ajout d'adresse avec validation : champs requis, format email, et envoi au backend. Mockez la requête POST /api/addresses et vérifiez le message de succès.

🔍 Voir corrigé 
test('ajout adresse avec validation', async () => {
  server.use(http.post('/api/addresses', () => HttpResponse.json({ id: 45 })));
  render(<AddressForm />);
  await userEvent.type(screen.getByLabelText(/ville/i), 'Douala');
  await userEvent.click(screen.getByRole('button', { name: /enregistrer/i }));
  expect(await screen.findByText(/adresse ajoutée/i)).toBeInTheDocument();
});
🧠 N'oubliez pas : cleaner les mocks après chaque test (resetHandlers).

✅ Bonnes pratiques & récapitulatif

  • Évitez de tester des détails internes (props, state) – privilégiez ce que l'utilisateur perçoit.
  • Utilisez waitFor ou findBy* pour les opérations asynchrones.
  • Structurez vos mocks par feature (auth, cart, products) pour les réutiliser.
  • Pour les formulaires complexes, privilégiez @testing-library/user-event.
  • Intégrez ces tests dans votre CI (GitHub Actions) pour éviter les régressions.
