🔐 Gestion des tokens JWT

Access token · Refresh token · Rafraîchissement automatique

localStorage / sessionStorage Refresh automatique Déconnexion propre

Une leçon pour sécuriser votre e-commerce et offrir une expérience utilisateur fluide.

🎯 Objectifs de la leçon

  • Stocker correctement les tokens (access & refresh) après connexion.
  • Implémenter le rafraîchissement automatique du token avant expiration (sans intervention utilisateur).
  • Nettoyer les données sensibles lors de la déconnexion.
  • Comprendre la différence entre access token (court) et refresh token (long).

À la fin, votre application saura gérer des sessions utilisateur robustes, même après plusieurs heures d'inactivité.

Access token vs Refresh token

🔑 Access token
  • Durée de vie courte (15 min – 1 heure)
  • Contient les claims (userId, rôles)
  • Envoyé dans chaque requête API (header Authorization)
  • Stockage : localStorage ou sessionStorage
🔄 Refresh token
  • Durée de vie longue (7 jours – 1 mois)
  • Utilisé uniquement pour obtenir un nouveau access token
  • Ne circule pas sur chaque requête
  • Stockage : localStorage (moins sécurisé) ou httpOnly cookie (recommandé)
Note de sécurité : En environnement de production, préférez stocker le refresh token dans un cookie httpOnly pour éviter les attaques XSS. Pour ce cours (contexte pédagogique), nous utiliserons localStorage.

Stocker les tokens après connexion

Exemple d'action login dans un store Zustand :

// authStore.js (Zustand + persist)
const useAuthStore = create(
  persist(
    (set) => ({
      user: null,
      accessToken: null,
      refreshToken: null,
      login: async (email, password) => {
        const res = await api.post('/auth/login', { email, password });
        set({
          user: res.data.user,
          accessToken: res.data.accessToken,
          refreshToken: res.data.refreshToken
        });
        // Stockage manuel (le middleware persist le fait déjà)
      },
      logout: () => set({ user: null, accessToken: null, refreshToken: null })
    }),
    { name: 'auth-storage' }
  )
);

📦 Le middleware persist synchronise automatiquement avec localStorage. On peut également utiliser sessionStorage pour une session fermée à l'onglet.

Interceptor Axios : Ajout automatique du token

import axios from 'axios';
import { useAuthStore } from '@/stores/authStore';

const api = axios.create({ baseURL: import.meta.env.VITE_API_URL });

// Interceptor request : ajouter le token
api.interceptors.request.use((config) => {
  const { accessToken } = useAuthStore.getState();
  if (accessToken) {
    config.headers.Authorization = `Bearer ${accessToken}`;
  }
  return config;
});

✅ Ainsi, chaque requête sortante inclut le token sans répétition de code.

Refresh automatique du token

Intercepteur de réponse : si le serveur retourne 401 Unauthorized → tenter un refresh.

let isRefreshing = false;
let failedQueue = [];

api.interceptors.response.use(
  (response) => response,
  async (error) => {
    const originalRequest = error.config;
    if (error.response?.status === 401 && !originalRequest._retry) {
      if (isRefreshing) {
        return new Promise((resolve) => {
          failedQueue.push(resolve);
        });
      }
      originalRequest._retry = true;
      isRefreshing = true;
      const { refreshToken } = useAuthStore.getState();
      try {
        const res = await api.post('/auth/refresh', { refreshToken });
        const { accessToken: newToken, refreshToken: newRefresh } = res.data;
        useAuthStore.setState({ accessToken: newToken, refreshToken: newRefresh });
        // Rejouer les requêtes en attente
        failedQueue.forEach(cb => cb());
        failedQueue = [];
        return api(originalRequest);
      } catch (refreshError) {
        useAuthStore.getState().logout();
        window.location.href = '/login';
      } finally { isRefreshing = false; }
    }
    return Promise.reject(error);
  }
);

✨ Ce pattern évite plusieurs appels de refresh simultanés et garantit que les requêtes échouées soient rejouées après l'obtention du nouveau token.

Déconnexion et nettoyage

logout: () => {
  // 1. Appel au backend pour révoquer le refresh token (optionnel)
  api.post('/auth/logout', { refreshToken: get().refreshToken })
    .catch(err => console.warn);
  // 2. Réinitialiser le store
  set({ user: null, accessToken: null, refreshToken: null });
  // 3. Nettoyer le cache React Query (si utilisé)
  queryClient.clear();
  // 4. Redirection vers login
  window.location.href = '/login';
}
Bonnes pratiques : vider aussi les données du panier local, les favoris temporaires, et toute autre donnée privée.

📌 Synthèse : bonnes pratiques

✔️ Access token

Durée courte (15 min), stocké en mémoire/localStorage, inclus dans chaque requête.

✔️ Refresh token

Durée longue, utilisé uniquement pour renouveler l'access token.

✔️ Intercepteurs Axios

Ajout automatique du token + gestion 401 + file d'attente des requêtes.

✔️ Déconnexion

Nettoyer store, localStorage, cache et rediriger.
